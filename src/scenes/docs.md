# Noridoc: scenes

Path: @/src/scenes

### Overview
- Phaser Scene subclasses that wire pure game systems to Phaser's rendering, physics, and input APIs
- Contains five scenes: `TitleScene` (main menu: displays "BACKROOMS" title with atmospheric pulsing text, provides continue/new game/delete save options based on save state, ambient drone audio, and a settings overlay for audio volume controls), `GameScene` (gameplay: movement, flashlight/darkness with a range-capped cone and an on/off toggle (F key / LIGHT button, auto-off while hiding), enemy AI, combat, healing via medkit pickup, shooting, weapon switching/pickups, battery drain (only while the flashlight is lit), item collection, enemy loot drops on death, lore note collection, closable doors, light switches, automatic ceiling lamps (always-on local light pools, see `@/src/systems/lamps.js`), furniture hiding, exploration minimap, an effectively-unbounded lazily-generated procedural world with deep-link stairs between floors, location-themed entrance room whose only opening into the backrooms is a sizzling dimensional rift (sealed against light, with proximity crackle audio and a first-crossing flash), exit condition, HUD, desktop keyboard/mouse and mobile twin-stick touch controls, procedural audio with ambient drone and event-driven sound effects with volume-controlled playback, per-run statistics tracking), `RunSummaryScene` (end-of-run stats display: shows "RETURNED" or "YOU DIED" with animated stat reveals, supports both desktop and mobile continue input, passes through to ShopScene), `ShopScene` (between-run upgrade shop: displays gold balance, upgrade categories with buy buttons, starting location selector, lore journal overlay, "Enter Backrooms" button, finger-sized tap targets for mobile, and volume-controlled UI sound feedback), and `PauseScene` (overlay scene launched on top of a frozen GameScene, displaying a controls reference, audio volume controls, and resume instructions)
- The game boots into TitleScene (main menu), which routes to ShopScene (continue with existing save) or GameScene (new game with cleared state). The primary gameplay loop then cycles through GameScene -> RunSummaryScene (on exit or death) -> ShopScene (on continue) -> GameScene (on "Enter Backrooms"). PauseScene is an overlay that pauses and resumes GameScene without interrupting this loop

### How it fits into the larger codebase
- `@/src/main.js` registers scene classes with the Phaser.Game instance (TitleScene, GameScene, ShopScene, PauseScene, RunSummaryScene); TitleScene is listed first (auto-starts as the boot screen). PauseScene and RunSummaryScene are registered but never auto-started -- PauseScene is launched on demand by GameScene via `scene.launch('PauseScene')`, and RunSummaryScene is transitioned to by GameScene via `scene.start('RunSummaryScene')` on exit or death
- Scenes import pure functions from `@/src/systems/` for game math (including maze generation from `@/src/systems/maze.js`, contiguous liminal-space zone assignment from `@/src/systems/zones.js`, per-room visual + furniture vibes from `@/src/systems/roomVibes.js`, the per-prop billboard light model from `@/src/systems/lighting.js`, lore spawning from `@/src/systems/lore.js`, enemy loot drops from `@/src/systems/lootDrops.js`, per-room persistent state from `@/src/systems/roomState.js` (consumed items, dead bodies, light on/off, plus the day-varying enemy-spawn seed), gunfire noise alerts from `@/src/systems/gunfireNoise.js`, coordinate-keyed unbounded room generation from `@/src/systems/worldgen.js`, AI-art helpers (texture-source selection and shop-icon mapping) from `@/src/systems/art.js`, location definitions from `@/src/systems/locations.js`, BFS room graph pathfinding from `@/src/systems/pathfinding.js`, time-based danger escalation from `@/src/systems/danger.js`, light spill gradient zones from `@/src/systems/lightSpill.js`, pause controls data from `@/src/systems/pause.js`, run statistics tracking and summary formatting from `@/src/systems/runStats.js`, audio config and per-enemy-type sound logic from `@/src/systems/audio.js`, sound synthesis from `@/src/systems/audioEngine.js`, audio settings and volume computation from `@/src/systems/settings.js`, procedural pixel art sprite/animation definitions AND the 8-way directional-facing helpers (`getDirectionalFrame`) from `@/src/systems/sprites.js`) and handle all Phaser-specific concerns: sprite creation, animation playback, physics bodies, input binding, camera, collision callbacks, Graphics drawing, and Web Audio integration
- **AI-generated art vs procedural fallback**: `GameScene` and `ShopScene` each have a `preload()` that loads the committed AI-generated PNGs under `@/src/assets/sprites/` (discovered at build time via Vite `import.meta.glob`). `GameScene` loads every non-`shop_` PNG; `ShopScene` loads only `shop_`-prefixed PNGs (the shop background and per-upgrade icons). These PNGs are produced offline and committed -- the game itself never calls any image API and ships no API key. The PNGs come from TWO offline generators: `@/scripts/generate-art.mjs` (the `ART_MANIFEST` furniture-top/shop sprites) and `@/scripts/gen-character-directional.mjs` (the player/enemy `{down,down_right,right,up_right,up}` directional frames plus their `_walk0`/`_walk1` step variants -- NOT in `ART_MANIFEST`). `GameScene.preload()` loads them all the same way via the filename glob, so both pipelines feed the same `this.textures.exists(key)` decision. For every texture key, the PNG (if present) wins; any missing key transparently falls back to the procedural pixel-grid texture from `@/src/systems/sprites.js`, so the game is always renderable even with no generated art. This per-key decision is the inline `this.textures.exists(key)` check in `generateSpriteTextures()` (and the `shop_bg`/icon checks in `ShopScene`), NOT the `getTextureSource()` helper in `@/src/systems/art.js`, which expresses the same policy as a pure function but has no production caller. The player and basic zombie are now directional PNGs (above); the remaining hand-authored procedural grids that are deliberately excluded from `ART_MANIFEST` (so they always render from `sprites.js`) are crawler/spitter and `furniture_armoire`/`furniture_couch`/`furniture_counter` -- all of which the image model cannot draw top-down (see `@/src/systems/sprites.js` / `@/src/systems/art.js`)
- Scenes are the only place in the codebase that depends on Phaser -- this boundary is intentional to keep game logic testable
- `GameScene` reads upgrade levels from `game.registry` in its `init()` method, using `getUpgradeValue()` from `@/src/systems/shop.js` to compute upgrade bonuses as deltas from pistol base values (e.g., `getUpgradeValue('weaponDamage', level) - 25` yields a +N damage bonus). These bonuses are stored in `this.upgradeBonuses` and applied additively to whatever weapon is active via `getEffectiveStats()` from `@/src/systems/weapons.js`. Binary unlock upgrades (`startingPistol`, `startingShotgun`, `startingRifle`, `minimap`, `rechargeBattery`) are checked as boolean flags: the three starting weapon flags are passed to `createWeaponState()`, which uses priority-based slot filling (rifle > shotgun > pistol) to equip up to 2 weapon slots -- when none are purchased the player starts unarmed, when all three are purchased the cheapest (pistol) is dropped; `minimap` sets `this.hasMinimap` which gates minimap Graphics creation and rendering; `rechargeBattery` sets `this.hasRechargeBattery` (same pattern as `hasMinimap`), which gates passing `BATTERY_RECHARGE_PER_MS` into the per-frame `updateBatteryWithFlashlight()` so the battery slowly refills while the flashlight is off. When unarmed, `getActiveWeapon()` returns null and firing is disabled until the player picks up a weapon. Survival stats (max HP, max battery, flashlight angle, speed) are still computed as absolute values. Enemy difficulty is NOT precomputed in `init()`: `getEnemyStats(type, distance)` computes each enemy's HP and contact damage on demand at spawn time from the spawn room's `distance` via `getEnemyHP()`/`getEnemyDamage()` from `@/src/systems/scaling.js` (each type has its own base constants), freezing the values on the per-enemy state object -- no global per-run difficulty state exists, and `runCount` no longer drives world generation at all: it affects only save bookkeeping and the first-run tutorial toast (difficulty is distance-keyed, not day-keyed). The world is instead seeded from a persisted per-save `worldSeed` (see **GameScene.init()** and `@/src/systems/persistence.js`), so the level LAYOUT is stable day-to-day for a save -- while a room's player-caused CONTENTS persist and its enemy spawns re-roll daily (`@/src/systems/roomState.js`). Upgrade level fallback uses `createShopState().upgrades` as defaults, ensuring forward compatibility when new upgrade keys are added to saves that predate them
- `ShopScene` reads/writes `shopState` from `game.registry`, using pure functions from `@/src/systems/shop.js` for all purchase logic. It receives `treasureEarned` and optionally `newLocation` as scene data from GameScene, converts treasure to gold via `addGold()`, and if `newLocation` is provided, adds it to `unlockedLocations` in the registry. ShopScene also calls `saveGame()` from `@/src/systems/persistence.js` at every point where state is mutated: after adding treasure gold in `init()`, before entering backrooms, and after purchasing an upgrade. All `saveGame()` calls pass `collectedLore`, `unlockedLocations`, `activeLocation`, `worldSeed`, the live `roomState`, and the live `weaponInventory` from `game.registry` to ensure all cross-run progress is persisted (passing the live values means returning to the shop never wipes a room's persisted player-caused changes nor the carried weapon loadout). ShopScene imports `getLocation()` and `LOCATIONS` from `@/src/systems/locations.js` to display location names in the location selector UI. ShopScene imports `getJournalEntries()` and `getJournalPage()` from `@/src/systems/lore.js` to power the lore journal overlay, which lets players review collected and uncollected lore notes between runs. ShopScene imports `generateAllSounds()` from `@/src/systems/audioEngine.js` to initialize procedural audio, playing UI feedback sounds for purchases, denied purchases, and button clicks
- `GameScene` calls `saveGame()` from `@/src/systems/persistence.js` after incrementing `runCount` in `init()`, persisting the updated run counter, shop state, location data, the resolved `worldSeed`, AND the `roomState` (the per-room player-caused changes, after the day's light flip has been applied) to localStorage before the run begins -- so the per-save world seed and the day's flipped light state are persisted the moment they are first minted/applied. `GameScene` also re-saves `roomState` to the registry at each in-run mutation point (item consumed, enemy killed, switch toggled) and writes `weaponInventory` to the registry on the day-cycle endpoints (`onDayComplete` saves the serialized loadout, `onPlayerDeath` clears it), and `@/src/main.js`'s lifecycle listeners flush both (along with the rest of the registry state) to localStorage on tab hide/close -- so `@/src/main.js`'s `saveGame()` call passes `registry.get('weaponInventory')` alongside `roomState`
- Data flows between scenes via two mechanisms: `game.registry` for in-memory cross-scene state (shopState, runCount, worldSeed, unlockedLocations, activeLocation, roomState, weaponInventory) and scene `init(data)` for per-transition data. `worldSeed`, `roomState`, and `weaponInventory` are threaded through the registry exactly like `runCount`: `@/src/main.js` hydrates all of them from the loaded save and re-flushes them on the lifecycle autosave; `ShopScene` passes `registry.get('worldSeed') ?? null`, the live `registry.get('roomState')`, and `registry.get('weaponInventory')` in each `saveGame()` call (so returning to the shop never wipes a room's persisted changes nor the carried loadout); `TitleScene.resetSaveState()` sets `worldSeed`, `roomState`, AND `weaponInventory` to `null` so New Game / Delete Save mints a fresh world layout and starts in a pristine, fully-stocked, fully-lit state with only the shop starting weapons, instead of reusing the deleted save's layout, room changes, or carried guns. `weaponInventory` is the serialized carry-over weapon loadout (found weapons survive the night) -- it is WRITTEN by `GameScene` on the day-cycle endpoints (`onDayComplete` saves the live loadout; `onPlayerDeath` clears it to `null`) and READ back by `GameScene.init()`. GameScene passes run statistics and outcome data to RunSummaryScene (`survived`, `treasureEarned`, `runStats`, `roomsExplored`, `hp`, `maxHp`, `newLocation`, `touchMode`). RunSummaryScene passes through `treasureEarned` and `newLocation` to ShopScene unchanged. Registry values are persisted to localStorage via `@/src/systems/persistence.js` so state survives browser reloads

### Core Implementation
- **TitleScene lifecycle**: `create()` fades in the camera, renders the "BACKROOMS" title text with a pulsing alpha tween (yellow #D4C073, oscillating between 1.0 and 0.3 over 3 seconds), a subtitle, and interactive menu options. Calls `loadGame()` from `@/src/systems/persistence.js` directly (not via registry) to check for an existing save, then `getMenuOptions(hasSave)` from `@/src/systems/titleMenu.js` to determine which menu items to display. When a save exists: Continue, New Game, and Delete Save are shown. Without a save: only New Game appears. A "SETTINGS" button is always shown below the dynamic menu options. Continue fades out and transitions to `ShopScene` (the returning player picks up from the shop). New Game calls `clearSave()` from persistence, resets all registry values (`shopState`, `runCount`, `worldSeed` to null, `roomState` to null so the new game starts pristine/fully-stocked/fully-lit, `weaponInventory` to null so the new game starts with only the shop starting weapons rather than a carried-over found gun, `collectedLore`, `unlockedLocations`, `activeLocation`) to defaults -- nulling `worldSeed` is what makes New Game / Delete Save mint a FRESH world layout instead of reusing the deleted save's seed, and transitions to `GameScene`. Delete Save shows a confirmation dialog overlay; on confirmation, calls `clearSave()`, resets the registry, and rebuilds the menu (now showing only New Game). Settings opens a full-screen overlay with three volume cycle buttons (Master, SFX, Ambient) that read and write `audioSettings` from the registry and persist changes immediately to localStorage via `saveSettings()` from `@/src/systems/settings.js`. Volume changes immediately update the title screen's ambient drone gain via `setVolume()`. The overlay uses the same depth-based modal pattern as the delete confirmation (depth 100-101 with an interactive background that blocks clicks through to the menu). Menu selection is blocked while either overlay is open (`this.confirmOverlay || this.settingsOverlay`). Uses the same responsive camera pattern as ShopScene (`fitCamera()` with `min(width/1024, height/768)` zoom), the same `enlargeHitArea()` pattern for finger-sized tap targets, and initializes ambient drone audio via `generateAllSounds()` and `createAmbientDrone()` from `@/src/systems/audioEngine.js`. The drone volume is set on creation using `getEffectiveVolume()` from `@/src/systems/settings.js` with the current master and ambient settings. The drone is stopped on scene shutdown. The `isTransitioning` flag prevents double-transitions (same pattern as RunSummaryScene)
- **GameScene lifecycle**: `init()` reads upgrade state from registry and computes per-run stats, `create()` sets up the level and all game objects, `update()` runs every frame
- **GameScene.init()**: reads `shopState`, `collectedLore`, `activeLocation`, `unlockedLocations`, and `audioSettings` from `game.registry` (converting `collectedLore` from a persisted array to a Set; falling back to `createDefaultSettings()` from `@/src/systems/settings.js` for audioSettings). Resolves the active location data via `getLocation()` from `@/src/systems/locations.js`, falling back to `'store'` if the location ID is not found. Spreads `createShopState().upgrades` as defaults before overlaying saved upgrade levels (for forward compatibility with saves predating new upgrades), then calls `getUpgradeValue()` for each upgrade to set survival stats (`this.maxCharge`, `this.flashlightAngle`, `this.maxHp`, `this.playerSpeed`), weapon upgrade bonuses (`this.upgradeBonuses = { damage, fireRate, range }` computed as deltas from pistol base values), and inventory capacity (`this.maxItems` from the backpack upgrade). Checks the three starting weapon upgrade levels (`startingPistol`, `startingShotgun`, `startingRifle`) and initializes `this.weaponState` as `fillStartingWeapons(deserializeWeaponState(registry.weaponInventory) || createWeaponState(), { startingPistol, startingShotgun, startingRifle })` from `@/src/systems/weapons.js`: it first RESTORES the carry-over loadout saved from a survived day (found weapons survive the night), then BACKFILLS only the empty slots with the player's owned starting weapons. So a gun found yesterday persists into today, while the shop "start each run with a X" upgrades act as a guaranteed-MINIMUM fallback rather than the whole loadout (priority rifle > shotgun > pistol, never overwriting a carried weapon, never duplicating, two-slot cap). When `registry.weaponInventory` is `null` (a fresh save, or after a death cleared it) `deserializeWeaponState` returns `null` and the loadout falls back to a fresh `createWeaponState()` -- so death reverts the player to exactly their shop starting weapons. Sets `this.hasMinimap` from the `minimap` upgrade level (binary flag: true when level > 0) -- this gates both `minimapGraphics` creation in `createHUD()` and `drawMinimap()` rendering. Sets `this.hasRechargeBattery` from the `rechargeBattery` upgrade level by the identical boolean-flag pattern -- this gates whether the per-frame `updateBatteryWithFlashlight()` is given `BATTERY_RECHARGE_PER_MS` (recharge-while-off) or 0. Calls `this.updateWeaponStats()` to derive effective bullet stats from the active weapon plus bonuses. Resolves the **world seed** for the lazy world generator: `resolveWorldSeed(this.registry.get('worldSeed'))` from `@/src/systems/persistence.js` returns the stored per-save seed if present, otherwise mints a fresh one; that value is written back to the registry and stored as `this.levelSeed` (the seed `floorSeed()` derives every floor's generation from). Because the seed is persisted once per save and reused every run, the world is byte-identical day-to-day for a player (so they can build a mental map) while differing between players. Also reads and increments `runCount`, storing `this.runCount` -- `runCount` is now decoupled from world generation entirely: it is retained ONLY for save persistence and the first-run tutorial toast, and feeds no enemy difficulty computation (no scaled stats are precomputed in `init()` -- see `getEnemyStats(type, distance)`). Initializes `this.dangerState` via `createDangerState()` from `@/src/systems/danger.js` -- the danger state is per-run and not persisted. Initializes `this.runStats` via `createRunStats()` from `@/src/systems/runStats.js` -- the run stats state is per-run and not persisted, tracking kills, elapsed time, and deepest floor reached for the end-of-run summary screen. Initializes `this.gunshotAlertCooldown = 0` -- tracks the cooldown between gunfire noise alerts (consumed by `@/src/systems/gunfireNoise.js` integration in the update loop). Loads `this.roomState` from the registry (`@/src/systems/roomState.js`, defaulting to an empty `createRoomState()`) -- the per-room player-caused changes (consumed items, dead bodies, light on/off) that persist day-to-day. On the FIRST `create()` of a new day (detected via `runCount > roomState.lastDayProcessed`), it runs `flipLights()` ONCE over every known light (seeded from `levelSeed + 70000 + runCount`) before any room is streamed in, so activation reads the post-flip state -- the bidirectional drift that means a lit room is never a permanent guarantee, then stamps `lastDayProcessed = runCount`. After incrementing, calls `saveGame()` to persist the updated `runCount`, `shopState`, `collectedLore`, `unlockedLocations`, `activeLocation`, the resolved `worldSeed`, `roomState`, AND the `weaponInventory` (the serialized carry-over loadout) to localStorage
- **Lazy world generation (replaces whole-level setup)**: `GameScene` no longer builds the entire multi-floor level in `create()`. Instead it keeps a growing `this.level = { rooms, stairs }` (only activated rooms) plus lazy-generation bookkeeping: `cellId` (Map from `"floor:gx,gy"` to a numeric id), `nextRoomId`, `cellRooms` (generated room cache), `roomsById`, `activatedRoomIds` (Set), `doorConnectionKeys` (Set for door dedup), `returnStack` (stair return path), and `weaponedFloors` (per-floor weapon guard). `floorSeed(floor)` is `levelSeed + floor * 1000`. `genRoom(floor, gx, gy)` builds a room via `getRoomAt()` from `@/src/systems/worldgen.js`, then re-stamps it with a numeric `id` (via `idForCell()`, assigned on first sight so the entrance cell `(0,0)` on floor 0 is always id 0), this floor's world-space Y (`floor * FLOOR_Y_OFFSET + gy * ROOM_HEIGHT`, `FLOOR_Y_OFFSET = 5_000_000`), a `distance = floor * FLOOR_DEPTH + localDistance(gx, gy)` (`FLOOR_DEPTH = 6`, the per-room exploration depth that replaces the old `computeRoomDistances`/`this.roomDistances` -- and the single axis BOTH loot quality and enemy difficulty scale on; see `@/src/systems/scaling.js`), and door `targetRoomId`s rewritten from worldgen string ids to numeric ids. `activateRoom(room)` runs all per-room scene wiring exactly once (guarded by `activatedRoomIds`): appends to `this.level.rooms`/`roomsById`, resolves the room's zone (`zoneId = getZoneAt(floorSeed, gridX, gridY)` from `@/src/systems/zones.js`, stored on `room.zoneId`; room 0 forced to `'store'`) and its theme (`getRoomVibe(zoneId, room.seed)` cached in `roomThemes`), draws floor/walls/doorways, builds room physics, pushes the room's perimeter occluder segments into `wallSegments` via `createRoomWalls()` from `@/src/systems/room.js` (with gaps at the doors -- EXCEPT the floor-0 rift, identified by `isRiftDoor()` from `@/src/systems/worldgen.js`, which is left out of the gap list so no light passes through it in either direction), then plans the room's SET PIECE via `planScene()` from `@/src/systems/furniture.js` (one coherent scene chosen by seed, fitted into a CLEARING clear of the `reserved` rects = door zones + stair + landing keep-outs; returns `{ items, clearing }`, or empty when nothing fits -- see **Scene-based room furnishing** below), then generates the internal maze rects (via `generateMazeWalls()`, skipped for id 0) carving around the stair `stairKeepOut()`/`stairLandingKeepOut()` zones AND the scene's `clearing` -- so the maze OPENS INTO the set piece (the player winds through bare maze and emerges into a furnished pocket) -- then assembles the room's furniture as `[...scene.items, ...ambient]` where `ambient` is a small `AMBIENT_SCATTER_COUNT`-capped scatter of the zone's `ambient` props via `generateRoomFurniture()` (empty clusters, the scene items + clearing + maze + door + landing rects all passed as obstacles so the scatter avoids them; hero props never scatter), passes that list to `createRoomFurniture(this.furnitureGfx, room, furniture)`, then renders the precomputed maze rects via `renderRoomMaze()` (which replaced the old `createRoomMaze()` that re-generated its own rects; it draws each rect, adds its physics zone, and pushes the SOLID rect into `this.mazeWallRects` -- the interior-wall occluder store expanded per-frame by both the flashlight and enemy LOS, NOT into `this.wallSegments`), then `applyRoomTextures(room, theme, mazeRects)` overlays the zone's tileable floor/wall textures as world-aligned TileSprites on top of the flat-colour fill (floor at depth 1, walls at depth 2, both below furniture and darkness; no-op when a zone texture is missing -- see **Zone floor/wall texture overlay** under Things to Know), then per-room doors, switch, lamp, items, lore, enemies, and weapon -- `activateRoomLamps(room)` runs in this sequence (after `activateRoomSwitch`) and is the lamps wiring: it re-derives the SAME seeded stream the pure `@/src/systems/lamps.js` module uses but PER-CELL (`mulberry32(this.floorSeed(room.floor) + LAMP_SEED_OFFSET + room.gridX*131 + room.gridY*977)`, matching `activateRoomSwitch`'s per-cell streaming for the lazy infinite world), gates on `LAMP_CHANCE`, then tracks the lamp's persisted on/off state per cell via `trackLamp()` from `@/src/systems/roomState.js` (a lamp starts lit but the between-days light flip may turn it off). If `isLampOn()` is false it returns immediately -- an off ceiling lamp draws no fixture and contributes no light (it is simply not there). For a lit lamp it picks a cool-vs-warm tint from the room's vibe id (`LAMP_COOL_VIBES` -- cool for `office`, warm otherwise, read from `this.roomThemes`; the set still lists the retired `spaceship` id, now a dead entry that never matches a live zone), pushes a lamp state into `this.lampStates`, and draws a static glowing fluorescent fixture via `drawLampFixture()` at `LAMP_FIXTURE_DEPTH = PLAYER_DEPTH + 5` -- ABOVE the player, so the player passes BENEATH the hanging fixture while the light pool stays below the player. A lamp is a LIGHT SOURCE ONLY -- it does NOT suppress enemy spawns or change any gameplay state (unlike a light switch). The three ground-pickup activators (`activateRoomItems/Lore/Weapon`, which all accept `(room, mazeRects = [])`) are each handed the same `mazeRects` and forward it as the generator's `obstacles` arg, so coins/notes/weapons are rejected out of solid maze walls where they would be uncollectable (maze walls only, not door zones -- doorways are walkable; see **Intra-room placement clearance** in the systems docs) -- and finally sets `roomGraphDirty = true`. Whether a room has a stair (and which way) is decided once by the shared `roomStairDirection(room)` helper, used by both the keep-out computation here (gating BOTH the stair and the landing keep-outs) and `activateRoomStairs()`, so the two never disagree. `ensureRoomsAround(floor, gx, gy)` activates every room returned by `roomsWithinRadius(floorSeed, gx, gy, 2)`; `ensureRoomsAroundPlayer()` (called each frame) recomputes the player's grid cell on the current floor and re-runs activation only when the player crosses into a new cell. Doors are deduped per shared edge (created when the second endpoint room activates, keyed by `doorConnectionKeys`); door closability and switch placement are seeded from canonical cell coordinates so they are independent of the order the player reaches a cell -- preserving determinism (same `worldSeed` produces the same world; the seed is persisted per save, so the world is stable day-to-day)
- **create() setup sequence**: `generateSpriteTextures()` runs first, iterating all sprite keys from `@/src/systems/sprites.js` and calling `this.textures.generate()` for each **only when `this.textures.exists(key)` is false** -- so any AI-art PNG already loaded in `preload()` is kept and only the missing keys get a procedural fallback texture. It then iterates all animation keys from `ANIM_DEFS` and calls `this.anims.create()` for each -- this pre-generates all pixel art textures and animations before any other creation method references them. Texture KEYS are identical whether sourced from PNG or procedural generation, so no gameplay/rendering code downstream changes. Then it creates the shared scene **infrastructure** once (room graphics, the batched `furnitureGfx` flat-furniture/fallback layer at depth 50 plus the `shadowGfx` billboard contact-shadow layer at depth 49 and the empty `this.billboardProps` tracking list (upright props the player can stand behind, depth-sorted each frame -- see **Player-vs-upright-prop occlusion**), door group + interact prompt, switch sprite map, lamp infra via `createLampInfra()` which initializes the empty `this.lampStates` / `this.nextLampId`, item/lore static groups, enemy group + colliders, weapon-pickup list, stair glow graphics), then activates the ENTRANCE room ALONE (`this.activateRoom(this.genRoom(0, 0, 0))`), captures `this.entranceRoom` (id 0), creates the player (at the entrance center, registering player-vs-item/lore/enemy overlaps), and ONLY THEN calls `ensureRoomsAround(0, 0, 0)` to generate and activate everything within 2 door-hops (the entrance is already active, so this skips it). This ORDER IS A HARD INVARIANT: a neighbour room with a stair registers a `physics.add.overlap(this.player, zone)` inside `activateRoomStairs()`, so the player MUST already exist when the surrounding rooms activate -- otherwise the overlap is created against an undefined `this.player` and Arcade physics dereferences it every step (the intermittent "reading 'isParent'" crash, seen only for seeds where an initially-activated room happened to have a stair). The fix was to split the prior single `ensureRoomsAround(0, 0, 0)` (which ran BEFORE `createPlayer()`) into entrance-activate -> create player -> activate neighbourhood. Then creates player bullets, enemy bullets, exit zone, crack visual, input, camera, darkness overlay, HUD, touch controls, and audio. `createTouchControls()` is called between HUD and audio setup and is a no-op on desktop (it early-returns when `this.touchMode` is false), so it only builds the touch UI on touch devices. Audio is initialized last via `createSounds()`. Alternate exit portals were removed (the lazy world's only world-exit is the floor-0 entrance zone); the shop's location cycling is unaffected. Combat state is initialized with `createCombatState(this.maxHp)`, battery state with `createBatteryState(this.maxCharge)`, inventory state with `createInventoryState(this.maxItems)` (where `maxItems` comes from the backpack upgrade computed in `init()`), hiding state with `createHidingState()` from `@/src/systems/hiding.js`, exploration state with `createExplorationState()` from `@/src/systems/exploration.js`, and run stats state with `createRunStats()` from `@/src/systems/runStats.js` (initialized in `init()` rather than `create()`). Two additional flags are initialized: `this.isTeleporting = false` (gates update loop during stair transitions) and `this.currentFloor = 0` (tracks which floor the player is currently on). Pathfinding state is initialized with `this.roomGraph = null` and `this.roomGraphDirty = true`, so the single closed-door-respecting room graph is lazily built on the first `updateEnemies()` call. Notable setup details:

- **Enemy integration pattern**: `GameScene` maintains a parallel `enemyStates` array alongside the Phaser physics sprites in `enemyGroup`. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, room-tracking fields (`spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`), `soundCooldown` (for per-enemy ambient sound timing), a `facingAngle` (used by basic enemies to choose their directional PNG frame -- updated from velocity while moving, retained while stopped), and standard AI state machine fields including the `lastKnownX/Y` last-seen-position memory. Enemies are added to `enemyStates` both at level creation (via `createEnemies()`) and dynamically during gameplay (via `spawnDangerWave()`, which creates new sprites in `enemyGroup` and pushes new state objects with the same structure, including stagger-initialized `soundCooldown`). Dynamic spawns automatically inherit existing colliders because Phaser group colliders cover all current and future group members. Each frame, `updateEnemies()` rebuilds the single room graph if dirty (via `buildRoomGraph()` from `@/src/systems/pathfinding.js`, cached and invalidated when `onToggleDoor()` or `activateRoom()` sets `roomGraphDirty = true`), counts enemies per room, detects room transitions (updating `currentRoomId` and resetting cooldown/targetDoorway when an enemy enters a new room), evaluates idle doorway-seeking eligibility (gated by `DOORWAY_SEEK_CHANCE`, cooldown, `MAX_ROOM_DISTANCE` from spawn via `getRoomDistance()`, and `MAX_ROOM_ENEMIES` cap), and constructs a slim per-enemy `navContext` of `{ roomTransitionCooldown, targetDoorway, doorway, seekDoorway }` -- this carries ONLY idle-wander data, passed as the 7th argument to `updateEnemyAI()`. The room graph is never consulted for chase routing: pursuit is strictly sight-based inside `updateEnemyAI()` (the AI stamps `lastKnownX/Y` while it has LOS and searches toward that stored point when LOS breaks -- see `@/src/systems/enemy.js` and its docs). After the AI update, state machine and room-tracking fields (including `lastKnownX/Y`, `roomTransitionCooldown`, `targetDoorway`) are copied back from the result. If `wantsToFire` is true on the result, GameScene calls `fireEnemyBullet()` -- pure function signals intent, scene executes side effect. After AI and side-effect processing, the enemy's visual facing is driven by TYPE: **basic** enemies update `es.facingAngle` from velocity (while moving) and call `applyDirectionalFrame('enemy', es.facingAngle, moving)` -- an 8-way PNG snap with `flipX` mirroring and a 4-frame `_walk0`/`_walk1` walk cycle, NO rotation and NO Phaser animation; **crawler/spitter** keep the procedural path -- the sprite's rotation is set from velocity (`Math.atan2`) and its animation is switched via `getAnimKeyForState(type, state)` from `@/src/systems/sprites.js` (chase -> walk animation, attack -> spitter attack animation). The `play(key, true)` call with `ignoreIfPlaying=true` prevents animation restarts when the state hasn't changed. On death, `enemySprite.stop()` freezes the current animation frame before applying corpse visuals. This keeps the AI logic testable in `@/src/systems/enemy.js` while letting Phaser handle physics resolution, animation playback, and projectile management
- **ShopScene lifecycle**: `init(data)` receives `treasureEarned` and optionally `newLocation` from GameScene, reads existing `shopState` from registry (or creates fresh state on first visit), adds the treasure as gold via `addGold()`, writes back to registry. If `newLocation` is provided and not already unlocked, it is added to `unlockedLocations` in the registry. Calls `saveGame()` to persist all state including location data. Also reads `collectedLore` from the registry (as an array) and converts it to a Set for use by the journal. `create()` first calls `fitShopCamera()` to zoom-fit the design space into the viewport (see below). If the AI-generated `shop_bg` PNG was loaded in `preload()` (`this.textures.exists('shop_bg')`), it is drawn as a dimmed full-design-space background image at depth -10 behind all UI. It then builds the UI: title, treasure earned display, gold balance, upgrade rows, location selector (when multiple locations are unlocked), a "NOTES X/20" journal button (top-right), and "Enter Backrooms" button. Each upgrade row renders a per-upgrade pixel-art icon when one is present: the row resolves a texture key via `shopIconKey(upgrade.id)` from `@/src/systems/art.js` (`shop_icon_{upgradeId}`) and draws it when `this.textures.exists(iconKey)` -- missing icons are simply omitted. If a `newLocation` was just discovered, a "New location discovered!" message is shown. When multiple locations are unlocked, `<` and `>` buttons let the player cycle through unlocked starting locations via `cycleLocation()`, which updates `activeLocation` in the registry. Each upgrade row shows name, description, level dots (filled/empty circles), next cost, and a buy button. Purchases call `purchaseUpgrade()`, write back to registry, and call `saveGame()` to persist. The "Enter Backrooms" button writes shopState and activeLocation to registry, calls `saveGame()`, and then calls `this.scene.start('GameScene')` to transition back to gameplay. The save-at-every-mutation pattern ensures no progress is lost even if the browser crashes between user actions. The **lore journal** is a full-screen overlay activated by clicking the "NOTES X/20" button. `showJournal()` creates a background Graphics at depth 2000 covering the full screen (1024x768), with all text elements at depth 2001 to layer above all shop content. The journal displays entries paginated at 5 per page via `getJournalPage()` from `@/src/systems/lore.js`. Each entry shows "Note #N" with the lore text (if collected) or "???" (if uncollected), using color to distinguish collected (white/gray) from uncollected (dark gray). Prev/Next buttons navigate pages; `getJournalPage()` handles page clamping so out-of-bounds page indices are safely corrected. A close button and ESC key handler (via `input.keyboard.on('keydown-ESC')`) call `hideJournal()`, which destroys all overlay objects. The journal is read-only -- it does not modify any game state
- **RunSummaryScene lifecycle**: `init(data)` receives run data from GameScene (`survived`, `treasureEarned`, `runStats`, `roomsExplored`, `hp`, `maxHp`, `newLocation`, `touchMode`), with safe defaults for all fields. `create()` calls `getSummaryData()` from `@/src/systems/runStats.js` to compute the display data (title, colors, stat lines), then renders a black background with the title ("RETURNED" in gold or "YOU DIED" in red -- never "escaped", and never an "X / total rooms" ratio, since the unbounded world has no total) faded in over 800ms. Stat lines (time, rooms explored, enemies killed, deepest floor, treasure, and HP remaining on survival) are rendered in two columns (labels in gray, values in the accent color) and staggered-revealed with 200ms delay per stat using `tweens.stagger()`. After all stats are revealed, a "Press any key to continue" (or "Tap to continue" on mobile) prompt fades in. `enableContinue()` wires keyboard and pointer input to `goToShop()`, which transitions to ShopScene with `{ treasureEarned, newLocation }` -- the same data contract ShopScene expects. A 150ms delay on touch pointer registration prevents the death/exit tap from immediately triggering continue. The `isTransitioning` flag prevents double-transitions. RunSummaryScene does not modify any game state, does not interact with the registry, and does not persist anything -- it is a pure display-and-passthrough scene
- **PauseScene**: an overlay scene launched on top of GameScene via `scene.launch('PauseScene', { touchMode })`. GameScene's `pauseGame()` method guards against invalid states (dead, transitioning, teleporting, already paused) before calling `scene.pause()` (which freezes GameScene's update loop, physics, tweens, and timers while continuing to render) and then `scene.launch('PauseScene')`. PauseScene's `init()` receives `touchMode` to select the correct controls list. `create()` draws a semi-transparent dark overlay, "PAUSED" title text, a two-column controls reference from `getControlsList()` in `@/src/systems/pause.js`, an "AUDIO" section with three volume cycle buttons (Master, SFX, Ambient) below the controls reference, and a resume instruction. The audio section reads `audioSettings` from the registry (falling back to `createDefaultSettings()`) and displays each volume level as a clickable cycle button. Clicking a button calls `cycleVolume()` from `@/src/systems/settings.js`, updates the registry, calls `saveSettings()` for localStorage persistence, and immediately applies volume changes to the running GameScene by calling `gameScene.applyVolumeSettings()` (looked up via `this.scene.get('GameScene')`). ESC key resumes on desktop; on mobile a tap-to-resume handler is registered with a 150ms delay (to avoid the pause-triggering tap from immediately resuming). `resumeGame()` calls `scene.resume('GameScene')` then `scene.stop()` on itself. The scene handles viewport resize by restarting itself to re-layout. This is the same pattern used by ShopScene's lore journal (overlay with ESC-to-close), but implemented as a separate Phaser scene rather than an in-scene overlay because it needs to freeze GameScene's update loop. Desktop triggers pause via ESC key (registered in `GameScene.setupInput()`); mobile triggers pause via an on-screen `| |` button (created by `GameScene.createTouchControls()` using `makeTouchButton` at the position from `computeTouchLayout().pauseButton` in `@/src/systems/touchControls.js`, placed top-left corner of the screen)

  Items at depth 5, enemies at depth 60, and furniture at depth 50 are all hidden by the darkness overlay and only revealed by the flashlight. Bullets at depth 150 are always visible (above darkness), giving visual feedback even when firing into dark areas. The HUD uses `setScrollFactor(0)` so it does not move with the camera AND is rendered by the separate un-zoomed `this.uiCamera` (see **Two-camera zoom / HUD split**) so the `CAMERA_ZOOM` applied to the world camera does not also scale it
- **Two-camera zoom / HUD split**: `GameScene` renders through TWO cameras. `cameras.main` is the world camera, zoomed in to `CAMERA_ZOOM` (`2.0`, from `@/src/systems/camera.js`) so the player and world read bigger and "closer" -- the deliberate fix for the binding spec's repeated "make the player bigger / rooms smaller / everything closer / claustrophobic" feedback. `this.uiCamera` is a second, un-zoomed camera added at viewport size that renders the HUD at 1:1. A dedicated UI camera is REQUIRED because Phaser's `setZoom()` also scales `scrollFactor(0)` objects, so HUD elements would otherwise zoom along with the world. `setupCameraLayers()` seeds the split: `partitionSceneObjects(this.children.list)` from `@/src/systems/camera.js` classifies every existing object (HUD = pinned on BOTH axes; everything else = world), then `cameras.main.ignore(hud)` and `uiCamera.ignore(world)`. The classification invariant is **scrollFactor**, not depth: the only one-axis-pinned WORLD object is the "Press E" `interactText` (`scrollFactorX = 1`, follows the world even though it sits at a HUD-like depth), which therefore renders on the zoomed main camera. Because `camera.ignore(group)` does NOT cover objects created later, an `ADDED_TO_SCENE` listener (`this._routeNewObject`, registered on `this.events`, removed on `shutdown`) routes every newly-created object -- pooled bullets, danger-wave enemies, loot drops, the lore popup -- to the correct camera. The listener keys off a `this._buildingHud` flag (default false, so new objects default to world) rather than reading `scrollFactor`, because `ADDED_TO_SCENE` fires BEFORE chained `.setScrollFactor()` runs and the value is not yet readable; `showLorePopup()` wraps its two `this.add.*` calls in `this._buildingHud = true/false` so the dynamic lore popup routes to the UI camera
- **Camera effects span both cameras vs. one**: the three scene-transition fades -- stair teleport (`onStairEnter()` fadeOut/fadeIn), day-complete exit (`onDayComplete()` fadeOut), and death (`onPlayerDeath()` fadeOut) -- are mirrored onto `this.uiCamera` so the HUD fades together with the world during transitions. Camera shake (including on combat damage) and the remaining `cameras.main.flash()` effects (the rift-crossing flash, the medkit green flash) stay on `cameras.main` ONLY, so they shake/flash the world without disturbing the fixed HUD. Combat DAMAGE feedback is deliberately NOT a `cameras.main.flash()` anymore -- it is a sub-player world tint (see **Damage feedback is a sub-player world tint, not a camera flash**)
- **Camera must snap on a player TELEPORT (discontinuous move), not smooth-follow** -- this is a hard invariant. `setupCamera()` installs a smooth follow (`cameras.main.startFollow(this.player, true, 0.1, 0.1)`, lerp 0.1), which eases the camera toward the player each frame. That easing is correct for ordinary walking, but it is wrong when the player is moved discontinuously: it would ease in across the entire jump distance, revealing the player far off-centre and slowly drifting in. The ONLY large discontinuous move is the stair transition: `onStairEnter()` reuses `getStairLandingPosition()` to `body.reset()` the player onto a landing on a different floor band (floor bands are `FLOOR_Y_OFFSET = 5_000_000` px apart), so without an explicit snap the fade-in would show the player ~1,000,000 px off-centre -- the spec's "player is often dropped in a random place nowhere near the stairs". The fix is the single `cameras.main.centerOn(destX, destY)` call inside the `camerafadeoutcomplete` handler, immediately after `body.reset()` and while the screen is still black, so the fade-in reveals the player dead-centre on the landing. The other two player relocations do NOT need a snap and deliberately omit one: `onEnterHiding()` `body.reset()`s the player to the furniture centre, but that is an in-view short hop, and the rift crossing (`updateRiftCrossing()`) is a normal walk-through, not a teleport. A regression check in `@/scripts/playtest.mjs` drives a real stair via the physics overlap and asserts the player is within 20px of screen centre after the transition
- **Darkness system architecture**: the BitmapMask has `invertAlpha = true`, so white pixels in `maskGraphics` punch transparent holes in `darkGraphics`, revealing the room beneath. The `updateDarkness(time)` method first draws lit room rectangles (from `getLitRoomIds()`, which always includes room 0 since it is permanently lit) into both `maskGraphics` (white, to punch holes in darkness) and `lightGraphics` (faint yellow tint). After lit rooms, it computes light spill zones via `getLightSpillZones()` from `@/src/systems/lightSpill.js` and draws each zone as a gradient rectangle into both `maskGraphics` (using `fillGradientStyle` for per-corner alpha that creates gradient visibility through the BitmapMask) and `lightGraphics` (warm tint at 5% of the spill alpha). The gradient fades from full brightness at the doorway edge to zero over 150px (`LIGHT_SPILL_DEPTH`). Closed doors block spill entirely; both-lit room pairs are skipped (already fully revealed); and any spill zone touching `this.riftRect` is filtered out via `zoneTouchesRift()`, so no light leaks through the rift in either direction (the permanently-lit store would otherwise reveal the first backroom before the player ever crosses). Next, **automatic ceiling lamps** light their local pools: each lamp in `this.lampStates` that survives a camera-`worldView` cull is drawn as a soft radial pool via `drawRadialGlow()` into BOTH `maskGraphics` (white `0xffffff`, to punch a hole in the darkness) and `lightGraphics` (the lamp's warm/cool `tint`), with the ring alpha multiplied by `lampBrightness(lamp, time)` from `@/src/systems/lamps.js` (steady lamps return 1; flickering lamps buzz between `LAMP_FLICKER_FLOOR` and full). A lamp lights its pool regardless of whether the flashlight is on, off, or the battery is dead -- it is a genuinely automatic light source (a permanent playtest guard in `@/scripts/playtest.mjs` drives a real lamp-bearing room with the flashlight forced OFF and the battery dead and asserts the lamp still reveals its pool). The flashlight cone is drawn only if `isFlashlightLit()` from `@/src/systems/flashlight.js` is true (light switched on and not hiding) and the battery-adjusted `getFlashlightConeAngle()` is positive and `shouldFlicker()` is false. The polygon itself comes from `getFlashlightPolygon()` whose occluder array is built each frame from three sources, all capped at `FLASHLIGHT_RANGE`: `this.wallSegments` (perimeter walls, closed doors) prefiltered by `filterSegmentsNear()`, plus `this.mazeWallRects` (interior walls) AND `this.furnitureOccluderRects` (light-blocking furniture), both prefiltered by `filterRectsNear()` and expanded to only their FAR faces via `wallRectOccluders(rect, origin)` (back-face culling, all from `@/src/systems/visibility.js` and `@/src/systems/maze.js`). The back-face cull is the fix for the spec's "walls have no thickness" / "furniture pitch black under the flashlight" bugs: skipping each rect's near faces lets the cone reach across the band and light its near surface (so the wall/furniture reads as a solid piece of visible thickness instead of black under the darkness overlay), while the far faces still block light so nothing leaks to the floor beyond. Both prefilters keep sight range-limited and the raycast cost bounded as the world grows. Lit rooms and their spill zones remain visible even when the flashlight is dark -- they use the same BitmapMask mechanism but draw rectangles instead of the flashlight polygon. CRITICALLY, `updateDarkness()` was restructured so the flashlight branch no longer EARLY-RETURNS: instead it CAPTURES the lit-room rects, the cone polygon, the player near-field pool, and the lamp pools into a "light field" object EVERY frame (even with the flashlight off), then calls `updateBillboardLighting(field, view)` at the end (see **Per-prop billboard lighting**). This is required because upright props render ABOVE the darkness and must be tinted by sampled light rather than revealed by the floor mask, so the per-prop lighting must run whether or not the flashlight is lit
- **Enemy physics uses a dynamic group** (unlike walls and furniture which are static groups). Enemies collide with walls, furniture, and each other. Player-enemy interaction uses `overlap` (not `collider`), triggering `onEnemyContact(player, enemySprite)` which looks up the specific enemy state object to read its per-enemy `contactDamage` (frozen at spawn from the spawn room's distance; early-returns if the state is not found), applies it via `applyDamage()`, triggers camera shake plus the sub-player damage tint via `triggerDamageFlash()` (see **Damage feedback is a sub-player world tint, not a camera flash**), and checks for player death
- **Player bullet physics group** uses pooling (`maxSize: 30`). Bullet-wall and bullet-door use `collider` (instant deactivation via `onBulletHitWall`). Bullet-furniture also uses `collider` but is gated by the `this.bulletHitsFurniture` process callback so only tall/solid furniture stops it (see **Selective bullet-vs-furniture collision**). Bullet-enemy uses `overlap` -- on hit, the bullet is deactivated and the enemy takes damage read from per-bullet data (`bullet.getData('damage')`, falling back to `this.bulletDamage`). When an enemy dies, the kill is recorded via `recordKill()` from `@/src/systems/runStats.js`, and a per-type death sound is played via `playSoundAtDistance(getEnemyDeathSound(type), distance)` from `@/src/systems/audio.js` (e.g., zombie_death for basic, crawler_death for crawler, spitter_death for spitter), with volume attenuated by the enemy's distance from the player. The sprite is kept visible as a corpse: `setActive(false)` (removes from physics/overlap checks), `body.enable = false`, and corpse visual constants from `@/src/systems/combat.js` are applied (gray tint, faded alpha, toppled angle, depth 5). The kill is ALSO recorded as a PERSISTENT corpse in `roomState`: `addCorpse()` from `@/src/systems/roomState.js` stores a `{x, y, type, angle}` body for the enemy's current cell (looked up via `this.roomsById`, FIFO-capped at `CORPSE_CAP = 15` per room), so the body re-appears on later days even after the room re-generates (see `activateRoomCorpses` below). After corpse visuals, `rollEnemyDrop()` from `@/src/systems/lootDrops.js` is called with the enemy's type, the room's exploration depth (the per-room `room.distance` field stamped during lazy generation -- see `@/src/scenes/GameScene.js` `genRoom()`), and `Math.random`. If a drop is returned, a sprite is created in `this.itemGroup` at the enemy's position with the item's type and value set via `setData()` -- the existing `onItemPickup` overlap collider auto-registers the dropped item for pickup with no additional wiring. A `loot_drop` sound plays at the death distance. The sprite remains in `enemyGroup` but Phaser skips inactive sprites in overlap checks, so corpses cannot trigger contact damage. The enemy state is removed from `enemyStates`. Each bullet stores its origin, damage, and range via Phaser's `setData()` at spawn time in `spawnBullet()`. This per-bullet approach ensures that bullets fired with one weapon retain their stats even after the player switches weapons mid-flight. `fireBullet()` selects the correct texture (`bullet_{weaponId}`) based on the active weapon and dispatches to either `calculateShotgunSpread()` (for multi-pellet weapons with `bulletCount > 1`) or `calculateBulletVelocity()` (for single-bullet weapons). Fire cooldown resets to `this.fireRate` and range expiry checks per-bullet range -- both derived from the active weapon via `updateWeaponStats()`. **Enemy bullet physics group** (`enemyBulletGroup`) mirrors this pattern with `maxSize: 30`, using wall/furniture/door colliders and a player overlap callback (`onEnemyBulletHitPlayer`). Enemy bullets use `SPITTER_PROJECTILE_SPEED` and `SPITTER_PROJECTILE_RANGE` constants directly (no upgrades). Enemy projectile damage is blocked while the player is hiding
- **Selective bullet-vs-furniture collision** (the spec's binding "gunfire goes over low furniture, blocked by armoires" feedback): all furniture stays in the single static `furnitureGroup`, but each furniture zone is tagged at creation in `createRoomFurniture()` via `zone.setData('blocksBullets', blocksBullets(item.type))` (from `@/src/systems/furniture.js`, true iff the type also blocks light/LoS). Both bullet-vs-furniture colliders (player `bulletGroup` and `enemyBulletGroup`) pass `this.bulletHitsFurniture` as the Arcade physics `processCallback` (the collider's 4th arg). `bulletHitsFurniture(bullet, furnitureZone)` returns `furnitureZone.getData('blocksBullets')`: returning a truthy value lets `World.separate()` proceed (the bullet is separated/stopped and `onBulletHitWall` fires), returning `false` makes Arcade skip separation entirely so the bullet flies straight over low furniture (tables, desks, beds) without deactivating. This is the only `processCallback` in the codebase -- it is the chosen mechanism because it preserves the single-group invariant (furniture stays in one group, so the UNCHANGED player-body collider `this.playerFurnitureCollider` and the enemy-body furniture collider still block ALL furniture -- the player still cannot walk through a table; only bullets pass selectively). Phaser's strict `=== false` check means the callback must return a real boolean, which `getData()` provides (`true`/`false`)
- **Player death flow**: when `combatState.isDead` becomes true, `onPlayerDeath()` CLEARS the carry-over loadout (`registry.set('weaponInventory', null)` -- death drops everything, so the next day's `GameScene.init()` reverts to the shop starting weapons), stops the player body, disables physics, triggers a 500ms camera fade-out (mirrored onto `this.uiCamera` so the HUD fades with the world -- see **Two-camera zoom / HUD split**), and transitions to RunSummaryScene with `{ survived: false, treasureEarned: 0, runStats, roomsExplored, hp: 0, maxHp, newLocation: null, touchMode }`. RunSummaryScene then passes through to ShopScene with `{ treasureEarned: 0 }`. The death penalty is losing all treasure collected that run AND every found weapon -- gold already in the shop balance is not affected. The `update()` loop early-returns while dead, preventing further input processing
- **Stair teleportation flow (deep-link + return stack)**: stairs are deep-links rather than fixed pairs. On overlap, `onStairEnter()` reads the zone's `direction`. A DOWN stair pushes `{ floor, gx, gy }` (its own room) onto `this.returnStack` and targets floor `+1` cell `(0,0)`; an UP stair (at a floor's `(0,0)`) pops the stack and targets that recorded floor/cell (defensively falling back to the floor-0 entrance if the stack is empty). It then sets `this.isTeleporting = true`, stops/disables the player body, and triggers a 300ms camera fade-out (mirrored onto `this.uiCamera`). On fade-out complete, it sets `this.currentFloor` to the destination floor, calls `ensureRoomsAround(destFloor, destGx, destGy)` to lazily generate the destination neighborhood, teleports the player (`body.reset()`) to `getStairLandingPosition(destRoom)` from `@/src/systems/stairs.js` (single-sourced -- the same point `activateRoom()` reserves via `stairLandingKeepOut()`, ~140px below the centered stair so the overlap does not re-fire), re-enables the body, **clears a stair-landing safe zone via `clearLandingZone(destRoom, destX, destY)`** (see below), **SNAPS the world camera onto the landing via `cameras.main.centerOn(destX, destY)` while the screen is still black** (the spec's "stairs ... player is dropped in a random place nowhere near the stairs" fix -- see **Camera must snap on a player TELEPORT**), grants a short invulnerability window via `grantInvulnerability(this.combatState, STAIR_SPAWN_PROTECTION_MS)` from `@/src/systems/combat.js` so an enemy that wandered near the landing cannot hit the player before they can react (the spec's "stairs ... vulnerable to attack immediately after spawning" fix, paired with the enemy stair/landing keep-out in `activateRoomEnemies()`), updates `this.runStats` via `updateMaxFloor()` from `@/src/systems/runStats.js`, and begins a 300ms fade-in. `isTeleporting` is cleared on fade-in complete. The `update()` loop early-returns while `isTeleporting` is true, freezing all game logic during the transition. This follows the same camera-fade + boolean-gate pattern as `dayEnding`

- **Wall dual-purpose**: walls exist as both Phaser Arcade Physics static zones (for player collision) and as line segments from `createRoomWalls()` (for raycasting). Closable doors follow the same segment pattern -- each is both a physics zone and a raycasting segment. Light-blocking furniture (any type with `blocksLight: true` in `@/src/systems/furniture.js`, e.g., shelf, bookcase, armoire, closet) is ALSO both a physics zone and an occluder, but its occluder is now a SOLID RECT in `this.furnitureOccluderRects` (parallel to `mazeWallRects`), NOT an outline in `wallSegments` -- expanded per consumer per frame exactly like a maze wall (flashlight to far faces via `wallRectOccluders()` for back-face culling, enemy LOS to the full outline via `wallRectSegments()`). For a light-blocking UPRIGHT billboard prop (vending machine, lockers) the rect is instead the small floor FOOTPRINT, NOT the tall standee art -- so the prop occludes only the ground it stands on (the footprint-decoupling invariant, see the render-path bullet below). Putting furniture in a back-face-culled rect store (instead of a full outline in `wallSegments`) is the fix for the spec's "furniture is pitch black under the flashlight" / "infinitely-thin tall black walls" bugs: the old outline stopped the cone at the piece's near face and excluded its footprint from the lit area, so the sprite (depth 50) sat under the darkness overlay (depth 100) and rendered black. Non-blocking furniture (types with `blocksLight: false`, e.g., table, desk, bed, vent) still has physics zones (so the player and enemies cannot walk through it) but does not produce any occluder -- and, via the same `blocksLight` classification surfaced as `blocksBullets()`, both light AND gunfire pass over it (see **Selective bullet-vs-furniture collision**). Maze internal walls are solid rects from `generateMazeWalls()` in `@/src/systems/maze.js`: `activateRoom()` generates the rects (passing stair and stair-landing keep-out zones where applicable) and `renderRoomMaze()` draws each rect exactly with `fillRect`, adds a single physics zone matching the rect, and stores the SOLID rect in `this.mazeWallRects` -- a separate occluder store from `wallSegments`, kept as rects so each consumer can expand them differently per frame (enemy LOS to the FULL outline via `wallRectSegments()`, the flashlight to the FAR faces only via `wallRectOccluders(rect, origin)`, the back-face cull that gives interior walls visible thickness -- see **Darkness system architecture** and the **Occluders are split** note in the src docs). So for a maze wall the drawn fill, the physics zone, and the occluder geometry are all the same rectangle, but the flashlight and enemy LOS see different subsets of its edges. (Previously the scene pushed each maze wall's full outline into `wallSegments`, the single occluder array both consumers shared, so the flashlight stopped at the wall's near face and the wall body stayed in shadow -- rendering interior walls as infinitely-thin black lines; the rect split is the fix.) Doors are unique in that they can toggle between active/inactive at runtime via `body.enable` and `wallSegments` push/splice. Enemies, items, and light switches do not produce segments; enemies are dynamic actors, items are small collectibles using staticGroup + overlap, and light switches are proximity-based interactables with no physics body
- **Scene-based room furnishing** (`activateRoom()`): each non-entrance room is furnished as ONE coherent hand-authored SCENE dropped into a CLEARING plus a little small-prop ambient scatter -- this REPLACED the prior random clusters + scattered singletons (`generateRoomFurniture()` with the zone's full profile), which scattered big "hero" props (a lone reception desk) incoherently. - **Furniture render path branches on `upright`** -- `createRoomFurniture()` has two paths keyed on the type's `FURNITURE_TYPES` def (from `@/src/systems/furniture.js`):
  - **Flat top-down furniture** (the default): the sprite is drawn centered on the logical rect at `setDisplaySize(item.width, item.height)`, depth 50. The collision/bullet zone AND the light occluder track the VISIBLE ART (not the padded texture). This is the fix for the spec's "furniture hitboxes do not match visually" bug: the AI furniture PNGs carry large transparent padding, so the visible art fills only part of the texture. `GameScene.getFurnitureArtFrac(key)` computes each texture's NORMALIZED opaque bounding box ONCE (draws the texture to an offscreen canvas, reads the pixels, passes them to `opaqueBounds()` from `@/src/systems/furniture.js`), caching the fraction in `this.furnitureArtFrac` (full rect when the texture cannot be inspected or has no PNG). `createRoomFurniture()` maps that fraction onto the world rect via `visibleFurnitureRect()` and sizes BOTH the collision/bullet zone AND the occluder rect to this VISIBLE sub-rect, while the sprite is still drawn stretched to the full logical rect. Procedural-fallback furniture (no PNG) uses the full rect.
  - **Upright 2.5D billboard props** (types with `upright: true` AND a loaded PNG): the standee is base-anchored -- `setOrigin(0.5, 1)` planted at the bottom-center (`baseY = item.y + item.height`, the footprint's near/south edge), displayed at the def's `spriteWidth`/`spriteHeight` (the VISUAL size, NOT the footprint), depth from `billboardDepth(baseY, room.floor * FLOOR_Y_OFFSET)` so props sort among themselves by base world-Y while staying ABOVE the darkness overlay (see the depth table and `@/src/systems/furniture.js`). Because the prop draws above the darkness, the floor reveal mask does NOT light it; instead it is given an INITIAL dark tint at creation (`brightnessToGray(0, BILLBOARD_MIN_LIGHT)` from `@/src/systems/lighting.js`, so a freshly spawned prop never flashes full-bright before the first lighting pass) and re-tinted every frame by `updateBillboardLighting()` (see **Per-prop billboard lighting**). The prop's texture is chosen via `propVariantKey(item.type, item.x, item.y, def.variants)` so variant types (trash cans, mannequins) pick one of their N images deterministically from position. Each prop is also tracked in `this.billboardProps` (as `{ sprite, cx, baseY, spriteWidth, spriteHeight, baseDepth }`) at creation, so the per-frame `updateBillboardOcclusion()` can y-sort it against the player (see **Player-vs-upright-prop occlusion**) and `updateBillboardLighting()` can tint it. A soft contact-shadow ellipse is drawn under the base on the dedicated `this.shadowGfx` layer (depth 49), sized to the VISIBLE base width (`def.spriteWidth * 0.9`, NOT the footprint) so it grounds the cropped standee on its feet without the wide smear the old footprint-sized ellipse produced. CRITICAL FOOTPRINT-DECOUPLING INVARIANT: for an upright prop the collision zone and the light occluder use the small FOOTPRINT rect (`item.x/y/width/height`), NOT the opaque-art bounds (`visibleFurnitureRect`) -- so a tall standee never becomes a tall collision wall or a tall light occluder. This is the whole point of the 2.5D treatment: the player perceives a tall object they walk past, but it only physically occupies and occludes its small floor footprint. An upright type that has no PNG falls through to the flat Graphics `fillRect` path on its footprint rect.
- **Player-vs-upright-prop occlusion** (`updateBillboardOcclusion()`, called from `update()` after `drawPlayer()`): each frame, walk the tracked `this.billboardProps` and call `shouldPropDrawOverPlayer(prop, playerX, playerFeetY)` from `@/src/systems/furniture.js` (the player's feet point is `this.player.y + PLAYER_FOOT_Y_OFFSET`). A prop the helper reports the player is standing BEHIND and overlapping on screen has its depth bumped to `BILLBOARD_OVER_PLAYER_DEPTH` (201, just above the player at `PLAYER_DEPTH` 200) so it occludes the player; otherwise its depth is restored to the `baseDepth` captured at creation. This exists because the player previously ALWAYS rendered on top of tall props (player depth 200 >> prop depth [110,120)), so standing beside/behind a locker looked like walking ON TOP of it instead of being occluded behind it. The player CANNOT be lowered into the billboard band to fix this -- it is pinned at 200 above the darkness/damage-flash/bullet layers (lowering it would re-introduce the "player disappears when shot" bug and make the player vanish when the flashlight is off), so the contained fix raises the SPECIFIC overlapped prop above the player instead. The lift is GATED TO OVERLAP on purpose: a lifted prop sits above the darkness (depth 201 > 100), so a prop lifted while the player stands behind it in the dark would glow; restricting the lift to the prop(s) the player actually overlaps keeps a lifted prop inside the player's own lit near-field rather than glowing alone in the dark. (A non-lifted prop is also above the darkness but its per-frame light tint -- which darkens an unlit prop to ~5% -- keeps it from glowing.) TRADEOFF: occlusion is player-vs-upright-props ONLY -- enemies still draw at their fixed depth (60) and are not y-sorted against props.
- **Per-prop billboard lighting** (`updateBillboardLighting(field, view)`, called at the end of `updateDarkness()`): upright billboard props render ABOVE the darkness overlay (depth band `[110, 120)`), so the floor reveal mask does NOT light them -- it only ever revealed a prop's floor FOOTPRINT, which left a tall standee a black silhouette lit only at its feet (the bug this fixes). Instead, `updateDarkness()` assembles a light field `{ litRoomRects, cone, nearField, lamps }` every frame (the cone/nearField are captured even when the flashlight is on, and the field is built even when it is off) and this method walks the on-screen `this.billboardProps`, samples `sampleBrightness(prop.cx, prop.baseY, field)` from `@/src/systems/lighting.js` at each prop's base, and applies a gray multiply tint via `brightnessToGray(b, BILLBOARD_MIN_LIGHT)`. A prop in a lit room, the flashlight cone, the near-field pool, or a lamp pool brightens its whole front face like a maze pillar; an unlit prop is darkened to ~5% (`BILLBOARD_MIN_LIGHT`) so it matches the darkened floor (the darkness is ~0.95 opaque) and fades into the dark rather than reading as a pure-black cutout. Props are culled to the viewport (with padding) so off-screen props on other floors are not tinted
- **Player rendering**: the player is a PNG-backed 8-direction TOP-DOWN sprite at `PLAYER_DEPTH` (200), above the darkness/damage-flash/bullet layers, so the player is always visible regardless of flashlight direction and is never erased or reddened-over. This depth is load-bearing and the player CANNOT be moved below those layers (it would re-introduce the "player disappears when shot" / "vanishes with the flashlight off" bugs), which is the reason upright-prop occlusion is implemented by lifting the prop above the player rather than lowering the player into the billboard band (see **Player-vs-upright-prop occlusion**). The 64px native directional texture is displayed at 48px (scale 0.75) with a smaller physics body (`body.setSize(32)`, so the effective ~24px world hitbox is unchanged from the pre-directional 40px/19 player -- Arcade bodies are multiplied by display/texture scale). `drawPlayer()` does NOT rotate the sprite: it calls `applyDirectionalFrame('player', this.playerAngle, isMoving)`, which snaps the body to one of 8 facings via `getDirectionalFrame()` from `@/src/systems/sprites.js`, mirrors the three left-side facings with `setFlipX`, and while moving cycles a 4-frame walk (`base -> _walk0 -> base -> _walk1`, where `_walk0`/`_walk1` are AI-generated left/right leg-step poses registered to the base by the head) on a `CHARACTER_STEP_MS` (130ms) per-frame timer. The flashlight light cone is drawn from `this.playerAngle` independently of the sprite (so it aims continuously 360° while the body snaps to 8 directions), and its apex is pushed `FLASHLIGHT_MUZZLE_OFFSET` px forward along the aim in `updateDarkness()` so the beam emanates from the held flashlight in front of the player rather than the body center. `drawPlayer()` also applies visual states: hiding (muted green tint, alpha 0.3), hurt (a red multiply tint via `getHurtFlash()` from `@/src/systems/combat.js`, applied with `setTint` so the silhouette never washes out), or normal (no tint, alpha 1)
- Camera and physics world bounds are a large fixed region (`WORLD_BOUND`, a `GameScene` constant) rather than computed from room positions, because the lazily-generated world is effectively unbounded and has no finite extent. The darkness overlay and its flashlight BitmapMask are sized from `cameras.main.worldView` each frame for the same reason. `worldView` already accounts for the main camera's `CAMERA_ZOOM`, so the darkness/flashlight mask -- which draws in WORLD coordinates -- stays correctly aligned under zoom with no zoom-specific code (this is why the camera zoom required no change to the darkness system)
- **Audio integration**: both scenes initialize audio via a shared pattern: check for `this.sound.context`, handle browser autoplay policy (if `this.sound.locked`, defer initialization to the `'unlocked'` event), then call `generateAllSounds()` from `@/src/systems/audioEngine.js` to render all `SOUND_CONFIGS` from `@/src/systems/audio.js` into Phaser's audio cache via `OfflineAudioContext`. The `this.audioReady` flag gates all `playSound()` calls to prevent errors before sounds are rendered. `GameScene.createSounds()` additionally starts a continuous ambient drone via `createAmbientDrone()`, starts the looping rift static via `createRiftCrackle()` (volume driven per-frame by `updateRiftCrackle()` from player distance to the rift, so it grows louder on approach), and schedules recurring ambient sounds (distant_bang, drip, whisper) at pseudo-random intervals via `scheduleAmbientSound()`. `createSounds()` also initializes the per-scene `this._shotCounter` (seeds per-shot pitch variation) and the live threat-signal fields (`this.nearestEnemyDist`, `this.chasingCount`, `this.inSafeRoom`) the adaptive drone reads. Sound effects are wired to game events throughout GameScene (weapon fire, bullet hit, per-type enemy death/alert/ambient/fire sounds, player damage/death, item/lore/ammo pickup, battery recharge, door open/close, switch click, hiding enter/exit, stair transition, day complete, weapon switch) and ShopScene (purchase, denied, button clicks). All audio playback is modulated by audio settings from `@/src/systems/settings.js`: `GameScene.playSound(key)` multiplies by `getEffectiveVolume(master, sfx)` and skips playback when the result is zero. `GameScene.playSoundAtDistance(key, distance)` computes distance-based volume via `getDistanceVolume()` from `@/src/systems/audio.js` (linear falloff over `ENEMY_SOUND_RANGE` of 400px), then multiplies by `getEffectiveVolume(master, sfx)` and skips playback when either factor is zero. Both `playSound`/`playSoundAtDistance` ALSO apply `getShotVariation()` from `@/src/systems/audio.js` when a sound's config has `vary: true`: they pull a deterministic `{ detune, volumeScale }` off the incrementing `this._shotCounter` and pass `detune` plus a scaled `volume` into `this.sound.play()`, so repeated gunfire/footsteps/growls are not mechanically identical. `ShopScene.playSound()` multiplies `0.5 * getEffectiveVolume(master, sfx)`. **Adaptive ambient intensity** (the data path GameScene -> `computeIntensity` -> `drone.setIntensity`): each frame `updateEnemies()` tracks the nearest ACTIVE (non-lit-room) enemy distance and a chasing/attacking count and stores them as `this.nearestEnemyDist`/`this.chasingCount`/`this.inSafeRoom`; then `updateAmbientIntensity()` (called per-frame in the `if (this.audioReady)` block alongside `updateRiftCrackle()`) feeds those stored signals plus the player's health fraction, the rift distance, and the danger `waveCount` into `computeIntensity()` from `@/src/systems/audio.js` and calls `this.ambientDrone.setIntensity(...)` -- so the drone's dread pad, sub-rumble, heartbeat, and filter brightness rise with live threat and ebb back in a lit safe room. The ambient drone's gain is set on creation and updated at runtime via `setVolume(getEffectiveVolume(master, ambient))` (independent of `setIntensity`). Scheduled ambient sounds pass `getEffectiveVolume(master, ambient)` as the `volumeMultiplier` to `playAmbientSound()`. `GameScene.applyVolumeSettings()` re-reads audioSettings from the registry and updates the drone gain -- this is called by PauseScene when the player changes volume mid-game. TitleScene applies the same volume pattern to its own drone instance (the title drone is never given an intensity -- it stays at the calm evolving bed). The ambient drone, rift crackle, and ambient sound timer are stopped on scene exit (`stopAmbientAudio()` called in `onDayComplete()` and `onPlayerDeath()`). All `playSound()` and `playSoundAtDistance()` calls are wrapped in try-catch as a system boundary safety measure
