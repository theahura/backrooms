# Noridoc: systems

Path: @/src/systems

### Overview
- Pure functions implementing game math: player movement, 2D raycasting visibility, room wall geometry, coordinate-keyed deterministic unbounded room generation (`worldgen.js`, generated lazily a few cells ahead of the player), legacy multi-room/multi-floor level generation (`level.js`/`stairs.js`, retained for tests and the per-floor PRNG offsets but no longer driving `GameScene`), room interior variety (six archetypes: open, maze, columns, corridor, storage, cubicles), room visual theming (one desaturated sickly-yellow palette family across all room types), semantic-cluster furniture placement with type-dependent light blocking, starting room layout, enemy spawning/AI with cross-room pathfinding, enemy difficulty scaling, time-based danger escalation, combat/health/healing management, shooting/bullet logic, weapon types/inventory/switching/ammo management, battery/flashlight drain and recharge, flashlight on/off toggle state with hiding auto-off (`flashlight.js`), item spawning (including ammo pickups, with distance-from-entrance loot tier scaling), enemy loot drops on death (with per-type drop chances and distance-based quality scaling), gunfire noise alerts (weapon-range-based enemy alerting on firing), BFS hop-distance from the entrance over the door/stair graph, inventory management (with capacity constraints), shop/upgrade state management (including unlock-style upgrades), door state management, light switch state management, light spill through doorways, hiding state management, exploration/minimap tracking, per-run statistics tracking, touch/desktop input resolution, pause controls data, title menu option generation, lore note spawning and definitions, floor trap spawning (spike and noise traps with distance-gated placement), localStorage persistence, audio settings with volume control state management, seeded random number generation, procedural audio configuration, Web Audio sound synthesis, pixel art sprite definitions for texture generation, character animation definitions, and the AI-art manifest/helpers (`art.js`: prompts, target sizes, PNG-then-procedural texture-source selection, chroma-key background-removal decision, and shop-icon mapping) that drive the offline art generator and the runtime PNG/fallback choice
- Deliberately free of Phaser dependencies so they can be unit-tested in Node without a browser
- Tests live in `@/src/systems/__tests__/`

### How it fits into the larger codebase
- `@/src/scenes/GameScene.js` imports and calls all system modules during scene creation and on every frame during `update()`. Combat, shooting, weapon, battery, and run stats systems are called in the frame loop (cooldown ticking, battery drain -- routed through `updateBatteryWithFlashlight()` from `flashlight.js` so the battery only drains while the flashlight is effectively lit, bullet expiry, weapon switching, ammo consumption, time accumulation) and combat/shooting/runStats are also called in collision callbacks (damage application, bullet deactivation, kill recording). Inventory state is updated in item pickup callbacks and battery-use input. Weapon state is updated on Q-key or scroll-wheel switch, E-key pickup, firing (ammo consumption via `consumeAmmo()`), and ammo item pickup (`addAmmo()`). Firing is gated by both `canFire()` (cooldown) and `hasAmmo()` (ammo available) and `getActiveWeapon()` returning non-null. At scene init, `getUpgradeValue()` from `shop.js` is called to compute upgrade bonuses (weapon damage, fire rate, bullet range) as deltas from pistol base values, and these bonuses are applied additively to whatever weapon is active via `getEffectiveStats()` from `weapons.js`. The three starting weapon upgrade levels (`startingPistol`, `startingShotgun`, `startingRifle`) are checked as boolean flags and passed to `createWeaponState()`, which uses priority-based slot filling (rifle > shotgun > pistol) to fill up to 2 weapon slots -- when all three are purchased, the cheapest (pistol) is dropped. Scaling functions from `scaling.js` compute run-scaled enemy stats (HP, count, chase speed, contact damage). During room creation, `generateMazeWalls()` from `@/src/systems/maze.js` is called per non-starting room to produce internal wall segments (from one of six room archetypes) that integrate into the raycasting and physics systems, and `getRoomTheme()` from `@/src/systems/roomThemes.js` is called (via `getRoomType()` from `maze.js`) to determine per-room-type floor and wall colors for rendering. Audio timing functions from `@/src/systems/audio.js` (`getFootstepInterval`, `getAmbientSoundDelay`) are called each frame and on ambient scheduling; per-enemy-type sound functions (`getEnemySoundEvent`, `getEnemyDeathSound`, `getEnemyFireSound`, `getDistanceVolume`) are called from `updateEnemies()` and `onBulletHitEnemy()` for proximity-based enemy audio; sound synthesis from `@/src/systems/audioEngine.js` (`generateAllSounds`, `createAmbientDrone`, `playAmbientSound`) is called at scene creation and during gameplay for sound playback. `rollEnemyDrop()` from `@/src/systems/lootDrops.js` is called in `onBulletHitEnemy()` when an enemy dies to determine if items drop at the enemy's position. After firing, `getAlertedEnemies()` from `@/src/systems/gunfireNoise.js` is called with the active weapon's `noiseRange` and the player's room ID to determine which idle/search enemies within BFS hop distance should be alerted -- alerted enemy states are directly mutated to chase, gated by a 5-second cooldown (`GUNSHOT_ALERT_COOLDOWN`)
- `@/src/scenes/TitleScene.js` imports `loadGame()` and `clearSave()` from `persistence.js` to detect save state and handle new game/delete save actions. Imports `getMenuOptions()` and `getTitleText()` from `@/src/systems/titleMenu.js` for menu option generation. Imports `generateAllSounds()` and `createAmbientDrone()` from `@/src/systems/audioEngine.js` for ambient audio on the title screen. Imports `getEffectiveVolume()`, `cycleVolume()`, `saveSettings()`, `createDefaultSettings()`, and `VOLUME_LEVELS` from `@/src/systems/settings.js` for the settings overlay that controls audio volume
- `@/src/scenes/ShopScene.js` imports the shop system for upgrade definitions, pricing, purchase validation, and state management. ShopScene reads/writes `shopState` from `game.registry` and uses the pure functions from `shop.js` for all purchase logic. ShopScene also imports `saveGame()` from `persistence.js` to persist state to localStorage at each mutation point. ShopScene imports `generateAllSounds()` from `@/src/systems/audioEngine.js` to initialize sound synthesis on scene creation, playing UI feedback sounds (purchase, denied, click) for shop interactions
- `@/src/main.js` imports `loadGame()` and `saveGame()` from `persistence.js` to hydrate `game.registry` on boot (including `unlockedLocations` and `activeLocation`) and to flush state to localStorage on browser lifecycle events (`visibilitychange`, `beforeunload`). It also imports `loadSettings()` from `settings.js` to load audio volume preferences into `game.registry` under the key `'audioSettings'` on boot (from a separate localStorage key `backrooms_settings`). It also imports `detectTouchPrimary()` from `touchControls.js` at boot to decide the Phaser Scale mode (touch devices get `Scale.RESIZE` at native viewport size; desktop keeps `Scale.FIT` at fixed 1024x768)
- The world is generated lazily. `GameScene` no longer builds a whole multi-floor level up front; instead it generates ~2 rooms ahead of the player via `worldgen.js` and activates each room exactly once (see the scenes docs for the lazy-activation pipeline). Rooms stay static once generated. Floors are stacked as non-overlapping horizontal bands in a single physics world (`GameScene` constant `FLOOR_Y_OFFSET = 5_000_000`, large enough that unbounded floors never collide), and camera/physics world bounds are a large fixed region (`WORLD_BOUND`) rather than a computed level extent. The legacy `generateMultiFloorLevel()`/`getFloorRoomCounts()` path in `stairs.js` is retained for tests but unused at runtime
- Furniture segments from `furniture.js` are conditionally appended to the level's wall segments array based on the `blocksLight` property of each furniture type -- furniture with `blocksLight: true` (e.g., shelf, bookcase, armoire, closet) is added to the segment array, while furniture with `blocksLight: false` (e.g., table, desk, bed, vent) does not occlude the flashlight. Maze wall segments from `maze.js` are also appended to the wall segments array so internal room walls participate in raycasting. Closed door segments from `doors.js` are also pushed/spliced into the same wall segments array dynamically when doors are toggled. Items from `items.js`, lore notes from `lore.js`, traps from `traps.js`, light switches from `lightswitch.js`, the light spill system (`lightSpill.js`), the hiding system (`hiding.js`), and the exploration system (`exploration.js`) do not produce segments -- items and lore notes are small collectibles, switches are small interactable objects, light spill is a visual-only gradient system, hiding is a player state system that reuses existing furniture objects (any with `canHide: true`), and exploration is a tracking system that monitors which rooms the player has visited. Light switches interact with the enemy and darkness systems via lit room IDs rather than wall segments. The light spill system extends the lit room visual by drawing gradient rectangles through open doorways into the BitmapMask and light tint layer, consuming lit room IDs from `lightswitch.js` and door states from `doors.js`. The hiding system interacts with enemy AI via an optional `playerHidden` flag on `updateEnemyAI()`. The exploration system reuses `isPointInRoom()` from `lightswitch.js` for room detection
- Movement system output (`{x, y}` velocity) is passed directly to `player.setVelocity()`
- Visibility system output (polygon as `{x, y}` point array) is drawn into a Phaser Graphics object used as a BitmapMask to punch a hole in the darkness overlay
- Enemy system (`enemy.js`) reuses `raySegmentIntersection` from `visibility.js` for line-of-sight checks and `mulberry32` from `random.js` for deterministic spawning -- it follows the same pure-function, segment-based integration pattern as furniture. The pathfinding system (`pathfinding.js`) provides BFS-based cross-room navigation that feeds into enemy AI via an optional `navContext` parameter on `updateEnemyAI()`. `GameScene` builds two room graphs from `pathfinding.js`: a standard graph (via `buildRoomGraph()`, excludes closed doors, used for basic/spitter pathfinding) and a full graph (via `buildFullRoomGraph()`, includes closed doors with `closedDoorId` annotations, used for crawler pathfinding). BFS runs on both graphs from the player's room, and `GameScene` constructs per-enemy `navContext` objects that tell the enemy AI which doorway to move toward and (for crawlers) whether a closed door blocks the path. The pathfinding module consumes `room.doors` data and door states from `doors.js` to build its adjacency graphs, and reuses `getCurrentRoom()` from `exploration.js` for room detection
- The dependency flow is: `GameScene.activateRoom()` calls `createRoomWalls()` from `room.js` for each activated room to produce perimeter wall segments with door gaps (on floor 0 the rift door is deliberately left OUT of the gap list, so the rift stays sealed against light in both directions) -> `GameScene` adds light-blocking furniture segments from `furniture.js` (or `locations.js` for room 0, via `getLocationLayout()`) -> `GameScene` adds maze wall segments from `maze.js` per non-starting room -> `GameScene` adds closed door segments from `doors.js` -> all segments accumulate in a single mutable array that grows forever as the unbounded world is explored -> `visibility.js` and `enemy.js` both consume this segment array for raycasting, each frame through the `filterSegmentsNear()` bounding-box prefilter from `visibility.js` so per-frame raycast cost stays proportional to nearby segments. Door segments are dynamically added/removed when the player toggles a door. (The legacy `level.js` -> `room.js` whole-level flow is retained for tests but no longer runs at runtime.) Separately, `locations.js` provides the themed layout, colors, and exit position for room 0 (based on the active starting location), while `startroom.js` provides the crack visual points; `items.js` spawns collectibles per room (consuming the furniture list for overlap avoidance), including medkits that bypass inventory and heal via `applyHeal()` from `combat.js`; `inventory.js` tracks what the player has picked up (excluding medkits, which never enter inventory); `lightswitch.js` manages per-room switch states that feed into the darkness mask and enemy AI freeze logic; `hiding.js` manages whether the player is hidden inside `canHide` furniture (feeding into enemy AI via the `playerHidden` parameter on `updateEnemyAI()`); and `exploration.js` tracks visited rooms and computes minimap rendering data from player position and level geometry

### Core Implementation
- **Unbounded room generation** (`@/src/systems/worldgen.js`): the live world generator. The world is an infinite grid of rooms; everything about the room at grid cell `(gx, gy)` -- its content `seed` and its four possible doors -- derives from a hash of `(floorSeed, gx, gy)`, so a room is identical regardless of when or in what order it is generated, and adjacent cells always agree on the door they share (a door is bond-percolation-decided once for the shared edge). `getRoomAt(seed, gx, gy)` returns `{ id: "gx,gy", gridX, gridY, x, y, width, height, doors: [{ wall, offset, width, targetRoomId }], seed }` with door open-probability `DOOR_PROB = 0.62` (above the 2D percolation threshold so the open-door graph forms one infinite connected cluster -- the player is never boxed in). The entrance cell `(0,0)` (the store) has EXACTLY one opening: the rift in its east wall; its other three edges are forced sealed, so the store's sole connection to the backrooms is the rift. `isRiftDoor(room, door)` identifies that single opening from either side (the store's east door, or cell `(1,0)`'s west door) -- consumed by `GameScene` to seal the rift against light and treat it as a dimensional transition rather than a doorway. `roomsWithinRadius(seed, cx, cy, radius)` BFS-walks the open-door graph and returns all rooms reachable within `radius` door-hops (GameScene uses `radius = 2` for activation). `hasStairDown(seed, gx, gy)` is a deterministic `STAIR_PROB = 0.08` roll, never true at `(0,0)`. `localDistance(gx, gy)` is Manhattan distance from `(0,0)`. Constants: `ROOM_WIDTH = 720`, `ROOM_HEIGHT = 600`, `DOOR_WIDTH = 80` -- rooms were deliberately shrunk (from 1200x1000) in the liminal-scale pass so that, combined with larger character sprites and dense furniture, the world reads as claustrophobic rather than empty. This module is pure (no Phaser, no mutable state) and is the deterministic backbone of the lazy world -- `GameScene` re-stamps the string `id`/`targetRoomId` into numeric ids on activation but never re-derives geometry. Tests live in `@/src/systems/__tests__/worldgen.test.js`
- **Legacy multi-floor level generation** (`@/src/systems/stairs.js`): no longer drives `GameScene` (which now generates the world lazily via `worldgen.js`), but retained because its tests still pass and several seed/offset conventions are shared. `getFloorRoomCounts(runCount)` and `generateMultiFloorLevel(seed, floorRoomCounts)` are no longer called by the scene. `getFloorRoomCounts(runCount)` is a pure function that returns the floor room-count array based on the player's run count -- run 0-1 returns `[4, 3]` (2 floors), run 2-3 returns `[4, 3, 3]` (3 floors), run 4+ returns `[4, 3, 3, 2]` (4 floors). This is the run-scaling difficulty escalator: more floors means more battery drain to explore and return. `generateMultiFloorLevel(seed, floorRoomCounts)` wraps `generateLevel()` once per floor, offsetting each floor's rooms and wall segments by `floor * FLOOR_Y_OFFSET` (10000px) in world-space Y. Room IDs are re-numbered globally so they remain unique across floors. Each room gets a `floor` field. After generating all floors, `createStairConnections()` uses a seeded PRNG (seed offset +50000) to create up to 2 stair connections per adjacent floor pair (floor i to floor i+1), iterating all consecutive floor pairs generically. For floor 0, room 0 (the starting room) is excluded from stair candidates. Each stair pair uses `usedFrom` and `usedTo` Sets to avoid reusing rooms within a floor pair. Each stair object carries `fromFloor` and `toFloor` fields alongside room IDs and positions. Each stair's position is centered within its room (with 100px margin). Returns `{ rooms, wallSegments, stairs }`. `getFloorBounds(rooms, floor)` computes the axis-aligned bounding box for all rooms on a given floor -- retained and tested but no longer used by `GameScene` (the lazy world has no finite floor extent; the minimap now uses the player-centered window from `getWindowedMinimapData()` in `@/src/systems/exploration.js`). The legacy constant `FLOOR_ROOM_COUNTS=[4,3]` is still exported for backward compatibility. Constants: `FLOOR_Y_OFFSET=10000`, `STAIR_SIZE=60` (the legacy module's own offsets -- the live world uses `GameScene`'s `FLOOR_Y_OFFSET = 5_000_000` and `STAIR_SIZE`)
- **Single-floor level generation** (`@/src/systems/level.js`): `generateLevel(seed, roomCount)` places rooms on a grid using a seeded growth algorithm. Starting from room 0 at grid position (0,0), each subsequent room picks a random existing room, picks a random cardinal direction, and places a new room in the adjacent grid cell if it is empty. **Exception**: the first room placed (i=1) always uses `DIRECTIONS[1]` (east), so room 0's first connection (the crack to the backrooms) is deterministically on the east wall regardless of seed. The PRNG still consumes a direction roll for i=1 (to preserve sequence stability for subsequent rooms) but discards it in favor of the forced east direction. Paired doorways are added to both rooms at the same wall offset. After all rooms are placed, `addExtraDoors(rooms, grid, seed)` iterates over all rooms and their grid neighbors, and for each unconnected adjacent pair, probabilistically (~50%) creates a new door connection between them using a separate seeded PRNG (seed offset +80000) to preserve determinism of the primary generation. Returns `{ rooms, wallSegments }` where `wallSegments` is the flattened array of all room wall segments (with door gaps from both primary and extra doors). This is a pure function -- no Phaser dependency. It is called by `stairs.js` as the per-floor building block
- **Room geometry** (`@/src/systems/room.js`): `createRoomWalls(x, y, width, height, doors)` returns line segments forming a rectangular boundary with gaps for doorways. When `doors` is empty or omitted, it returns four simple wall segments (backward-compatible). When doors are present, each wall is split into sub-segments around the door gaps by `splitWall()`. Segments maintain clockwise winding order -- south and west walls are reversed via `reverseSegments()` so endpoint connectivity is preserved. This module is live at runtime: `GameScene.activateRoom()` calls it for every activated room to push perimeter occluder segments into the shared `wallSegments` array. (An earlier unbounded-world rework had left perimeters physics-only with no occluder segments, which let the flashlight shine through walls and enemies see through them -- restoring this call was the fix.)
- **Random** (`@/src/systems/random.js`): exports `mulberry32(seed)` which returns a stateful function that produces deterministic floats in [0, 1). Used by `level.js`, `furniture.js`, `enemy.js`, and `items.js`
- **Movement** (`@/src/systems/movement.js`): `calculateVelocity(keys, speed)` takes a key-state object and speed scalar, returns `{x, y}` velocity. Normalizes diagonal movement to match cardinal speed
- **Touch controls** (`@/src/systems/touchControls.js`): pure decision functions that fork desktop vs mobile input without any Phaser dependency, so the branching is unit-tested in Node. They exist as a separate module precisely so the input-mode logic does not get tangled into `GameScene.update()`. The functions are: `getMoveVelocity(angle, force, speed)` converts a joystick angle + normalized force into `{x, y}` velocity (force clamped to 0..1 so partial deflection is analog and full deflection never overshoots max speed); `resolveAimAngle({touchMode, aimActive, aimAngle, pointerAngle, lastAngle})` returns the mouse pointer angle on desktop, and on touch returns the aim-stick angle while the stick is deflected, otherwise the last angle (so the flashlight holds its direction when the thumb lifts off the aim stick to tap FIRE); `resolveFireIntent({touchMode, fireButtonDown, leftButtonDown})` fires on left button on desktop but on touch fires ONLY from the dedicated FIRE button (the browser reports touching the move/aim sticks as a left-button press, so this guard prevents stick-touches from firing -- the key invariant of the touch scheme); `resolveMoveVelocity({touchMode, moveActive, stickVelocity, keyboardVelocity})` picks the move-stick velocity when active in touch mode, otherwise the keyboard velocity; `detectTouchPrimary({maxTouchPoints, coarsePointer})` returns true if the primary pointer is coarse or the hardware reports touch points -- it is the single touch/desktop discriminator, called both by `@/src/main.js` (to choose the Phaser Scale mode at boot) and by `GameScene.setupInput()` (to set `this.touchMode`); `computeTouchLayout({width, height, stickRadius})` is the pure layout calculator that maps the live viewport size to screen-space positions for every touch control, returning `{ moveStick, aimStick, fireButton, useButton, actionRow, fullscreenButton }`. Its layout rules encode the mobile feedback fixes: the move stick is inset from the left edge by `stickRadius + 30` and the aim stick is symmetric on the right, both **vertically centered** at `round(height/2)` (this is the deliberate fix for downward thumb travel being eaten by the browser's bottom gesture zone -- bottom-anchored sticks could not deflect fully down); the fire button (radius 52) sits just inside the aim stick on the right-thumb side; the USE button is placed directly below the right aim stick (sharing `aimStick.x`, just under the stick and clamped to stay on-screen on short landscape phones) so it falls within right-thumb reach -- this is the deliberate response to feedback that USE needed to be closer to the aim stick; the action row holds the WPN, BATT, and LIGHT buttons (LIGHT is the touch equivalent of the desktop F-key flashlight toggle) spaced around the horizontal center and sitting 72px above the bottom edge; the fullscreen button sits top-center. Because touch devices run under `Scale.RESIZE` (canvas == device viewport), `GameScene` calls this on create and on every `resize` event so the UI re-lays-out on orientation flips, URL-bar hide/show, and fullscreen toggles. The portrait rotate prompt is implemented CSS-only in `@/index.html` via a `(orientation: portrait) and (pointer: coarse)` media query, not in this module. The module has no Phaser dependency and no seeded PRNG
- **Furniture** (`@/src/systems/furniture.js`): `generateRoomFurniture()` uses a seeded PRNG to fill a room with semantic furniture CLUSTERS rather than uniform scatter -- recognizable arrangements (e.g., an office pod of desks, a waiting row of couches, a storage aisle of shelves) anchored at random positions, then topped up with singleton pieces. The piece target scales with room area (with a floor minimum), so rooms read as densely furnished-but-abandoned: this is the deliberate fix for the prior uniform scatter, which made rooms feel empty and incoherent rather than liminal. All placement rejects overlap with already-placed pieces and the player spawn zone. `createFurnitureSegments()` produces four `{x1,y1,x2,y2}` segments per furniture item in the same format as wall segments. Each furniture type defines a `canHide` property and a `blocksLight` property. These two properties are independent: some furniture is hideable but does not block light (e.g., table, bed -- low-profile, the player hides underneath), some blocks light but is not hideable (e.g., shelf, bookcase -- tall but not enclosed), and some is both hideable and blocks light (e.g., armoire, closet -- tall enclosed spaces the player hides inside). The `blocksLight` property controls whether `GameScene` pushes the furniture's segments into the `wallSegments` array for raycasting -- only light-blocking furniture participates in flashlight occlusion and enemy LOS. The `canHide` property is consumed by `@/src/systems/hiding.js` to determine which furniture items the player can hide inside. `FURNITURE_TYPES` is a static object (including couch and counter, added for the cluster templates and room 0); type selection comes from the cluster templates plus a small fixed singleton list, so adding a new type to `FURNITURE_TYPES` does NOT automatically make it spawn -- it must appear in a template or the singleton list. Room 0 (the starting room) does not use `generateRoomFurniture()` -- it uses `getLocationLayout()` from `@/src/systems/locations.js` instead, which produces location-specific furniture in the same `{ type, x, y, width, height, color, canHide }` format
- **Room interior variety / maze** (`@/src/systems/maze.js`): generates internal walls and obstacles inside rooms to create varied layouts. `getRoomType(seed)` uses a seeded PRNG (seed offset +70000) to classify each room into one of six types: open (deliberately rare, ~5% -- nearly every room has internal structure), maze (shallow recursive-division walls), columns (staggered column grid), corridor (the most common, deep recursive-division creating winding passages), storage (rejection-sampled rectangular crates), cubicles (grid-based U-shaped partition templates). `generateMazeWalls(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, doors)` returns an array of `{x1,y1,x2,y2}` segments matching the universal wall segment format. All room types share the same door-zone rejection logic via `segmentCrossesDoorZone()` to ensure doors remain accessible. The original `subdivide()` function was unified with corridor generation into `subdivideGeneric(bounds, segments, rand, doorZones, depth, maxDepth, minSize, gapWidth)` -- maze rooms subdivide shallowly with wider passages while corridor rooms subdivide deeply with narrower ones. Storage rooms use rejection-sampling to place rectangular crates, producing 4 segments per crate in the same format as columns. Cubicle rooms use a grid-template approach where each cell is either skipped or rendered as a U-shaped partition with one randomly chosen open side, producing 3 wall segments per active cell. All sizing/gap constants were rescaled for the 720x600 liminal rooms: passage gaps are roughly 64-72px (about 1.5x the player's rendered width), so interiors feel like narrow corridors rather than open halls. Room 0 (the starting room) is never given internal walls -- `GameScene` skips the `createRoomMaze()` call for `room.id === 0`
- **Room visual theming** (`@/src/systems/roomThemes.js`): pure-data module that maps room types to floor and wall colors. `getRoomTheme(roomType)` accepts a room type string (open, maze, columns, corridor, storage, cubicles) and returns `{ floorColor, wallColor }`. All six types now share ONE desaturated, sickly-yellow color family with only subtle per-type variation -- the monotone sameness is deliberate: repeated rooms should feel familiar-but-off (liminal) rather than visually distinct. (An earlier version gave each type its own contrasting palette; that read as a normal dungeon, not the backrooms.) Unknown room types fall back to a default theme in the same family. This module is consumed by `@/src/scenes/GameScene.js` in three places: `drawRoom()` for floor and wall rendering, `drawRoomDoorways()` for doorway floor color, and `createRoomMaze()` for internal maze wall color. Room 0 (the starting room) does not use this module -- it uses `@/src/systems/locations.js` colors instead. The room type input comes from `getRoomType()` in `@/src/systems/maze.js`, which derives the type deterministically from the room's seed. This module has no Phaser dependency, no seeded PRNG, and no state -- it is a pure lookup table
- **Starting room** (`@/src/systems/startroom.js`): contains legacy store layout data and utility functions for room 0 visuals. `generateStoreLayout()` and `getExitPosition()` are still exported but are no longer called by `GameScene` -- room 0's furniture layout and exit position are now driven by `@/src/systems/locations.js` (via `getLocationLayout()` and `getLocationExitPosition()`). `STORE_FLOOR_COLOR` and `STORE_WALL_COLOR` are still exported but are no longer imported by `GameScene`. The one function still actively used by `GameScene` is `generateCrackPoints(startX, startY, length, segments, vertical)`, which produces a jagged polyline (alternating +/-3px offsets) for the rift visual at room 0's east opening -- this visual is the same regardless of the active starting location
- **Locations** (`@/src/systems/locations.js`): defines starting location themes for room 0 and (in a now-unused helper) spawns alternate exit portals. The `LOCATIONS` array contains location definitions (e.g., furniture store, office building, parking garage, hotel lobby, laundromat), each with an `id`, `name`, `floorColor`, `wallColor`, and `furniture` array (using relative positioning via `relX`/`relY`). `getLocation(id)` looks up a location by ID. `getLocationLayout(locationId, roomX, roomY, roomWidth, roomHeight, wallThickness)` converts relative furniture positions to absolute pixel coordinates within the room's inner bounds, skipping items that overlap the center spawn zone (60x60px) or exceed room bounds -- this is the same algorithm as the legacy `generateStoreLayout()` in `startroom.js`, now parameterized by location. `getLocationExitPosition(locationId, room, wallThickness)` returns the exit zone position (west-center of the room). `generateAlternateExit(rooms, stairs, seed, unlockedLocationIds)` uses a seeded PRNG (seed offset +100000) to probabilistically (20% chance) place an alternate exit portal in a non-stair room on a deeper floor (floor > 0). It is retained and tested but **no longer called by `GameScene`** -- the unbounded-world rewrite removed mid-run alternate exits, so the only world exit is the floor-0 entrance zone. The default starting location is `'store'`; `unlockedLocations` (which the shop cycles through) is unchanged but no longer grows via in-game portals
- **Pathfinding** (`@/src/systems/pathfinding.js`): pure functions providing BFS-based navigation on the room connectivity graph for cross-room enemy movement. Two graph builders exist: `buildRoomGraph(rooms, doorStates)` constructs an adjacency list from `room.doors`, excluding connections where a closable door is closed (used for basic/spitter pathfinding); `buildFullRoomGraph(rooms, doorStates)` includes all connections including through closed doors, annotating each edge with a `closedDoorId` (null if the door is open or no closable door exists) -- used for crawler pathfinding so crawlers can path through closed doors. `bfsFromRoom(graph, startRoomId)` runs breadth-first search and returns a `cameFrom` Map for path reconstruction. `getNextDoorway(graph, cameFrom, enemyRoomId, playerRoomId)` traces the BFS path backward from the enemy's room to find the immediate next room toward the player, then looks up the corresponding doorway's world-space center. `getClosedDoorOnPath(fullGraph, cameFrom, enemyRoomId, playerRoomId, doorStates, rooms)` checks whether the next step on the BFS path from the enemy toward the player crosses a closed door; returns `{ doorId, center }` if so, or null if the path is clear, same-room, or unreachable -- consumed by `GameScene` to provide `closedDoorOnPath` in the crawler's `navContext`. `getDoorwayCenter(room, door)` computes the pixel center of a door opening based on its wall, offset, and width. `getRoomDistance(graph, fromId, toId)` returns the BFS hop count between two rooms (or -1 if unreachable). Constants: `DOORWAY_SEEK_CHANCE` (0.15, probability an idle enemy will seek a doorway during a wander tick), `ROOM_TRANSITION_COOLDOWN` (10000ms, cooldown after an enemy enters a new room before it can seek another), `MAX_ROOM_DISTANCE` (2, maximum BFS distance from spawn room an idle enemy will wander), `MAX_ROOM_ENEMIES` (4, cap on enemies per room to prevent crowding). The module does not use a seeded PRNG and does not produce wall segments -- it is a read-only graph traversal system that operates on room geometry and door states produced by other systems
- **Enemy spawning and AI** (`@/src/systems/enemy.js`): `generateRoomEnemies()` places enemies per room using seeded PRNG (room 0 is always empty -- it is the player spawn room). It accepts optional `extraCount` (default 0) and `runCount` (default 0) parameters. `extraCount` adds enemies beyond the base 1-2, capped at 5 total per room. `runCount` controls enemy type distribution via `getEnemyType()`: run 0 = 100% basic, run 1 = 80/20 basic/crawler, run 2+ = 50/25/25 basic/crawler/spitter. Each spawned enemy carries a `type` field. Three enemy types exist with different speed constants: basic (chase 80, wander 30), crawler (chase 112, wander 50), and spitter (chase 60, wander 20). `hasLineOfSight()` casts a single ray from enemy to player against the wall segment array to determine visibility. `updateEnemyAI()` is a pure state machine with five states: idle/chase/search (used by all types), attack (spitter-only), and opening_door (crawler-only). The 7th optional parameter `navContext` (default null) enables cross-room movement when provided. In the **chase** state, if the enemy loses LOS and is in a different room from the player with a known doorway path, it moves toward the next doorway at chase speed instead of transitioning to search -- this lets enemies pursue the player through connected rooms. For crawlers specifically, if `navContext.closedDoorOnPath` is set and the crawler is within `CRAWLER_DOOR_INTERACT_RANGE` (60px) of the closed door, it transitions to `opening_door` instead of continuing to chase. In the **idle** state, when the wander timer fires and `navContext.seekDoorway` is true with cooldown expired, the enemy sets `targetDoorway` and moves toward it at wander speed. The `roomTransitionCooldown` field is decremented each tick when navContext is provided. Without navContext, behavior is identical to before (backward compatible). The **attack** state activates when a spitter has LOS and is within `SPITTER_ATTACK_RANGE` (250px) -- the spitter stops, counts down `attackCooldown`, and sets `wantsToFire = true` when the cooldown expires (GameScene reads this flag to fire an enemy projectile). The **opening_door** state is entered only by crawlers from chase state: the crawler stops moving (zero velocity), counts down `doorOpenTimer` from `DOOR_OPEN_TIME` (1000ms), and when the timer expires sets `wantsToOpenDoor = true` and `targetDoorId` on the result object before transitioning back to chase. If the crawler gains LOS to the player while in `opening_door`, it interrupts and transitions directly to chase. This `wantsToOpenDoor`/`targetDoorId` signal pattern mirrors the spitter's `wantsToFire` pattern: the pure function signals an intent, and GameScene executes the side effect (calling `onToggleDoor()`). `getTypeSpeeds()` maps the enemy type to the correct speed constants, falling back to the `chaseSpeed` parameter for basic enemies so that scaling from `@/src/systems/scaling.js` is preserved. Constants: `DOOR_OPEN_TIME` (1000ms), `CRAWLER_DOOR_INTERACT_RANGE` (60px)
- **Danger escalation** (`@/src/systems/danger.js`): pure functions implementing a time-based enemy wave spawning system that creates escalating pressure as the player spends time outside safe rooms. `createDangerState()` returns `{ unsafeTime: 0, waveCount: 0 }`. `updateDangerTime(state, delta, isInSafeRoom)` accumulates `unsafeTime` when the player is not in a safe room (room 0 or any lit room); returns the state unchanged when in a safe room -- the timer pauses but does not reset. `shouldSpawnWave(state)` returns true when accumulated unsafe time crosses the next wave threshold (`(waveCount + 1) * DANGER_WAVE_INTERVAL`), unless `waveCount` has reached `MAX_DANGER_WAVES`. `advanceWave(state)` increments `waveCount` to prevent the same wave from re-triggering. `getWaveComposition(waveCount)` determines both the count and types of enemies per wave: count scales as `min(1 + floor(waveCount / 3), 4)`, and types shift from all-basic (waves 0-1) to basic/crawler mix (waves 2-3) to crawler/spitter mix (waves 4-5) to a mixed rotation of all three types (waves 6+). `getSpawnRoom(rooms, playerRoomId, litRoomIds, currentFloor)` selects the first candidate room that is not room 0, not the player's current room, not lit, and on the same floor as the player; returns null if no valid room exists. Constants: `DANGER_WAVE_INTERVAL = 30000` (30 seconds), `MAX_DANGER_WAVES = 8`. The danger state is per-run (initialized in `GameScene.init()`, not persisted). This module follows the same pure-function immutable-state pattern as combat, battery, and hiding. It has no Phaser dependency and no seeded PRNG -- spawn positions within the chosen room use `Math.random()` in `GameScene.spawnDangerWave()` rather than deterministic seeding, since dynamically spawned enemies do not need cross-session reproducibility
- **Run statistics** (`@/src/systems/runStats.js`): pure functions tracking per-run gameplay statistics for the end-of-run summary screen. Follows the same immutable-state pattern as combat, battery, and danger. `createRunStats()` returns `{ enemiesKilled: 0, timeElapsed: 0, maxFloor: 0 }`. `recordKill(stats)` increments the kill counter. `updateTime(stats, delta)` accumulates elapsed time from the frame delta. `updateMaxFloor(stats, floor)` tracks the highest floor reached (uses `Math.max` so returning to a lower floor does not decrease the value). `formatTime(ms)` converts milliseconds to a `"M:SS"` display string. `getSummaryData(stats, survived, treasureValue, roomsExplored, hp, maxHp)` produces a structured summary object `{ title, titleColor, accentColor, lines }` consumed by `@/src/scenes/RunSummaryScene.js` -- the title is "RETURNED" (gold) on survival or "YOU DIED" (red) on death ("escaped" was deliberately dropped: the backrooms are explored of one's own volition), and `lines` is an array of `{ label, value }` pairs covering time, rooms explored, enemies killed, deepest floor, treasure, and (on survival only) HP remaining. The summary intentionally shows only a raw rooms-explored count, never an "X / total" ratio -- the world is unbounded, so there is no total (the old `totalRooms` parameter was removed). The runStats state is initialized in `GameScene.init()` alongside other per-run state objects and updated in three places: `update()` (time accumulation each frame), `onBulletHitEnemy()` (kill recording when `isEnemyDead()` is true), and stair teleportation (max floor tracking when `this.currentFloor` changes). The state object is passed to RunSummaryScene at end-of-run and is not persisted across runs. This module has no Phaser dependency and no seeded PRNG
- **Enemy difficulty scaling** (`@/src/systems/scaling.js`): pure functions that compute run-scaled enemy stats from base values and `runCount`. `getEnemyHP(baseHP, runCount)` applies +25% per run capped at 3x base. `getEnemyCount(runCount)` returns extra enemies per room (floor of runCount/2, capped at 3). `getEnemyChaseSpeed(baseSpeed, runCount)` applies +5% per run capped at run 6. `getEnemyDamage(baseDamage, runCount)` adds +5 per run capped at run 4. All functions are type-agnostic -- they accept base values and return scaled values, so GameScene calls them once per enemy type in `init()` with each type's base constants (e.g., `ENEMY_MAX_HP`, `CRAWLER_MAX_HP`, `SPITTER_MAX_HP`). Precomputed scaled values are stored as instance properties (e.g., `this.scaledEnemyHP`, `this.scaledCrawlerHP`, `this.scaledSpitterHP`, and similarly for chase speed and damage). The module has no Phaser dependency and no seeded PRNG
- **Hiding** (`@/src/systems/hiding.js`): manages player hiding state for furniture-based stealth. Follows the same pure-function pattern as `doors.js` and `lightswitch.js`. `createHidingState()` returns `{ isHiding: false, furnitureRef: null }`. `enterHiding(state, furnitureRef)` returns a new state with `isHiding: true` and the furniture reference (which carries `x, y, width, height` used by GameScene to set physics bounds). `exitHiding()` returns the default un-hidden state. `findNearestHideable(furnitureByRoom, px, py, maxRange)` iterates over all rooms' furniture, filtering for items with `canHide: true` (any furniture type in `@/src/systems/furniture.js` that has this flag set), and returns the nearest one within `HIDE_INTERACT_RANGE` (80px) or null. Exports `HIDING_SPEED_MULTIPLIER = 0.5` consumed by `GameScene` to halve player movement speed while hidden. The hiding system does not produce wall segments or consume the seeded PRNG -- it reuses furniture objects already placed by `furniture.js`
- **Exploration** (`@/src/systems/exploration.js`): tracks room visitation and computes minimap rendering data. Follows the same pure-function pattern as other system modules. `createExplorationState()` returns `{ visitedRoomIds: Set, currentRoomId: null }`. `getCurrentRoom(playerX, playerY, rooms)` iterates rooms and uses `isPointInRoom()` from `@/src/systems/lightswitch.js` to find which room contains the player. `updateExploration(state, playerX, playerY, rooms)` returns unchanged state if the player is in the same already-visited room (performance optimization), otherwise returns a new state with the room added to the visited set. `getMinimapData(state, rooms, levelBounds, playerPos, exitPos, screenWidth, floorFilter)` computes all rendering coordinates for the minimap by scaling a whole-floor bounding box into a fixed-size minimap box (160x130 pixels) -- this whole-floor variant is retained and tested but is **not** used by `GameScene` anymore (an unbounded world has no finite floor bounds). `getWindowedMinimapData(state, rooms, playerPos, exitPos, screenWidth, floorFilter, floorYOffset, windowCells)` is the live variant: it renders a **player-centered grid window** of `windowCells` (default `MINIMAP_WINDOW_CELLS = 4`) cells in every direction around the player's current cell on the current floor, mapping world coordinates relative to the window origin at a fixed scale and clamping dots to the box. It only includes visited rooms within the window and on the current floor (using `room.gridX/gridY` and the floor's Y-band offset). Both variants return `{ bgRect, roomRects, playerDot, exitDot }` with an `isCurrent` flag per room. The module exports dimension and color constants consumed by `GameScene.drawMinimap()`. Note: the exploration system always tracks visited rooms even when the minimap upgrade has not been purchased -- the rendering is gated by `this.hasMinimap` in `GameScene`, but the tracking data remains available for future features. The exploration system does not use a seeded PRNG and does not produce wall segments -- it is a read-only tracking system that reacts to player position
- **Persistence** (`@/src/systems/persistence.js`): manages localStorage save/load for cross-session game state. `saveGame(shopState, runCount, collectedLore, unlockedLocations, activeLocation)` serializes a versioned envelope `{ version: 3, shopState, runCount, collectedLore, unlockedLocations, activeLocation }` to localStorage under the key `backrooms_save`. Parameters default to safe values if omitted: `collectedLore` to `[]`, `unlockedLocations` to `['store']`, `activeLocation` to `'store'`. `loadGame()` parses the saved JSON, returns `null` if the data is missing, corrupted, or from a future save version (higher than `SAVE_VERSION`), and defaults missing fields to safe fallback values -- backward-compatible with v1 saves (lacking `collectedLore`) and v2 saves (lacking `unlockedLocations` and `activeLocation`). `clearSave()` removes the save key. All three functions wrap localStorage access in try-catch so the game degrades gracefully when localStorage is unavailable (e.g., private browsing, storage quota exceeded). The module has no Phaser dependency and no seeded PRNG -- it is a system boundary module that bridges in-memory `game.registry` state to browser storage
- **Audio configuration** (`@/src/systems/audio.js`): pure config module defining all sound effect parameters, per-enemy-type sound logic, and behavioral functions for audio timing. `SOUND_CONFIGS` is a flat object mapping sound keys (e.g., `pistol_fire`, `zombie_growl`, `crawler_skitter`, `spitter_gurgle`, `door_open`, `footstep`, `shop_purchase`, `spike_trap`, `noise_trap_alarm`, `loot_drop`) to synthesis parameters -- each entry specifies a `type` (one of: gunshot, noise_burst, chime, tone, sweep_down, growl, skitter, gurgle, door, door_thud, filtered_noise) plus type-specific fields (frequencies, Q values, decay rates, note arrays, etc.). Each enemy type has a distinct set of sounds: ambient (zombie_growl, crawler_skitter, spitter_gurgle), alert (zombie_alert, crawler_alert, spitter_alert), death (zombie_death, crawler_death, spitter_death), and spitters have a fire sound (spitter_fire). An internal `ENEMY_TYPE_PREFIX` map translates enemy types to sound key prefixes (basic->zombie, crawler->crawler, spitter->spitter), and `AMBIENT_SOUND_SUFFIX` maps types to their ambient sound name (basic->growl, crawler->skitter, spitter->gurgle). `AMBIENT_SOUND_TYPES` defines the three ambient background sounds (distant_bang, drip, whisper) used for atmospheric audio between gameplay events. Exported constants `ENEMY_SOUND_RANGE` (400px), `ENEMY_SOUND_COOLDOWN_MIN` (4000ms), and `ENEMY_SOUND_COOLDOWN_MAX` (8000ms) control proximity-based enemy audio. `getFootstepInterval(speed)` computes milliseconds between footstep sounds as a function of player velocity (`BASE_FOOTSTEP_DISTANCE / speed * 1000`), returning `Infinity` when speed is zero (no footsteps when stationary). `getAmbientSoundDelay(time)` produces a pseudo-random delay between 5-30 seconds using a Knuth multiplicative hash of the current time, creating irregular ambient sound spacing without a seeded PRNG. `getSoundConfig(key)` is a lookup returning the config for a given key or null. `getEnemySoundEvent(oldState, newState, type, soundCooldown, delta)` is the per-enemy sound state machine: returns an alert sound on any transition to chase state, returns the type's ambient sound when the cooldown expires, and otherwise decrements the cooldown -- returning `{ soundKey, newCooldown }`. `getEnemyDeathSound(type)` returns the per-type death sound key (e.g., `'zombie_death'` for basic). `getEnemyFireSound(type)` returns `'spitter_fire'` for spitters, null for other types. `getDistanceVolume(distance, maxRange, baseGain)` computes linear distance-based volume attenuation, returning 0 when beyond `maxRange`. `getRiftCrackleVolume(distance)` reuses that falloff (over `RIFT_HEAR_RANGE`) to map the player's distance from the entrance rift to the gain of the looping rift-static sound -- `GameScene` calls it every frame to drive the crackle that grows louder as the player approaches the rift. This module has no Phaser dependency and no browser API dependency -- it is fully testable in Node.js. It does not use a seeded PRNG and audio does not affect level determinism
- **Sprite definitions** (`@/src/systems/sprites.js`): centralized pixel art sprite data for all game entities rendered via Phaser's `textures.generate()` data-array format. `SPRITE_DEFS` is an object mapping texture keys to `{ data, palette, pixelWidth, pixelHeight }` definitions. `data` is an array of strings where each character indexes into `palette` (a map of characters to hex colors); `.` characters are transparent. `pixelWidth`/`pixelHeight` are scaling multipliers that control the rendered size of each pixel in the data array. Two palette groups exist: shared `PALETTES` (wood, metal, items, weapons, bullets, door, switchPl, lore, trap) for environment/object sprites, and `CHARACTER_PALETTES` (player, enemy, crawler, spitter) for character sprites. Sprite keys follow a naming convention: `furniture_{type}` for furniture, `item_{type}` for collectibles, `pickup_{weaponId}` for weapon pickups, `bullet_{weaponId}` and `enemyBullet` for projectiles, `door_closed` for doors, `switch_on`/`switch_off` for light switches, `lore_note` for lore items, `trap_{type}` for floor traps (e.g., `trap_spike`, `trap_noise`), and `{entityType}_{state}_{frame}` for character animation frames (e.g., `player_idle_0`, `enemy_walk_1`, `crawler_walk_0`, `spitter_attack_0`). Furniture sprites use `pixelWidth: 5` so their rendered pixel dimensions match the `width`/`height` values in `FURNITURE_TYPES` from `@/src/systems/furniture.js` (e.g., a table data array of 16x10 characters at pixelWidth 5 renders to 80x50 pixels). Character sprites use `pixelWidth: 1` -- the player and basic enemy/spitter frames are 20x20 pixel data arrays (rendering to 20x20 pixels), crawlers are 14x14 pixel data arrays (matching the crawler's smaller body size). Item, weapon pickup, bullet, switch, and lore sprites also use `pixelWidth: 1`. The door sprite uses `pixelWidth: 2` and `pixelHeight: 2`. `ANIM_DEFS` is an exported object mapping animation keys (e.g., `player_idle`, `enemy_walk`, `spitter_attack`) to `{ frames, frameRate, repeat }` configuration -- this is pure data consumed by `GameScene.generateSpriteTextures()` to create Phaser animations alongside textures. Walk animations use higher frame rates than idle (e.g., crawler walk at 8fps vs idle at 1fps). `repeat: -1` means loop forever; `repeat: 0` means play once. `getSpriteConfig(key)` looks up a sprite definition by key. `getAllSpriteKeys()` returns all defined sprite keys. `getAllAnimKeys()` returns all defined animation keys. `getEnemyTextureKey(type)` maps an enemy type string ('basic', 'crawler', 'spitter') to the initial texture key for that enemy (e.g., 'crawler_idle_0'). `getAnimKeyForState(entityType, aiState)` maps an entity type and AI state to the correct animation key (e.g., `('spitter', 'attack')` -> `'spitter_attack'`, `('basic', 'chase')` -> `'enemy_walk'`); for basic enemies, the prefix is remapped from 'basic' to 'enemy' to match sprite key conventions. This module has no Phaser dependency -- it is pure data. These procedural definitions are the **fallback** layer: `GameScene.generateSpriteTextures()` only calls `this.textures.generate()` for a key when no AI-art PNG was already loaded for it (see `@/src/systems/art.js` and the GameScene/ShopScene `preload()`), then calls `this.anims.create()` once per animation key, all before any creation method references the textures. Every furniture type (including couch and counter, used both by `@/src/systems/locations.js` for room 0 and by the cluster templates in `@/src/systems/furniture.js`) now has a procedural definition here, so all furniture renders as a sprite even with no PNG; the Graphics `fillRect` path in `GameScene.createRoomFurniture()` remains only as a last-resort fallback for a type with neither
- **AI-art manifest and helpers** (`@/src/systems/art.js`): pure module that is the single source of truth for the AI-generated pixel art, consumed by BOTH the offline generator (`@/scripts/generate-art.mjs`) and the runtime scenes. `ART_MANIFEST` is an array where each entry produces ONE generated image written to every texture key in its `keys` (character entries fan a single sprite out across all their animation frames -- AI multi-frame walk cycles are not consistent enough for dimly-lit top-down sprites -- and `chroma: false` marks opaque entries like the shop background that skip background removal); each entry carries the model `prompt`, target `width`/`height`, and `keys`. Only meaningful sprites are listed; tiny micro-sprites (bullets, small coins) are intentionally absent and stay procedural. Prompts use strict-orthographic phrasing (ceiling-mounted camera, 90-degree overhead, describing ONLY the object's top surface -- e.g., "the flat rectangular top panel", "no face visible") because the model otherwise tends to return front-facing or perspective views even when asked for "top-down"; this wording was the fix for the front-facing player and furniture sprites. `getArtSpriteKeys()` flattens all manifest keys. `getTextureSource(hasTexture, key)` returns `'png'` or `'procedural'` given a `hasTexture` predicate (the scenes pass Phaser's `this.textures.exists`) -- this encodes the PNG-then-procedural fallback decision. `colorDistance()` and `isBackgroundSample()` are the per-pixel chroma-key decision used by the generator's flood-fill background remover: nano banana cannot output transparency and its solid background colour is inconsistent (green, grey-green, sometimes purple), so the generator samples the image's OWN corner colour and keys out pixels within `BG_TOLERANCE` of it, stopping at the sprite's dark outline. `shopIconKey(upgradeId)` maps a shop upgrade id (validated against `UPGRADES` from `@/src/systems/shop.js`) to its `shop_icon_{id}` texture key, or null -- consumed by `@/src/scenes/ShopScene.js` for per-row upgrade icons. This module has no Phaser dependency and no seeded PRNG. Tests live in `@/src/systems/__tests__/art.test.js`
- **Audio settings** (`@/src/systems/settings.js`): pure function module managing audio volume settings independently from game saves. Settings are persisted to localStorage under a separate key (`backrooms_settings`, defined as `SETTINGS_KEY`) so volume preferences survive game save deletion. `VOLUME_LEVELS` is the ordered cycle array `['OFF', 'LOW', 'MED', 'HIGH', 'FULL']`. `VOLUME_VALUES` maps each level to an exponential perceptual value: OFF=0, LOW=0.05, MED=0.15, HIGH=0.4, FULL=1.0. `createDefaultSettings()` returns `{ masterVolume: 'FULL', sfxVolume: 'FULL', ambientVolume: 'FULL' }` -- defaults preserve pre-settings behavior (everything at full volume). `cycleVolume(currentLevel)` advances to the next level in the cycle (wraps from FULL back to OFF). `getVolumeValue(level)` returns the numeric multiplier for a named level (falls back to 1.0 for unknown levels). `getEffectiveVolume(masterLevel, categoryLevel)` multiplies the master and category numeric values to produce the final multiplier applied to all audio calls -- this is the core function consumed by all scenes. `saveSettings(settings)` and `loadSettings()` are the localStorage boundary, wrapped in try-catch per the project's system boundary convention. `loadSettings()` validates all three fields against `VOLUME_LEVELS` and returns defaults if any field is invalid or localStorage is unavailable, handling corruption gracefully. This module follows the same pure-function pattern as `@/src/systems/persistence.js` and sits at the same system boundary (browser storage). Settings flow through `game.registry` (key `'audioSettings'`) for in-memory cross-scene access, loaded once at boot by `@/src/main.js`. Scenes read settings from the registry and call `getEffectiveVolume()` to compute volume multipliers for every `playSound()`, `playSoundAtDistance()`, and ambient drone gain call. Tests live in `@/src/systems/__tests__/settings.test.js`
- **Audio engine** (`@/src/systems/audioEngine.js`): Web Audio synthesis engine that renders sound buffers from the configs defined in `@/src/systems/audio.js` and manages the ambient drone. Unlike other system modules, this module depends on browser Web Audio APIs (`OfflineAudioContext`, `AudioContext` nodes) and cannot be unit-tested in Node.js. `generateAllSounds(audioContext, cache)` iterates all entries in `SOUND_CONFIGS`, dispatches each to a type-specific renderer function via a `RENDERERS` map, renders the sound via `OfflineAudioContext`, and injects the resulting `AudioBuffer` into Phaser's audio cache so sounds can be played via `this.sound.play(key)`. All rendering happens in parallel via `Promise.all`. The `RENDERERS` map covers all synthesis types including `renderSkitter` (rapid highpass-filtered noise clicks for crawler ambient sounds) and `renderGurgle` (resonant bandpass-filtered noise with LFO modulation for spitter ambient/death sounds). `createAmbientDrone(audioContext, destination)` builds a continuous low-frequency drone using three detuned oscillators (two sawtooth at 120/120.5 Hz for beating, one sine sub-bass at 60 Hz), LFO-modulated gain (0.3 Hz), and a 400 Hz lowpass filter -- returns `{ start(), stop(), setVolume(value) }`. The `setVolume(value)` method multiplies the base gain (0.06) by the provided value, allowing runtime adjustment of the drone's volume without stopping/restarting oscillators -- consumed by scenes when audio settings change mid-play. `playAmbientSound(soundManager, time, volumeMultiplier)` selects an ambient sound type deterministically from the current time (using the same Knuth hash approach as `getAmbientSoundDelay`) and plays it via Phaser's sound manager at `0.5 * volumeMultiplier` volume, skipping playback when the computed volume is zero. `createRiftCrackle(audioContext, destination)` builds the entrance rift's crackling static: a looped white-noise buffer through a bandpass (the hiss) plus a sparse impulse buffer through a highpass (the pops), returning `{ start(), stop(), setVolume(value) }`; unlike the drone's multiplier, `setVolume` takes the absolute gain -- `GameScene` drives it per-frame from `getRiftCrackleVolume()` in `@/src/systems/audio.js` times the ambient volume setting. This module is a system boundary -- it bridges the pure config in `audio.js` to browser audio APIs, similar to how `persistence.js` bridges game state to localStorage
- **Visibility / raycasting** (`@/src/systems/visibility.js`): `getFlashlightPolygon(origin, angle, coneAngle, segments, maxRange)` builds the flashlight polygon by casting rays at the two cone edges, toward every in-cone segment endpoint (plus tiny offset angles to peek around corners), AND at fixed arc samples roughly every 10 degrees across the cone. Any ray that misses all segments -- or whose hit lies beyond `maxRange` -- contributes a point capped at `maxRange` along the ray. The arc samples plus range capping make the polygon structurally collapse-proof: it always has enough points even when aimed where no segments exist (the old endpoint-only version collapsed to fewer than 3 points and the cone vanished -- the reported "flashlight disappears into itself while rotating" bug). `FLASHLIGHT_RANGE` (300) is the gameplay sight cap consumed by `GameScene`, enforcing claustrophobic short sight lines. `filterSegmentsNear(origin, range, segments)` is a bounding-box prefilter that keeps per-frame raycast cost proportional to nearby segments -- essential because `wallSegments` accumulates forever in the unbounded world; `GameScene` uses it both for the flashlight (at `FLASHLIGHT_RANGE`) and for enemy line-of-sight (at the enemy `DETECTION_RANGE`). With range capping, dropping far segments is exact, not approximate. `getVisibilityPolygon()` is the full 360-degree endpoint-cast version. Also exports `raySegmentIntersection()` which is reused by `enemy.js` for LOS checks
- **Combat** (`@/src/systems/combat.js`): manages player and enemy health as immutable state. `createCombatState(maxHp?)` produces a state object with `hp`, `maxHp`, `damageCooldown`, and `isDead`. The optional `maxHp` parameter defaults to `PLAYER_MAX_HP` for backward compatibility but is overridden by the health upgrade value when called from GameScene. `applyDamage(state, amount)` returns a new state with reduced HP and sets a damage cooldown (invulnerability window) -- damage is ignored while the cooldown is active or the player is dead. `applyHeal(state, amount)` returns a new state with increased HP clamped to `maxHp` -- healing is ignored when the player is dead. This is the inverse of `applyDamage` and follows the same immutable-state pattern but does not set a cooldown. `MEDKIT_HEAL_AMOUNT` (25) is the constant consumed by `GameScene.onItemPickup()` for medkit healing. GameScene passes per-enemy `contactDamage` (read from the enemy state object, which stores a run-scaled type-specific value) as the damage amount. `updateCombat(state, delta)` ticks the cooldown down each frame. `applyEnemyDamage(health, damage)` and `isEnemyDead(health)` are simpler stateless helpers for enemy health (enemies have no invulnerability). GameScene passes `this.bulletDamage` (from the weaponDamage upgrade) as the damage amount. Base constants are exported per enemy type: `ENEMY_MAX_HP` (50), `CRAWLER_MAX_HP` (30), `SPITTER_MAX_HP` (40) for HP; `ENEMY_CONTACT_DAMAGE` (20), `CRAWLER_CONTACT_DAMAGE` (15), `SPITTER_CONTACT_DAMAGE` (10) for contact damage; and `SPITTER_PROJECTILE_DAMAGE` (15) for ranged damage. These serve as base values that `scaling.js` scales from. Corpse visual constants are also exported: `CORPSE_TINT` (0x666666, gray), `CORPSE_ALPHA` (0.6, faded), `CORPSE_ANGLE` (90, toppled sideways), and `CORPSE_DEPTH` (5, same depth as items -- below living enemies at 60 and below the darkness overlay at 100). These are consumed by `GameScene.onBulletHitEnemy()` to render dead enemies as persistent corpses rather than hiding them. `getHurtFlash(cooldownRemaining)` maps the remaining damage cooldown to a high-contrast white blink (`{ alpha, flash }`, alpha never below 0.7) consumed by `GameScene.drawPlayer()` for the hurt state -- it replaced a constant half-alpha dark-red tint that made the player effectively vanish against dark floors when taking damage (the reported "player randomly disappears" bug)
- **Shooting** (`@/src/systems/shooting.js`): `calculateBulletVelocity(originX, originY, targetX, targetY, speed)` normalizes the direction vector and scales to the given speed. `getBulletVelocityFromAngle(angle, speed)` produces `{vx, vy}` directly from an aim angle rather than a target point -- used by both the desktop mouse fallback (when the cursor sits exactly on the player and the target-point velocity is zero) and the touch aim path, where firing is driven by the aim-stick angle (`this.playerAngle`) rather than a pointer position. `canFire(cooldown)` and `updateFireCooldown(cooldown, delta)` manage fire rate. `isBulletExpired(originX, originY, currentX, currentY, maxRange)` checks if a bullet has traveled beyond its maximum range. `calculateShotgunSpread(originX, originY, targetX, targetY, speed, bulletCount, spreadAngle)` computes multiple pellet velocity vectors evenly distributed across a spread arc -- used by `GameScene.fireBullet()` when the active weapon has `bulletCount > 1`. Player bullet constants `BULLET_SPEED=500`, `FIRE_RATE_MS=200`, `BULLET_MAX_RANGE=600` now serve only as legacy reference values -- actual bullet stats come from the active weapon via `getEffectiveStats()` in `@/src/systems/weapons.js`. Enemy projectile constants `SPITTER_PROJECTILE_SPEED` (200) and `SPITTER_PROJECTILE_RANGE` (400) are also exported -- these are consumed by `GameScene.fireEnemyBullet()` and `updateEnemyBullets()` directly (not upgrade-modified, since enemies do not have upgrades)
- **Gunfire noise alerts** (`@/src/systems/gunfireNoise.js`): pure function module that determines which enemies should be alerted when the player fires a weapon, creating a tactical cost to shooting. `getAlertedEnemies(enemyStates, roomGraph, sourceRoomId, noiseRange)` iterates over all enemies, filtering for those in idle or search state (via the `ALERTABLE_STATES` Set -- chase, attack, and opening_door enemies are already engaged and not re-alerted), then uses `getRoomDistance()` from `@/src/systems/pathfinding.js` to check if each enemy's `currentRoomId` is within `noiseRange` BFS hops of the source room. Returns an array of enemy indices that should be alerted. The `noiseRange` parameter comes from the active weapon's `noiseRange` property in `WEAPON_TYPES` (pistol: 1, shotgun: 2, rifle: 1 -- shotguns are louder). `GUNSHOT_ALERT_COOLDOWN` (5000ms) is exported and consumed by `@/src/scenes/GameScene.js` to prevent rapid re-alerting. This module follows the same pure-function pattern as `@/src/systems/traps.js` noise trap alerting but through a testable function rather than inline logic. It imports `getRoomDistance` from `@/src/systems/pathfinding.js` and has no Phaser dependency, no seeded PRNG, and no state. Tests live in `@/src/systems/__tests__/gunfireNoise.test.js`
- **Weapons** (`@/src/systems/weapons.js`): manages weapon types, a 2-slot inventory, weapon switching, per-weapon ammo, and effective stat computation. `WEAPON_TYPES` is the array of all weapon definitions (pistol, shotgun, rifle), each specifying `damage`, `fireRate`, `speed`, `range`, `bulletCount`, `spreadAngle`, `color`, `pickupColor`, `maxAmmo`, `startAmmo`, and `noiseRange` (BFS hop distance for gunfire noise alerts -- consumed by `@/src/systems/gunfireNoise.js`). `createWeaponState(options)` uses priority-based slot filling to determine the player's starting loadout. It collects all purchased starting weapons in priority order (rifle > shotgun > pistol, i.e., most expensive first) and fills up to 2 weapon slots. When no starting weapons are purchased, the player starts unarmed (`slots: [null, null], ammo: [0, 0]`). When all three are purchased, only the top two (rifle, shotgun) fill the slots and the pistol is dropped from the starting loadout. Each filled slot's ammo is initialized to that weapon's `startAmmo`. The `ammo` array is parallel to `slots` -- `ammo[i]` tracks the ammo count for the weapon in `slots[i]`. `switchWeapon(state)` returns a new state with the active slot toggled (no-op if the other slot is empty). `pickupWeapon(state, weaponType)` fills the first empty slot if one exists, otherwise swaps slot 1's weapon with the picked-up weapon and returns the dropped weapon (with its ammo count attached) for re-placement in the world. Picked-up weapons start with their `startAmmo`. `getActiveWeapon(state)` returns the weapon in the active slot or `null` if the slot is empty -- all callers must handle null. `hasAmmo(state)` returns whether the active weapon has ammo remaining. `consumeAmmo(state)` decrements active slot ammo by 1 (no-op if already 0). `addAmmo(state, amount)` adds ammo to the active slot clamped to `maxAmmo`. `AMMO_PER_PICKUP` maps weapon IDs to how much ammo the `ammo` item type restores (pistol: 10, shotgun: 4, rifle: 3). `getEffectiveStats(weapon, bonuses)` applies additive upgrade bonuses (damage, fireRate, range) to the weapon's base stats, clamping fireRate to a minimum of 50ms. `generateLevelWeapons(rooms, seed, furnitureByRoom)` is retained and tested but no longer used by `GameScene` (it needs the full floor room list up front, which the lazy world does not have). `generateRoomWeapon(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, furnitureItems, roomId)` places a single weapon pickup in a room using seeded PRNG (seed offset +60000), skipping room 0 -- this is the live path: `GameScene` calls it as each non-entrance room activates and places at most one weapon per floor (first valid placement in activation order, tracked via a per-floor guard set). All weapon types including pistol can appear as floor drops. Uses the same retry-on-overlap placement pattern as other spawning systems. The weapon system follows the same pure-function immutable-state pattern as combat, inventory, and hiding
- **Battery** (`@/src/systems/battery.js`): manages flashlight battery drain and recharge as immutable state following the same pattern as combat. `createBatteryState(maxCharge?)` produces `{ charge, maxCharge, isDepleted }`. The optional `maxCharge` parameter defaults to `BATTERY_MAX` for backward compatibility but is overridden by the battery upgrade value when called from GameScene. `updateBattery(state, delta)` drains charge each frame and clamps at zero, setting `isDepleted` when charge reaches 0. `getBatteryFraction(state)` returns 0-1 charge ratio. `getFlashlightConeAngle(state, baseConeAngle)` scales the flashlight cone from 100% at full charge to 25% at empty using linear interpolation (`0.25 + 0.75 * fraction`), returning 0 when depleted. `rechargeBattery(state, amount)` adds charge back (clamped to max), clearing the `isDepleted` flag if charge rises above 0 -- this is called when the player uses a battery item. `shouldFlicker(state, time)` returns true during portions of a deterministic sine-wave pattern when charge is below `FLICKER_THRESHOLD` (25%). `getBatteryHint(state, inventory)` returns a contextual HUD hint string once charge drops to half (telling the player to insert a battery if they have one, or that the beam narrows as it drains if not) -- part of the battery-UX pass making the drain mechanic self-explanatory. Note that `GameScene` no longer calls `updateBattery()` directly each frame; drain is routed through `updateBatteryWithFlashlight()` in `@/src/systems/flashlight.js` so charge only drains while the flashlight is effectively lit. Constants: `BATTERY_MAX=100`, `BATTERY_DRAIN_PER_MS` (drains full charge in 150 seconds), `FLICKER_THRESHOLD=0.25`, `BATTERY_RECHARGE_AMOUNT=30`
- **Flashlight toggle** (`@/src/systems/flashlight.js`): pure immutable state `{ on, hiding }` for whether the flashlight is switched on and whether hiding is forcing it dark. `toggleFlashlight()` flips the switch (F key on desktop, LIGHT button on touch -- both via `GameScene.toggleFlashlightAction()`); `setFlashlightHiding(state, hiding)` is set/cleared by `GameScene.onEnterHiding()`/`onExitHiding()`, so hiding auto-darkens the light but the switch state is preserved and restored on exit (per spec: the flashlight turns off automatically while hiding); `isFlashlightLit(state)` is `on && !hiding` and gates both the flashlight polygon draw in `GameScene.updateDarkness()` and battery drain; `updateBatteryWithFlashlight(batteryState, flashlightState, delta)` wraps `updateBattery()` from `battery.js`, returning the state unchanged when unlit -- turning the light off is the player's battery-conservation tool. Follows the same immutable-state pattern as combat and battery
- **Light Switches** (`@/src/systems/lightswitch.js`): manages light switch state for creating safe zones. `createSwitchStates(rooms, seed, wallThickness)` (the whole-level builder) is retained and tested but **no longer used by `GameScene`** -- the lazy world places switches per room in `GameScene.activateRoomSwitch()` (15% chance per non-entrance room, seeded from the room's CELL coordinates so placement is path-order-independent), positioned near the center of a randomly chosen wall, inset by `wallThickness + 4` pixels. Each switch stores `{ id, roomId, x, y, isOn }` and starts off. `toggleSwitch(switchStates, switchId)` returns a new array with the targeted switch's `isOn` flipped (immutable). `findNearestSwitch(switchStates, px, py, maxRange)` returns the closest switch within `SWITCH_INTERACT_RANGE` (80px), or null. `getLitRoomIds(switchStates)` returns the room IDs of all switches that are on -- consumed by GameScene to drive both darkness mask reveal and enemy AI freezing. `isPointInRoom(room, x, y)` is a bounds check helper used by GameScene to determine if an enemy is inside a lit room
- **Light Spill** (`@/src/systems/lightSpill.js`): pure-function module that computes gradient light spill zones through open doorways from lit rooms into adjacent dark rooms. `isDoorOpen(roomIdA, roomIdB, doorStates)` normalizes room IDs (min/max) to match the deduplication invariant in `@/src/systems/doors.js` (`roomId < targetRoomId`), then checks if a matching door state exists -- returns true if no state exists (permanently open passageway) or if the state has `isClosed: false`. `getLightSpillZones(rooms, litRoomIds, doorStates)` iterates each lit room's `doors[]`, skipping doors where the target room is also lit (both already fully revealed) or the door is closed, and calls `computeSpillZone(room, door)` for each qualifying doorway. `computeSpillZone()` produces a zone descriptor `{ x, y, width, height, alphas: { topLeft, topRight, bottomLeft, bottomRight } }` where the rectangle extends `LIGHT_SPILL_DEPTH` (150px) outward from the doorway into the adjacent dark room. The `alphas` encode the gradient direction: 1 at the doorway edge (full brightness), 0 at the far edge (fading to dark). This is consumed by `@/src/scenes/GameScene.js` in `updateDarkness()` via Phaser 3's `fillGradientStyle` WebGL API, which sets per-corner alpha on the subsequent `fillRect` call. Two draws per zone: one into `maskGraphics` (white, varying alpha to create gradient visibility through the BitmapMask) and one into `lightGraphics` (warm tint at 5% of the spill alpha for atmospheric color). The module has no Phaser dependency, no seeded PRNG, and no state -- it is a pure geometric computation consumed each frame
- **Pause controls data** (`@/src/systems/pause.js`): pure-data module exporting `DESKTOP_CONTROLS` and `MOBILE_CONTROLS` arrays, each containing `{ key, action }` descriptors for the controls reference displayed on the pause screen. `getControlsList(touchMode)` returns the appropriate array based on input mode. Desktop controls include WASD, mouse aim, left/right click, E interact, Q/scroll weapon switch, and ESC to resume. Mobile controls list the equivalent touch UI elements (sticks and on-screen buttons). This module has no Phaser dependency, no state, and no seeded PRNG -- it is a pure data lookup consumed by `@/src/scenes/PauseScene.js`
- **Title menu** (`@/src/systems/titleMenu.js`): pure-data module that generates menu option arrays for the title screen based on save state. `getMenuOptions(hasSave)` returns an array of `{ key, label, color }` objects -- when `hasSave` is true, returns Continue (green), New Game (green), and Delete Save (red); when false, returns only New Game (green). Continue is always placed before New Game when present. `getTitleText()` returns `{ title: 'BACKROOMS', subtitle: "Don't let the lights go out." }`. This module has no Phaser dependency, no state, and no seeded PRNG -- it is a pure data lookup consumed by `@/src/scenes/TitleScene.js`, following the same pattern as `@/src/systems/pause.js`
- **Doors** (`@/src/systems/doors.js`): manages closable door state for tactical gameplay. `createDoorStates(rooms, seed)` (the whole-level builder) is retained and tested but **no longer used by `GameScene`** -- the lazy world creates door states per connection in `GameScene.activateRoomDoors()` (one state per shared edge, created when the second of the two rooms activates, deduped by a canonical connection key, closability decided deterministically from the connection's canonical CELL coordinates so it is path-order-independent). The runtime mutation helpers below are still used. Closable doors start closed (`isClosed: true`) and each stores a precomputed `segment` in `{x1,y1,x2,y2}` format matching the wall segment contract. `toggleDoor(doorStates, doorId)` returns a new array with the targeted door's `isClosed` flipped (immutable). `getClosedDoorSegments(doorStates)` filters for closed doors and returns their segments -- used by GameScene to manage the `wallSegments` array. `getDoorCenter(doorState, rooms)` computes the world-space center point of a door for rendering and proximity checks. `findNearestDoor(doorStates, rooms, px, py, maxRange)` returns the closest door within `DOOR_INTERACT_RANGE` (80px) of the player, or null. The segment winding matches `room.js` conventions: south and west walls are reversed
- **Items** (`@/src/systems/items.js`): `generateRoomItems(..., distance = 0)` places collectible items per room using a seeded PRNG (seed offset +20000 to avoid correlation with furniture and enemies). Room 0 (starting room) always returns empty. **Item sparsity**: a single seeded roll decides the room is empty (~55%), has 1 item (~35%), or has 2 items (~10%, the hard cap) -- so most non-entrance rooms are empty, making each find feel earned. **Distance-from-entrance loot scaling** is the core reward mechanism: each item type's `weight` is a `{ base, perRoom }` shape rather than a scalar, and `weightForDistance()` computes the effective weight as `max(0, base + perRoom * distance)` where `distance` is the room's exploration depth from the entrance (the `room.distance` field, threaded through from `GameScene.activateRoomItems()`). `pickWeightedType(rand, distance)` then does the usual weighted roll over those distance-adjusted weights. Utility items (battery, ammo, medkit) use `perRoom: 0` so they appear at constant rates everywhere (survival supplies are not gated by exploration). Treasure tiers shift with depth: `copper_coin` has a negative `perRoom` (falls off as you go deeper), while `silver_coin`/`gold_coin`/`gem` have positive `perRoom` (rise with depth). `gold_coin` and `gem` have negative `base`, so their effective weight is clamped to 0 near the entrance -- `gem` only becomes possible at roughly `distance >= 5`, making gems a deep-exploration reward that never spawns near the entrance. In the unbounded world, `distance` is the per-room exploration depth stamped in `GameScene.genRoom()` (`floor * FLOOR_DEPTH + localDistance(gx, gy)`), which grows without bound as the player explores deeper. Each item type defines a color, size, and treasure value (battery, ammo, and medkit have value 0). Three item categories exist with different pickup behavior: **inventory items** (battery, treasure coins/gems) require inventory capacity and are stored via `@/src/systems/inventory.js`; **ammo items** restore ammo to the active weapon (amount determined by `AMMO_PER_PICKUP` in `@/src/systems/weapons.js`, ignored if the player has no weapon equipped); **medkit items** bypass inventory entirely and are consumed immediately on contact, healing the player via `applyHeal()` from `@/src/systems/combat.js` (ignored at full health, leaving the medkit on the ground). Item positions avoid furniture overlap using the same 8px-radius exclusion check and retry pattern as other placement systems. This distance-based tiering is independent of `runCount` -- run count reseeds the unbounded world and scales enemies, distance governs which loot tier spawns within a room
- **Distance** (`@/src/systems/distance.js`): `computeRoomDistances(rooms, stairs)` returns a `Map<roomId, hopDistance>` of each room's BFS hop-distance from the entrance (room id 0). It builds an undirected adjacency graph from two edge sources: each room's `doors[].targetRoomId` (added bidirectionally) and each stair's `fromRoomId`<->`toRoomId` (also bidirectional), then runs breadth-first search seeded at room 0. Because stairs are single edges, a room reachable only by climbing a stair is one hop farther than its same-floor neighbors -- so depth accumulates across floors. Unreachable rooms are simply absent from the returned map. This is a pure, read-only graph traversal (no seeded PRNG, no wall segments, no Phaser). It is retained and tested but **no longer used by `GameScene`**: in the lazy world, per-room exploration depth is computed directly as `room.distance = floor * FLOOR_DEPTH + localDistance(gridX, gridY)` when a room is generated, and that value is threaded into item tier scaling, distance-gated trap placement, and enemy loot drop quality (via `@/src/systems/lootDrops.js`). It remains the spatial counterpart to the run-count difficulty escalator, rewarding exploration depth rather than cumulative play
- **Lore** (`@/src/systems/lore.js`): spawns collectible lore notes in rooms for atmospheric storytelling, and provides journal data functions for between-run lore review. `generateRoomLore(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, furnitureItems, roomId, collectedLoreIds)` uses a seeded PRNG (seed offset +90000) to determine if a room gets a lore note (~25% chance, `LORE_CHANCE`). Room 0 always returns empty. At most one lore note spawns per room. A lore entry is selected from `LORE_ENTRIES` (an array of 20 Backrooms-themed text entries, each with a unique `id`). If the selected entry's ID is in `collectedLoreIds` (a Set of already-collected IDs), the function returns empty -- this is the meta-progression mechanism that prevents re-spawning collected lore across runs. Placement uses the same retry-on-overlap pattern (20 attempts, 8px radius furniture exclusion) as other spawning systems. Returns an array of zero or one `{ x, y, loreId, text }` objects. `getJournalEntries(collectedLoreIds)` maps all `LORE_ENTRIES` to `{ id, text, collected }` objects (where `collected` is true if the ID is in the provided Set), and returns `{ entries, collectedCount, totalCount }` -- this is consumed by `@/src/scenes/ShopScene.js` to render the journal overlay and the "NOTES X/20" button. `getJournalPage(entries, page, perPage)` paginates an entries array, clamping the page index to valid bounds, and returns `{ pageEntries, currentPage, totalPages }` -- consumed by ShopScene's `renderJournalPage()` to display one page at a time. Both journal functions are pure (no PRNG, no side effects). The module does not produce wall segments and does not manage state -- state management (the `collectedLore` Set) lives in `game.registry` and is persisted via `@/src/systems/persistence.js`
- **Floor traps** (`@/src/systems/traps.js`): pure function module spawning environmental hazards in rooms, following the same seeded placement pattern as `@/src/systems/items.js`. `TRAP_TYPES` defines two trap types: spike (damage: 15, cooldown: 3000ms, reusable) and noise (damage: 0, single-use). `generateRoomTraps(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, furnitureItems, roomId, distance)` uses a seeded PRNG (seed offset +110000) to place at most one trap per room. Room 0 always returns empty. **Distance gating**: rooms with distance < 2 from the entrance never get traps; noise traps require distance >= 3 while spike traps require distance >= 2. Trap type selection uses a single roll: noise has ~15% chance (when distance qualifies), spike has ~20% chance, and the remaining ~65% produces no trap. Placement uses the same retry-on-overlap pattern as other spawning systems (20 attempts, 10px radius furniture exclusion, 40px margin from walls). Returns an array of zero or one `{ x, y, type, damage, cooldown, singleUse }` objects. This module is consumed by `@/src/scenes/GameScene.js` in `createTraps()`, which creates a staticGroup and wires player overlap detection. Spike traps apply damage via `applyDamage()` from `@/src/systems/combat.js` with a per-trap cooldown tracked via Phaser `setData('lastTriggeredAt')`. Noise traps alert nearby enemies (within BFS distance <= 1 via `getRoomDistance()` from `@/src/systems/pathfinding.js`) by directly setting `es.state = 'chase'` and `es.lastKnownX/Y` to the player position -- gunfire noise alerts (via `@/src/systems/gunfireNoise.js`) use the same direct mutation pattern after firing. These are the only two places enemy states are mutated from outside the normal `updateEnemyAI()` loop. After triggering, noise traps are disabled (`body.enable = false`) and visually dimmed. This module has no Phaser dependency and no state
- **Enemy loot drops** (`@/src/systems/lootDrops.js`): pure function module determining what items enemies drop on death, creating a risk-reward dynamic where combat is rewarded with resources. `DROP_CHANCES` defines per-enemy-type drop probability (basic 30%, crawler 45%, spitter 55% -- harder enemies drop more often). `ENEMY_DISTANCE_BONUS` gives per-enemy-type bonus to effective room distance (basic +0, crawler +1, spitter +2 -- harder enemies bias toward higher-value loot). `getDropChance(enemyType)` and `getDistanceBonus(enemyType)` are lookups with fallback to basic/0. `rollEnemyDrop(enemyType, distance, rand)` is the main entry point: first roll against `getDropChance()` to decide if any drop occurs; if so, compute `effectiveDistance = distance + getDistanceBonus(enemyType)`, then select an item type via `pickLootType(rand, effectiveDistance)` using the same distance-based weighted selection formula as room item spawning in `@/src/systems/items.js`: `max(0, base + perRoom * effectiveDistance)`. Returns `{ type, value }` or null. The loot pool is `ITEM_TYPES` from `@/src/systems/items.js` with medkits filtered out (excluded from drops to preserve horror tension -- combat should not be self-sustaining with healing). The `weightForDistance` formula is duplicated from `items.js` intentionally -- keeping it private in each module avoids coupling them. The `rand` function is caller-provided for testability (`GameScene` passes `Math.random` since drops are runtime events, not deterministic level generation). This module has no Phaser dependency, no seeded PRNG, and no state. Consumed by `@/src/scenes/GameScene.js` in `onBulletHitEnemy()` at the point where an enemy dies: drops are placed as sprites in `this.itemGroup` at the enemy's position, so the existing `onItemPickup` overlap collider auto-registers them for pickup with no additional wiring. Tests live in `@/src/systems/__tests__/lootDrops.test.js`
- **Inventory** (`@/src/systems/inventory.js`): manages collected items as immutable state with a capacity constraint. `createInventoryState(maxItems?)` produces `{ batteries, treasureValue, itemCount, maxItems }` where `maxItems` defaults to `BASE_INVENTORY_CAPACITY` (8). `canPickupItem(state)` returns true when `itemCount < maxItems` -- GameScene calls this before every pickup attempt and leaves items on the ground when full. `pickupItem(state, item)` either increments the battery counter (for battery type) or adds the item's value to `treasureValue` (for all treasure types), and always increments `itemCount`. `useBattery(state)` returns `{ used, state }` -- decrements battery count and `itemCount` if available (consuming a battery frees an inventory slot), otherwise returns `used: false` with unchanged state. GameScene calls `rechargeBattery()` from `battery.js` when `useBattery()` succeeds. Constants: `BASE_INVENTORY_CAPACITY=8`, `INVENTORY_CAPACITY_PER_LEVEL=4`
- **Shop** (`@/src/systems/shop.js`): manages the between-run upgrade economy as immutable state. `createShopState()` produces `{ gold, upgrades }` where upgrades maps each upgrade id to its current level (starting at 0). The `UPGRADES` array is the single source of truth for all upgrade categories -- ShopScene auto-generates UI rows from it, and `createShopState()` derives initial keys from it. `addGold(state, amount)` increases the gold balance. `canPurchase(state, upgradeId)` checks affordability and max-level constraints. `purchaseUpgrade(state, upgradeId)` deducts gold and increments the upgrade level, returning unchanged state if the purchase is invalid. `getUpgradeValue(upgradeId, level)` computes the stat value as `base + perLevel * level` -- this is called by `GameScene.init()` to derive all player stats from upgrade levels. `getUpgradeCost(upgradeId, currentLevel)` returns the next tier's cost or null at max level. The `UPGRADES` array defines survival, weapon, inventory, and unlock upgrade categories (e.g., battery capacity, flashlight width, max health, movement speed, weapon damage, fire rate, bullet range, backpack size, starting pistol, minimap). Most upgrades share a 3-tier structure with costs 200/600/1500. Binary unlock upgrades use a single-tier pattern (maxLevel 1, base 0, perLevel 1): three starting weapon upgrades (`startingPistol` at 1000g, `startingShotgun` at 1500g, `startingRifle` at 2000g) cause `GameScene.init()` to pass the corresponding flags to `createWeaponState()`, which fills up to 2 weapon slots using priority-based selection (rifle > shotgun > pistol); `minimap` (cost 3000) enables the fog-of-war minimap display in `GameScene`. Note that fire rate uses a negative `perLevel` (-30ms) since lower cooldown means faster firing. The backpack upgrade controls inventory capacity (base 8, +4 per level, giving 8/12/16/20 at levels 0/1/2/3). Save compatibility: old saves missing new upgrade keys are handled by `GameScene.init()` spreading `createShopState().upgrades` as defaults before overlaying the saved upgrade levels

### Things to Know
- **Segment format is the universal integration contract**: any system that produces `{x1, y1, x2, y2}` segments can participate in both the visibility/raycasting pipeline and enemy LOS checks with zero changes. Room perimeter walls (pushed by `GameScene.activateRoom()` via `createRoomWalls()` from `room.js`), light-blocking furniture, maze internal walls, and closed doors all share this format. Doors are the only system to dynamically add/remove segments at runtime (via `push`/`splice` on the mutable `wallSegments` array); all other segments are static once their room activates, and the array only ever grows as the unbounded world is explored -- which is why all per-frame raycasting goes through the `filterSegmentsNear()` prefilter from `visibility.js`. The floor-0 rift is the one perimeter opening with NO door gap in the occluder segments (light is blocked both ways even though the player can walk through). Bullets do not produce segments -- they are dynamic projectiles handled by Phaser's Arcade Physics
- **Combat, battery, flashlight, inventory, weapon, door, light switch, hiding, exploration, danger, run stats, and shop state are immutable**: `applyDamage()`, `applyHeal()`, `updateCombat()`, `updateBattery()`, `updateBatteryWithFlashlight()`, `toggleFlashlight()`, `setFlashlightHiding()`, `rechargeBattery()`, `pickupItem()`, `useBattery()`, `switchWeapon()`, `pickupWeapon()`, `consumeAmmo()`, `addAmmo()`, `toggleDoor()`, `toggleSwitch()`, `enterHiding()`, `exitHiding()`, `updateExploration()`, `updateDangerTime()`, `advanceWave()`, `recordKill()`, `updateTime()`, `updateMaxFloor()`, `addGold()`, and `purchaseUpgrade()` all return new state objects/arrays rather than mutating (note: `canPickupItem()`, `hasAmmo()`, `isFlashlightLit()`, and `shouldSpawnWave()` are read-only predicates, not state transitions). `GameScene` reassigns `this.combatState`, `this.batteryState`, `this.flashlightState`, `this.inventoryState`, `this.weaponState`, `this.doorStates`, `this.switchStates`, `this.hidingState`, `this.explorationState`, `this.dangerState`, and `this.runStats` on each call. `ShopScene` reassigns `this.shopState` and writes it back to `game.registry`. Enemy health, by contrast, is a mutable field on the enemy state objects in `enemyStates[]` -- this asymmetry exists because player combat has richer state (cooldowns, death flag) while enemy health is a simple integer decrement
- **Door pairing invariant**: a door is a single physical opening on a shared edge, and both adjacent rooms must describe it identically (same `offset` and `width`, mirrored `wall`/`targetRoomId`). In the live generator (`worldgen.js`) this holds by construction -- each shared edge's open/closed state and offset are hashed once from the edge's canonical coordinates, so both rooms always agree regardless of generation order. (The legacy `level.js` enforced the same invariant by writing paired door entries explicitly.) This ensures the gaps in adjacent rooms' walls align perfectly, allowing flashlight rays and player movement to pass through
- **Room coordinates are grid-based**: room pixel position is `gridX * ROOM_WIDTH` and `gridY * ROOM_HEIGHT`, so adjacent rooms share edges exactly. The live generator uses `ROOM_WIDTH = 720` and `ROOM_HEIGHT = 600` with `DOOR_WIDTH = 80` (the legacy `level.js` still uses the old 1200x1000 cells)
- **Enemy AI state machine** (idle -> chase -> search -> idle, plus attack for spitters, plus opening_door for crawlers): idle enemies wander randomly and transition to chase on LOS; chasing enemies pursue (speed varies by type: basic 80, crawler 112, spitter 60) and transition to search when LOS breaks; searching enemies walk to the last known position and revert to idle on arrival or after a 3-second timeout. Spitters have an `attack` state, entered from chase when within 250px range -- the spitter stops moving, counts down an attack cooldown, and signals `wantsToFire = true` when ready (GameScene reads this to spawn an enemy projectile). Attack transitions back to chase if the player moves out of range, or to search if LOS breaks. Crawlers have an `opening_door` state, entered from chase when the crawler is within 60px of a closed door blocking the path -- the crawler stops, counts down a 1-second timer, then signals `wantsToOpenDoor = true` with `targetDoorId` (GameScene reads this to call `onToggleDoor()`). The crawler transitions back to chase after opening, or immediately to chase if LOS is reacquired during the timer. Any state transitions to chase if LOS is reacquired. When `navContext` is provided, the chase state gains a doorway-routing fallback: if LOS breaks but the enemy is in a different room from the player and a doorway path exists, the enemy moves toward the next doorway at chase speed instead of dropping to search -- this creates cross-room pursuit. Idle enemies may also seek doorways probabilistically (15% chance per wander tick, gated by `roomTransitionCooldown` and `MAX_ROOM_DISTANCE`/`MAX_ROOM_ENEMIES` constraints), causing gradual enemy migration between rooms. Each enemy state object carries `spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`, `doorOpenTimer`, and `targetDoorId` fields for room and door interaction tracking
- **Closed doors block pathfinding (with crawler exception)**: `buildRoomGraph()` in `@/src/systems/pathfinding.js` excludes closed doors from the adjacency graph, so closing a door cuts off that connection for basic and spitter enemy navigation. However, crawlers use `buildFullRoomGraph()` which includes closed-door edges with `closedDoorId` annotations, allowing crawlers to pathfind through closed doors and open them after a 1-second delay (see enemy AI `opening_door` state). Both room graphs are cached in GameScene and rebuilt together when `onToggleDoor()` sets `roomGraphDirty = true`
- **Enemy, furniture, and item placement share the same retry pattern**: all attempt `count * 20` random placements and discard candidates that overlap existing objects. Fewer items may be placed than requested if the room is crowded
- `mulberry32` was extracted from `furniture.js` into `random.js` so that `level.js`, `enemy.js`, `items.js`, `doors.js`, `lightswitch.js`, `stairs.js`, `weapons.js`, `maze.js`, and `locations.js` could also use a seeded PRNG. All modules import from the same source to avoid duplication. Each spawning system uses a different seed offset (furniture +0, enemies +10000, items +20000, doors +30000, light switches +40000, stairs +50000, weapons +60000, maze +70000, extra doors +80000, lore +90000, alternate exits +100000, traps +110000) so that adding or removing items in one system does not change the random sequence of another. The lazy world (`GameScene`) likewise derives a per-floor seed as `levelSeed + floor * 1000` (mirroring the legacy stairs module) so each floor has a distinct layout, and `worldgen.js` hashes that floor seed with cell coordinates to keep room/door generation order-independent. The `generateAlternateExit()` path is no longer wired into the lazy `GameScene`. The starting room system (`startroom.js`), location definitions in `locations.js` (the static `LOCATIONS` array and layout functions), room theming (`roomThemes.js`), hiding system (`hiding.js`), exploration system (`exploration.js`), scaling system (`scaling.js`), pathfinding system (`pathfinding.js`), distance system (`distance.js`, a pure BFS over the door/stair graph), persistence system (`persistence.js`), light spill system (`lightSpill.js`, a pure geometric computation from lit rooms and door states), pause controls data (`pause.js`, a pure data lookup), title menu data (`titleMenu.js`, a pure data lookup), and audio systems (`audio.js`, `audioEngine.js`) do not use a seeded PRNG -- startroom uses a fixed hand-crafted layout, location definitions are static data, room theming is a pure lookup table, hiding operates on furniture objects already placed by `furniture.js`, exploration is deterministic based on player position, scaling applies deterministic formulas based on `runCount`, persistence is a browser storage boundary, and audio uses a Knuth multiplicative hash of game time for ambient scheduling (no seeded PRNG, no effect on level determinism). The `generateAlternateExit()` function in `locations.js` is the exception -- it uses a seeded PRNG (offset +100000) for probabilistic exit placement
- **System boundary modules**: three system modules interact with browser APIs rather than being pure game math. `persistence.js` wraps `localStorage` access for game saves -- all three exported functions use try-catch because localStorage can throw (SecurityError in private browsing, QuotaExceededError). `settings.js` wraps `localStorage` access for audio settings under a separate key (`backrooms_settings`), using the same try-catch pattern and following the same graceful-degradation approach. `audioEngine.js` wraps Web Audio APIs (`OfflineAudioContext`, oscillator/filter/gain nodes) -- its `playAmbientSound()` and `generateAllSounds()` use try-catch because audio playback can fail when the browser blocks autoplay or the audio context is not running. All three use try-catch appropriately per the project's coding guidelines since they sit at system boundaries. The companion config module `audio.js` is pure (no browser dependency, Node-testable) and now includes per-enemy-type sound logic (state transition sounds, ambient cooldown sounds, death sounds, fire sounds, and distance-based volume attenuation) -- the config/engine split follows the same pattern as the rest of the systems directory. The versioned save format in persistence.js (`SAVE_VERSION = 3`) enables future migrations -- `loadGame()` rejects saves with a higher version than the running code understands, and defaults missing fields to safe fallback values (e.g., `collectedLore` to `[]` for v1 saves, `unlockedLocations` to `['store']` and `activeLocation` to `'store'` for v1/v2 saves). Audio settings intentionally use a separate localStorage key from game saves so that volume preferences are independent of save lifecycle (creating/deleting saves does not affect volume settings)
