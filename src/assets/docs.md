# Noridoc: assets

Path: @/src/assets

### Overview
- Holds the committed, AI-generated pixel-art PNGs that the game ships (`sprites/`), one file per texture key
- These are build artifacts of THREE offline generators -- `@/scripts/generate-art.mjs` (top-down furniture sprites, front-elevation 2.5D upright billboard props, + shop sprites, all driven by `ART_MANIFEST`), `@/scripts/gen-character-directional.mjs` (the player/enemy 8-direction character frames, NOT in `ART_MANIFEST`), and `@/scripts/gen-textures.mjs` (the tileable per-zone floor/wall textures `tex_<zone>_floor` / `tex_<zone>_wall`, driven by `TEXTURE_MANIFEST`); they are committed so the running game never calls any image API

### How it fits into the larger codebase
- Each PNG is named exactly after a Phaser texture key (e.g. `furniture_table.png`, `furniture_vending_machine.png`, `shop_bg.png`, `player_down.png`, `enemy_up_right_walk0.png`, `tex_mall_floor.png`). For the furniture/prop/shop set the key list is driven by `ART_MANIFEST` in `@/src/systems/art.js`. The 2.5D upright billboard props (`furniture_vending_machine`, `furniture_lockers`, and the full per-zone prop set -- mall directory/bench/trash/mannequin, hospital wheelchair/iv_stand/..., hotel reception_desk/ice_machine/..., office filing_cabinet/cubicle_partition/..., warehouse pallet_rack/forklift/oil_barrel) live in the same `furniture_*` keyspace but are drawn FRONT-ON (the `prop()` elevation prompt) rather than top-down -- they have NO procedural grid fallback (only a flat-rect Graphics fallback), so unlike top-down furniture they MUST ship as PNGs to render as standees (see `@/src/systems/art.js` and `@/src/systems/furniture.js`). A prop type that declares `variants: N` ships N images keyed `furniture_<type>_0..N-1` (e.g. `furniture_trash_can_0`, `furniture_mannequin_1`); `propVariantKey()` in `@/src/systems/furniture.js` picks one per placement so repeated props in a room are varied. The character set is the player/basic-zombie directional frames: `{player,enemy}_{down,down_right,right,up_right,up}.png` plus two walk-step variants of each (`_walk0`/`_walk1` -- AI-generated left/right leg-step poses, registered to the base by the head) -- only FIVE base facings per character, because `@/src/systems/sprites.js` mirrors the three left-side facings with `flipX` (so there are no `*_left`/`*_down_left`/`*_up_left` files). The zone-texture set is `tex_<zone>_floor.png` / `tex_<zone>_wall.png` (one floor + one wall per liminal zone), driven by `TEXTURE_MANIFEST` in `@/src/systems/art.js` and matching the `floorTextureKey`/`wallTextureKey` in `@/src/systems/roomVibes.js`. Crawler/spitter and `furniture_armoire`/`furniture_couch`/`furniture_counter` have NO files here -- they are hand-authored procedural top-down grids in `@/src/systems/sprites.js`, because the image model cannot draw them from directly overhead
- The PNGs are discovered at build time via Vite `import.meta.glob` and loaded in the scenes' `preload()`: `@/src/scenes/GameScene.js` loads every non-`shop_` file (furniture tops AND character directional frames, via the same glob); `@/src/scenes/ShopScene.js` loads only the `shop_`-prefixed files (background + per-upgrade icons)
- At runtime a loaded PNG always wins for its key; any SPRITE key WITHOUT a PNG falls back to the procedural pixel-grid texture defined in `@/src/systems/sprites.js`. The texture keys are identical either way, so no gameplay code depends on the art source. The per-key choice is the scenes' inline `this.textures.exists(key)` check, NOT the `getTextureSource()` helper in `@/src/systems/art.js` (which states the same policy but has no production caller). The zone FLOOR/WALL textures have NO procedural fallback: when a `tex_<zone>_*` key is missing, `GameScene.applyRoomTextures()` simply skips the overlay and leaves the flat jittered palette colour visible (`@/src/scenes/GameScene.js`)
- Only the sprites the model can actually draw are generated; anything the model gets wrong is intentionally left procedural and has no file in this folder -- tiny micro-sprites (bullets, small coins), the crawler/spitter creatures, and the armoire/couch/counter furniture. The player and basic zombie ARE PNG-backed, but as 8-direction frames from `gen-character-directional.mjs` (a separate pipeline that builds the facings consistently via image-to-image), not from the `generate-art.mjs`/`ART_MANIFEST` path

### Things to Know
- These files are regenerated only at build time (requires `GEMINI_KEY` in the repo-root `.env`); the key is NEVER shipped to the browser. The furniture/shop set comes from `npm run generate-art` (`@/scripts/generate-art.mjs`); the player/enemy directional frames come from `node scripts/gen-character-directional.mjs`; the per-zone floor/wall textures come from `npm run gen-textures` (`@/scripts/gen-textures.mjs`). See the root `@/docs.md` "Offline AI-art pipeline"
- The zone textures are made seamlessly tileable in POST (nano banana has no tiling mode): the generator rolls each image by half so its continuous-interior borders move to the edges and match when repeated, then blurs a thin band over the resulting centre seam. Raw model outputs are cached under the gitignored `screenshots/texture-raw/` so the tiling can be re-tuned via `gen-textures.mjs --reprocess` with no new API calls
- nano banana cannot output transparency, so committed sprite PNGs have already had their background flood-filled to alpha by the generator's chroma-key step (`isBackgroundSample` in `@/src/systems/art.js`); the opaque `shop_bg` is the exception (`chroma: false`)
- Furniture/icon PNGs are generated at a target size and stretched to logical in-game dimensions via `setDisplaySize()`, so they do not need to match gameplay dimensions exactly
