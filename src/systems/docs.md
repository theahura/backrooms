# Noridoc: systems

Path: @/src/systems

### Overview
- Pure functions implementing game math: player movement, 2D raycasting visibility, room wall geometry, multi-room level generation, furniture/obstacle placement, enemy spawning/AI, combat/health management, shooting/bullet logic, battery/flashlight drain and recharge, item spawning, inventory management, shop/upgrade state management, door state management, light switch state management, and seeded random number generation
- Deliberately free of Phaser dependencies so they can be unit-tested in Node without a browser
- Tests live in `@/src/systems/__tests__/`

### How it fits into the larger codebase
- `@/src/scenes/GameScene.js` imports and calls all system modules during scene creation and on every frame during `update()`. Combat, shooting, and battery systems are called in the frame loop (cooldown ticking, battery drain, bullet expiry) and combat/shooting are also called in collision callbacks (damage application, bullet deactivation). Inventory state is updated in item pickup callbacks and battery-use input. At scene init, `getUpgradeValue()` from `shop.js` is called to compute upgrade-modified stats (max HP, max battery, flashlight angle, speed)
- `@/src/scenes/ShopScene.js` imports the shop system for upgrade definitions, pricing, purchase validation, and state management. ShopScene reads/writes `shopState` from `game.registry` and uses the pure functions from `shop.js` for all purchase logic
- At scene start, `GameScene` calls `generateLevel()` from `level.js`, which internally calls `createRoomWalls()` from `room.js` for each room. The returned `{ rooms, wallSegments }` structure drives all subsequent setup (rendering, physics, furniture, enemies, items)
- Furniture segments from `furniture.js` are appended to the level's wall segments array so the visibility system treats them identically to walls. Closed door segments from `doors.js` are also pushed/spliced into the same wall segments array dynamically when doors are toggled. Items from `items.js` and light switches from `lightswitch.js` do not produce segments -- items are small collectibles and switches are small interactable objects, not occluders. Light switches interact with the enemy and darkness systems via lit room IDs rather than wall segments
- Movement system output (`{x, y}` velocity) is passed directly to `player.setVelocity()`
- Visibility system output (polygon as `{x, y}` point array) is drawn into a Phaser Graphics object used as a BitmapMask to punch a hole in the darkness overlay
- Enemy system (`enemy.js`) reuses `raySegmentIntersection` from `visibility.js` for line-of-sight checks and `mulberry32` from `random.js` for deterministic spawning -- it follows the same pure-function, segment-based integration pattern as furniture
- The dependency flow is: `level.js` calls `room.js` to produce wall segments -> `GameScene` adds furniture segments from `furniture.js` -> `GameScene` adds closed door segments from `doors.js` -> all segments are concatenated into a single mutable array -> `visibility.js` and `enemy.js` both consume this segment array for raycasting. Door segments are dynamically added/removed when the player toggles a door. Separately, `items.js` spawns collectibles per room (consuming the furniture list for overlap avoidance), `inventory.js` tracks what the player has picked up, and `lightswitch.js` manages per-room switch states that feed into the darkness mask and enemy AI freeze logic

### Core Implementation
- **Level generation** (`@/src/systems/level.js`): `generateLevel(seed, roomCount)` places rooms on a grid using a seeded growth algorithm. Starting from room 0 at grid position (0,0), each subsequent room picks a random existing room, picks a random cardinal direction, and places a new room in the adjacent grid cell if it is empty. Paired doorways are added to both rooms at the same wall offset. Returns `{ rooms, wallSegments }` where `wallSegments` is the flattened array of all room wall segments (with door gaps). This is a pure function -- no Phaser dependency
- **Room geometry** (`@/src/systems/room.js`): `createRoomWalls(x, y, width, height, doors)` returns line segments forming a rectangular boundary with gaps for doorways. When `doors` is empty or omitted, it returns four simple wall segments (backward-compatible). When doors are present, each wall is split into sub-segments around the door gaps by `splitWall()`. Segments maintain clockwise winding order -- south and west walls are reversed via `reverseSegments()` so endpoint connectivity is preserved
- **Random** (`@/src/systems/random.js`): exports `mulberry32(seed)` which returns a stateful function that produces deterministic floats in [0, 1). Used by `level.js`, `furniture.js`, `enemy.js`, and `items.js`
- **Movement** (`@/src/systems/movement.js`): `calculateVelocity(keys, speed)` takes a key-state object and speed scalar, returns `{x, y}` velocity. Normalizes diagonal movement to match cardinal speed
- **Furniture** (`@/src/systems/furniture.js`): `generateRoomFurniture()` uses a seeded PRNG to place furniture items within a room's inner bounds, enforcing no-overlap with 20px padding. `createFurnitureSegments()` produces four `{x1,y1,x2,y2}` segments per furniture item in the same format as wall segments
- **Enemy spawning and AI** (`@/src/systems/enemy.js`): `generateRoomEnemies()` places 1-2 enemies per room using seeded PRNG (room 0 is always empty -- it is the player spawn room). `hasLineOfSight()` casts a single ray from enemy to player against the wall segment array to determine visibility. `updateEnemyAI()` is a pure state machine (idle/chase/search) that takes current enemy state and returns updated state with new velocity -- GameScene applies the returned velocity to physics bodies
- **Visibility / raycasting** (`@/src/systems/visibility.js`): `getFlashlightPolygon(origin, angle, coneAngle, segments)` casts rays toward every segment endpoint (plus tiny offset angles to peek around corners), finds closest intersections, clips to a cone, and returns a polygon. `getVisibilityPolygon()` is the full 360-degree version. Also exports `raySegmentIntersection()` which is reused by `enemy.js` for LOS checks
- **Combat** (`@/src/systems/combat.js`): manages player and enemy health as immutable state. `createCombatState(maxHp?)` produces a state object with `hp`, `maxHp`, `damageCooldown`, and `isDead`. The optional `maxHp` parameter defaults to `PLAYER_MAX_HP` for backward compatibility but is overridden by the health upgrade value when called from GameScene. `applyDamage(state, amount)` returns a new state with reduced HP and sets a damage cooldown (invulnerability window) -- damage is ignored while the cooldown is active or the player is dead. `updateCombat(state, delta)` ticks the cooldown down each frame. `applyEnemyDamage(health, damage)` and `isEnemyDead(health)` are simpler stateless helpers for enemy health (enemies have no invulnerability). Constants: `PLAYER_MAX_HP=100`, `ENEMY_CONTACT_DAMAGE=20`, `DAMAGE_COOLDOWN_MS=1000`, `ENEMY_MAX_HP=50`, `BULLET_DAMAGE=25`
- **Shooting** (`@/src/systems/shooting.js`): `calculateBulletVelocity(originX, originY, targetX, targetY, speed)` normalizes the direction vector and scales to the given speed. `canFire(cooldown)` and `updateFireCooldown(cooldown, delta)` manage fire rate. `isBulletExpired(originX, originY, currentX, currentY, maxRange)` checks if a bullet has traveled beyond its maximum range. Constants: `BULLET_SPEED=500`, `FIRE_RATE_MS=200`, `BULLET_MAX_RANGE=600`
- **Battery** (`@/src/systems/battery.js`): manages flashlight battery drain and recharge as immutable state following the same pattern as combat. `createBatteryState(maxCharge?)` produces `{ charge, maxCharge, isDepleted }`. The optional `maxCharge` parameter defaults to `BATTERY_MAX` for backward compatibility but is overridden by the battery upgrade value when called from GameScene. `updateBattery(state, delta)` drains charge each frame and clamps at zero, setting `isDepleted` when charge reaches 0. `getBatteryFraction(state)` returns 0-1 charge ratio. `getFlashlightConeAngle(state, baseConeAngle)` scales the flashlight cone from 100% at full charge to 25% at empty using linear interpolation (`0.25 + 0.75 * fraction`), returning 0 when depleted. `rechargeBattery(state, amount)` adds charge back (clamped to max), clearing the `isDepleted` flag if charge rises above 0 -- this is called when the player uses a battery item. `shouldFlicker(state, time)` returns true during portions of a deterministic sine-wave pattern when charge is below `FLICKER_THRESHOLD` (25%). Constants: `BATTERY_MAX=100`, `BATTERY_DRAIN_PER_MS` (drains full charge in 90 seconds), `FLICKER_THRESHOLD=0.25`, `BATTERY_RECHARGE_AMOUNT=30`
- **Light Switches** (`@/src/systems/lightswitch.js`): manages light switch state for creating safe zones. `createSwitchStates(rooms, seed, wallThickness)` iterates over all rooms (skipping room 0), rolls a seeded random (seed offset +40000) with 40% chance to place a switch. Switches are positioned near the center of a randomly chosen wall (north/south/east/west), inset by `wallThickness + 4` pixels. Each switch stores `{ id, roomId, x, y, isOn }` and starts off. `toggleSwitch(switchStates, switchId)` returns a new array with the targeted switch's `isOn` flipped (immutable). `findNearestSwitch(switchStates, px, py, maxRange)` returns the closest switch within `SWITCH_INTERACT_RANGE` (80px), or null. `getLitRoomIds(switchStates)` returns the room IDs of all switches that are on -- consumed by GameScene to drive both darkness mask reveal and enemy AI freezing. `isPointInRoom(room, x, y)` is a bounds check helper used by GameScene to determine if an enemy is inside a lit room
- **Doors** (`@/src/systems/doors.js`): manages closable door state for tactical gameplay. `createDoorStates(rooms, seed)` iterates over all rooms, and for each door where `room.id < door.targetRoomId` (deduplication -- ensures one physical door per bidirectional connection), rolls a seeded random (seed offset +30000) with 50% chance to create a closable door. Closable doors start closed (`isClosed: true`) and each stores a precomputed `segment` in `{x1,y1,x2,y2}` format matching the wall segment contract. `toggleDoor(doorStates, doorId)` returns a new array with the targeted door's `isClosed` flipped (immutable). `getClosedDoorSegments(doorStates)` filters for closed doors and returns their segments -- used by GameScene to manage the `wallSegments` array. `getDoorCenter(doorState, rooms)` computes the world-space center point of a door for rendering and proximity checks. `findNearestDoor(doorStates, rooms, px, py, maxRange)` returns the closest door within `DOOR_INTERACT_RANGE` (80px) of the player, or null. The segment winding matches `room.js` conventions: south and west walls are reversed
- **Items** (`@/src/systems/items.js`): `generateRoomItems()` places 2-5 collectible items per room using a seeded PRNG (seed offset +20000 to avoid correlation with furniture and enemies). Room 0 (starting room) always returns empty. Item types are selected via weighted random: battery (weight 15), copper_coin (weight 40), silver_coin (weight 25), gold_coin (weight 15), gem (weight 5). Each item type defines a color, size, and treasure value (battery has value 0). Item positions avoid furniture overlap using the same 8px-radius exclusion check and retry pattern as other placement systems
- **Inventory** (`@/src/systems/inventory.js`): manages collected items as immutable state. `createInventoryState()` produces `{ batteries, treasureValue }`. `pickupItem(state, item)` either increments the battery counter (for battery type) or adds the item's value to `treasureValue` (for all treasure types). `useBattery(state)` returns `{ used, state }` -- decrements battery count if available, otherwise returns `used: false` with unchanged state. GameScene calls `rechargeBattery()` from `battery.js` when `useBattery()` succeeds
- **Shop** (`@/src/systems/shop.js`): manages the between-run upgrade economy as immutable state. `createShopState()` produces `{ gold, upgrades }` where upgrades maps each upgrade id to its current level (starting at 0). `addGold(state, amount)` increases the gold balance. `canPurchase(state, upgradeId)` checks affordability and max-level constraints. `purchaseUpgrade(state, upgradeId)` deducts gold and increments the upgrade level, returning unchanged state if the purchase is invalid. `getUpgradeValue(upgradeId, level)` computes the stat value as `base + perLevel * level` -- this is called by `GameScene.init()` to derive battery capacity, flashlight angle, max HP, and movement speed from the current upgrade levels. `getUpgradeCost(upgradeId, currentLevel)` returns the next tier's cost or null at max level. The `UPGRADES` array defines all upgrade types:

  | Upgrade ID | Stat | Base | Per Level | Costs (3 tiers) |
  |------------|------|------|-----------|-----------------|
  | battery | Max charge | 100 | +30 | 100 / 300 / 800 |
  | flashlight | Cone angle (rad) | PI/4 | +PI/12 | 100 / 300 / 800 |
  | health | Max HP | 100 | +25 | 100 / 300 / 800 |
  | speed | Movement speed | 200 | +20 | 100 / 300 / 800 |

### Things to Know
- **Segment format is the universal integration contract**: any system that produces `{x1, y1, x2, y2}` segments can participate in both the visibility/raycasting pipeline and enemy LOS checks with zero changes. Walls, furniture, and closed doors all share this format. Doors are the first system to dynamically add/remove segments at runtime (via `push`/`splice` on the mutable `wallSegments` array), while walls and furniture segments are static after level creation. Bullets do not produce segments -- they are dynamic projectiles handled by Phaser's Arcade Physics
- **Combat, battery, inventory, door, light switch, and shop state are immutable**: `applyDamage()`, `updateCombat()`, `updateBattery()`, `rechargeBattery()`, `pickupItem()`, `useBattery()`, `toggleDoor()`, `toggleSwitch()`, `addGold()`, and `purchaseUpgrade()` all return new state objects/arrays rather than mutating. `GameScene` reassigns `this.combatState`, `this.batteryState`, `this.inventoryState`, `this.doorStates`, and `this.switchStates` on each call. `ShopScene` reassigns `this.shopState` and writes it back to `game.registry`. Enemy health, by contrast, is a mutable field on the enemy state objects in `enemyStates[]` -- this asymmetry exists because player combat has richer state (cooldowns, death flag) while enemy health is a simple integer decrement
- **Door pairing invariant**: when `level.js` connects two rooms, both rooms receive a door entry at the same `offset` and `width` on their shared wall. The parent gets `{ wall: 'east', offset, width, targetRoomId }` and the child gets the corresponding `{ wall: 'west', offset, width, targetRoomId }`. This ensures the gaps in adjacent rooms' walls align perfectly, allowing flashlight rays and player movement to pass through
- **Room coordinates are grid-based**: room pixel position is `gridX * ROOM_WIDTH` and `gridY * ROOM_HEIGHT`, so adjacent rooms share edges exactly. The level generator uses `ROOM_WIDTH = 1200` and `ROOM_HEIGHT = 1000` with `DOOR_WIDTH = 80`
- **Enemy AI state machine** (idle -> chase -> search -> idle): idle enemies wander randomly and transition to chase on LOS; chasing enemies pursue at 80px/s and transition to search when LOS breaks (recording last known player position); searching enemies walk to the last known position at 30px/s and revert to idle on arrival or after a 3-second timeout. Any state transitions to chase if LOS is reacquired
- **Enemy, furniture, and item placement share the same retry pattern**: all attempt `count * 20` random placements and discard candidates that overlap existing objects. Fewer items may be placed than requested if the room is crowded
- `mulberry32` was extracted from `furniture.js` into `random.js` so that `level.js`, `enemy.js`, `items.js`, `doors.js`, and `lightswitch.js` could also use a seeded PRNG. All modules import from the same source to avoid duplication. Each spawning system uses a different seed offset (furniture +0, enemies +10000, items +20000, doors +30000, light switches +40000) so that adding or removing items in one system does not change the random sequence of another
