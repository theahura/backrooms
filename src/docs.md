# Noridoc: src

Path: @/src

### Overview
- Contains all game source code, organized into scenes (Phaser integration), systems (pure game logic), and `@/src/assets/` (committed AI-generated pixel-art PNGs loaded in the scenes' `preload()`, with a procedural fallback)
- Entry point is `@/src/main.js`, which creates the Phaser.Game instance, loads saved state from localStorage, loads audio settings from a separate localStorage key, registers scenes (TitleScene first, so it auto-starts as the boot screen), and installs browser lifecycle listeners for save safety nets

### How it fits into the larger codebase
- `@/index.html` loads `@/src/main.js` as an ES module via `<script type="module">`. Its `#game-container` is sized to `100dvh` (with a `100vh` fallback) so the canvas tracks the dynamic mobile viewport as the browser URL bar hides/shows, and it carries `mobile-web-app-capable`/`theme-color` meta tags alongside the existing Apple web-app tags
- `@/src/main.js` wires Phaser config (physics, renderer), picks the Scale mode at boot based on whether the device is touch-primary (see Core Implementation), registers all scene classes (TitleScene, GameScene, ShopScene, PauseScene, RunSummaryScene) with the engine, loads persisted game state (including `unlockedLocations` and `activeLocation`) into `game.registry`, and loads audio settings from a separate localStorage key (`backrooms_settings`) via `loadSettings()` from `@/src/systems/settings.js` into `game.registry` under the key `'audioSettings'`
- `@/src/scenes/` contains Phaser Scene subclasses: `TitleScene` for the main menu (game entry point with continue/new game/delete save options), `GameScene` for gameplay, `ShopScene` for the between-run upgrade shop, `PauseScene` for the in-game pause overlay (launched on top of GameScene, freezes gameplay and displays a controls reference), and `RunSummaryScene` for the end-of-run stats display (shown between GameScene and ShopScene)
- `@/src/systems/` contains pure functions that scenes call for game math (movement, visibility, room geometry, room interior variety via six room archetypes (open/maze/columns/corridor/storage/cubicles), per-room visual + furniture "vibes" (distinct-but-desaturated palettes paired with furniture profiles via `@/src/systems/roomVibes.js`, on a seed axis orthogonal to the structural archetype), profile-driven semantic-cluster furniture placement with a type-dependent tall/solid classification that drives light blocking AND bullet blocking (via `blocksBullets()` -- tall/solid furniture stops gunfire, low furniture is shot over), starting room layout, starting location definitions, coordinate-keyed deterministic unbounded room generation (`@/src/systems/worldgen.js`, generated lazily ahead of the player), legacy multi-floor/level generation (`@/src/systems/stairs.js`/`level.js`, retained for tests), enemy spawning/AI with strictly sight-based pursuit (last-seen-position memory), enemy difficulty scaling, time-based danger escalation, combat/health/healing with per-type constants, shooting/bullets including shotgun spread and enemy projectile constants, weapon types/inventory/switching/stat computation, battery/flashlight drain, flashlight on/off toggle state with hiding auto-off (`@/src/systems/flashlight.js`), item spawning with distance-from-entrance loot density and tier scaling, enemy loot drops on death (`@/src/systems/lootDrops.js`), gunfire noise alerts (`@/src/systems/gunfireNoise.js`), legacy BFS hop-distance from the entrance (`@/src/systems/distance.js`, test-only -- rooms now carry a precomputed `distance`), inventory management with capacity constraints, shop/upgrade logic, door state management, light switch state management, light spill through doorways (`@/src/systems/lightSpill.js`), hiding state management, exploration/minimap tracking, BFS-based room graph pathfinding, pause controls data (`@/src/systems/pause.js`), title menu option generation (`@/src/systems/titleMenu.js`), localStorage persistence, audio settings with volume control state management, procedural audio configuration with per-enemy-type sound logic and distance-based volume attenuation, Web Audio sound synthesis, pixel art sprite definitions, and character animation definitions)
- The architecture enforces a one-way dependency: scenes import from systems, but systems never import from scenes or Phaser. Systems may import from each other (e.g., `enemy.js` imports `raySegmentIntersection` from `visibility.js`)

### Core Implementation
- **Game initialization** (`@/src/main.js`): creates a `Phaser.Game` whose Scale mode is chosen at module load by `detectTouchPrimary()` from `@/src/systems/touchControls.js` (coarse-pointer media query OR `navigator.maxTouchPoints > 0`). Registers all five scene classes: TitleScene, GameScene, ShopScene, PauseScene, and RunSummaryScene. TitleScene is listed first so it auto-starts as the boot screen. PauseScene and RunSummaryScene are never auto-started; PauseScene is launched on demand by GameScene, and RunSummaryScene is transitioned to by GameScene on exit or death. The two Scale branches are a deliberate desktop/touch split:
  - **Desktop (FIT)**: a fixed internal 1024x768 resolution scaled via `Scale.FIT` + `CENTER_BOTH` (the canvas auto-scales to fill any screen and letterboxes with black bars, which are invisible against the black game). Because the internal resolution is fixed, every world and HUD coordinate is unchanged regardless of window size. This is the original behavior and is preserved exactly.
  - **Touch (RESIZE)**: `Scale.RESIZE` sized to `window.innerWidth`/`innerHeight`, so the canvas fills the entire device viewport at native resolution (fixes the prior pillarboxed/width-limited mobile layout). The render surface now equals the screen, so touch UI cannot use hardcoded design-space coordinates -- it is positioned at runtime from the live `this.scale.width`/`height` via `computeTouchLayout()` (see `@/src/scenes` and `@/src/systems`), and the ShopScene zoom-fits its 1024x768 layout into the viewport (see `@/src/scenes`). Under RESIZE the viewport changes on orientation flip, URL-bar hide/show, and fullscreen toggles, so scenes re-layout on `this.scale` `resize` events.
  Also configured: Arcade physics (zero gravity for top-down), `pixelArt: true` for nearest-neighbor texture sampling (prevents anti-aliasing blur on pixel art sprites), and the `rexVirtualJoystick` global plugin (from `phaser3-rex-plugins`) consumed by the mobile touch sticks. TitleScene is listed first, so it auto-starts as the boot screen (the main menu). Immediately after creating the game, `main.js` calls `loadGame()` from `@/src/systems/persistence.js` and, if a valid save exists, populates `game.registry` with the saved `shopState`, `runCount`, `collectedLore`, `unlockedLocations`, and `activeLocation`. It also calls `loadSettings()` from `@/src/systems/settings.js` and stores the result in `game.registry` under `'audioSettings'` -- this is loaded unconditionally (not gated by whether a game save exists) because audio settings are stored in a separate localStorage key (`backrooms_settings`) and persist independently. This happens before any scene's `init()` runs, so scenes always see persisted state in the registry. `main.js` also installs two browser lifecycle listeners (`visibilitychange` with `hidden` check, and `beforeunload`) that flush the current registry state (including `collectedLore`, `unlockedLocations`, and `activeLocation`) to localStorage via `saveGame()` as safety nets for unexpected tab closures. Note that audio settings are NOT flushed by lifecycle listeners -- they are saved immediately on each user interaction via `saveSettings()` in the scenes that provide volume controls. `main.js` also exposes the created game instance as `window.__game`, consumed only by the Playwright playtest script (`@/scripts/playtest.mjs`) for runtime introspection during visual verification
- **Scene flow**: the game boots into TitleScene (main menu), which provides continue/new game/delete save options based on save state. Continue transitions to ShopScene (returning player resumes from the shop). New Game clears the save and registry, then transitions to GameScene (fresh start). From there, the game cycles through GameScene (gameplay), RunSummaryScene (end-of-run stats), and ShopScene (upgrade shop). On exit or death, GameScene transitions to RunSummaryScene, passing run statistics (`runStats`, `survived`, `treasureEarned`, `roomsExplored`, `hp`, `maxHp`, `newLocation`, `touchMode`). RunSummaryScene displays animated stat reveals and then transitions to ShopScene, passing through `treasureEarned` and `newLocation` unchanged. On "Enter Backrooms", ShopScene transitions to GameScene. During gameplay, PauseScene can be launched as an overlay on top of GameScene -- GameScene is frozen via `scene.pause()` while PauseScene runs, and on resume PauseScene calls `scene.resume('GameScene')` and stops itself. PauseScene does not modify any game state or interact with the registry. Cross-scene state (shop upgrades, gold, run counter, unlocked locations, active location) flows through `game.registry` in memory and is backed by localStorage via `@/src/systems/persistence.js` for cross-session persistence. Scenes call `saveGame()` at every point where they mutate registry state (e.g., after adding gold, after purchasing upgrades, after incrementing runCount, after unlocking a location), and `main.js` adds lifecycle listeners as safety nets
- **Frame loop**: Phaser calls `GameScene.update()` every frame, which reads input, resolves the aim angle and movement velocity through pure resolvers in `@/src/systems/touchControls.js` (so the desktop mouse/keyboard path and the touch joystick path share a single branch point keyed on `this.touchMode`), computes velocity via the movement system (at half speed while hiding via `HIDING_SPEED_MULTIPLIER`, full speed otherwise), ticks combat/fire/battery state (battery drain is gated through `@/src/systems/flashlight.js` so charge only drains while the flashlight is on and not auto-darkened by hiding; F key or the touch LIGHT button toggles it), processes shooting input (gated through `resolveFireIntent` so touch fires only from the dedicated FIRE button; disabled while hiding; fires per-weapon bullets using weapon-specific textures, speeds, and spread patterns), resolves E-key interaction (nearest door, switch, hideable furniture, or weapon pickup by distance), processes Q-key weapon switching, checks bullet expiry for both player and enemy bullets (per-bullet range via `setData`), runs enemy AI updates (with lit-room freezing, hiding-based detection suppression, and spitter projectile firing), updates exploration state (tracking which rooms the player has visited), redraws the player/HUD/minimap (including inventory counts and capacity indicator, weapon name, hiding visual state, per-floor fog-of-war room reveal, and stair glow animation), toggles door sprite visibility, swaps switch sprite textures, and recomputes the flashlight/darkness overlay (including lit rooms; the flashlight cone is range-capped at `FLASHLIGHT_RANGE` and raycast against a `filterSegmentsNear()`-prefiltered occluder set from `@/src/systems/visibility.js`, and is skipped entirely while toggled off or hiding). The update loop early-returns if the player is dead, a day-ending transition is in progress, or a stair teleportation is in progress (`isTeleporting`)
- **Level generation**: the world is effectively unbounded and generated lazily. At scene creation, `GameScene` generates and activates the entrance plus everything within 2 door-hops via `@/src/systems/worldgen.js` (a coordinate-keyed deterministic generator seeded per-run from `runCount`), and continues generating ~2 rooms ahead as the player explores -- rooms stay static once generated. The entrance store's only opening into the backrooms is the east-wall rift (its other edges are forced sealed by `worldgen.js`), presented as a sizzling light-sealed tear rather than a doorway. Each room is activated exactly once (rendering, physics, furniture, internal structure via `@/src/systems/maze.js`, enemies, items, lore, doors, switch, weapon). Floors are stacked as non-overlapping horizontal bands in a single physics world (`FLOOR_Y_OFFSET = 5_000_000`), connected by deep-link stairs (a per-cell down-stair from `hasStairDown()`, and an up-stair at each non-entrance floor's entrance cell) with a return stack. The legacy `generateMultiFloorLevel()`/`level.js` path is retained for tests but no longer drives the scene

### Things to Know
- **Texture source: AI PNG first, procedural fallback**: meaningful sprites have committed AI-generated PNGs under `@/src/assets/sprites/` (produced offline by `@/scripts/generate-art.mjs`). `GameScene.preload()` loads every present non-`shop_` PNG (and `ShopScene.preload()` the `shop_` ones) via Vite `import.meta.glob`. Then `GameScene.generateSpriteTextures()` (run once at the very start of `create()`) procedurally generates a texture from `@/src/systems/sprites.js` only for keys where `this.textures.exists(key)` is false -- so the PNG wins when present and any missing key transparently falls back to the procedural pixel-grid. Texture KEYS are identical either way, so all downstream rendering/animation code is unchanged regardless of source. All animations are pre-generated by the same method from `ANIM_DEFS`. Furniture, doors, switches, items, weapon pickups, lore notes, and the player/enemy character frames all participate; tiny micro-sprites (bullets, small coins) are intentionally left procedural (no manifest entry in `@/src/systems/art.js`). Furniture rendering gates on `this.textures.exists('furniture_{type}')`, falling back to Graphics `fillRect()` only when neither a PNG nor a procedural definition exists. Doors and switches use static sprites whose visibility or texture is toggled per frame. Facing is split by entity: the player and basic zombie are PNG-backed 8-direction sprites whose texture + `flipX` are snapped to the facing each frame (via `getDirectionalFrame()` from `@/src/systems/sprites.js`, no `setRotation`), while crawlers and spitters are animated procedural sprites that play looping animation sequences and rotate with `setRotation()` toward their velocity vector
- Camera follows the player with lerp 0.1 for smooth tracking. Because the world is unbounded, camera and physics world bounds are a large fixed region (`WORLD_BOUND`) rather than a computed level extent, and the darkness overlay is sized from `cameras.main.worldView` each frame
- Mouse coordinates must use `pointer.worldX`/`worldY` (not `pointer.x`/`pointer.y`) because the camera moves with the player. However, Phaser 3 only updates these cached properties on mouse events -- they go stale when the camera scrolls without mouse movement (e.g., WASD movement with a stationary mouse). `GameScene.update()` calls `pointer.updateWorldPoint(this.cameras.main)` at the top of every frame to keep world coordinates current. Any new code reading `pointer.worldX`/`worldY` in the frame loop can rely on this, but code outside the frame loop (e.g., event callbacks) should call `updateWorldPoint()` itself if camera position may have changed since the last mouse event
- Floors are stacked as non-overlapping horizontal bands in world-space Y, separated by `FLOOR_Y_OFFSET = 5_000_000` (a `GameScene` constant, large enough that unbounded floors never collide and that raycasting/physics/enemy AI cannot interact across floors without explicit stair teleportation). Each room object carries a `floor` field (integer). The minimap renders a player-centered grid window on the current floor via `getWindowedMinimapData()` (it no longer scales a whole-floor bounding box, since an unbounded floor has none)
- **Occluders are split into two accumulating stores, and the maze-wall store is expanded differently per consumer.** `wallSegments` (a `{x1,y1,x2,y2}` segment array) holds room perimeter walls (`room.js` via `GameScene.activateRoom()`, with door gaps -- except the floor-0 rift, which gets no gap so light cannot pass through it), light-blocking furniture (`furniture.js`, only items with `blocksLight: true`), and closed door segments (the door system add/removes these dynamically). `mazeWallRects` (a `{x,y,width,height}` rect array) holds the room interior structures from `maze.js` (walls, columns, crates, cubicle partitions, etc.) as SOLID rects rather than pre-expanded outlines. Each frame the consumers expand the maze rects in different ways: enemy line-of-sight expands each rect to its FULL four-segment outline via `wallRectSegments()` (walls block sight from both sides -- byte-identical to the old behavior when maze outlines lived in `wallSegments`), while the flashlight expands each rect to only its FAR faces via `wallRectOccluders(rect, origin)` (back-face culling -- the cone reaches across the wall band to light its near face, so the wall reads as a solid band of visible thickness instead of an infinitely-thin shadow line, the fix for the spec's "walls have no thickness" bug). Non-blocking furniture (types with `blocksLight: false`, e.g., table, desk, bed, vent) does NOT contribute occluders. Both stores grow forever in the unbounded world and are never cleared; per frame the flashlight and enemy LOS prefilter `wallSegments` via `filterSegmentsNear()` and `mazeWallRects` via `filterRectsNear()` (both from `@/src/systems/visibility.js`) so raycast cost stays proportional to nearby occluders. Physics collision for maze walls (the static zones added in `renderRoomMaze()`) is unaffected by this split. Items and lore notes contribute no occluders -- they are small ground-level objects using staticGroup + overlap physics
- **Touch input follows the same systems/scenes split as the rest of the game**: pure, Phaser-free decision functions live in `@/src/systems/touchControls.js` (which input source to use for aim/move, whether to fire, touch detection, and `computeTouchLayout()` for screen-space placement of the touch UI) and are unit-tested in Node; the Phaser wiring (rex virtual joysticks, FIRE/USE/WPN/BATT/FS buttons, pointer polling, resize relayout, fullscreen requests) lives in `@/src/scenes/GameScene.js`. `this.touchMode` is set once during input setup and gates every input fork. Desktop behavior is unchanged when `touchMode` is false because each resolver returns the original mouse/keyboard value in that case
- **Touch UI is responsive, not hardcoded**: because touch devices run under `Scale.RESIZE` (see Game initialization), the canvas equals the device viewport and changes size on orientation flips, URL-bar hide/show, and fullscreen toggles. Touch control positions therefore come from the pure `computeTouchLayout()` (in `@/src/systems/touchControls.js`) keyed on the live viewport, and `GameScene` re-runs it on every `this.scale` `resize` event to reposition every control. Vertically centering the sticks (rather than anchoring them to the bottom) is the deliberate fix for downward thumb travel being eaten by the mobile browser's bottom gesture zone. `ShopScene` is laid out around the fixed 1024x768 design space and stays aligned under RESIZE by zoom-fitting its camera to the viewport (see `@/src/scenes`)
- Light switches and the hiding system do not produce wall segments. Switches are small interactable objects placed near walls that interact with the darkness system by drawing white rectangles into the BitmapMask (same mask the flashlight uses) and with the enemy system by providing lit room IDs that freeze enemies in place (velocity zeroed, AI skipped). Hiding operates on existing furniture objects (those with `canHide: true`) and affects enemy detection via the `playerHidden` flag on `updateEnemyAI()`
