# Noridoc: scenes

Path: @/src/scenes

### Overview
- Phaser Scene subclasses that wire pure game systems to Phaser's rendering, physics, and input APIs
- Contains two scenes: `GameScene` (gameplay: movement, flashlight/darkness, enemy AI, combat, shooting, battery drain, item collection, closable doors, light switches, furniture hiding, exit condition, HUD) and `ShopScene` (between-run upgrade shop: displays gold balance, upgrade categories with buy buttons, and "Enter Backrooms" button)
- The two scenes alternate in a loop: GameScene -> ShopScene (on exit or death) -> GameScene (on "Enter Backrooms")

### How it fits into the larger codebase
- `@/src/main.js` registers scene classes with the Phaser.Game instance; GameScene is listed first (initial scene)
- Scenes import pure functions from `@/src/systems/` for game math and handle all Phaser-specific concerns: sprite creation, physics bodies, input binding, camera, collision callbacks, and Graphics drawing
- Scenes are the only place in the codebase that depends on Phaser -- this boundary is intentional to keep game logic testable
- `GameScene` reads upgrade levels from `game.registry` in its `init()` method, using `getUpgradeValue()` from `@/src/systems/shop.js` to compute upgrade-modified stats (max HP, max battery charge, flashlight cone angle, movement speed). These values are passed to `createCombatState()` and `createBatteryState()` in `create()`, and used directly for movement and flashlight calculations
- `ShopScene` reads/writes `shopState` from `game.registry`, using pure functions from `@/src/systems/shop.js` for all purchase logic. It receives `treasureEarned` as scene data from GameScene and converts it to gold via `addGold()`. ShopScene also calls `saveGame()` from `@/src/systems/persistence.js` at every point where `shopState` is mutated: after adding treasure gold in `init()`, before entering backrooms, and after purchasing an upgrade
- `GameScene` calls `saveGame()` from `@/src/systems/persistence.js` after incrementing `runCount` in `init()`, persisting the updated run counter and shop state to localStorage before the run begins
- Data flows between scenes via two mechanisms: `game.registry` for in-memory cross-scene state (shopState, runCount) and scene `init(data)` for per-transition data (treasureEarned). Both registry values are also persisted to localStorage via `@/src/systems/persistence.js` so state survives browser reloads

### Core Implementation
- **GameScene lifecycle**: `init()` reads upgrade state from registry and computes per-run stats, `create()` sets up the level and all game objects, `update()` runs every frame
- **GameScene.init()**: reads `shopState` from `game.registry` (or uses zeroed defaults on first run), calls `getUpgradeValue()` for each upgrade to set `this.maxCharge`, `this.flashlightAngle`, `this.maxHp`, and `this.playerSpeed`. Also reads and increments `runCount` in the registry to derive a unique level seed for this run. After incrementing, calls `saveGame()` to persist the updated `runCount` and `shopState` to localStorage (using `createShopState()` as a fallback if no shop state exists in the registry)
- **create() setup sequence**: initializes level geometry, doors, switches, player, enemies, item textures, items, bullets, exit zone, input, camera, darkness overlay, and HUD. Door creation happens after rooms but before player creation so that door colliders are ready. Switch creation happens after doors. Combat state is initialized with `createCombatState(this.maxHp)`, battery state with `createBatteryState(this.maxCharge)`, inventory state with `createInventoryState()`, and hiding state with `createHidingState()` from `@/src/systems/hiding.js`. Notable setup details:
  - `createEnemies()` adds a `health` field (from `ENEMY_MAX_HP`) to each enemy state object and wires a player-enemy overlap callback (`onEnemyContact`) for contact damage
  - `createItemTextures()` generates a Phaser texture for each item type (rectangles for coins/batteries, triangle for gems), keyed as `item_{type}`
  - `createItems()` creates a static physics group, calls `generateRoomItems()` per room, and places item sprites at depth 5 (below darkness). Wires a player-item overlap callback (`onItemPickup`) for auto-collection
  - `createDoors()` calls `createDoorStates()` from `@/src/systems/doors.js` to determine which connections have closable doors. For each closable door, creates a static physics zone in `doorGroup` sized to fill the doorway gap. Closed doors have their segment pushed into `wallSegments` and their physics body enabled; open doors have `body.enable = false`. Also creates a "Press E" interaction prompt text element (hidden by default)
  - `createSwitches()` calls `createSwitchStates()` from `@/src/systems/lightswitch.js` to place light switches in ~40% of non-starting rooms. Creates a Graphics object at depth 50 for switch rendering. Switches have no physics bodies -- they are purely visual objects interacted with via proximity checks
  - `createBullets()` creates a pooled physics group (`maxSize: 20`), generates a bullet texture, and wires bullet-wall/furniture/door colliders and bullet-enemy overlap
  - `createExitZone()` places a 64x64 physics zone in the top-left corner of room 0 with a green visual indicator, wiring a player overlap callback (`onDayComplete`) for the day-cycle win condition
  - `createHUD()` creates a Graphics object with `setScrollFactor(0)` at depth 1000 so health and battery bars stay fixed on screen, plus text elements for battery count and treasure value
- **update() frame loop**: early-returns if player is dead or `this.dayEnding` is true (exit transition in progress). Then: reads WASD key state -> if hiding and any movement key is pressed, exits hiding -> if hiding, zeroes velocity; otherwise, `calculateVelocity()` -> sets player velocity -> computes aim angle from mouse -> ticks combat cooldown (`updateCombat`), battery state (`updateBattery`), and fire cooldown (`updateFireCooldown`) -> checks for left-click + `canFire()` to fire bullets (blocked while hiding) -> `updateInteractPrompt()` resolves the nearest interactable (door, switch, or hideable furniture by Euclidean distance) and shows/hides "Press E" -> checks E key (`JustDown`) and dispatches to `onToggleDoor()`, `onToggleSwitch()`, or `enterHiding()` based on the resolved interactable type (pressing E while already hiding exits hiding) -> `updateBullets()` expires out-of-range bullets -> `updateEnemies(delta)` runs AI (with lit-room freeze check and `this.hidingState.isHiding` passed as the `playerHidden` flag) -> redraws player (with hiding, invulnerability, or normal visual state) -> recomputes darkness (flashlight cone + lit room rectangles drawn into BitmapMask) -> `drawDoors()` renders closed doors -> `drawSwitches()` renders switch visuals (gray when off, yellow when on) -> redraws HUD (health bar + battery bar + battery count + treasure value). Right-click input triggers `onUseBattery()` outside the frame loop via a `pointerdown` event listener (blocked while hiding). Item pickup happens via Phaser overlap callbacks (auto-collected on contact)
- **Enemy integration pattern**: `GameScene` maintains a parallel `enemyStates` array alongside the Phaser physics sprites in `enemyGroup`. Each frame, `updateEnemies()` syncs sprite positions into the state objects, calls the pure `updateEnemyAI()` function, copies the returned state back, and applies velocity to the sprite. This keeps the AI logic testable in `@/src/systems/enemy.js` while letting Phaser handle physics resolution
- **ShopScene lifecycle**: `init(data)` receives `treasureEarned` from GameScene, reads existing `shopState` from registry (or creates fresh state on first visit), adds the treasure as gold via `addGold()`, writes back to registry, and calls `saveGame()` to persist. `create()` builds the UI: title, treasure earned display, gold balance, upgrade rows, and "Enter Backrooms" button. Each upgrade row shows name, description, level dots (filled/empty circles), next cost, and a buy button. Purchases call `purchaseUpgrade()`, write back to registry, and call `saveGame()` to persist. The "Enter Backrooms" button writes shopState to registry, calls `saveGame()`, and then calls `this.scene.start('GameScene')` to transition back to gameplay. The save-at-every-mutation pattern ensures no progress is lost even if the browser crashes between user actions

### Things to Know
- **Depth layering order** controls what is visible through the darkness:

  | Layer | Depth | Purpose |
  |-------|-------|---------|
  | item sprites | 5 | Collectibles hidden by darkness (only visible in flashlight) |
  | exit zone graphics | 10 | Green exit indicator visible under darkness |
  | furniture graphics | 50 | Furniture rendered below enemies |
  | door graphics | 50 | Closed doors rendered at same depth as furniture |
  | switch graphics | 50 | Light switches rendered at same depth as doors/furniture |
  | enemy sprites | 60 | Enemies hidden by darkness (below mask) |
  | `lightGraphics` | 99 | Faint yellow tint drawn under the flashlight cone |
  | `darkGraphics` | 100 | Black rectangle covering the entire level (the darkness) |
  | bullet sprites | 150 | Bullets visible above darkness but below player |
  | `playerGraphics` | 200 | Player always visible (above darkness) |
  | `hudGraphics` + HUD text | 1000 | Health bar, battery bar, battery count, treasure value (above everything) |

  Items at depth 5, enemies at depth 60, and furniture at depth 50 are all hidden by the darkness overlay and only revealed by the flashlight. Bullets at depth 150 are always visible (above darkness), giving visual feedback even when firing into dark areas. The HUD uses `setScrollFactor(0)` so it does not move with the camera
- **Darkness system architecture**: the BitmapMask has `invertAlpha = true`, so white pixels in `maskGraphics` punch transparent holes in `darkGraphics`, revealing the room beneath. The `updateDarkness(time)` method first draws lit room rectangles (from `getLitRoomIds()`) into both `maskGraphics` (white, to punch holes in darkness) and `lightGraphics` (faint yellow tint). Then it calls `getFlashlightConeAngle()` each frame to get a battery-adjusted cone angle and `shouldFlicker()` to determine if the flashlight should momentarily go dark. When the cone angle is zero (battery depleted) or flickering is active, the method returns early after drawing lit rooms but without drawing the flashlight polygon. Lit rooms remain visible even when the flashlight is dead -- they use the same BitmapMask mechanism but draw rectangles instead of the flashlight polygon
- **Enemy physics uses a dynamic group** (unlike walls and furniture which are static groups). Enemies collide with walls, furniture, and each other. Player-enemy interaction uses `overlap` (not `collider`), triggering `onEnemyContact()` which applies contact damage, camera shake/flash, and checks for player death
- **Bullet physics group** uses pooling (`maxSize: 20`). Bullet-wall/furniture uses `collider` (instant deactivation). Bullet-enemy uses `overlap` -- on hit, the bullet is deactivated and the enemy takes `BULLET_DAMAGE`. When enemy health reaches 0, the enemy is deactivated and removed from `enemyStates[]`. Bullets store their origin position via `setData()` for range-expiry checks each frame
- **Player death flow**: when `combatState.isDead` becomes true, `onPlayerDeath()` stops the player body, disables physics, triggers a 500ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: 0 }`. The death penalty is losing all treasure collected that run -- gold already in the shop balance is not affected. The `update()` loop early-returns while dead, preventing further input processing
- **Day-complete flow (exit)**: when the player overlaps the exit zone, `onDayComplete()` sets `this.dayEnding = true`, stops the player, triggers a 1000ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: this.inventoryState.treasureValue }`. The player keeps all collected treasure. The `update()` loop early-returns while `dayEnding` is true, freezing game state during the transition
- **Visual feedback for combat, battery, hiding, and inventory**: taking damage triggers `cameras.main.shake()` and `cameras.main.flash()`. `drawPlayer()` selects from three visual states in priority order: hiding (muted green, alpha 0.3), invulnerability (red tint, alpha 0.5), or normal (green, alpha 1). The HUD health bar turns red when below 30% HP. The battery bar sits below the health bar and transitions from yellow to orange (at 50%) to red (at 25%). Below the bars, text elements display the current battery item count (`BAT xN`) and accumulated treasure value (`$N`)
- **E-key interaction resolution**: `updateInteractPrompt()` runs each frame and finds the nearest door, nearest switch, and nearest hideable furniture independently. All candidates are collected into a candidates array, sorted by Euclidean distance to the player, and the closest one wins. The resolved interactable is stored in `this.nearestInteractable` (with `type` and `item` fields), and E-key press dispatches to `onToggleDoor()`, `onToggleSwitch()`, or `enterHiding()` based on the type. Pressing E while already hiding exits hiding regardless of nearby interactables. This candidates-array approach is extensible for future interactable types
- **Hiding mechanics**: when the player presses E near `canHide` furniture (tables and desks), `enterHiding()` from `@/src/systems/hiding.js` sets `this.hidingState.isHiding` to true. While hidden: movement is disabled (velocity set to 0), shooting is disabled, right-click battery use is disabled, the player is immune to enemy contact damage (`onEnemyContact()` early-returns), and enemies cannot detect the player (the `playerHidden` flag forces `canSee = false` in `updateEnemyAI()`). The player graphic renders semi-transparent (alpha 0.3) with muted green colors. The player exits hiding by pressing any WASD key or pressing E again
- **Closable door toggle flow**: when the player presses E near a closable door, `onToggleDoor()` does three things: (1) if the door was closed, finds and splices its segment out of `wallSegments` via `indexOf`; (2) calls `toggleDoor()` from `doors.js` to produce new immutable door states; (3) enables/disables the Phaser physics body on the door zone. This ensures physics, raycasting, and enemy LOS all update atomically. Player, enemy, and bullet groups all have colliders against `doorGroup`, so closed doors block all three entity types
- **Light switch toggle flow**: when the player presses E near a switch, `onToggleSwitch()` calls `toggleSwitch()` from `lightswitch.js` to produce new immutable switch states. If the switch was toggled ON, it finds all enemies currently in the switch's room (via `isPointInRoom()`) and despawns them (deactivate sprite, stop body, remove from `enemyStates`). On subsequent frames, `updateEnemies()` checks `getLitRoomIds()` and freezes (zero velocity, skip AI) any enemy that is inside a lit room. This means enemies are despawned when the light first turns on, and any enemy that later wanders into the lit room is frozen but not destroyed
- **Multi-room physics with door gaps**: `createRoomPhysics()` splits each wall into physics zones that skip over door offsets. This mirrors the raycasting segments produced by `createRoomWalls()` so that physics collision and flashlight occlusion agree on where doors are. Closable doors fill these gaps with their own physics zones that can be toggled
- **Right-click input**: `setupInput()` disables the browser context menu (`this.input.mouse.disableContextMenu()`) and listens for `pointerdown` events. Right-click triggers `onUseBattery()`, which calls `useBattery()` from `inventory.js` and, if successful, `rechargeBattery()` from `battery.js`. This input is guarded by `!isDead && !dayEnding` checks
- **Item pickup flow**: items are a static physics group with player overlap detection. On overlap, `onItemPickup()` updates `this.inventoryState` via `pickupItem()` and destroys the item sprite. Items store their type and value via Phaser's `setData()` on the sprite. Items do not produce wall segments -- they are collectibles, not occluders
- **Wall dual-purpose**: walls exist as both Phaser Arcade Physics static zones (for player collision) and as line segments from `createRoomWalls()` (for raycasting). Furniture and closable doors follow the same pattern -- each is both a physics zone and a raycasting segment. Doors are unique in that they can toggle between active/inactive at runtime via `body.enable` and `wallSegments` push/splice. Enemies, items, and light switches do not produce segments; enemies are dynamic actors, items are small collectibles, and light switches are proximity-based interactables with no physics body
- **Player rendering**: the player is a Graphics object (`playerGraphics`) drawn at depth 200 (above the darkness overlay at depth 100), so the player is always visible regardless of flashlight direction
- Camera and physics bounds are computed dynamically from the level's room positions, so they adapt to however many rooms the level generator produces
