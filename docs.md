# Noridoc: Backrooms (Project Root)

Path: @/

### Overview
- Survival horror top-down 2D twin-stick shooter built with Phaser 3, Vite, and plain JavaScript
- Based on the "Backrooms" creepypasta: player explores pitch-black rooms using only a flashlight, collects treasure, fights zombies, and must return to the exit before the battery dies
- Implements a Motherload-inspired roguelike loop: explore rooms, collect treasure, exit (or die), spend gold on upgrades in the shop, re-enter with stronger stats

### How it fits into the larger codebase
- `@/index.html` is the single HTML entry point, loading `@/src/main.js` as an ES module
- `@/src/main.js` creates the Phaser.Game instance and registers both scenes (GameScene and ShopScene)
- Game logic is split between pure systems (`@/src/systems/`) and Phaser scene classes (`@/src/scenes/`) to keep game math testable without a browser or Phaser runtime. Audio follows this same split: pure config in `@/src/systems/audio.js` (testable in Node) and browser-dependent synthesis in `@/src/systems/audioEngine.js` -- all sounds are procedurally generated via Web Audio API with no audio file assets
- Cross-run state (gold balance, upgrade levels, run counter, collected lore IDs, unlocked locations, active starting location) lives in Phaser's `game.registry` for in-memory cross-scene access, and is backed by `localStorage` via `@/src/systems/persistence.js` for cross-session persistence. On boot, `@/src/main.js` loads saved state from localStorage into the registry before any scene starts
- `@/APPLICATION_SPEC.md` is the source of truth for game design requirements
- `@/CURRENT-PROGRESS.md` tracks what has been implemented vs. what remains
- `@/RESEARCH-NOTES.md` captures technology decisions and design rationale

### Core Implementation
- **Build tooling**: Vite dev server on port 8080, Vite build targeting ES2020, Phaser chunked separately via Rollup `manualChunks` for cache efficiency
- **Testing**: Vitest with Node environment -- systems are pure functions that do not require a DOM or Phaser
- **Phaser config**: 1024x768 canvas, Arcade physics with zero gravity (top-down game), auto-renderer selection
- **Controls**: WASD movement, mouse aim, left-click shoot, right-click use battery item, E key to interact with nearby doors, light switches, hideable furniture, or weapon pickups (nearest one wins by distance). E key exits hiding (WASD moves at half speed within the furniture while hidden). Q key or scroll wheel switches between the two weapon slots (only works when both slots are occupied; scroll wheel has a 200ms cooldown)
- **Game loop**: the game alternates between two scenes. `GameScene` is the gameplay scene where the player explores, fights, and collects treasure. On exit or death, the game transitions to `ShopScene` where the player spends accumulated gold on upgrades (battery capacity, flashlight width, max health, movement speed, weapon damage, fire rate, bullet range). Clicking "Enter Backrooms" transitions back to `GameScene` with upgraded stats. Difficulty scales with `runCount` in two ways: enemy stats (HP, count, chase speed, contact damage) scale via `@/src/systems/scaling.js`, and the number of dungeon floors increases via `getFloorRoomCounts()` in `@/src/systems/stairs.js` (2 floors for runs 0-1, 3 floors for runs 2-3, 4 floors for runs 4+). More floors means more battery drain to explore and return -- a natural difficulty escalator. Death forfeits all treasure earned that run; exiting keeps it. Additionally, alternate exit portals may appear on deeper floors (floor 1+), granting the player a new starting location when used; players can cycle through unlocked starting locations in the shop screen
- **Level generation**: a seeded procedural level generator (`@/src/systems/level.js`) produces multiple connected rooms with doorways, including a post-placement pass that adds extra doors between grid-adjacent rooms (~50% chance per unconnected neighbor pair). The first room placed (room 1) is always east of room 0, so the crack/entry to the backrooms is deterministically on room 0's east wall regardless of seed. The seed increments each run via a `runCount` stored in `game.registry`, so each run produces a different layout. Non-starting rooms get internal structure via `@/src/systems/maze.js` which assigns one of six room types: open, maze (shallow recursive division), columns (staggered grid), corridor (deep recursive division), storage (rejection-sampled crates), or cubicles (U-shaped grid partitions). `GameScene` renders and wires physics for the entire level at startup. Room 0 (the starting/spawn room) is special: its visual theme (floor/wall colors, furniture layout) is determined by the active starting location via `@/src/systems/locations.js`, which defines multiple themed locations (e.g., furniture store, office building, parking garage, hotel lobby, laundromat). Room 0 always has permanent lighting (no darkness overlay), a crack-in-the-wall visual at its first doorway with a pulsing glow (using `generateCrackPoints` from `@/src/systems/startroom.js`), and the exit zone at its west-center position
- **Closable doors**: approximately half of room connections (seeded random, 50% chance) have closable doors managed by `@/src/systems/doors.js`. Closable doors start closed, blocking movement (physics colliders), flashlight rays (wall segments), enemy line-of-sight, and standard enemy cross-room pathfinding (the standard room graph from `@/src/systems/pathfinding.js` excludes closed door connections). Crawlers bypass this defense -- they pathfind using a separate "full" room graph that includes closed-door edges, and can open closed doors after a 1-second delay (see Enemy system). The player can toggle them with the E key when within range. Opening a door removes its segment from the `wallSegments` array; closing it re-adds the segment -- following the same mutable segment-array pattern as walls and light-blocking furniture. Toggling a door also invalidates the cached room graph so enemy pathfinding reflects the new connectivity
- **Light switches**: ~15% of non-starting rooms get a light switch placed on a random wall, managed by `@/src/systems/lightswitch.js`. The player toggles switches with the E key (same as doors -- nearest interactable wins by Euclidean distance). Turning a switch on reveals the entire room by drawing a white rectangle into the BitmapMask (same system the flashlight uses) and freezes any enemies currently in or later wandering into that room (velocity zeroed, AI skipped). Enemies in lit rooms remain alive and visible but frozen in place -- turning the switch off lets them resume AI. Switches use a seeded PRNG with offset +40000
- **Enemy system**: enemies spawn in every room except the starting room, using a pure-function AI state machine (`@/src/systems/enemy.js`). Three enemy types exist: basic (standard zombie), crawler (fast melee, smaller, lower HP), and spitter (ranged, fires projectiles, has an `attack` AI state). Type distribution is run-gated: run 0 spawns only basic enemies, run 1 introduces crawlers, and run 2+ adds spitters. Each type carries its own speed constants, HP, and contact damage. The AI state machine has five states: idle/chase/search (shared by all types), attack (spitter-only, fires projectiles when in range), and opening_door (crawler-only, pauses to open a closed door blocking the path). Enemies detect the player via line-of-sight raycasting against the same wall segment array used by the flashlight system. Detection is suppressed when the player is hiding (via the `playerHidden` parameter on `updateEnemyAI()`). Room 0 is permanently included in the lit room list, so any enemies that wander in are frozen (same behavior as rooms with an active light switch). Enemy stats (HP, count per room, chase speed, contact damage) scale with `runCount` via pure functions in `@/src/systems/scaling.js` -- scaling is applied per-type using each type's base values, and all scaling is capped to prevent infinite growth. **Cross-room movement**: enemies can navigate between rooms via BFS-based pathfinding (`@/src/systems/pathfinding.js`). `GameScene` builds a room connectivity graph from `room.doors` data (excluding closed doors), runs BFS from the player's room each frame, and provides per-enemy `navContext` to `updateEnemyAI()`. In chase state, if an enemy loses LOS but is in a different room from the player and a doorway path exists, it moves toward the next doorway at chase speed instead of transitioning to search. Idle enemies probabilistically (15% chance per wander tick) seek nearby doorways to migrate between rooms, constrained by a 10-second cooldown after room transitions, a maximum BFS distance of 2 from spawn, and a cap of 4 enemies per room. Closing doors removes connections from the standard pathfinding graph, making door-closing a valid defensive strategy against basic and spitter enemies. However, crawlers can open closed doors: `GameScene` maintains a second "full" room graph (via `buildFullRoomGraph()` from `@/src/systems/pathfinding.js`) that includes closed-door connections, allowing crawlers to pathfind through closed doors. When a crawler in chase state reaches a closed door within range, it enters the `opening_door` state, pauses for 1 second, then signals `wantsToOpenDoor` -- GameScene executes the door toggle. This creates asymmetric threat: doors block most enemies but not crawlers. **Danger escalation**: beyond per-run stat scaling and static enemy placement, a time-based danger system (`@/src/systems/danger.js`) spawns waves of enemies as the player spends time in unsafe rooms (any room that is not room 0 and not lit). The timer pauses but does not reset in safe rooms, so hiding in a safe room delays but cannot prevent escalation. Every 30 seconds of cumulative unsafe time triggers a wave, with enemy count and type composition shifting toward harder enemies as waves progress (basic-only early, mixed crawler/spitter late), up to a cap of 8 waves per run. Wave enemies spawn in a valid room on the same floor (not room 0, not lit, not the player's current room) and have the same state structure and behavior as pre-placed enemies. The HUD shows a red "DANGER N" indicator when waves have been spawned
- **Hiding mechanics**: the player can hide inside `canHide` furniture (any furniture type with `canHide: true` in `@/src/systems/furniture.js`, e.g., table, desk, bed, armoire, closet, vent) by pressing E when near them, managed by `@/src/systems/hiding.js`. While hidden: the player moves at half speed (`HIDING_SPEED_MULTIPLIER = 0.5`) constrained to the furniture bounds, shooting and battery use are disabled, the player is invisible to enemy line-of-sight detection and immune to contact damage, and the player graphic renders semi-transparent. Only the E key exits hiding (WASD moves the player within the furniture). The hiding system uses Phaser's `body.setBoundsRectangle()` to constrain the player within the furniture area, and the player-furniture collider is disabled while hidden (otherwise physics would push the player out of the furniture body). Hiding adds strategic depth by creating mobile safe spots within enemy-occupied rooms
- **Exploration and minimap**: a fog-of-war minimap renders in the top-right corner of the screen, managed by `@/src/systems/exploration.js`. The minimap is a shop upgrade (id: `minimap`, 3000 gold, single tier) -- it is not available by default. When purchased, rooms are revealed as the player visits them. The minimap shows visited rooms (dark gray), the current room (lighter gray), the player position (green dot), and the exit position (green dot). Exploration state tracks which rooms have been visited using `isPointInRoom()` from `@/src/systems/lightswitch.js` for room detection regardless of whether the minimap is purchased (future features may use visitation data). The minimap is drawn via Phaser Graphics at depth 1000 with `setScrollFactor(0)`, same as the HUD. Exploration state resets each run -- no persistence needed
- **Combat loop**: enemies damage the player on contact (with invulnerability cooldown) and spitter enemies additionally fire projectiles. Each enemy type has its own contact damage value stored on its state object, so `onEnemyContact` reads damage per-enemy rather than using a single global value. The player has a 2-slot weapon inventory (managed by `@/src/systems/weapons.js`) with three weapon types: Pistol (default, balanced), Shotgun (multi-pellet spread, close range, slow fire), and Rifle (high damage, long range, slow fire). Weapon pickups spawn in rooms and are collected via E-key. Q key or scroll wheel switches weapons. Upgrade bonuses (damage, fire rate, range) from the shop are applied additively to whichever weapon is active. Bullets store per-bullet damage and range so in-flight bullets retain correct stats after weapon switches. When an enemy dies, its sprite persists as a visible corpse (gray tint, faded alpha, toppled angle, depth 5) with physics disabled (`setActive(false)`, `body.enable = false`) -- corpses are safe because Phaser skips inactive sprites in overlap checks. Enemy projectiles use a separate physics group (`enemyBulletGroup`) that mirrors the player bullet pattern (pooling, wall/furniture collision, range expiry). Player death triggers scene restart. All combat/shooting/weapon math lives in pure-function systems (`@/src/systems/combat.js`, `@/src/systems/shooting.js`, `@/src/systems/weapons.js`), with Phaser wiring in `GameScene`
- **Survival loop**: the flashlight battery drains over 150 seconds, progressively narrowing the flashlight cone from 100% to 25% of its base angle (base angle is PI/6 or 30 degrees at upgrade level 0). Below 25% charge, the flashlight flickers. At 0%, the flashlight goes completely dark. The player can collect battery items and use them (right-click) to recharge the flashlight by 30 charge. The player must reach the exit zone (in room 0) before the battery dies. Reaching the exit transitions to the shop with all treasure earned. Battery math lives in `@/src/systems/battery.js`, following the same pure-function immutable-state pattern as combat and shooting
- **Items and inventory**: collectible items (batteries, coins, gems) spawn in every room except the starting room via a seeded PRNG. Items are only collected on player overlap when inventory has capacity (controlled by `canPickupItem()` from `@/src/systems/inventory.js`) -- items remain on the ground when the inventory is full. Battery items are stored in inventory and consumed with right-click to recharge the flashlight (consuming a battery also frees an inventory slot). Treasure items (coins, gems) accumulate a value total that converts to gold in the shop. Inventory capacity defaults to 8 items and is increased by the "Backpack Size" shop upgrade (+4 per level, max 20). Item spawning lives in `@/src/systems/items.js`, inventory state management in `@/src/systems/inventory.js`, both following the pure-function pattern
- **Lore items**: collectible lore notes spawn in ~25% of non-starting rooms (max 1 per room) via `@/src/systems/lore.js`, a pure-function module using a seeded PRNG (seed offset +90000). Lore items are auto-collected on player overlap (they do NOT consume inventory space) and display a tween-animated text popup at the bottom of the screen that auto-dismisses after 5 seconds. Lore is a meta-progression collectible -- collected lore IDs are persisted across runs via `collectedLore` in `game.registry` and localStorage, and already-collected entries do not respawn. The module defines 20 Backrooms-themed lore entries (atmospheric/horror text). Lore items use a separate physics group (`loreGroup`) from regular items (`itemGroup`). Between runs, the ShopScene provides a **lore journal** overlay (activated via a "NOTES X/20" button) that lets players review all collected and uncollected lore entries with pagination. The journal data logic (`getJournalEntries`, `getJournalPage`) lives in `@/src/systems/lore.js` as pure functions, while the overlay rendering lives in `@/src/scenes/ShopScene.js`
- **Persistence**: game progress (shopState, runCount, collectedLore, unlockedLocations, activeLocation) is persisted to localStorage via `@/src/systems/persistence.js`, a pure function module that uses a versioned save format (`{ version: 3, shopState, runCount, collectedLore, unlockedLocations, activeLocation }`) under the key `backrooms_save`. State is saved at every mutation point in both scenes (after adding gold, purchasing upgrades, incrementing runCount, collecting lore, unlocking locations, and before entering backrooms) and also by browser lifecycle listeners (`visibilitychange`, `beforeunload`) installed in `@/src/main.js`. On boot, `main.js` loads saved state and populates `game.registry` (including `collectedLore`, `unlockedLocations`, and `activeLocation`) before scenes start. All localStorage access is try-catch wrapped for graceful degradation. Older saves are backward-compatible -- `loadGame()` defaults missing fields to safe values (e.g., `collectedLore` to `[]`, `unlockedLocations` to `['store']`, `activeLocation` to `'store'`)
- **Shop and upgrades**: between runs, the player visits a shop (`@/src/scenes/ShopScene.js`) where treasure earned converts to gold. Gold can be spent on upgrades across several categories (survival: battery, flashlight, health, speed; weapon: damage, fire rate, bullet range; inventory: backpack size; unlocks: starting pistol, minimap) with three tiers each for most upgrades (costs: 200/600/1500) and single-tier binary unlocks for others (starting pistol: 1000, minimap: 3000). Upgrade definitions, pricing, and purchase logic live in the pure-function module `@/src/systems/shop.js`. The UPGRADES array is the single source of truth -- ShopScene auto-generates UI rows from it. Shop state persists in `game.registry` across scene transitions. The shop also displays a lore journal button ("NOTES X/20") that opens a full-screen overlay for reviewing collected lore entries (see Lore items section)

### Things to Know
- The project uses Phaser 3 (~3.90.x) deliberately over Phaser 4, because Phaser 4's BitmapMask API changed with incomplete docs -- and the flashlight system is the core visual mechanic (see `@/RESEARCH-NOTES.md`)
- Systems in `@/src/systems/` are designed as pure functions with no Phaser dependency, so they can be tested with `vitest run` in a Node environment
- The `vite.config.js` test environment is set to `node`, not `jsdom`, because the testable code has no DOM dependencies
- `package.json` uses `"type": "module"` -- all imports use ES module syntax throughout
- All procedural generation (level layout, furniture placement, enemy spawning, item placement, door closability, light switch placement, weapon pickup placement, room interior generation, extra door connections, alternate exit spawning) uses a shared seeded PRNG (`@/src/systems/random.js`) so levels are deterministic for a given seed. Each system uses a different seed offset to avoid correlation: furniture +0, enemies +10000, items +20000, doors +30000, light switches +40000, stairs +50000, weapons +60000, maze +70000, extra doors +80000, lore +90000, alternate exits +100000. The seed changes each run via an incrementing `runCount` stored in `game.registry` (and persisted to localStorage via `@/src/systems/persistence.js`). The starting room system (`@/src/systems/startroom.js`), locations system (`@/src/systems/locations.js` for static location definitions), hiding system (`@/src/systems/hiding.js`), exploration system (`@/src/systems/exploration.js`), scaling system (`@/src/systems/scaling.js`), danger system (`@/src/systems/danger.js`), pathfinding system (`@/src/systems/pathfinding.js`), persistence system (`@/src/systems/persistence.js`), and audio systems (`@/src/systems/audio.js`, `@/src/systems/audioEngine.js`) do not use a seeded PRNG -- startroom uses a fixed hand-crafted layout, location definitions are static data, hiding operates on furniture objects already placed by the furniture system, exploration is deterministic based on player position, scaling applies deterministic formulas based on `runCount`, danger uses `Math.random()` for spawn positions because dynamic mid-run spawns do not need cross-session reproducibility, persistence is a browser storage boundary, and audio uses a Knuth multiplicative hash of game time for ambient scheduling (no effect on level determinism). The `generateAlternateExit()` function in `locations.js` is the one part of that module that does use a seeded PRNG (offset +100000)
