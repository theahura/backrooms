# Noridoc: systems

Path: @/src/systems

### Overview
- Pure functions implementing game math: player movement, 2D raycasting visibility, room wall geometry, coordinate-keyed deterministic unbounded room generation (`worldgen.js`, generated lazily a few cells ahead of the player), legacy multi-room/multi-floor level generation (`level.js`/`stairs.js`, retained for tests and the per-floor PRNG offsets but no longer driving `GameScene`), room interior variety (six archetypes: open, maze, columns, corridor, storage, cubicles), per-room visual + furniture "vibes" (`roomVibes.js`: distinct-but-still-desaturated identities like furniture store, library, office, derelict ship, overgrowth, warehouse, picked on a seed axis orthogonal to the structural archetype), profile-driven semantic-cluster furniture placement with type-dependent light blocking, starting room layout, enemy spawning/AI with strictly sight-based pursuit (last-seen-position memory), enemy difficulty scaling, time-based danger escalation, combat/health/healing management, shooting/bullet logic, weapon types/inventory/switching/ammo management, battery/flashlight drain and recharge, flashlight on/off toggle state with hiding auto-off (`flashlight.js`), item spawning (including ammo pickups, with distance-from-entrance loot density and tier scaling), enemy loot drops on death (with per-type drop chances and distance-based quality scaling), gunfire noise alerts (weapon-range-based enemy alerting on firing), legacy BFS hop-distance from the entrance over the door/stair graph (`distance.js`, test-only -- rooms now carry a precomputed `distance`), inventory management (with capacity constraints), shop/upgrade state management (including unlock-style upgrades), door state management, light switch state management, light spill through doorways, hiding state management, exploration/minimap tracking, per-run statistics tracking, touch/desktop input resolution, pause controls data, title menu option generation, lore note spawning and definitions, localStorage persistence, audio settings with volume control state management, seeded random number generation, gameplay/HUD camera-layer classification (`camera.js`: which display objects belong to the zoomed world camera vs. the fixed UI camera), procedural audio configuration, Web Audio sound synthesis, pixel art sprite definitions for texture generation, character animation definitions, and the AI-art manifest/helpers (`art.js`: prompts, target sizes, PNG-then-procedural texture-source selection helper, chroma-key background-removal decision, and shop-icon mapping) that drive the offline art generator -- noting that the live runtime PNG/fallback choice is actually the scenes' inline `textures.exists()` check, not the art.js helper
- Deliberately free of Phaser dependencies so they can be unit-tested in Node without a browser
- Tests live in `@/src/systems/__tests__/`

### How it fits into the larger codebase
- `@/src/scenes/GameScene.js` imports and calls all system modules during scene creation and on every frame during `update()`. Combat, shooting, weapon, battery, and run stats systems are called in the frame loop (cooldown ticking, battery drain -- routed through `updateBatteryWithFlashlight()` from `flashlight.js` so the battery only drains while the flashlight is effectively lit, bullet expiry, weapon switching, ammo consumption, time accumulation) and combat/shooting/runStats are also called in collision callbacks (damage application, bullet deactivation, kill recording). Inventory state is updated in item pickup callbacks and battery-use input. Weapon state is updated on Q-key or scroll-wheel switch, E-key pickup, firing (ammo consumption via `consumeAmmo()`), and ammo item pickup (`addAmmo()`). Firing is gated by both `canFire()` (cooldown) and `hasAmmo()` (ammo available) and `getActiveWeapon()` returning non-null. At scene init, `getUpgradeValue()` from `shop.js` is called to compute upgrade bonuses (weapon damage, fire rate, bullet range) as deltas from pistol base values, and these bonuses are applied additively to whatever weapon is active via `getEffectiveStats()` from `weapons.js`. The three starting weapon upgrade levels (`startingPistol`, `startingShotgun`, `startingRifle`) are checked as boolean flags and passed to `createWeaponState()`, which uses priority-based slot filling (rifle > shotgun > pistol) to fill up to 2 weapon slots -- when all three are purchased, the cheapest (pistol) is dropped. Scaling functions from `scaling.js` compute distance-scaled enemy stats at spawn time: `GameScene.getEnemyStats(type, distance)` derives HP and contact damage from the spawn room's `distance`, and `generateRoomEnemies()` derives per-room enemy count and type mix from the same value (movement speeds never scale). During room creation, `generateMazeWalls()` from `@/src/systems/maze.js` is called per non-starting room to produce solid internal wall rects (from one of six room archetypes) -- each rect is simultaneously the drawn fill, the physics zone, and a light/sight occluder, but the occluder is expanded per consumer: enemy LOS uses the rect's full outline (`wallRectSegments()`) while the flashlight uses only its far faces (`wallRectOccluders(rect, origin)`, back-face culling so interior walls show visible thickness) -- and `getRoomVibe(room.seed)` from `@/src/systems/roomVibes.js` is called to determine each room's floor/wall colors AND its furniture profile (the vibe seed axis is independent of `getRoomType()`, so structural layout and visual identity vary separately). Audio timing functions from `@/src/systems/audio.js` (`getFootstepInterval`, `getAmbientSoundDelay`) are called each frame and on ambient scheduling; per-enemy-type sound functions (`getEnemySoundEvent`, `getEnemyDeathSound`, `getEnemyFireSound`, `getDistanceVolume`) are called from `updateEnemies()` and `onBulletHitEnemy()` for proximity-based enemy audio; sound synthesis from `@/src/systems/audioEngine.js` (`generateAllSounds`, `createAmbientDrone`, `playAmbientSound`) is called at scene creation and during gameplay for sound playback. `rollEnemyDrop()` from `@/src/systems/lootDrops.js` is called in `onBulletHitEnemy()` when an enemy dies to determine if items drop at the enemy's position. After firing, `getAlertedEnemies()` from `@/src/systems/gunfireNoise.js` is called with the active weapon's `noiseRange` and the player's room ID to determine which idle/search enemies within BFS hop distance should be alerted -- alerted enemy states are directly mutated to chase, gated by a 5-second cooldown (`GUNSHOT_ALERT_COOLDOWN`)
- `@/src/scenes/TitleScene.js` imports `loadGame()` and `clearSave()` from `persistence.js` to detect save state and handle new game/delete save actions. Imports `getMenuOptions()` and `getTitleText()` from `@/src/systems/titleMenu.js` for menu option generation. Imports `generateAllSounds()` and `createAmbientDrone()` from `@/src/systems/audioEngine.js` for ambient audio on the title screen. Imports `getEffectiveVolume()`, `cycleVolume()`, `saveSettings()`, `createDefaultSettings()`, and `VOLUME_LEVELS` from `@/src/systems/settings.js` for the settings overlay that controls audio volume
- `@/src/scenes/ShopScene.js` imports the shop system for upgrade definitions, pricing, purchase validation, and state management. ShopScene reads/writes `shopState` from `game.registry` and uses the pure functions from `shop.js` for all purchase logic. ShopScene also imports `saveGame()` from `persistence.js` to persist state to localStorage at each mutation point. ShopScene imports `generateAllSounds()` from `@/src/systems/audioEngine.js` to initialize sound synthesis on scene creation, playing UI feedback sounds (purchase, denied, click) for shop interactions
- `@/src/main.js` imports `loadGame()` and `saveGame()` from `persistence.js` to hydrate `game.registry` on boot (including `unlockedLocations` and `activeLocation`) and to flush state to localStorage on browser lifecycle events (`visibilitychange`, `beforeunload`). It also imports `loadSettings()` from `settings.js` to load audio volume preferences into `game.registry` under the key `'audioSettings'` on boot (from a separate localStorage key `backrooms_settings`). It also imports `detectTouchPrimary()` from `touchControls.js` at boot to decide the Phaser Scale mode (touch devices get `Scale.RESIZE` at native viewport size; desktop keeps `Scale.FIT` at fixed 1024x768)
- The world is generated lazily. `GameScene` no longer builds a whole multi-floor level up front; instead it generates ~2 rooms ahead of the player via `worldgen.js` and activates each room exactly once (see the scenes docs for the lazy-activation pipeline). Rooms stay static once generated. Floors are stacked as non-overlapping horizontal bands in a single physics world (`GameScene` constant `FLOOR_Y_OFFSET = 5_000_000`, large enough that unbounded floors never collide), and camera/physics world bounds are a large fixed region (`WORLD_BOUND`) rather than a computed level extent. The legacy `generateMultiFloorLevel()`/`getFloorRoomCounts()` path in `stairs.js` is retained for tests but unused at runtime
- Light-blocking furniture from `furniture.js` (types with `blocksLight: true`, e.g., shelf, bookcase, armoire, closet) is NO LONGER appended to the wall segments array. It now occludes EXACTLY like a maze wall: `@/src/scenes/GameScene.js` keeps each piece's VISIBLE rect (sized to the opaque art, not the padded texture) in a separate store (`furnitureOccluderRects`, parallel to `mazeWallRects`) and expands it per consumer per frame -- the flashlight to far faces only via `wallRectOccluders()` (back-face culling, so the cone reaches the near face and the piece reads as illuminated with visible thickness instead of pitch-black), enemy LOS to the full outline via `wallRectSegments()`. Furniture with `blocksLight: false` (e.g., table, desk, bed, vent) does not occlude at all. That same `blocksLight` set is the unified "tall/solid cover" classification: via the exported `blocksBullets()` helper it ALSO stops gunfire, so light occlusion, enemy/player line of sight, and bullet blocking are all driven by one property (`@/src/scenes/GameScene.js` tags furniture zones with a `blocksBullets` data flag and uses it as a bullet-vs-furniture process callback). Maze internal walls from `maze.js` are likewise kept as solid rects in their own `GameScene` store (`mazeWallRects`) and expanded the same way per consumer; see the maze module entry. Closed door segments from `doors.js` ARE pushed/spliced into the wall segments array dynamically when doors are toggled. Items from `items.js`, lore notes from `lore.js`, light switches from `lightswitch.js`, the light spill system (`lightSpill.js`), the hiding system (`hiding.js`), and the exploration system (`exploration.js`) do not produce segments -- items and lore notes are small collectibles, switches are small interactable objects, light spill is a visual-only gradient system, hiding is a player state system that reuses existing furniture objects (any with `canHide: true`), and exploration is a tracking system that monitors which rooms the player has visited. Light switches interact with the enemy and darkness systems via lit room IDs rather than wall segments. The light spill system extends the lit room visual by drawing gradient rectangles through open doorways into the BitmapMask and light tint layer, consuming lit room IDs from `lightswitch.js` and door states from `doors.js`. The hiding system interacts with enemy AI via an optional `playerHidden` flag on `updateEnemyAI()`. The exploration system reuses `isPointInRoom()` from `lightswitch.js` for room detection
- Movement system output (`{x, y}` velocity) is passed directly to `player.setVelocity()`
- Visibility system output (polygon as `{x, y}` point array) is drawn into a Phaser Graphics object used as a BitmapMask to punch a hole in the darkness overlay
- Enemy system (`enemy.js`) reuses `raySegmentIntersection` from `visibility.js` for line-of-sight checks and `mulberry32` from `random.js` for deterministic spawning -- it follows the same pure-function, segment-based integration pattern as furniture. Enemy pursuit is strictly sight-based: per the binding spec, enemies must not follow the player once they lose line of sight (walls, closed doors, and hiding all break LOS), so the AI tracks only a stored last-seen position rather than the player's live coordinates (see the Enemy spawning and AI bullet). The pathfinding system (`pathfinding.js`) is deliberately NOT used for chase routing -- `GameScene` builds a single closed-door-respecting room graph (via `buildRoomGraph()`) and uses it only for idle wander filtering (how far from spawn an idle enemy may drift, via `getRoomDistance()`) and for the gunfire sound-alert radius (via `gunfireNoise.js`). The optional `navContext` parameter on `updateEnemyAI()` carries only idle doorway-seek wandering data. The pathfinding module consumes `room.doors` data and door states from `doors.js` to build its adjacency graph, and `GameScene` reuses `getCurrentRoom()` from `exploration.js` for room detection
- The dependency flow is: `GameScene.activateRoom()` calls `createRoomWalls()` from `room.js` for each activated room to produce perimeter wall segments with door gaps (on floor 0 the rift door is deliberately left OUT of the gap list, so the rift stays sealed against light in both directions) -> `GameScene` adds closed door segments from `doors.js` -> these segments accumulate in a single mutable `wallSegments` array that grows forever as the unbounded world is explored. The maze wall rects from `maze.js` AND the light-blocking furniture rects from `furniture.js` (or `locations.js` for room 0, via `getLocationLayout()`) accumulate SEPARATELY in two parallel rect stores (`mazeWallRects` and `furnitureOccluderRects`, both unbounded, never cleared); per frame each consumer expands the nearby ones differently -- `enemy.js` LOS to the full outline (`wallRectSegments()`), `visibility.js`'s flashlight to the far faces only (`wallRectOccluders()`). `wallSegments` (via `filterSegmentsNear()`) and both rect stores (via `filterRectsNear()`, all from `visibility.js`) are bounding-box prefiltered each frame so per-frame raycast cost stays proportional to nearby occluders. Door segments are dynamically added/removed when the player toggles a door. (The legacy `level.js` -> `room.js` whole-level flow is retained for tests but no longer runs at runtime.) Separately, `locations.js` provides the themed layout, colors, and exit position for room 0 (based on the active starting location), while `startroom.js` provides the crack visual points; `items.js` spawns collectibles per room (consuming the furniture list for overlap avoidance), including medkits that bypass inventory and heal via `applyHeal()` from `combat.js`; `inventory.js` tracks what the player has picked up (excluding medkits, which never enter inventory); `lightswitch.js` manages per-room switch states that feed into the darkness mask and enemy AI freeze logic; `hiding.js` manages whether the player is hidden inside `canHide` furniture (feeding into enemy AI via the `playerHidden` parameter on `updateEnemyAI()`); and `exploration.js` tracks visited rooms and computes minimap rendering data from player position and level geometry

### Core Implementation
- **Unbounded room generation** (`@/src/systems/worldgen.js`): the live world generator. The world is an infinite grid of rooms; everything about the room at grid cell `(gx, gy)` -- its content `seed` and its four possible doors -- derives from a hash of `(floorSeed, gx, gy)`, so a room is identical regardless of when or in what order it is generated, and adjacent cells always agree on the door they share (a door is bond-percolation-decided once for the shared edge). `getRoomAt(seed, gx, gy)` returns `{ id: "gx,gy", gridX, gridY, x, y, width, height, doors: [{ wall, offset, width, targetRoomId }], seed }` with door open-probability `DOOR_PROB = 0.62` (above the 2D percolation threshold so the open-door graph forms one infinite connected cluster -- the player is never boxed in). The entrance cell `(0,0)` (the store) has EXACTLY one opening: the rift in its east wall; its other three edges are forced sealed, so the store's sole connection to the backrooms is the rift. `isRiftDoor(room, door)` identifies that single opening from either side (the store's east door, or cell `(1,0)`'s west door) -- consumed by `GameScene` to seal the rift against light and treat it as a dimensional transition rather than a doorway. `roomsWithinRadius(seed, cx, cy, radius)` BFS-walks the open-door graph and returns all rooms reachable within `radius` door-hops (GameScene uses `radius = 2` for activation). `hasStairDown(seed, gx, gy)` is a deterministic `STAIR_PROB = 0.08` roll, never true at `(0,0)`. `localDistance(gx, gy)` is Manhattan distance from `(0,0)`. Constants: `ROOM_WIDTH = 720`, `ROOM_HEIGHT = 600`, `DOOR_WIDTH = 80` -- rooms were deliberately shrunk (from 1200x1000) in the liminal-scale pass so that, combined with larger character sprites and dense furniture, the world reads as claustrophobic rather than empty. This module is pure (no Phaser, no mutable state) and is the deterministic backbone of the lazy world -- `GameScene` re-stamps the string `id`/`targetRoomId` into numeric ids on activation but never re-derives geometry. The top-level `seed` that flows into `floorSeed` is the **persisted per-save world seed** (`resolveWorldSeed()` from `@/src/systems/persistence.js`, threaded through `GameScene.init()` as `this.levelSeed`), NOT the run/day counter -- so the whole world is byte-identical day-to-day for a given save (the player builds a mental map) while differing between players, since the seed is randomly minted once per save. Tests live in `@/src/systems/__tests__/worldgen.test.js`
- **Legacy multi-floor level generation + canonical stair geometry** (`@/src/systems/stairs.js`): no longer drives `GameScene` (which now generates the world lazily via `worldgen.js`), but retained because its tests still pass, several seed/offset conventions are shared, AND it is the single source of truth for where a room's stair sits. The stair geometry helpers ARE live: `getStairTopLeft(room, stairSize, margin)` returns the canonical centered stair top-left (a fixed square inset by `STAIR_MARGIN` on every side), and `stairKeepOut(room, stairSize, margin, padding)` returns that footprint inflated by `padding` for use as a maze `keepOut` zone (see `@/src/systems/maze.js`). The stair is deliberately CENTERED rather than searched-for so the center-relative landing math stays valid -- rooms with a stair instead carve their maze walls away from the keep-out square. The **stair LANDING** (where the player materializes after a transition) is single-sourced here too: `getStairLandingPosition(room)` returns a point horizontally centered and dropped `STAIR_LANDING_DROP` (= `STAIR_SIZE + 80`) below the room center, far enough below the centered stair that re-entry does not instantly re-fire the stair overlap (which would bounce the player between floors). Because the landing sits ~140px below center -- outside the furniture room-center spawn zone -- it is NOT otherwise reserved, so `stairLandingKeepOut(room, size = STAIR_SIZE)` returns a `size`x`size` square around the landing for use as BOTH a maze keep-out (CARVE) and a furniture obstacle (REJECT), mirroring `stairKeepOut`. This is the residual fix for the spec's "stairs spawn inside walls" bug -- the landing spot itself was previously unreserved, so a maze wall or furniture piece could spawn exactly where the player lands (Arcade Physics then shoved the player out -- visible jank). `@/src/scenes/GameScene.js` consumes `getStairLandingPosition()` in `onStairEnter()` (replacing the formerly-hardcoded inline formula) and `stairLandingKeepOut()` in `activateRoom()`. `getFloorRoomCounts(runCount)` and `generateMultiFloorLevel(seed, floorRoomCounts)` have ZERO production callers (kept alive only by their tests). `getFloorRoomCounts(runCount)` returns a floor room-count array that grows with run count -- a legacy bounded-world escalator; the live game's world is unbounded and its difficulty scales on room distance instead (see `@/src/systems/scaling.js`). `generateMultiFloorLevel(seed, floorRoomCounts)` wraps `generateLevel()` once per floor, offsetting each floor's rooms and wall segments by `floor * FLOOR_Y_OFFSET` (10000px) in world-space Y. Room IDs are re-numbered globally so they remain unique across floors. Each room gets a `floor` field. After generating all floors, `createStairConnections()` uses a seeded PRNG (seed offset +50000) to create up to 2 stair connections per adjacent floor pair (floor i to floor i+1), iterating all consecutive floor pairs generically. For floor 0, room 0 (the starting room) is excluded from stair candidates. Each stair pair uses `usedFrom` and `usedTo` Sets to avoid reusing rooms within a floor pair. Each stair object carries `fromFloor` and `toFloor` fields alongside room IDs and positions. Each stair's position is centered within its room (with 100px margin). Returns `{ rooms, wallSegments, stairs }`. `getFloorBounds(rooms, floor)` computes the axis-aligned bounding box for all rooms on a given floor -- retained and tested but no longer used by `GameScene` (the lazy world has no finite floor extent; the minimap now uses the player-centered window from `getWindowedMinimapData()` in `@/src/systems/exploration.js`). The legacy constant `FLOOR_ROOM_COUNTS=[4,3]` is still exported for backward compatibility. Constants: `FLOOR_Y_OFFSET=10000`, `STAIR_SIZE=60` (the legacy module's own offsets -- the live world uses `GameScene`'s `FLOOR_Y_OFFSET = 5_000_000` and `STAIR_SIZE`)
- **Single-floor level generation** (`@/src/systems/level.js`): `generateLevel(seed, roomCount)` places rooms on a grid using a seeded growth algorithm. Starting from room 0 at grid position (0,0), each subsequent room picks a random existing room, picks a random cardinal direction, and places a new room in the adjacent grid cell if it is empty. **Exception**: the first room placed (i=1) always uses `DIRECTIONS[1]` (east), so room 0's first connection (the crack to the backrooms) is deterministically on the east wall regardless of seed. The PRNG still consumes a direction roll for i=1 (to preserve sequence stability for subsequent rooms) but discards it in favor of the forced east direction. Paired doorways are added to both rooms at the same wall offset. After all rooms are placed, `addExtraDoors(rooms, grid, seed)` iterates over all rooms and their grid neighbors, and for each unconnected adjacent pair, probabilistically (~50%) creates a new door connection between them using a separate seeded PRNG (seed offset +80000) to preserve determinism of the primary generation. Returns `{ rooms, wallSegments }` where `wallSegments` is the flattened array of all room wall segments (with door gaps from both primary and extra doors). This is a pure function -- no Phaser dependency. It is called by `stairs.js` as the per-floor building block
- **Room geometry** (`@/src/systems/room.js`): `createRoomWalls(x, y, width, height, doors)` returns line segments forming a rectangular boundary with gaps for doorways. When `doors` is empty or omitted, it returns four simple wall segments (backward-compatible). When doors are present, each wall is split into sub-segments around the door gaps by `splitWall()`. Segments maintain clockwise winding order -- south and west walls are reversed via `reverseSegments()` so endpoint connectivity is preserved. This module is live at runtime: `GameScene.activateRoom()` calls it for every activated room to push perimeter occluder segments into the shared `wallSegments` array. (An earlier unbounded-world rework had left perimeters physics-only with no occluder segments, which let the flashlight shine through walls and enemies see through them -- restoring this call was the fix.)
- **Random** (`@/src/systems/random.js`): exports `mulberry32(seed)` which returns a stateful function that produces deterministic floats in [0, 1). Used by `level.js`, `furniture.js`, `enemy.js`, and `items.js`
- **Movement** (`@/src/systems/movement.js`): `calculateVelocity(keys, speed)` takes a key-state object and speed scalar, returns `{x, y}` velocity. Normalizes diagonal movement to match cardinal speed
- **Touch controls** (`@/src/systems/touchControls.js`): pure decision functions that fork desktop vs mobile input without any Phaser dependency, so the branching is unit-tested in Node. They exist as a separate module precisely so the input-mode logic does not get tangled into `GameScene.update()`. The functions are: `getMoveVelocity(angle, force, speed)` converts a joystick angle + normalized force into `{x, y}` velocity (force clamped to 0..1 so partial deflection is analog and full deflection never overshoots max speed); `resolveAimAngle({touchMode, aimActive, aimAngle, pointerAngle, lastAngle})` returns the mouse pointer angle on desktop, and on touch returns the aim-stick angle while the stick is deflected, otherwise the last angle (so the flashlight holds its direction when the thumb lifts off the aim stick to tap FIRE); `resolveFireIntent({touchMode, fireButtonDown, leftButtonDown})` fires on left button on desktop but on touch fires ONLY from the dedicated FIRE button (the browser reports touching the move/aim sticks as a left-button press, so this guard prevents stick-touches from firing -- the key invariant of the touch scheme); `resolveMoveVelocity({touchMode, moveActive, stickVelocity, keyboardVelocity})` picks the move-stick velocity when active in touch mode, otherwise the keyboard velocity; `detectTouchPrimary({maxTouchPoints, coarsePointer})` returns true if the primary pointer is coarse or the hardware reports touch points -- it is the single touch/desktop discriminator, called both by `@/src/main.js` (to choose the Phaser Scale mode at boot) and by `GameScene.setupInput()` (to set `this.touchMode`); `computeTouchLayout({width, height, stickRadius})` is the pure layout calculator that maps the live viewport size to screen-space positions for every touch control, returning `{ moveStick, aimStick, fireButton, useButton, actionRow, fullscreenButton }`. Its layout rules encode the mobile feedback fixes: the move stick is inset from the left edge by `stickRadius + 30` and the aim stick is symmetric on the right, both **vertically centered** at `round(height/2)` (this is the deliberate fix for downward thumb travel being eaten by the browser's bottom gesture zone -- bottom-anchored sticks could not deflect fully down); the fire button (radius 52) sits just inside the aim stick on the right-thumb side; the USE button is placed directly below the right aim stick (sharing `aimStick.x`, just under the stick and clamped to stay on-screen on short landscape phones) so it falls within right-thumb reach -- this is the deliberate response to feedback that USE needed to be closer to the aim stick; the action row holds the WPN, BATT, and LIGHT buttons (LIGHT is the touch equivalent of the desktop F-key flashlight toggle) spaced around the horizontal center and sitting 72px above the bottom edge; the fullscreen button sits top-center. Because touch devices run under `Scale.RESIZE` (canvas == device viewport), `GameScene` calls this on create and on every `resize` event so the UI re-lays-out on orientation flips, URL-bar hide/show, and fullscreen toggles. The portrait rotate prompt is implemented CSS-only in `@/index.html` via a `(orientation: portrait) and (pointer: coarse)` media query, not in this module. The module has no Phaser dependency and no seeded PRNG
- **Camera layers** (`@/src/systems/camera.js`): pure classifier that decides which display objects belong to the zoomed gameplay camera and which belong to the fixed UI camera. The main gameplay camera is zoomed in (`CAMERA_ZOOM = 2.0`) so the player, enemies, furniture, weapons, and pickups all read bigger and "closer" -- a single proportional change that delivers the binding spec's repeated demand for a claustrophobic, "everything is closer" feel in the small `720x600` liminal rooms (it preserves the prior hitbox/display tuning rather than re-tuning each sprite). Because Phaser's `setZoom()` also scales `scrollFactor(0)` objects, a dedicated un-zoomed UI camera renders the HUD at 1:1 -- this module classifies the split. `isHudObject(obj)` is true ONLY when the object is pinned on BOTH axes (`scrollFactorX === 0 && scrollFactorY === 0`); `partitionSceneObjects(objects)` splits a list into `{ hud, world }` by that predicate. The split is along scrollFactor so the lone one-axis-pinned WORLD label (`interactText`, the "Press E" prompt) is correctly classified as world and stays on the zoomed camera. This module is pure (no Phaser, no state) and is consumed by `@/src/scenes/GameScene.js` `setupCameraLayers()` to seed the two-camera split at create time (see the scenes docs for the dynamic-routing of pooled/spawned objects). Tests live in `@/src/systems/__tests__/camera.test.js`
- **Furniture** (`@/src/systems/furniture.js`): `generateRoomFurniture(..., profile = DEFAULT_PROFILE)` uses a seeded PRNG to fill a room with semantic furniture CLUSTERS rather than uniform scatter -- recognizable arrangements (e.g., an office pod of desks, a forest grove of trees, a ship's console bank) anchored at random positions, then topped up with singleton pieces. The cluster templates and singleton list come from the `profile` ( `{ clusters, singletons }` ), which `@/src/scenes/GameScene.js` supplies from the room's vibe (`@/src/systems/roomVibes.js`) so each vibe leaves behind its own kind of furniture. When `profile` is omitted, `DEFAULT_PROFILE` (the generic cluster templates + singleton list defined in this module) is used -- this keeps the starting room (which is furnished separately) and the unit tests unaffected. The piece target scales with room area (with a floor minimum), so rooms read as densely furnished-but-abandoned: this is the deliberate fix for the prior uniform scatter, which made rooms feel empty and incoherent rather than liminal. The optional trailing `obstacles = []` parameter (an array of `{x,y,width,height}` rects) is the cross-system clearance hook: the `fits()` rejection predicate rejects any candidate that overlaps the player spawn zone, an already-placed piece, OR an obstacle (with a small `OBSTACLE_CLEARANCE`). `@/src/scenes/GameScene.js` passes the room's maze wall rects plus its door entry zones (`doorEntryZones()` from `@/src/systems/maze.js`) here, so furniture no longer spawns inside an internal wall or in a doorway -- the REJECT-and-resample half of the unified placement-clearance fix for the spec's "hiding places/stairs/doors spawn inside walls" bug. Determinism is preserved because the obstacle test runs AFTER the PRNG draws each candidate (no change to PRNG consumption), so calling with empty/omitted `obstacles` is byte-identical to before. `createFurnitureSegments()` produces four `{x1,y1,x2,y2}` segments per furniture item in the same format as wall segments. Two pure helpers map the AI art onto a tighter footprint: `opaqueBounds(rgbaData, width, height, threshold)` scans an RGBA pixel buffer and returns the bounding box of the non-transparent pixels (null if fully transparent), and `visibleFurnitureRect(item, frac)` maps a NORMALIZED opaque-art box (fractions of the texture, 0..1) onto the piece's world rect. The AI furniture PNGs carry large transparent padding (visible art fills only part of the texture), so `@/src/scenes/GameScene.js` scans each texture's opaque box once (cached) and sizes BOTH the collision/bullet zone AND the light occluder to this VISIBLE sub-rect -- the sprite is still drawn stretched to the full logical rect, but the hitbox and occluder now line up with the visible art instead of the empty padding (the fix for the spec's "furniture hitboxes do not match visually" bug). Procedural-fallback furniture (no PNG) fills its whole rect, so its visible rect is the full rect. Each furniture type defines a `canHide` property and a `blocksLight` property. These two properties are independent: some furniture is hideable but does not block light (e.g., table, bed -- low-profile, the player hides underneath), some blocks light but is not hideable (e.g., shelf, bookcase -- tall but not enclosed), and some is both hideable and blocks light (e.g., armoire, closet -- tall enclosed spaces the player hides inside). The `blocksLight` property controls whether `GameScene` stores the furniture's VISIBLE rect in its `furnitureOccluderRects` store for raycasting -- only light-blocking furniture participates in flashlight occlusion and enemy LOS. Light-blocking furniture now occludes EXACTLY like a maze wall: it is kept as a SOLID rect (no longer expanded to a four-segment outline pushed into `wallSegments`), and each consumer expands it differently per frame -- the flashlight uses far faces only (`wallRectOccluders(rect, origin)`, back-face culling, so the cone reaches the near face and the piece reads as illuminated with visible thickness instead of pitch-black under the darkness overlay), while enemy LOS uses the full outline (`wallRectSegments(rect)`, blocking sight from both sides). This is the fix for the spec's "furniture is pitch black under the flashlight" / "infinitely-thin tall black walls" bugs. The exported `blocksBullets(type)` helper returns `!!(def && def.blocksLight)` (unknown types -> false), so the same tall/solid `blocksLight` set is now the ONE unified "tall/solid cover" property driving three systems: flashlight light occlusion, enemy/player line of sight, AND bullet blocking. Bullet-blocking is deliberately kept IDENTICAL to light-blocking -- per the binding spec feedback ("gunfire should go over tables and desks and low furniture, should be blocked by armoires and other furniture"), a player never shoots through something opaque they cannot see through, so low furniture (table, desk, bed, vent, couch, bush, pod) lets bullets fly over while tall cover (armoire, closet, bookcase, ...) stops them. `blocksBullets` is consumed by `@/src/scenes/GameScene.js`, which tags each furniture zone with a `blocksBullets` data flag and uses it as a bullet-vs-furniture process callback. The `canHide` property is consumed by `@/src/systems/hiding.js` to determine which furniture items the player can hide inside. `FURNITURE_TYPES` is a static object spanning generic backrooms pieces (table, shelf, bookcase, couch, counter, ...) plus vibe-specific pieces (forest: tree/rock/bush; derelict ship: console/pod), each with `canHide` and `blocksLight` flags (e.g., tree/rock/console block light but are not hideable; bush/pod are hideable but do not block light). Type selection comes from whichever cluster templates + singleton list the active profile supplies (the per-vibe profiles in `@/src/systems/roomVibes.js`, or `DEFAULT_PROFILE`), so adding a new type to `FURNITURE_TYPES` does NOT automatically make it spawn -- it must be referenced by a profile's clusters or singletons (and needs a sprite in `@/src/systems/sprites.js` / an art manifest entry in `@/src/systems/art.js` to render as art). Room 0 (the starting room) does not use `generateRoomFurniture()` -- it uses `getLocationLayout()` from `@/src/systems/locations.js` instead, which produces location-specific furniture in the same `{ type, x, y, width, height, color, canHide }` format
- **Room interior variety / maze** (`@/src/systems/maze.js`): generates internal walls and obstacles inside rooms to create varied layouts. `getRoomType(seed)` uses a seeded PRNG (seed offset +70000) to classify each room into one of six types: open (deliberately rare, ~5% -- nearly every room has internal structure), maze (shallow recursive-division walls), columns (staggered column grid), corridor (the most common, deep recursive-division creating winding passages), storage (rejection-sampled rectangular crates), cubicles (grid-based U-shaped partition templates). `generateMazeWalls(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, doors, keepOut = [])` returns an array of solid wall RECTS in `{x, y, width, height}` format. Each rect is simultaneously the drawn footprint, the physics body, and the occluder source. Two exports expand a rect to occluder segments: `wallRectSegments(rect)` returns all four outline segments in the universal `{x1,y1,x2,y2}` format (used by enemy LOS -- walls block sight from both sides), while `wallRectOccluders(rect, origin)` returns only the FAR (back-facing) edges relative to a light `origin` outside the rect (used by the flashlight -- back-face culling). The far-face-only set is the fix for the spec's "walls have no thickness" bug: an interior wall's near faces are skipped so the flashlight cone reaches across the wall band and lights its near surface (the wall reads as a solid band of visible thickness instead of an infinitely-thin black line), while the far faces still block light so nothing leaks to the floor beyond -- enemy LOS keeps the full outline so its both-sides blocking is byte-identical to before. This rect contract is also the fix for the earlier "flashlight leaks through wall edges" bug: the previous contract returned zero-width centerline segments while the scene drew walls as thick bands, so subdivision-wall occluders ended at the inner face of the room's perimeter band (leaving an occluder-free slot under visually solid wall at every maze-to-perimeter junction that rays passed through diagonally, and tunneled lengthwise inside bands near their ends), and box/cubicle occluders met at zero-width corners that the visibility algorithm's deliberate corner-overshoot rays wrapped through inside drawn corner blocks (the leak was purely geometric -- the ray math in `visibility.js` was verified correct and needed no change). Subdivision walls (maze and corridor types, via the shared `subdivideGeneric()` -- maze subdivides shallowly with wider passages, corridor deeply with narrower ones) become wallThickness-wide bands centered on their split lines; a band end that touches the maze-area edge is extended through the perimeter band to the true room edge (sealing the junction slot), and stub walls shorter than wallThickness are dropped. Cubicle rooms use a grid-template approach where each cell is either skipped or rendered as a U-shaped partition with one randomly chosen open side; partition bands are centered on the cubicle's edge lines so perpendicular partitions of the same cubicle OVERLAP at corners (no see-through corner gap). Storage crates and columns return their exact footprint rects. All room types carve walls away from a combined keep-out list -- the per-door entry zones PLUS the optional `keepOut` rects the caller supplies -- via the internal `rectOverlapsZone()` (a generalization of the former door-only `rectOverlapsDoorZone()`) applied to the final (extended) rect, ensuring doors remain accessible and any caller-supplied zone stays wall-free. The exported `doorEntryZones(room, wallThickness)` returns the per-door entry zones (each reaching ~80px into the room) so OTHER placement systems can keep the same area clear -- `@/src/scenes/GameScene.js` passes these into `generateRoomFurniture()`. This CARVE strategy is used by `GameScene` for the fixed-position centered stair: it passes a `stairKeepOut()` rect (from `@/src/systems/stairs.js`) as `keepOut` so a room's stair is never embedded in an internal wall (the maze-side half of the unified placement-clearance fix for the spec's "stairs spawn inside walls" bug; the furniture side rejects-and-resamples). This mirrors the outline-equals-visual pattern furniture already used (`createFurnitureSegments()` in `@/src/systems/furniture.js`). The rect contract only changes the output representation -- `getRoomType()`, `generateColumns()`, and all seed offsets consume the same PRNG sequence, so room layouts and world generation determinism are unaffected. All sizing/gap constants were rescaled for the 720x600 liminal rooms: passage gaps are roughly 64-72px (about 1.5x the player's rendered width), so interiors feel like narrow corridors rather than open halls. Room 0 (the starting room) is never given internal walls -- `GameScene` skips maze generation for `room.id === 0` (no `generateMazeWalls()` call, so `renderRoomMaze()` receives no rects)
- **Room vibes** (`@/src/systems/roomVibes.js`): pure-data module giving each room a coherent identity -- a palette PLUS the kind of furniture left behind in it. `ROOM_VIBES` is keyed by vibe id (furnitureStore, library, office, spaceship, forest, warehouse); each vibe is `{ id, name, floorColor, wallColor, furniture: { clusters, singletons } }`. Every palette stays desaturated and dim so the world still reads as liminal, but each vibe is its OWN dim (warm tan, grey-green, blue-grey, steel-cyan, mossy green, brown) -- so adjacent rooms feel like different places rather than the same room twice. This intentionally reverses the prior monotone "single sickly-yellow family" approach (`roomThemes.js`, now deleted): per the binding spec feedback, rooms had felt "too samey"/"totally uncoherent", and each room should now have a coherent-but-different vibe. `getRoomVibe(seed)` picks a vibe deterministically using `VIBE_SEED_OFFSET = 130000`, an axis ORTHOGONAL to the structural layout from `getRoomType()` in `@/src/systems/maze.js` (which uses +70000) -- so a library can be a maze and a forest can be a column hall. The `furniture` profile is passed straight into `generateRoomFurniture()` (see Furniture) so forest rooms place groves/thickets, libraries place reading corners / book stacks, ship bays place console banks / cryo pods, etc. This module is consumed by `@/src/scenes/GameScene.js`: `getRoomVibe(room.seed)` is stored in the `this.roomThemes` Map (name unchanged) and feeds `drawRoom()`, `drawRoomDoorways()`, `renderRoomMaze()` (colors), and `createRoomFurniture()` (the furniture profile). Room 0 (the starting room) does not use this module -- it uses `@/src/systems/locations.js` colors/layout instead. This module has no Phaser dependency, no seeded PRNG beyond `mulberry32` for the deterministic pick, and no mutable state -- it is a pure lookup. Tests live in `@/src/systems/__tests__/roomVibes.test.js`
- **Starting room** (`@/src/systems/startroom.js`): contains legacy store layout data and utility functions for room 0 visuals, plus the pure rift-crossing state machine. `generateStoreLayout()` and `getExitPosition()` are still exported but are no longer called by `GameScene` -- room 0's furniture layout and exit position are now driven by `@/src/systems/locations.js` (via `getLocationLayout()` and `getLocationExitPosition()`). `STORE_FLOOR_COLOR` and `STORE_WALL_COLOR` are still exported but are no longer imported by `GameScene`. The functions `GameScene` actively imports are `generateCrackPoints(startX, startY, length, segments, vertical)`, which produces a jagged polyline (alternating +/-3px offsets) for the rift visual at room 0's east opening (the same regardless of the active starting location), and `nextRiftCrossing(playerX, playerY, room, riftRect, riftCrossed)`, the one-frame rift-crossing latch. `isThroughRift(playerX, playerY, edgeX, riftRect)` -- true ONLY when the player is east of the rift edge AND within the rift opening's vertical Y band -- is now an INTERNAL helper used only by `nextRiftCrossing` (no longer imported by `GameScene`). `nextRiftCrossing` returns `{ riftCrossed, triggered }`: the next one-shot latch state plus whether the dimensional transition (white flash + camera shake + `stair_transition` sound) should play this frame. It **fires once** when the player passes THROUGH the rift opening into the backrooms (delegating the vertical-band check to `isThroughRift`), and **re-arms** the one-shot ONLY when the player is genuinely back inside the entrance room (the store) -- gated on full containment in the store's vertical extent (`room.y .. room.y + room.height`) AND the `edge - 40` x-hysteresis margin. This is two DIFFERENT Y regions on purpose: the rift BAND (a slice of the east wall) for the trigger, the whole store RECT for the re-arm. The store-rect containment is the fix for the spec's "rift animation plays on an unrelated door crossing on the same horizontal line" bug: the previous re-arm condition was the single x-axis test `playerX < edge - 40` alone, which a column-0 room directly north/south of the store (reachable by looping back through the backrooms, `DOOR_PROB = 0.62`) also satisfies because it shares the store's east-edge x -- stepping into one wrongly re-armed the latch and replayed the transition the next time the player re-entered the rift's horizontal band. The `edge - 40` margin is kept as hysteresis against the `edge + 4` trigger so jitter at the rift edge cannot double-fire. `riftCrossed` stays a private latch read only inside `GameScene.updateRiftCrossing()` (see the scenes docs). This mirrors the established split: pure decision logic lives in `@/src/systems/*.js` and is unit-tested (cf. `isThroughRift`, `getStairLandingPosition`) while the scene wiring stays thin and is verified by the playtest harness
- **Locations** (`@/src/systems/locations.js`): defines starting location themes for room 0 and (in a now-unused helper) spawns alternate exit portals. The `LOCATIONS` array contains location definitions (e.g., furniture store, office building, parking garage, hotel lobby, laundromat), each with an `id`, `name`, `floorColor`, `wallColor`, and `furniture` array (using relative positioning via `relX`/`relY`). `getLocation(id)` looks up a location by ID. `getLocationLayout(locationId, roomX, roomY, roomWidth, roomHeight, wallThickness)` converts relative furniture positions to absolute pixel coordinates within the room's inner bounds, skipping items that overlap the center spawn zone (60x60px) or exceed room bounds -- this is the same algorithm as the legacy `generateStoreLayout()` in `startroom.js`, now parameterized by location. `getLocationExitPosition(locationId, room, wallThickness)` returns the exit zone position (west-center of the room). `generateAlternateExit(rooms, stairs, seed, unlockedLocationIds)` uses a seeded PRNG (seed offset +100000) to probabilistically (20% chance) place an alternate exit portal in a non-stair room on a deeper floor (floor > 0). It is retained and tested but **no longer called by `GameScene`** -- the unbounded-world rewrite removed mid-run alternate exits, so the only world exit is the floor-0 entrance zone. The default starting location is `'store'`; `unlockedLocations` (which the shop cycles through) is unchanged but no longer grows via in-game portals
- **Pathfinding** (`@/src/systems/pathfinding.js`): pure functions providing BFS hop-distance queries over the room connectivity graph. `buildRoomGraph(rooms, doorStates)` constructs an adjacency list from `room.doors`, excluding connections where a closable door is closed -- this is the ONLY graph builder; closed doors are opaque to it for every enemy type (the former second "full" graph that let crawlers route through closed doors was removed when enemy pursuit became strictly sight-based). `bfsFromRoom(graph, startRoomId)` runs breadth-first search and returns a `cameFrom` Map. `getDoorwayCenter(room, door)` computes the pixel center of a door opening based on its wall, offset, and width. `getRoomDistance(graph, fromId, toId)` returns the BFS hop count between two rooms (or -1 if unreachable) -- the main consumer-facing query, used by `GameScene` for the idle-wander distance filter and by `@/src/systems/gunfireNoise.js` for gunfire alert radii. The graph is never used to route a chasing enemy toward the player: enemies pursue only what they can see. Constants: `DOORWAY_SEEK_CHANCE` (0.15, probability an idle enemy will seek a doorway during a wander tick), `ROOM_TRANSITION_COOLDOWN` (10000ms, cooldown after an enemy enters a new room before it can seek another), `MAX_ROOM_DISTANCE` (2, maximum BFS distance from spawn room an idle enemy will wander), `MAX_ROOM_ENEMIES` (4, cap on enemies per room to prevent crowding). The module does not use a seeded PRNG and does not produce wall segments -- it is a read-only graph traversal system that operates on room geometry and door states produced by other systems
- **Enemy spawning and AI** (`@/src/systems/enemy.js`): `generateRoomEnemies()` places enemies per room using seeded PRNG (room 0 is always empty -- it is the player spawn room). It accepts an optional `distance` parameter (default 0), the spawn room's distance from the entrance (`room.distance`), and a trailing optional `keepOut = []` parameter -- an array of `{x,y,width,height}` rects a spawn must NOT land in (concatenated with the furniture blockers). `@/src/scenes/GameScene.js` passes the room's stair footprint and stair-landing keep-out rects (the same rects the maze carves around) when a room has a stair, so no enemy spawns on the stair or on the spot where the player materializes after a transition -- part of the fix for the spec's "stairs ... vulnerable to attack immediately after spawning" bug. Count: rooms within `SAFE_DISTANCE` of the entrance roll 0-1 enemies (so some entrance-adjacent rooms are empty -- a permanent safe-ish band), farther rooms roll 1-2 plus `getEnemyCount(distance)` extras from `@/src/systems/scaling.js`, capped at 5 total per room; the count still consumes exactly one PRNG draw, so the per-room seeded stream shape is unchanged from the old `extraCount`/`runCount` signature. `distance` also controls enemy type distribution via `getEnemyType()`: below 4 = 100% basic, 4+ = 80/20 basic/crawler, 8+ = 50/25/25 basic/crawler/spitter. Each spawned enemy carries a `type` field. Three enemy types exist with different speed constants: basic (chase 80, wander 30), crawler (chase 112, wander 50), and spitter (chase 60, wander 20). `hasLineOfSight()` casts a single ray from enemy to player against the wall segment array to determine visibility. `updateEnemyAI()` is a pure state machine with four states: idle/chase/search (used by all types) and attack (spitter-only). Pursuit follows the genre-standard **last-known-position pattern**: on every frame the enemy CAN see the player (LOS holds and the player is not hidden), `lastKnownX/Y` is stamped from the player's position -- uniformly, before the state switch -- and this stored position is the ONLY player-location memory the AI has. On LOS loss (wall, closed door, or the player hiding), **chase** and **attack** transition to **search**, which walks toward the STORED last-seen position at wander speed and gives up to idle on arrival (within 10px) or after a 3-second timeout. A chasing enemy never consults the live player position without sight, never routes through doorways toward the player, and crawlers no longer open closed doors -- this enforces the binding spec rule that enemies must not follow the player once line of sight is lost. The 7th optional parameter `navContext` (default null) carries only idle wandering data: in the **idle** state, when the wander timer fires and `navContext.seekDoorway` is true with cooldown expired, the enemy sets `targetDoorway` and moves toward it at wander speed (this is spawn-relative wandering, not player tracking). The `roomTransitionCooldown` field is decremented each tick when navContext is provided; without navContext, behavior is unchanged (backward compatible). The **attack** state activates when a spitter has LOS and is within `SPITTER_ATTACK_RANGE` (250px) -- the spitter stops, counts down `attackCooldown`, and sets `wantsToFire = true` when the cooldown expires (the pure function signals an intent; GameScene executes the side effect of firing an enemy projectile). `getTypeSpeeds()` maps the enemy type to its speed constants -- crawler and spitter speeds are hardcoded per type, and the `chaseSpeed` parameter applies only to basic enemies, for which `GameScene` always passes the constant `ENEMY_SPEED_CHASE` (movement speed deliberately does not scale with difficulty; see `@/src/systems/scaling.js`)
- **Danger escalation** (`@/src/systems/danger.js`): pure functions implementing a time-based enemy wave spawning system that creates escalating pressure as the player spends time outside safe rooms. `createDangerState()` returns `{ unsafeTime: 0, waveCount: 0 }`. `updateDangerTime(state, delta, isInSafeRoom)` accumulates `unsafeTime` when the player is not in a safe room (room 0 or any lit room); returns the state unchanged when in a safe room -- the timer pauses but does not reset. `shouldSpawnWave(state)` returns true when accumulated unsafe time crosses the next wave threshold (`(waveCount + 1) * DANGER_WAVE_INTERVAL`), unless `waveCount` has reached `MAX_DANGER_WAVES`. `advanceWave(state)` increments `waveCount` to prevent the same wave from re-triggering. `getWaveComposition(waveCount)` determines both the count and types of enemies per wave: count scales as `min(1 + floor(waveCount / 3), 4)`, and types shift from all-basic (waves 0-1) to basic/crawler mix (waves 2-3) to crawler/spitter mix (waves 4-5) to a mixed rotation of all three types (waves 6+). `getSpawnRoom(rooms, playerRoomId, litRoomIds, currentFloor)` selects the first candidate room that is not room 0, not the player's current room, not lit, and on the same floor as the player; returns null if no valid room exists. Constants: `DANGER_WAVE_INTERVAL = 30000` (30 seconds), `MAX_DANGER_WAVES = 8`. The danger state is per-run (initialized in `GameScene.init()`, not persisted). This module follows the same pure-function immutable-state pattern as combat, battery, and hiding. It has no Phaser dependency and no seeded PRNG -- spawn positions within the chosen room use `Math.random()` in `GameScene.spawnDangerWave()` rather than deterministic seeding, since dynamically spawned enemies do not need cross-session reproducibility. Wave enemies' HP and contact damage come from the spawn ROOM's `distance` (via `GameScene.getEnemyStats()`), while wave COMPOSITION scales with `waveCount` -- the time-pressure axis is deliberately separate from the distance-difficulty axis in `@/src/systems/scaling.js`
- **Run statistics** (`@/src/systems/runStats.js`): pure functions tracking per-run gameplay statistics for the end-of-run summary screen. Follows the same immutable-state pattern as combat, battery, and danger. `createRunStats()` returns `{ enemiesKilled: 0, timeElapsed: 0, maxFloor: 0 }`. `recordKill(stats)` increments the kill counter. `updateTime(stats, delta)` accumulates elapsed time from the frame delta. `updateMaxFloor(stats, floor)` tracks the highest floor reached (uses `Math.max` so returning to a lower floor does not decrease the value). `formatTime(ms)` converts milliseconds to a `"M:SS"` display string. `getSummaryData(stats, survived, treasureValue, roomsExplored, hp, maxHp)` produces a structured summary object `{ title, titleColor, accentColor, lines }` consumed by `@/src/scenes/RunSummaryScene.js` -- the title is "RETURNED" (gold) on survival or "YOU DIED" (red) on death ("escaped" was deliberately dropped: the backrooms are explored of one's own volition), and `lines` is an array of `{ label, value }` pairs covering time, rooms explored, enemies killed, deepest floor, treasure, and (on survival only) HP remaining. The summary intentionally shows only a raw rooms-explored count, never an "X / total" ratio -- the world is unbounded, so there is no total (the old `totalRooms` parameter was removed). The runStats state is initialized in `GameScene.init()` alongside other per-run state objects and updated in three places: `update()` (time accumulation each frame), `onBulletHitEnemy()` (kill recording when `isEnemyDead()` is true), and stair teleportation (max floor tracking when `this.currentFloor` changes). The state object is passed to RunSummaryScene at end-of-run and is not persisted across runs. This module has no Phaser dependency and no seeded PRNG
- **Enemy difficulty scaling** (`@/src/systems/scaling.js`): pure functions that compute distance-scaled enemy stats from base values and the spawn room's `distance` from the entrance (the `room.distance` field: door-hop Manhattan distance plus a fixed depth per floor descended, stamped in `GameScene.genRoom()`). This is the SAME axis loot quality already scales on (`@/src/systems/items.js`, `@/src/systems/lootDrops.js`), so risk and reward rise together -- a Diablo-style "area level" coupling. This deliberately replaced day-number (`runCount`) scaling, per the binding spec feedback that the game must not get harder on later days; difficulty must be a function of distance from the start. `SAFE_DISTANCE = 2` defines an entrance band that stays at base difficulty permanently. `getEnemyHP(baseHP, distance)` ramps HP linearly past the safe band (+10% per distance step), capped at 2.5x base. `getEnemyCount(distance)` returns extra enemies per room (0 near the entrance, then +1 per 4 distance, capped at +3). `getEnemyDamage(baseDamage, distance)` adds flat +5 steps past the safe band, capped at +20. There is deliberately NO speed-scaling function -- the former `getEnemyChaseSpeed` was deleted because speed scaling destroys dodge readability, and the scaled crawler/spitter speeds were silently dead code anyway (`getTypeSpeeds()` in `@/src/systems/enemy.js` hardcodes those types' constants); enemy movement speeds are pure per-type constants. All functions are type-agnostic -- they accept base values and return scaled values; `GameScene.getEnemyStats(type, distance)` calls them per spawn with each type's base constants (e.g., `ENEMY_MAX_HP`, `CRAWLER_MAX_HP`, `SPITTER_MAX_HP`) and freezes the results on the per-enemy state object. No global per-run difficulty state exists anywhere. The module has no Phaser dependency and no seeded PRNG
- **Hiding** (`@/src/systems/hiding.js`): manages player hiding state for furniture-based stealth. Follows the same pure-function pattern as `doors.js` and `lightswitch.js`. `createHidingState()` returns `{ isHiding: false, furnitureRef: null }`. `enterHiding(state, furnitureRef)` returns a new state with `isHiding: true` and the furniture reference (which carries `x, y, width, height` used by GameScene to set physics bounds). `exitHiding()` returns the default un-hidden state. `findNearestHideable(furnitureByRoom, px, py, maxRange)` iterates over all rooms' furniture, filtering for items with `canHide: true` (any furniture type in `@/src/systems/furniture.js` that has this flag set), and returns the nearest one within `HIDE_INTERACT_RANGE` (80px) or null. Exports `HIDING_SPEED_MULTIPLIER = 0.5` consumed by `GameScene` to halve player movement speed while hidden. The hiding system does not produce wall segments or consume the seeded PRNG -- it reuses furniture objects already placed by `furniture.js`
- **Exploration** (`@/src/systems/exploration.js`): tracks room visitation and computes minimap rendering data. Follows the same pure-function pattern as other system modules. `createExplorationState()` returns `{ visitedRoomIds: Set, currentRoomId: null }`. `getCurrentRoom(playerX, playerY, rooms)` iterates rooms and uses `isPointInRoom()` from `@/src/systems/lightswitch.js` to find which room contains the player. `updateExploration(state, playerX, playerY, rooms)` returns unchanged state if the player is in the same already-visited room (performance optimization), otherwise returns a new state with the room added to the visited set. `getMinimapData(state, rooms, levelBounds, playerPos, exitPos, screenWidth, floorFilter)` computes all rendering coordinates for the minimap by scaling a whole-floor bounding box into a fixed-size minimap box (160x130 pixels) -- this whole-floor variant is retained and tested but is **not** used by `GameScene` anymore (an unbounded world has no finite floor bounds). `getWindowedMinimapData(state, rooms, playerPos, exitPos, screenWidth, floorFilter, floorYOffset, windowCells)` is the live variant: it renders a **player-centered grid window** of `windowCells` (default `MINIMAP_WINDOW_CELLS = 4`) cells in every direction around the player's current cell on the current floor, mapping world coordinates relative to the window origin at a fixed scale and clamping dots to the box. It only includes visited rooms within the window and on the current floor (using `room.gridX/gridY` and the floor's Y-band offset). Both variants return `{ bgRect, roomRects, playerDot, exitDot }` with an `isCurrent` flag per room. The module exports dimension and color constants consumed by `GameScene.drawMinimap()`. Note: the exploration system always tracks visited rooms even when the minimap upgrade has not been purchased -- the rendering is gated by `this.hasMinimap` in `GameScene`, but the tracking data remains available for future features. The exploration system does not use a seeded PRNG and does not produce wall segments -- it is a read-only tracking system that reacts to player position
- **Persistence** (`@/src/systems/persistence.js`): manages localStorage save/load for cross-session game state AND mints/resolves the per-save world seed. `saveGame(shopState, runCount, collectedLore, unlockedLocations, activeLocation, worldSeed)` serializes a versioned envelope `{ version: 4, shopState, runCount, collectedLore, unlockedLocations, activeLocation, worldSeed }` to localStorage under the key `backrooms_save`. Parameters default to safe values if omitted: `collectedLore` to `[]`, `unlockedLocations` to `['store']`, `activeLocation` to `'store'`, `worldSeed` to `null`. `loadGame()` parses the saved JSON, returns `null` if the data is missing, corrupted, or from a future save version (higher than `SAVE_VERSION`), and defaults missing fields to safe fallback values -- including `worldSeed` to `null`, so older saves that predate the field load fine and get a fresh seed minted on next play (backward-compatible migration). `clearSave()` removes the save key. All localStorage access is wrapped in try-catch so the game degrades gracefully when localStorage is unavailable (e.g., private browsing, storage quota exceeded). The module also exports the **world seed** helpers, the single source of the persisted per-save world identity (consumed by `@/src/scenes/GameScene.js` `init()`): `generateWorldSeed(rng = Math.random)` returns a fresh non-zero integer in `[1, MAX_WORLD_SEED]` (`MAX_WORLD_SEED = 0x7fffffff`, a 31-bit ceiling that keeps the seed positive after the `| 0` truncation in the downstream PRNGs and leaves headroom for the per-floor `seed + floor*1000` derivation without wrapping; 0 is excluded because it is degenerate for mulberry32 and avoids falsy-seed confusion), and `resolveWorldSeed(savedSeed, rng = Math.random)` returns a stored seed when it is a valid positive integer (checked with `Number.isInteger`, NEVER truthiness, so a stored seed is never clobbered and the `seed === 0` falsy trap is avoided) and otherwise mints a fresh one. The module has no Phaser dependency -- it is a system boundary module that bridges in-memory `game.registry` state to browser storage. Tests live in `@/src/systems/__tests__/persistence.test.js`
- **Audio configuration** (`@/src/systems/audio.js`): pure config module defining all sound effect parameters, per-enemy-type sound logic, and behavioral functions for audio timing. `SOUND_CONFIGS` is a flat object mapping sound keys (e.g., `pistol_fire`, `zombie_growl`, `crawler_skitter`, `spitter_gurgle`, `door_open`, `footstep`, `shop_purchase`, `loot_drop`) to synthesis parameters -- each entry specifies a `type` (one of: gunshot, noise_burst, chime, tone, sweep_down, growl, skitter, gurgle, door, door_thud, filtered_noise) plus type-specific fields (frequencies, Q values, decay rates, note arrays, etc.). Each enemy type has a distinct set of sounds: ambient (zombie_growl, crawler_skitter, spitter_gurgle), alert (zombie_alert, crawler_alert, spitter_alert), death (zombie_death, crawler_death, spitter_death), and spitters have a fire sound (spitter_fire). An internal `ENEMY_TYPE_PREFIX` map translates enemy types to sound key prefixes (basic->zombie, crawler->crawler, spitter->spitter), and `AMBIENT_SOUND_SUFFIX` maps types to their ambient sound name (basic->growl, crawler->skitter, spitter->gurgle). `AMBIENT_SOUND_TYPES` defines the three ambient background sounds (distant_bang, drip, whisper) used for atmospheric audio between gameplay events. This module has no Phaser dependency and no browser API dependency -- it is fully testable in Node.js. It does not use a seeded PRNG and audio does not affect level determinism
- **Sprite definitions** (`@/src/systems/sprites.js`): centralized pixel art sprite data for procedurally-generated game entities, PLUS the pure 8-direction facing helpers for the PNG-backed characters. `SPRITE_DEFS` is an object mapping texture keys to `{ data, palette, pixelWidth, pixelHeight }` definitions rendered via Phaser's `textures.generate()` data-array format. `data` is an array of strings where each character indexes into `palette` (a map of characters to hex colors); `.` characters are transparent. `pixelWidth`/`pixelHeight` are scaling multipliers that control the rendered size of each pixel in the data array. Sprite keys follow a naming convention: `furniture_{type}` for furniture, `item_{type}` for collectibles, `pickup_{weaponId}` for weapon pickups, `bullet_{weaponId}` and `enemyBullet` for projectiles, `door_closed` for doors, `switch_on`/`switch_off` for light switches, `lore_note` for lore items, and `{entityType}_{state}_{frame}` for the remaining character animation frames (crawler/spitter only, e.g., `crawler_walk_0`, `spitter_attack_0` -- the old `player_idle_0/1`, `player_walk_0/1`, `enemy_idle_0`, and `enemy_walk_0/1` grids were REMOVED). Crawler/spitter character grids use `pixelWidth: 1` (crawler 24x24, spitter 32x32) -- small radial creatures that read fine when `GameScene` rotates them to their velocity vector (`setRotation`), so they are still procedural and still go through the animation path. Item, weapon pickup, bullet, switch, and lore sprites also use `pixelWidth: 1`. The door sprite uses `pixelWidth: 2` and `pixelHeight: 2`. `ANIM_DEFS` now maps ONLY crawler/spitter animation keys (`crawler_idle`/`crawler_walk`, `spitter_idle`/`spitter_walk`/`spitter_attack`) to `{ frames, frameRate, repeat }` -- the player/enemy `idle`/`walk` anim entries were removed (the directional characters do not use Phaser animations; see **Directional character facing** below). Walk animations use higher frame rates than idle (e.g., crawler walk at 8fps vs idle at 1fps). `repeat: -1` means loop forever; `repeat: 0` means play once. `getSpriteConfig(key)` looks up a sprite definition by key. `getAllSpriteKeys()`/`getAllAnimKeys()` return all defined keys. `getEnemyTextureKey(type)` maps an enemy type to its initial texture key: crawler -> `'crawler_idle_0'`, spitter -> `'spitter_idle_0'`, basic -> `'enemy_down'` (the directional PNG, NOT a procedural grid). `getAnimKeyForState(entityType, aiState)` maps an entity type and AI state to an animation key (e.g., `('spitter', 'attack')` -> `'spitter_attack'`); the `entityType === 'basic'` -> `'enemy'` prefix remap is retained, but in practice this function is now consumed ONLY for crawler/spitter -- basic enemies render via `getDirectionalFrame()` and never go through `getAnimKeyForState()`. This module has no Phaser dependency -- it is pure data + pure functions. For the procedural keys these definitions are the **fallback** layer: `GameScene.generateSpriteTextures()` only calls `this.textures.generate()` for a key when no PNG was already loaded for it (see `@/src/systems/art.js` and the GameScene/ShopScene `preload()`), then calls `this.anims.create()` once per animation key. Crawler/spitter and `furniture_armoire`/`furniture_couch`/`furniture_counter` are deliberately ABSENT from `ART_MANIFEST` (the image model cannot draw them top-down), so their grids are the SOLE source and always render. Every furniture type (including couch and counter, used both by `@/src/systems/locations.js` for room 0 and by the cluster templates in `@/src/systems/furniture.js`) has a procedural definition here, so all furniture renders as a sprite even with no PNG; the Graphics `fillRect` path in `GameScene.createRoomFurniture()` remains only as a last-resort fallback for a type with neither
- **Directional character facing** (`@/src/systems/sprites.js`): the player and basic zombie are PNG-backed 8-direction top-down sprites (GBA-Pokemon style) instead of a single grid rotated to the aim/movement vector -- the old rotated "strictly overhead" blob read poorly. The pure helpers here pick the right facing without any Phaser/rotation. `directionFromAngle(angle)` snaps a radian aim/velocity angle to one of `DIRECTIONS_8` (`right`, `down_right`, `down`, `down_left`, `left`, `up_left`, `up`, `up_right`) by rounding to the nearest `PI/4` step -- screen space is +x right and +y DOWN, so Phaser's `atan2(dy, dx)` increases clockwise from east. `getDirectionalFrame(prefix, angle)` returns `{ textureKey, flipX, base }`: only FIVE base facings are authored per character (`down`, `down_right`, `right`, `up_right`, `up`); the three left-side facings (`left`, `down_left`, `up_left`) are the corresponding right-side textures mirrored via `flipX: true` (the `MIRROR_FRAME` table), since a flashlight switching hands is invisible from overhead. - **AI-art manifest and helpers** (`@/src/systems/art.js`): pure module that is the single source of truth for the AI-generated pixel art, consumed by BOTH the offline generator (`@/scripts/generate-art.mjs`) and the runtime scenes. `ART_MANIFEST` is an array where each entry produces ONE generated image written to every texture key in its `keys` (`chroma: false` marks opaque entries like the shop background that skip background removal); each entry carries the model `prompt`, target `width`/`height`, and `keys`. Only the flat furniture-top + shop sprites the model can draw orthographically are listed here; everything else is intentionally ABSENT from `ART_MANIFEST`. Most of those absent keys are hand-authored procedurally in `@/src/systems/sprites.js` -- tiny micro-sprites (bullets, small coins), the crawler/spitter creatures, and `furniture_armoire`/`furniture_couch`/`furniture_counter`. The player and basic zombie are ALSO absent from `ART_MANIFEST` but are NOT procedural: they are PNG-backed 8-direction sprites from the separate `@/scripts/gen-character-directional.mjs` pipeline (see the **Directional character facing** bullet), kept out of this manifest because they need per-facing image-to-image consistency rather than the one-shot generation `ART_MANIFEST` entries get. The exclusions are deliberate: the model reverts to front/3-4 views for anything with vertical structure (creatures, wardrobes, couches, a counter's till) despite strict "top surface only" prompts -- a documented canonical-view bias re-prompting cannot fix -- and at sprite sizes the downscale destroys detail. This module has no Phaser dependency and no seeded PRNG. Tests live in `@/src/systems/__tests__/art.test.js`
- **Audio settings** (`@/src/systems/settings.js`): pure function module managing audio volume settings independently from game saves. Settings are persisted to localStorage under a separate key (`backrooms_settings`, defined as `SETTINGS_KEY`) so volume preferences survive game save deletion. `VOLUME_LEVELS` is the ordered cycle array `['OFF', 'LOW', 'MED', 'HIGH', 'FULL']`. `VOLUME_VALUES` maps each level to an exponential perceptual value: OFF=0, LOW=0.05, MED=0.15, HIGH=0.4, FULL=1.0. `createDefaultSettings()` returns `{ masterVolume: 'FULL', sfxVolume: 'FULL', ambientVolume: 'FULL' }` -- defaults preserve pre-settings behavior (everything at full volume). `cycleVolume(currentLevel)` advances to the next level in the cycle (wraps from FULL back to OFF). `getVolumeValue(level)` returns the numeric multiplier for a named level (falls back to 1.0 for unknown levels). `getEffectiveVolume(masterLevel, categoryLevel)` multiplies the master and category numeric values to produce the final multiplier applied to all audio calls -- this is the core function consumed by all scenes. `saveSettings(settings)` and `loadSettings()` are the localStorage boundary, wrapped in try-catch per the project's system boundary convention. `loadSettings()` validates all three fields against `VOLUME_LEVELS` and returns defaults if any field is invalid or localStorage is unavailable, handling corruption gracefully. This module follows the same pure-function pattern as `@/src/systems/persistence.js` and sits at the same system boundary (browser storage). Settings flow through `game.registry` (key `'audioSettings'`) for in-memory cross-scene access, loaded once at boot by `@/src/main.js`. Scenes read settings from the registry and call `getEffectiveVolume()` to compute volume multipliers for every `playSound()`, `playSoundAtDistance()`, and ambient drone gain call. Tests live in `@/src/systems/__tests__/settings.test.js`
- **Audio engine** (`@/src/systems/audioEngine.js`): Web Audio synthesis engine that renders sound buffers from the configs defined in `@/src/systems/audio.js` and manages the ambient drone. Unlike other system modules, this module depends on browser Web Audio APIs (`OfflineAudioContext`, `AudioContext` nodes) and cannot be unit-tested in Node.js. `generateAllSounds(audioContext, cache)` iterates all entries in `SOUND_CONFIGS`, dispatches each to a type-specific renderer function via a `RENDERERS` map, renders the sound via `OfflineAudioContext`, and injects the resulting `AudioBuffer` into Phaser's audio cache so sounds can be played via `this.sound.play(key)`. All rendering happens in parallel via `Promise.all`. The `RENDERERS` map covers all synthesis types including `renderSkitter` (rapid highpass-filtered noise clicks for crawler ambient sounds) and `renderGurgle` (resonant bandpass-filtered noise with LFO modulation for spitter ambient/death sounds). `createAmbientDrone(audioContext, destination)` builds a continuous low-frequency drone using three detuned oscillators (two sawtooth at 120/120.5 Hz for beating, one sine sub-bass at 60 Hz), LFO-modulated gain (0.3 Hz), and a 400 Hz lowpass filter -- returns `{ start(), stop(), setVolume(value) }`. The `setVolume(value)` method multiplies the base gain (0.06) by the provided value, allowing runtime adjustment of the drone's volume without stopping/restarting oscillators -- consumed by scenes when audio settings change mid-play. `playAmbientSound(soundManager, time, volumeMultiplier)` selects an ambient sound type deterministically from the current time (using the same Knuth hash approach as `getAmbientSoundDelay`) and plays it via Phaser's sound manager at `0.5 * volumeMultiplier` volume, skipping playback when the computed volume is zero. `createRiftCrackle(audioContext, destination)` builds the entrance rift's crackling static: a looped white-noise buffer through a bandpass (the hiss) plus a sparse impulse buffer through a highpass (the pops), returning `{ start(), stop(), setVolume(value) }`; unlike the drone's multiplier, `setVolume` takes the absolute gain -- `GameScene` drives it per-frame from `getRiftCrackleVolume()` in `@/src/systems/audio.js` times the ambient volume setting. This module is a system boundary -- it bridges the pure config in `audio.js` to browser audio APIs, similar to how `persistence.js` bridges game state to localStorage
- **Visibility / raycasting** (`@/src/systems/visibility.js`): `getFlashlightPolygon(origin, angle, coneAngle, segments, maxRange)` builds the flashlight polygon by casting rays at the two cone edges, toward every in-cone segment endpoint (plus tiny offset angles to peek around corners), AND at fixed arc samples roughly every 10 degrees across the cone. Any ray that misses all segments -- or whose hit lies beyond `maxRange` -- contributes a point capped at `maxRange` along the ray. The arc samples plus range capping make the polygon structurally collapse-proof: it always has enough points even when aimed where no segments exist (the old endpoint-only version collapsed to fewer than 3 points and the cone vanished -- the reported "flashlight disappears into itself while rotating" bug). `FLASHLIGHT_RANGE` (300) is the gameplay sight cap consumed by `GameScene`, enforcing claustrophobic short sight lines. `filterSegmentsNear(origin, range, segments)` is a bounding-box prefilter that keeps per-frame raycast cost proportional to nearby segments -- essential because `wallSegments` accumulates forever in the unbounded world; `GameScene` uses it both for the flashlight (at `FLASHLIGHT_RANGE`) and for enemy line-of-sight (at the enemy `DETECTION_RANGE`). `filterRectsNear(origin, range, rects)` is the same AABB-vs-range prefilter for the `{x,y,width,height}` rect stores -- both interior maze walls (`mazeWallRects`) and light-blocking furniture (`furnitureOccluderRects`) in `GameScene` -- applied before each consumer expands the surviving rects to occluder segments (LOS to full outlines, flashlight to far faces). With range capping, dropping far occluders is exact, not approximate. `getVisibilityPolygon()` is the full 360-degree endpoint-cast version. Also exports `raySegmentIntersection()` which is reused by `enemy.js` for LOS checks. The corner-peek rays deliberately overshoot segment endpoints by a tiny angle, which means occluders for solid obstacles must be closed outlines around the drawn geometry: the spec's "flashlight leaks through wall edges" bug was geometric (zero-width maze occluder segments mismatching the drawn wall bands -- fixed in `@/src/systems/maze.js` by the wall-rect contract), not a numerical bug in this module's ray math, which was verified correct; the captured-leak regression fixtures (lengthwise band tunneling, maze-to-perimeter junction, corner wrap between two bands) live in the thick-wall-rects block of `@/src/systems/__tests__/visibility.test.js`. A load-bearing tolerance of the brute-force raycaster: overlapping/coincident collinear segments are benign input (adjacent rooms double-push shared perimeter walls, and extended maze wall outlines overlap the room boundary line) -- third-party visibility-polygon libraries treat such input as illegal, so any replacement of the ray-casting core must preserve this tolerance
- **Combat** (`@/src/systems/combat.js`): manages player and enemy health as immutable state. `createCombatState(maxHp?)` produces a state object with `hp`, `maxHp`, `damageCooldown`, and `isDead`. The optional `maxHp` parameter defaults to `PLAYER_MAX_HP` for backward compatibility but is overridden by the health upgrade value when called from GameScene. `applyDamage(state, amount)` returns a new state with reduced HP and sets a damage cooldown (invulnerability window) -- damage is ignored while the cooldown is active or the player is dead. `applyHeal(state, amount)` returns a new state with increased HP clamped to `maxHp` -- healing is ignored when the player is dead. This is the inverse of `applyDamage` and follows the same immutable-state pattern but does not set a cooldown. `MEDKIT_HEAL_AMOUNT` (25) is the constant consumed by `GameScene.onItemPickup()` for medkit healing. GameScene passes per-enemy `contactDamage` (read from the enemy state object, which stores a type-specific value scaled by the spawn room's distance and frozen at spawn) as the damage amount. `updateCombat(state, delta)` ticks the cooldown down each frame. `grantInvulnerability(state, ms)` extends the damage cooldown to at least `ms` (never shortening an already-longer cooldown, no-op when dead) -- `GameScene.onStairEnter()` calls it the instant the player lands after a stair transition (`STAIR_SPAWN_PROTECTION_MS`) so a nearby enemy cannot hit them before they can react, the post-spawn half of the fix for the spec's "stairs ... vulnerable to attack immediately after spawning" bug (the keep-out half lives in `@/src/systems/enemy.js`). `applyEnemyDamage(health, damage)` and `isEnemyDead(health)` are simpler stateless helpers for enemy health (enemies have no invulnerability). GameScene passes `this.bulletDamage` (from the weaponDamage upgrade) as the damage amount. Base constants are exported per enemy type: `ENEMY_MAX_HP` (50), `CRAWLER_MAX_HP` (30), `SPITTER_MAX_HP` (40) for HP; `ENEMY_CONTACT_DAMAGE` (20), `CRAWLER_CONTACT_DAMAGE` (15), `SPITTER_CONTACT_DAMAGE` (10) for contact damage; and `SPITTER_PROJECTILE_DAMAGE` (15) for ranged damage. These serve as base values that `scaling.js` scales from. Corpse visual constants are also exported: `CORPSE_TINT` (0x666666, gray), `CORPSE_ALPHA` (0.6, faded), `CORPSE_ANGLE` (90, toppled sideways), and `CORPSE_DEPTH` (5, same depth as items -- below living enemies at 60 and below the darkness overlay at 100). These are consumed by `GameScene.onBulletHitEnemy()` to render dead enemies as persistent corpses rather than hiding them. **Two SEPARATE damage-feedback helpers live here -- one for the player SPRITE, one for the SCREEN tint:** `getHurtFlash(cooldownRemaining)` is the SPRITE-level feedback. It maps the remaining damage cooldown to a red blink, returning `{ alpha, tint }` where `tint` is the red multiply-tint colour `HURT_TINT` (`0xff4444`) on flash frames or `null` otherwise, with alpha never below 0.85. `GameScene.drawPlayer()` applies it as a MULTIPLY tint (`setTint`), never a flat fill (`setTintFill`): a multiply tint keeps the sprite's dark outline so the silhouette stays visible against any background -- both dark floors and the bright flashlight cone. This is the centralized hurt-visual policy and the fix for the recurring "player disappears when shot" bug: the old `setTintFill(white)` replaced the sprite with a flat white silhouette that vanished against the lit cone, and an even earlier constant half-alpha dark-red tint vanished against dark floors. `getDamageFlashAlpha(remainingMs, duration?, peak?)` is the SCREEN-level counterpart -- the instant "I got hit" red wash. It returns a tint alpha that linearly fades from `peak` (default `DAMAGE_FLASH_PEAK_ALPHA = 0.45`) down to 0 over `duration` (default `DAMAGE_FLASH_DURATION_MS = 200`), clamped so it is never negative and never exceeds the peak (even if more than `duration` is somehow remaining). The peak is capped well below 1 ON PURPOSE: this is the math behind the fix for the spec's "being shot makes the player fully disappear" bug. The OLD damage feedback was a full-opacity (peak alpha 1.0) red `cameras.main.flash()` overlay that covered the WHOLE main camera and so erased the player (sitting in the dark backrooms) on every hit. `GameScene` now drives a sub-player world-tint Graphics (depth 190, below the player at 200) repainted each frame at this capped, fading alpha (see **Damage feedback is a sub-player world tint, not a camera flash** in the scenes docs) -- so the world reddens for feedback but the player stays visible through it. Both helpers are pure and stateless; the cooldown/timer state they read is owned by `GameScene`

- **Gunfire noise alerts** (`@/src/systems/gunfireNoise.js`): pure function module that determines which enemies should be alerted when the player fires a weapon, creating a tactical cost to shooting. `getAlertedEnemies(enemyStates, roomGraph, sourceRoomId, noiseRange)` iterates over all enemies, filtering for those in idle or search state (via the `ALERTABLE_STATES` Set -- chase and attack enemies are already engaged and not re-alerted), then uses `getRoomDistance()` from `@/src/systems/pathfinding.js` to check if each enemy's `currentRoomId` is within `noiseRange` BFS hops of the source room. This is now the ONLY place enemy AI state is mutated from outside `updateEnemyAI()` -- `GameScene` directly sets the returned enemies to chase. It imports `getRoomDistance` from `@/src/systems/pathfinding.js` and has no Phaser dependency, no seeded PRNG, and no state. Tests live in `@/src/systems/__tests__/gunfireNoise.test.js`
- **Weapons** (`@/src/systems/weapons.js`): manages weapon types, a 2-slot inventory, weapon switching, per-weapon ammo, and effective stat computation. `WEAPON_TYPES` is the array of all weapon definitions (pistol, shotgun, rifle), each specifying `damage`, `fireRate`, `speed`, `range`, `bulletCount`, `spreadAngle`, `color`, `pickupColor`, `maxAmmo`, `startAmmo`, and `noiseRange` (BFS hop distance for gunfire noise alerts -- consumed by `@/src/systems/gunfireNoise.js`). `createWeaponState(options)` uses priority-based slot filling to determine the player's starting loadout. It collects all purchased starting weapons in priority order (rifle > shotgun > pistol, i.e., most expensive first) and fills up to 2 weapon slots. When no starting weapons are purchased, the player starts unarmed (`slots: [null, null], ammo: [0, 0]`). When all three are purchased, only the top two (rifle, shotgun) fill the slots and the pistol is dropped from the starting loadout. Each filled slot's ammo is initialized to that weapon's `startAmmo`. The `ammo` array is parallel to `slots` -- `ammo[i]` tracks the ammo count for the weapon in `slots[i]`. `switchWeapon(state)` returns a new state with the active slot toggled (no-op if the other slot is empty). `pickupWeapon(state, weaponType)` fills the first empty slot if one exists, otherwise swaps slot 1's weapon with the picked-up weapon and returns the dropped weapon (with its ammo count attached) for re-placement in the world. Picked-up weapons start with their `startAmmo`. `getActiveWeapon(state)` returns the weapon in the active slot or `null` if the slot is empty -- all callers must handle null. `hasAmmo(state)` returns whether the active weapon has ammo remaining. `consumeAmmo(state)` decrements active slot ammo by 1 (no-op if already 0). `addAmmo(state, amount)` adds ammo to the active slot clamped to `maxAmmo`. `AMMO_PER_PICKUP` maps weapon IDs to how much ammo the `ammo` item type restores (pistol: 10, shotgun: 4, rifle: 3). `getEffectiveStats(weapon, bonuses)` applies additive upgrade bonuses (damage, fireRate, range) to the weapon's base stats, clamping fireRate to a minimum of 50ms. `generateLevelWeapons(rooms, seed, furnitureByRoom)` is retained and tested but no longer used by `GameScene` (it needs the full floor room list up front, which the lazy world does not have). `generateRoomWeapon(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, furnitureItems, roomId, obstacles = [])` places a single weapon pickup in a room using seeded PRNG (seed offset +60000), skipping room 0 -- this is the live path: `GameScene` calls it as each non-entrance room activates and places at most one weapon per floor (first valid placement in activation order, tracked via a per-floor guard set). All weapon types including pistol can appear as floor drops. Uses the same retry-on-overlap placement pattern as other spawning systems, and the same optional trailing `obstacles = []` parameter -- when non-empty, candidates are rejected against `furnitureItems.concat(obstacles)` so a weapon never spawns inside a solid maze wall where it could never be picked up (`GameScene.activateRoomWeapon()` threads the room's `mazeRects`; see **Intra-room placement clearance** in this doc). The test-only `generateLevelWeapons` path keeps the empty default. The empty/omitted default is byte-identical to before. The weapon system follows the same pure-function immutable-state pattern as combat, inventory, and hiding
- **Battery** (`@/src/systems/battery.js`): manages flashlight battery drain and recharge as immutable state following the same pattern as combat. `createBatteryState(maxCharge?)` produces `{ charge, maxCharge, isDepleted }`. The optional `maxCharge` parameter defaults to `BATTERY_MAX` for backward compatibility but is overridden by the battery upgrade value when called from GameScene. `updateBattery(state, delta)` drains charge each frame and clamps at zero, setting `isDepleted` when charge reaches 0. `getBatteryFraction(state)` returns 0-1 charge ratio. `getFlashlightConeAngle(state, baseConeAngle)` scales the flashlight cone from 100% at full charge to 25% at empty using linear interpolation (`0.25 + 0.75 * fraction`), returning 0 when depleted. `rechargeBattery(state, amount)` adds charge back (clamped to max), clearing the `isDepleted` flag if charge rises above 0 -- it is called both when the player uses a battery item (with `BATTERY_RECHARGE_AMOUNT`) AND, with `BATTERY_RECHARGE_PER_MS * delta`, by the "Rechargeable Battery" store-upgrade trickle-recharge while the flashlight is off (see `updateBatteryWithFlashlight()` in `@/src/systems/flashlight.js`). `shouldFlicker(state, time)` returns true during portions of a deterministic sine-wave pattern when charge is below `FLICKER_THRESHOLD` (25%). `getBatteryHint(state, inventory)` returns a contextual HUD hint string once charge drops to half (telling the player to insert a battery if they have one, or that the beam narrows as it drains if not) -- part of the battery-UX pass making the drain mechanic self-explanatory. Note that `GameScene` no longer calls `updateBattery()` directly each frame; drain (and the optional recharge-while-off) is routed through `updateBatteryWithFlashlight()` in `@/src/systems/flashlight.js` so charge only drains while the flashlight is effectively lit. Constants: `BATTERY_MAX=100`, `BATTERY_DRAIN_PER_MS` (drains full charge in 150 seconds), `FLICKER_THRESHOLD=0.25`, `BATTERY_RECHARGE_AMOUNT=30`, and `BATTERY_RECHARGE_PER_MS = BATTERY_DRAIN_PER_MS / 3` -- the recharge-while-off rate is deliberately a third of the drain rate so the upgrade eases battery anxiety without ever letting the player keep the light on indefinitely (regaining a second of light costs ~3 seconds in the dark, preserving the turn-it-off-to-conserve tension)
- **Flashlight toggle** (`@/src/systems/flashlight.js`): pure immutable state `{ on, hiding }` for whether the flashlight is switched on and whether hiding is forcing it dark. `toggleFlashlight()` flips the switch (F key on desktop, LIGHT button on touch -- both via `GameScene.toggleFlashlightAction()`); `setFlashlightHiding(state, hiding)` is set/cleared by `GameScene.onEnterHiding()`/`onExitHiding()`, so hiding auto-darkens the light but the switch state is preserved and restored on exit (per spec: the flashlight turns off automatically while hiding); `isFlashlightLit(state)` is `on && !hiding` and gates both the flashlight polygon draw in `GameScene.updateDarkness()` and battery drain; `updateBatteryWithFlashlight(batteryState, flashlightState, delta, rechargeRatePerMs = 0)` wraps `updateBattery()` from `battery.js`: when lit it drains as before, and when unlit it returns the state unchanged UNLESS a positive `rechargeRatePerMs` is supplied (the "Rechargeable Battery" upgrade), in which case it trickle-recharges via `rechargeBattery(state, rechargeRatePerMs * delta)` -- so turning the light off is the player's battery-conservation tool, and with the upgrade it also slowly refills. The `rechargeRatePerMs` default of 0 keeps every pre-upgrade caller byte-identical (backward compatible); `GameScene` passes `BATTERY_RECHARGE_PER_MS` only when `this.hasRechargeBattery` is set. Follows the same immutable-state pattern as combat and battery
- **Light Switches** (`@/src/systems/lightswitch.js`): manages light switch state for creating safe zones. `createSwitchStates(rooms, seed, wallThickness)` (the whole-level builder) is retained and tested but **no longer used by `GameScene`** -- the lazy world places switches per room in `GameScene.activateRoomSwitch()` (15% chance per non-entrance room, seeded from the room's CELL coordinates so placement is path-order-independent). Placement is now MAZE-AWARE, mirroring how items/lore/weapons/furniture already receive and reject against the room's interior `mazeRects` (see **Intra-room placement clearance**). Two exports drive it. `chooseSwitchWall(room, wallThickness, mazeRects, startWallIndex)` scans the four walls from a seeded start index and returns the FIRST whose largest clear span is big enough for the switch footprint, falling back to the largest-span wall -- so a wall an interior maze wall has fully buried is skipped rather than chosen. `computeSwitchPosition(room, wall, wallThickness, mazeRects = [])` then seats the switch flush to that wall (inset by `wallThickness + 4`) at the CENTER OF THE LARGEST CLEAR SPAN, falling back to the wall midpoint only when no clear span exists. The free-span search removes the corners, the door gaps (inflated by clearance), AND the along-wall projection of any interior maze wall that reaches the switch's thin mounting band (each blocked interval inflated by the sprite half-extent so the placed FOOTPRINT clears the obstacle, not just its center). The `mazeRects` argument is optional and defaults to empty, so legacy three-argument callers (and `createSwitchStates`) are byte-identical to before. This is the fix for the spec's "light switches spawn inside walls / on doors" bug. Door-gap avoidance alone was not enough: `subdivisionWallRect` in `@/src/systems/maze.js` extends interior maze/corridor walls OUT through the perimeter band to the true room edge, so a switch mounted flush on that perimeter could land buried inside a maze wall band (the switch sprite at depth 50 then drew over the wall fill, reading as a switch floating in a black band). The old inline `GameScene` formula always used the bare wall midpoint with no door OR maze check. Each switch stores `{ id, roomId, x, y, isOn }` and starts off. `toggleSwitch(switchStates, switchId)` returns a new array with the targeted switch's `isOn` flipped (immutable). `findNearestSwitch(switchStates, px, py, maxRange)` returns the closest switch within `SWITCH_INTERACT_RANGE` (80px), or null. `getLitRoomIds(switchStates)` returns the room IDs of all switches that are on -- consumed by GameScene to drive both darkness mask reveal and enemy AI freezing. `isPointInRoom(room, x, y)` is a bounds check helper used by GameScene to determine if an enemy is inside a lit room
- **Light Spill** (`@/src/systems/lightSpill.js`): pure-function module that computes gradient light spill zones through open doorways from lit rooms into adjacent dark rooms. `isDoorOpen(roomIdA, roomIdB, doorStates)` normalizes room IDs (min/max) to match the deduplication invariant in `@/src/systems/doors.js` (`roomId < targetRoomId`), then checks if a matching door state exists -- returns true if no state exists (permanently open passageway) or if the state has `isClosed: false`. `getLightSpillZones(rooms, litRoomIds, doorStates)` iterates each lit room's `doors[]`, skipping doors where the target room is also lit (both already fully revealed) or the door is closed, and calls `computeSpillZone(room, door)` for each qualifying doorway. `computeSpillZone()` produces a zone descriptor `{ x, y, width, height, alphas: { topLeft, topRight, bottomLeft, bottomRight } }` where the rectangle extends `LIGHT_SPILL_DEPTH` (150px) outward from the doorway into the adjacent dark room. The `alphas` encode the gradient direction: 1 at the doorway edge (full brightness), 0 at the far edge (fading to dark). This is consumed by `@/src/scenes/GameScene.js` in `updateDarkness()` via Phaser 3's `fillGradientStyle` WebGL API, which sets per-corner alpha on the subsequent `fillRect` call. Two draws per zone: one into `maskGraphics` (white, varying alpha to create gradient visibility through the BitmapMask) and one into `lightGraphics` (warm tint at 5% of the spill alpha for atmospheric color). The module has no Phaser dependency, no seeded PRNG, and no state -- it is a pure geometric computation consumed each frame
- **Pause controls data** (`@/src/systems/pause.js`): pure-data module exporting `DESKTOP_CONTROLS` and `MOBILE_CONTROLS` arrays, each containing `{ key, action }` descriptors for the controls reference displayed on the pause screen. `getControlsList(touchMode)` returns the appropriate array based on input mode. Desktop controls include WASD, mouse aim, left/right click, E interact, Q/scroll weapon switch, and ESC to resume. Mobile controls list the equivalent touch UI elements (sticks and on-screen buttons). This module has no Phaser dependency, no state, and no seeded PRNG -- it is a pure data lookup consumed by `@/src/scenes/PauseScene.js`
- **Title menu** (`@/src/systems/titleMenu.js`): pure-data module that generates menu option arrays for the title screen based on save state. `getMenuOptions(hasSave)` returns an array of `{ key, label, color }` objects -- when `hasSave` is true, returns Continue (green), New Game (green), and Delete Save (red); when false, returns only New Game (green). Continue is always placed before New Game when present. `getTitleText()` returns `{ title: 'BACKROOMS', subtitle: "Don't let the lights go out." }`. This module has no Phaser dependency, no state, and no seeded PRNG -- it is a pure data lookup consumed by `@/src/scenes/TitleScene.js`, following the same pattern as `@/src/systems/pause.js`
- **Doors** (`@/src/systems/doors.js`): manages closable door state for tactical gameplay. `createDoorStates(rooms, seed)` (the whole-level builder) is retained and tested but **no longer used by `GameScene`** -- the lazy world creates door states per connection in `GameScene.activateRoomDoors()` (one state per shared edge, created when the second of the two rooms activates, deduped by a canonical connection key, closability decided deterministically from the connection's canonical CELL coordinates so it is path-order-independent). The runtime mutation helpers below are still used. Closable doors start closed (`isClosed: true`) and each stores a precomputed `segment` in `{x1,y1,x2,y2}` format matching the wall segment contract. `toggleDoor(doorStates, doorId)` returns a new array with the targeted door's `isClosed` flipped (immutable). `getClosedDoorSegments(doorStates)` filters for closed doors and returns their segments -- used by GameScene to manage the `wallSegments` array. `getDoorCenter(doorState, rooms)` computes the world-space center point of a door for rendering and proximity checks. `findNearestDoor(doorStates, rooms, px, py, maxRange)` returns the closest door within `DOOR_INTERACT_RANGE` (80px) of the player, or null. The segment winding matches `room.js` conventions: south and west walls are reversed
- **Items** (`@/src/systems/items.js`): `generateRoomItems(..., distance = 0)` places collectible items per room using a seeded PRNG (seed offset +20000 to avoid correlation with furniture and enemies). Room 0 (starting room) always returns empty. **Distance-banded item counts**: a single seeded roll decides the per-room item count, but the roll's thresholds depend on which distance band the room falls in (band edges exported as `NEAR_DISTANCE` and `DEEP_DISTANCE`, resolved by the internal `itemCountForRoll()`). Rooms near the entrance (distance at or below `NEAR_DISTANCE`) are generous -- rarely empty, up to 3 items; a mid band tapers down; deep rooms (distance at or past `DEEP_DISTANCE`) keep the original sparse distribution where most rooms are empty and 2 items is the cap, so each deep find feels earned. This implements the binding spec feedback that there should be "more lower rewards earlier on" given how high the shop prices are: the near band coincides with the all-basic-enemy zone from `@/src/systems/scaling.js`, and its type weights already favor low-tier loot (copper coins, batteries), so raising the COUNT there yields a dense-but-cheap surface (Motherload-style pacing) while the deep economy stays sparse and rarity-driven. Only the thresholds vary by band -- it is still exactly one PRNG draw for the count, followed by x/y/type draws per placement attempt, so the seeded stream shape is unchanged and deep-room generation is byte-identical to the old flat distribution. **Distance-from-entrance loot scaling** is the core reward mechanism: each item type's `weight` is a `{ base, perRoom }` shape rather than a scalar, and `weightForDistance()` computes the effective weight as `max(0, base + perRoom * distance)` where `distance` is the room's exploration depth from the entrance (the `room.distance` field, threaded through from `GameScene.activateRoomItems()`). `pickWeightedType(rand, distance)` then does the usual weighted roll over those distance-adjusted weights. Utility items (battery, ammo, medkit) use `perRoom: 0` so they appear at constant rates everywhere (survival supplies are not gated by exploration). Treasure tiers shift with depth: `copper_coin` has a negative `perRoom` (falls off as you go deeper), while `silver_coin`/`gold_coin`/`gem` have positive `perRoom` (rise with depth). `gold_coin` and `gem` have negative `base`, so their effective weight is clamped to 0 near the entrance -- `gem` only becomes possible at roughly `distance >= 5`, making gems a deep-exploration reward that never spawns near the entrance. In the unbounded world, `distance` is the per-room exploration depth stamped in `GameScene.genRoom()` (`floor * FLOOR_DEPTH + localDistance(gx, gy)`), which grows without bound as the player explores deeper. Each item type defines a color, size, and treasure value (battery, ammo, and medkit have value 0). Three item categories exist with different pickup behavior: **inventory items** (battery, treasure coins/gems) require inventory capacity and are stored via `@/src/systems/inventory.js`; **ammo items** restore ammo to the active weapon (amount determined by `AMMO_PER_PICKUP` in `@/src/systems/weapons.js`, ignored if the player has no weapon equipped); **medkit items** bypass inventory entirely and are consumed immediately on contact, healing the player via `applyHeal()` from `@/src/systems/combat.js` (ignored at full health, leaving the medkit on the ground). Item positions avoid furniture overlap using the same 8px-radius exclusion check and retry pattern as other placement systems, AND -- via the optional trailing `obstacles = []` parameter -- avoid the room's internal maze walls: when `obstacles` is non-empty the candidate is rejected against `furnitureItems.concat(obstacles)`, so coins/pickups never spawn embedded inside a solid maze wall where they would be uncollectable. `GameScene.activateRoomItems()` threads the room's `mazeRects` here (see **Intra-room placement clearance**). When `obstacles` is empty/omitted (the default) the path reuses the `furnitureItems` array reference unchanged, so determinism is byte-identical to before -- the obstacle test runs AFTER each PRNG draw. This distance-based tiering shares its axis with enemy difficulty scaling (`@/src/systems/scaling.js`), so deeper rooms pair better loot with harder fights; `runCount` only reseeds the unbounded world per day and affects neither loot nor difficulty
- **Distance** (`@/src/systems/distance.js`): `computeRoomDistances(rooms, stairs)` returns a `Map<roomId, hopDistance>` of each room's BFS hop-distance from the entrance (room id 0). It builds an undirected adjacency graph from two edge sources: each room's `doors[].targetRoomId` (added bidirectionally) and each stair's `fromRoomId`<->`toRoomId` (also bidirectional), then runs breadth-first search seeded at room 0. Because stairs are single edges, a room reachable only by climbing a stair is one hop farther than its same-floor neighbors -- so depth accumulates across floors. Unreachable rooms are simply absent from the returned map. This is a pure, read-only graph traversal (no seeded PRNG, no wall segments, no Phaser). It is retained and tested but has **ZERO production callers** (kept alive only by its tests): in the lazy world, per-room exploration depth is computed directly as `room.distance = floor * FLOOR_DEPTH + localDistance(gridX, gridY)` when a room is generated, and that value is threaded into item tier scaling, enemy loot drop quality (via `@/src/systems/lootDrops.js`), AND enemy difficulty (via `@/src/systems/scaling.js`). `room.distance` is now the single axis both reward (loot tiers, drop quality) and risk (enemy HP, damage, count, type mix) scale on
- **Lore** (`@/src/systems/lore.js`): spawns collectible lore notes in rooms for atmospheric storytelling, and provides journal data functions for between-run lore review. `generateRoomLore(roomX, roomY, roomWidth, roomHeight, wallThickness, seed, furnitureItems, roomId, collectedLoreIds)` uses a seeded PRNG (seed offset +90000) to determine if a room gets a lore note (~25% chance, `LORE_CHANCE`). Room 0 always returns empty. At most one lore note spawns per room. A lore entry is selected from `LORE_ENTRIES` (an array of 20 Backrooms-themed text entries, each with a unique `id`). If the selected entry's ID is in `collectedLoreIds` (a Set of already-collected IDs), the function returns empty -- this is the meta-progression mechanism that prevents re-spawning collected lore across runs. Placement uses the same retry-on-overlap pattern (20 attempts, 8px radius furniture exclusion) as other spawning systems, and takes the same optional trailing `obstacles = []` parameter -- when non-empty, candidates are rejected against `furnitureItems.concat(obstacles)` so a note never spawns inside a solid maze wall where it could never be picked up (`GameScene.activateRoomLore()` threads the room's `mazeRects`; see **Intra-room placement clearance**). The empty/omitted default is byte-identical to before. Returns an array of zero or one `{ x, y, loreId, text }` objects. `getJournalEntries(collectedLoreIds)` maps all `LORE_ENTRIES` to `{ id, text, collected }` objects (where `collected` is true if the ID is in the provided Set), and returns `{ entries, collectedCount, totalCount }` -- this is consumed by `@/src/scenes/ShopScene.js` to render the journal overlay and the "NOTES X/20" button. `getJournalPage(entries, page, perPage)` paginates an entries array, clamping the page index to valid bounds, and returns `{ pageEntries, currentPage, totalPages }` -- consumed by ShopScene's `renderJournalPage()` to display one page at a time. Both journal functions are pure (no PRNG, no side effects). The module does not produce wall segments and does not manage state -- state management (the `collectedLore` Set) lives in `game.registry` and is persisted via `@/src/systems/persistence.js`
- **Enemy loot drops** (`@/src/systems/lootDrops.js`): pure function module determining what items enemies drop on death, creating a risk-reward dynamic where combat is rewarded with resources. `DROP_CHANCES` defines per-enemy-type drop probability (basic 30%, crawler 45%, spitter 55% -- harder enemies drop more often). `ENEMY_DISTANCE_BONUS` gives per-enemy-type bonus to effective room distance (basic +0, crawler +1, spitter +2 -- harder enemies bias toward higher-value loot). `getDropChance(enemyType)` and `getDistanceBonus(enemyType)` are lookups with fallback to basic/0. `rollEnemyDrop(enemyType, distance, rand)` is the main entry point: first roll against `getDropChance()` to decide if any drop occurs; if so, compute `effectiveDistance = distance + getDistanceBonus(enemyType)`, then select an item type via `pickLootType(rand, effectiveDistance)` using the same distance-based weighted selection formula as room item spawning in `@/src/systems/items.js`: `max(0, base + perRoom * effectiveDistance)`. Returns `{ type, value }` or null. The loot pool is `ITEM_TYPES` from `@/src/systems/items.js` with medkits filtered out (excluded from drops to preserve horror tension -- combat should not be self-sustaining with healing). The `weightForDistance` formula is duplicated from `items.js` intentionally -- keeping it private in each module avoids coupling them. The `rand` function is caller-provided for testability (`GameScene` passes `Math.random` since drops are runtime events, not deterministic level generation). This module has no Phaser dependency, no seeded PRNG, and no state. Consumed by `@/src/scenes/GameScene.js` in `onBulletHitEnemy()` at the point where an enemy dies: drops are placed as sprites in `this.itemGroup` at the enemy's position, so the existing `onItemPickup` overlap collider auto-registers them for pickup with no additional wiring. Tests live in `@/src/systems/__tests__/lootDrops.test.js`
- **Inventory** (`@/src/systems/inventory.js`): manages collected items as immutable state with a capacity constraint. `createInventoryState(maxItems?)` produces `{ batteries, treasureValue, itemCount, maxItems }` where `maxItems` defaults to `BASE_INVENTORY_CAPACITY` (8). `canPickupItem(state)` returns true when `itemCount < maxItems` -- GameScene calls this before every pickup attempt and leaves items on the ground when full. `pickupItem(state, item)` either increments the battery counter (for battery type) or adds the item's value to `treasureValue` (for all treasure types), and always increments `itemCount`. `useBattery(state)` returns `{ used, state }` -- decrements battery count and `itemCount` if available (consuming a battery frees an inventory slot), otherwise returns `used: false` with unchanged state. GameScene calls `rechargeBattery()` from `battery.js` when `useBattery()` succeeds. Constants: `BASE_INVENTORY_CAPACITY=8`, `INVENTORY_CAPACITY_PER_LEVEL=4`
- **Shop** (`@/src/systems/shop.js`): manages the between-run upgrade economy as immutable state. `createShopState()` produces `{ gold, upgrades }` where upgrades maps each upgrade id to its current level (starting at 0). The `UPGRADES` array is the single source of truth for all upgrade categories -- ShopScene auto-generates UI rows from it, and `createShopState()` derives initial keys from it. `addGold(state, amount)` increases the gold balance. `canPurchase(state, upgradeId)` checks affordability and max-level constraints. `purchaseUpgrade(state, upgradeId)` deducts gold and increments the upgrade level, returning unchanged state if the purchase is invalid. `getUpgradeValue(upgradeId, level)` computes the stat value as `base + perLevel * level` -- this is called by `GameScene.init()` to derive all player stats from upgrade levels. `getUpgradeCost(upgradeId, currentLevel)` returns the next tier's cost or null at max level. The `UPGRADES` array defines survival, weapon, inventory, and unlock upgrade categories (e.g., battery capacity, flashlight width, max health, movement speed, weapon damage, fire rate, bullet range, backpack size, starting pistol, minimap). Most upgrades share a 3-tier structure with costs 200/600/1500. Binary unlock upgrades use a single-tier pattern (maxLevel 1, base 0, perLevel 1): three starting weapon upgrades (`startingPistol` at 1000g, `startingShotgun` at 1500g, `startingRifle` at 2000g) cause `GameScene.init()` to pass the corresponding flags to `createWeaponState()`, which fills up to 2 weapon slots using priority-based selection (rifle > shotgun > pistol); `minimap` (cost 3000) enables the fog-of-war minimap display in `GameScene`; `rechargeBattery` (cost 2500, deliberately very expensive) sets `GameScene.hasRechargeBattery`, which gates passing `BATTERY_RECHARGE_PER_MS` into `updateBatteryWithFlashlight()` so the battery slowly refills while the flashlight is off. Because the upgrade system is fully data-driven, adding an entry to `UPGRADES` auto-flows into `createShopState()`, the ShopScene UI rows, and persistence with no `SAVE_VERSION` bump -- the same `GameScene.hasMinimap` boolean-flag pattern (see `@/src/scenes/docs.md`). Note that fire rate uses a negative `perLevel` (-30ms) since lower cooldown means faster firing. The backpack upgrade controls inventory capacity (base 8, +4 per level, giving 8/12/16/20 at levels 0/1/2/3). Save compatibility: old saves missing new upgrade keys are handled by `GameScene.init()` spreading `createShopState().upgrades` as defaults before overlaying the saved upgrade levels

### Things to Know
- **Segment format is the universal integration contract, but occluders are split into a segment store and two rect stores by consumer**: `getFlashlightPolygon()` and enemy LOS both raycast against `{x1, y1, x2, y2}` segments. `wallSegments` (in `GameScene`) holds room perimeter walls (pushed by `GameScene.activateRoom()` via `createRoomWalls()` from `room.js`) and closed doors -- and is consumed identically by both the flashlight and enemy LOS. Maze internal walls live SEPARATELY in `mazeWallRects`, and light-blocking furniture in the parallel `furnitureOccluderRects`, both as `{x,y,width,height}` rects, NOT pre-expanded segments, precisely so the two consumers can expand them differently: enemy LOS to the full outline via `wallRectSegments()` (blocks sight from both sides), the flashlight to the far faces only via `wallRectOccluders(rect, origin)` (back-face culling, so the cone lights each rect's near face and it reads as a band/piece of visible thickness instead of pitch-black). Light-blocking furniture used to live in `wallSegments` as a full outline via `createFurnitureSegments()`, which left the cone stopping at the piece's near face so the footprint rendered pitch-black under the darkness overlay -- moving it into a back-face-culled rect store is the fix. Doors are the only system to dynamically add/remove from `wallSegments` at runtime (via `push`/`splice`); everything else in all three stores is static once its room activates, and all three only ever grow as the unbounded world is explored -- which is why per-frame raycasting prefilters `wallSegments` via `filterSegmentsNear()` and both rect stores via `filterRectsNear()` (both from `visibility.js`). The floor-0 rift is the one perimeter opening with NO door gap in the occluder segments (light is blocked both ways even though the player can walk through). Bullets do not produce segments -- they are dynamic projectiles handled by Phaser's Arcade Physics
- **Occluder = footprint invariant for interior obstacles**: maze wall rects (from `@/src/systems/maze.js`) and light-blocking furniture occlude along their drawn rectangle, so the raycast boundary, the drawn fill, and the physics zone are the same rectangle -- light stops at the visible face and cannot tunnel inside or wrap around a drawn solid (this invariant is what fixed the flashlight-leaking-through-wall-edges bug). One deliberate refinement for maze walls: the flashlight uses only each wall's FAR faces (`wallRectOccluders()`, back-face culling) so the cone reaches across the band and lights its near face (visible thickness), while enemy LOS still uses the full outline (`wallRectSegments()`) -- both still coincide with the drawn rectangle's edges, so neither lets light or sight tunnel inside the solid. Perimeter walls are the deliberate exception: they occlude along the room boundary line (the outer face of their drawn band) so lit wall faces still light up, leaving the band interiors occluder-free -- which is why maze subdivision walls are extended through the perimeter band to the room edge to seal the maze-to-perimeter junctions. The known accepted residual of the perimeter exception: light entering through an OPEN door at a shallow angle can clip a sliver (up to a band thickness) of the drawn door-jamb corner -- verified sub-visual in playtests and intentionally left out of scope
- **Combat, battery, flashlight, inventory, weapon, door, light switch, hiding, exploration, danger, run stats, and shop state are immutable**: `applyDamage()`, `applyHeal()`, `updateCombat()`, `updateBattery()`, `updateBatteryWithFlashlight()`, `toggleFlashlight()`, `setFlashlightHiding()`, `rechargeBattery()`, `pickupItem()`, `useBattery()`, `switchWeapon()`, `pickupWeapon()`, `consumeAmmo()`, `addAmmo()`, `toggleDoor()`, `toggleSwitch()`, `enterHiding()`, `exitHiding()`, `updateExploration()`, `updateDangerTime()`, `advanceWave()`, `recordKill()`, `updateTime()`, `updateMaxFloor()`, `addGold()`, and `purchaseUpgrade()` all return new state objects/arrays rather than mutating (note: `canPickupItem()`, `hasAmmo()`, `isFlashlightLit()`, and `shouldSpawnWave()` are read-only predicates, not state transitions). `GameScene` reassigns `this.combatState`, `this.batteryState`, `this.flashlightState`, `this.inventoryState`, `this.weaponState`, `this.doorStates`, `this.switchStates`, `this.hidingState`, `this.explorationState`, `this.dangerState`, and `this.runStats` on each call. `ShopScene` reassigns `this.shopState` and writes it back to `game.registry`. Enemy health, by contrast, is a mutable field on the enemy state objects in `enemyStates[]` -- this asymmetry exists because player combat has richer state (cooldowns, death flag) while enemy health is a simple integer decrement
- **Door pairing invariant**: a door is a single physical opening on a shared edge, and both adjacent rooms must describe it identically (same `offset` and `width`, mirrored `wall`/`targetRoomId`). In the live generator (`worldgen.js`) this holds by construction -- each shared edge's open/closed state and offset are hashed once from the edge's canonical coordinates, so both rooms always agree regardless of generation order. (The legacy `level.js` enforced the same invariant by writing paired door entries explicitly.) This ensures the gaps in adjacent rooms' walls align perfectly, allowing flashlight rays and player movement to pass through
- **Room coordinates are grid-based**: room pixel position is `gridX * ROOM_WIDTH` and `gridY * ROOM_HEIGHT`, so adjacent rooms share edges exactly. The live generator uses `ROOM_WIDTH = 720` and `ROOM_HEIGHT = 600` with `DOOR_WIDTH = 80` (the legacy `level.js` still uses the old 1200x1000 cells)
- **Enemy AI state machine** (idle -> chase -> search -> idle, plus attack for spitters): idle enemies wander randomly and transition to chase on LOS; chasing enemies pursue (speed varies by type: basic 80, crawler 112, spitter 60); searching enemies walk to the stored last-seen position and revert to idle on arrival or after a 3-second timeout. The AI stamps `lastKnownX/Y` from the player's position on every frame it has LOS; the moment LOS breaks (wall, closed door, or hiding), chase and attack drop to search toward that stored point -- there is no doorway routing toward the live player and no door-opening state (both were removed under the binding spec rule that enemies must not follow the player after losing sight). Spitters have an `attack` state, entered from chase when within 250px range -- the spitter stops moving, counts down an attack cooldown, and signals `wantsToFire = true` when ready (GameScene reads this to spawn an enemy projectile); attack transitions back to chase if the player moves out of range, or to search if LOS breaks. Any state transitions to chase if LOS is reacquired. Sound alerts are the one asymmetry: gunfire (`gunfireNoise.js`) externally sets idle/search enemies to chase with `lastKnownX/Y` snapshotted at noise time, which the FSM resolves as a fixed-point investigation. Idle enemies may also seek doorways probabilistically (15% chance per wander tick, gated by `roomTransitionCooldown` and `MAX_ROOM_DISTANCE`/`MAX_ROOM_ENEMIES` constraints), causing gradual enemy migration between rooms -- this is spawn-room-relative wandering, not player tracking. Each enemy state object carries `spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, and `targetDoorway` fields for room tracking
- **Closed doors block all enemies**: `buildRoomGraph()` in `@/src/systems/pathfinding.js` excludes closed doors from the adjacency graph, and a closed door's occluder segment breaks enemy LOS, so closing a door is a universal defense -- every enemy type loses sight and falls back to searching the last-seen position. (Crawlers previously used a second "full" graph and an `opening_door` state to open closed doors mid-chase; both were removed when pursuit became strictly sight-based.) The single room graph is cached in GameScene and rebuilt when `onToggleDoor()` or room activation sets `roomGraphDirty = true`
- **Enemy, furniture, and item placement share the same retry pattern**: all attempt `count * 20` random placements and discard candidates that overlap existing objects. Fewer items may be placed than requested if the room is crowded
- **Intra-room placement clearance** (the fix for the spec's "hiding places, stairs, and doors spawn inside walls" bug): the intra-room systems used to be generated independently and never checked each other's geometry, so furniture/stairs/doors -- and every loose ground pickup -- could end up embedded inside a solid maze wall, where a coin/note/weapon is unreachable (uncollectable, never picked up). They are now unified through three strategies. REJECT-and-resample: `generateRoomFurniture()` (`@/src/systems/furniture.js`) AND all three ground-pickup generators -- `generateRoomItems()` (`@/src/systems/items.js`), `generateRoomLore()` (`@/src/systems/lore.js`), `generateRoomWeapon()` (`@/src/systems/weapons.js`) -- take an optional trailing `obstacles` list and reject any candidate overlapping it (the pickups concat `obstacles` onto their existing `furnitureItems` rejection set, reusing each generator's own clearance radius). Furniture gets the maze rects PLUS `doorEntryZones()` PLUS the stair-landing keep-out; the three pickups get maze rects ONLY (doorways are walkable, so a pickup in a doorway is still collectable and is not the bug). `@/src/scenes/GameScene.js` threads the room's `mazeRects` (computed once in `activateRoom()`) into `activateRoomItems/Lore/Weapon`, each forwarding it as the generator's `obstacles` arg. FREE-SPAN-on-a-wall: the wall-mounted light switch cannot resample a free 2D point (it must sit flush on a wall), so `@/src/systems/lightswitch.js` instead folds the along-wall projection of the same `mazeRects` AND the room's door gaps into its largest-clear-span search and skips a fully-buried wall (`chooseSwitchWall`/`computeSwitchPosition`, threaded the same `mazeRects` via `activateRoomSwitch()`; see **Light Switches**). This is the fix for the spec's "light switches spawn inside walls / on doors and open entrances" bug -- switches were the one intra-room placement system never given `mazeRects`; doorways and open entrances are the same `doorEntryZones()` rectangles (worldgen has no separate "open entrance" concept). CARVE: `generateMazeWalls()` (`@/src/systems/maze.js`) takes a `keepOut` list and removes walls from those zones (in addition to the always-carved door zones) -- `GameScene` supplies a `stairKeepOut()` rect AND a `stairLandingKeepOut()` rect for rooms with a centered stair. The stair-landing keep-out (where the player materializes after a transition) is also fed to `generateRoomEnemies()` (`@/src/systems/enemy.js`) via its `keepOut` parameter, alongside the stair footprint, so no enemy spawns on the stair or on the landing spot -- pairing with the post-transition `grantInvulnerability()` window (see Combat) to fix the spec's "stairs ... vulnerable to attack immediately after spawning" bug. The shared invariant: the maze rects are generated FIRST so furniture, the stair, every pickup, and every enemy spawn can be kept clear of them; it is locked in by a full-pipeline invariant test (`@/src/systems/__tests__/placement.test.js`) that mirrors `activateRoom()`'s obstacle assembly over the real worldgen rooms -- the live switch path (`chooseSwitchWall` + maze-aware `computeSwitchPosition`, exactly as `activateRoomSwitch()` runs it) is guarded there against BOTH interior maze walls AND doorways/open entrances over real worldgen rooms, where the door-avoidance tests in `lightswitch.test.js` use only synthetic rooms (the dead `createSwitchStates` path and direct `computeSwitchPosition` unit checks), not the full generation pipeline. Determinism is unaffected (every obstacle/keep-out test runs AFTER the PRNG draws, and an empty/omitted `obstacles`/`keepOut` reuses the original array reference) -- so existing callers, the start room (id 0, no maze), and the test-only `generateLevelWeapons` are byte-identical to before
- `mulberry32` was extracted from `furniture.js` into `random.js` so that `level.js`, `enemy.js`, `items.js`, `doors.js`, `lightswitch.js`, `stairs.js`, `weapons.js`, `maze.js`, and `locations.js` could also use a seeded PRNG. All modules import from the same source to avoid duplication. Each spawning/classification system uses a different seed offset (furniture +0, enemies +10000, items +20000, doors +30000, light switches +40000, stairs +50000, weapons +60000, maze room type +70000, extra doors +80000, lore +90000, alternate exits +100000, room vibe +130000) so that adding or removing items in one system does not change the random sequence of another. The room vibe (`@/src/systems/roomVibes.js`, +130000) is deliberately a separate axis from the maze room type (+70000) so visual identity and structural layout vary independently. The lazy world (`GameScene`) likewise derives a per-floor seed as `levelSeed + floor * 1000` (mirroring the legacy stairs module) so each floor has a distinct layout, and `worldgen.js` hashes that floor seed with cell coordinates to keep room/door generation order-independent. The `generateAlternateExit()` path is no longer wired into the lazy `GameScene`. The starting room system (`startroom.js`), location definitions in `locations.js` (the static `LOCATIONS` array and layout functions), hiding system (`hiding.js`), exploration system (`exploration.js`), scaling system (`scaling.js`), pathfinding system (`pathfinding.js`), distance system (`distance.js`, a pure BFS over the door/stair graph), persistence system (`persistence.js`), light spill system (`lightSpill.js`, a pure geometric computation from lit rooms and door states), pause controls data (`pause.js`, a pure data lookup), title menu data (`titleMenu.js`, a pure data lookup), and audio systems (`audio.js`, `audioEngine.js`) do not use a seeded PRNG -- startroom uses a fixed hand-crafted layout, location definitions are static data, hiding operates on furniture objects already placed by `furniture.js`, exploration is deterministic based on player position, scaling applies deterministic formulas based on a room's distance from the entrance, persistence is a browser storage boundary, and audio uses a Knuth multiplicative hash of game time for ambient scheduling (no seeded PRNG, no effect on level determinism). The `generateAlternateExit()` function in `locations.js` is the exception -- it uses a seeded PRNG (offset +100000) for probabilistic exit placement
- **System boundary modules**: three system modules interact with browser APIs rather than being pure game math. `persistence.js` wraps `localStorage` access for game saves -- all three exported functions use try-catch because localStorage can throw (SecurityError in private browsing, QuotaExceededError). `settings.js` wraps `localStorage` access for audio settings under a separate key (`backrooms_settings`), using the same try-catch pattern and following the same graceful-degradation approach. `audioEngine.js` wraps Web Audio APIs (`OfflineAudioContext`, oscillator/filter/gain nodes) -- its `playAmbientSound()` and `generateAllSounds()` use try-catch because audio playback can fail when the browser blocks autoplay or the audio context is not running. All three use try-catch appropriately per the project's coding guidelines since they sit at system boundaries. The companion config module `audio.js` is pure (no browser dependency, Node-testable) and now includes per-enemy-type sound logic (state transition sounds, ambient cooldown sounds, death sounds, fire sounds, and distance-based volume attenuation) -- the config/engine split follows the same pattern as the rest of the systems directory. The versioned save format in persistence.js (`SAVE_VERSION = 4`) enables future migrations -- `loadGame()` rejects saves with a higher version than the running code understands, and defaults missing fields to safe fallback values (e.g., `collectedLore` to `[]`, `unlockedLocations`/`activeLocation` to `'store'`, and `worldSeed` to `null` for saves that predate those fields). The `worldSeed → null` default is the migration path for the bump from v3 to v4: a v3 save loads with no seed and `resolveWorldSeed()` mints a fresh one on the next run (see Persistence above). Audio settings intentionally use a separate localStorage key from game saves so that volume preferences are independent of save lifecycle (creating/deleting saves does not affect volume settings)
