# Noridoc: scenes

Path: @/src/scenes

### Overview
- Phaser Scene subclasses that wire pure game systems to Phaser's rendering, physics, and input APIs
- Contains two scenes: `GameScene` (gameplay: movement, flashlight/darkness, enemy AI, combat, shooting, weapon switching/pickups, battery drain, item collection, closable doors, light switches, furniture hiding, exploration minimap, multi-floor stairs with teleportation, starting room furniture store, exit condition, HUD) and `ShopScene` (between-run upgrade shop: displays gold balance, upgrade categories with buy buttons, and "Enter Backrooms" button)
- The two scenes alternate in a loop: GameScene -> ShopScene (on exit or death) -> GameScene (on "Enter Backrooms")

### How it fits into the larger codebase
- `@/src/main.js` registers scene classes with the Phaser.Game instance; GameScene is listed first (initial scene)
- Scenes import pure functions from `@/src/systems/` for game math (including maze generation from `@/src/systems/maze.js`) and handle all Phaser-specific concerns: sprite creation, physics bodies, input binding, camera, collision callbacks, and Graphics drawing
- Scenes are the only place in the codebase that depends on Phaser -- this boundary is intentional to keep game logic testable
- `GameScene` reads upgrade levels from `game.registry` in its `init()` method, using `getUpgradeValue()` from `@/src/systems/shop.js` to compute upgrade bonuses as deltas from pistol base values (e.g., `getUpgradeValue('weaponDamage', level) - 25` yields a +N damage bonus). These bonuses are stored in `this.upgradeBonuses` and applied additively to whatever weapon is active via `getEffectiveStats()` from `@/src/systems/weapons.js`. Survival stats (max HP, max battery, flashlight angle, speed) are still computed as absolute values. It also calls scaling functions from `@/src/systems/scaling.js` to compute run-scaled enemy stats per type -- each enemy type (basic, crawler, spitter) has its own base HP, contact damage, and chase speed constants, and scaling is applied independently to each set (e.g., `this.scaledEnemyHP`, `this.scaledCrawlerHP`, `this.scaledSpitterHP`). These values are stored as instance properties and used throughout `create()` and `update()`. Upgrade level fallback uses `createShopState().upgrades` as defaults, ensuring forward compatibility when new upgrade keys are added to saves that predate them
- `ShopScene` reads/writes `shopState` from `game.registry`, using pure functions from `@/src/systems/shop.js` for all purchase logic. It receives `treasureEarned` as scene data from GameScene and converts it to gold via `addGold()`. ShopScene also calls `saveGame()` from `@/src/systems/persistence.js` at every point where `shopState` is mutated: after adding treasure gold in `init()`, before entering backrooms, and after purchasing an upgrade
- `GameScene` calls `saveGame()` from `@/src/systems/persistence.js` after incrementing `runCount` in `init()`, persisting the updated run counter and shop state to localStorage before the run begins
- Data flows between scenes via two mechanisms: `game.registry` for in-memory cross-scene state (shopState, runCount) and scene `init(data)` for per-transition data (treasureEarned). Both registry values are also persisted to localStorage via `@/src/systems/persistence.js` so state survives browser reloads

### Core Implementation
- **GameScene lifecycle**: `init()` reads upgrade state from registry and computes per-run stats, `create()` sets up the level and all game objects, `update()` runs every frame
- **GameScene.init()**: reads `shopState` from `game.registry`, spreads `createShopState().upgrades` as defaults before overlaying saved upgrade levels (for forward compatibility with saves predating new upgrades), then calls `getUpgradeValue()` for each upgrade to set survival stats (`this.maxCharge`, `this.flashlightAngle`, `this.maxHp`, `this.playerSpeed`), weapon upgrade bonuses (`this.upgradeBonuses = { damage, fireRate, range }` computed as deltas from pistol base values), and inventory capacity (`this.maxItems` from the backpack upgrade). Initializes `this.weaponState` via `createWeaponState()` from `@/src/systems/weapons.js` and calls `this.updateWeaponStats()` to derive effective bullet stats (`this.bulletDamage`, `this.fireRate`, `this.bulletRange`, `this.bulletSpeed`, `this.bulletCount`, `this.bulletSpreadAngle`, `this.bulletColor`) from the active weapon plus bonuses. Also reads and increments `runCount` in the registry to derive a unique level seed for this run, and calls scaling functions from `@/src/systems/scaling.js` to precompute per-type scaled stats: HP (`this.scaledEnemyHP`, `this.scaledCrawlerHP`, `this.scaledSpitterHP`), chase speed (`this.scaledEnemyChaseSpeed`, `this.scaledCrawlerChaseSpeed`, `this.scaledSpitterChaseSpeed`), contact damage (`this.scaledEnemyDamage`, `this.scaledCrawlerDamage`, `this.scaledSpitterDamage`), and `this.extraEnemyCount`. After incrementing, calls `saveGame()` to persist the updated `runCount` and `shopState` to localStorage
- **create() setup sequence**: initializes level geometry (including per-room maze walls/columns via `createRoomMaze()`), doors, switches, player, enemies, item textures, items, player bullets, enemy bullets, stairs, exit zone, input, camera, darkness overlay, and HUD. Door creation happens after rooms but before player creation so that door colliders are ready. Switch creation happens after doors. Stairs are created after items. Combat state is initialized with `createCombatState(this.maxHp)`, battery state with `createBatteryState(this.maxCharge)`, inventory state with `createInventoryState(this.maxItems)` (where `maxItems` comes from the backpack upgrade computed in `init()`), hiding state with `createHidingState()` from `@/src/systems/hiding.js`, and exploration state with `createExplorationState()` from `@/src/systems/exploration.js`. Two additional flags are initialized: `this.isTeleporting = false` (gates update loop during stair transitions) and `this.currentFloor = 0` (tracks which floor the player is currently on). Notable setup details:
  - `createEnemyTextures()` generates distinct textures for each enemy type: basic (red circle with dark "eyes"), crawler (smaller purple/magenta circle), and spitter (blue/cyan circle with a "mouth" marking)
  - `createEnemies()` passes `this.extraEnemyCount` and `this.runCount` to `generateRoomEnemies()` for run-scaled enemy density and type distribution. `getEnemyStats(type)` maps each spawn's type to the correct scaled HP, contact damage, texture key, and body size. Each enemy state object carries `type`, `contactDamage`, and `attackCooldown` fields alongside the existing state machine fields. Wires a player-enemy overlap callback (`onEnemyContact`) for per-enemy contact damage
  - `createItemTextures()` generates a Phaser texture for each item type (rectangles for coins/batteries, triangle for gems), keyed as `item_{type}`
  - `createItems()` creates a static physics group, calls `generateRoomItems()` per room, and places item sprites at depth 5 (below darkness). Wires a player-item overlap callback (`onItemPickup`) for auto-collection
  - `createDoors()` calls `createDoorStates()` from `@/src/systems/doors.js` to determine which connections have closable doors. For each closable door, creates a static physics zone in `doorGroup` sized to fill the doorway gap. Closed doors have their segment pushed into `wallSegments` and their physics body enabled; open doors have `body.enable = false`. Also creates a "Press E" interaction prompt text element (hidden by default)
  - `createSwitches()` calls `createSwitchStates()` from `@/src/systems/lightswitch.js` to place light switches in ~40% of non-starting rooms. Creates a Graphics object at depth 50 for switch rendering. Switches have no physics bodies -- they are purely visual objects interacted with via proximity checks
  - `createBullets()` creates a pooled physics group (`maxSize: 30`), generates per-weapon bullet textures (pistol: yellow 3px, shotgun: orange 2px, rifle: cyan 3px), and wires bullet-wall/furniture/door colliders and bullet-enemy overlap. Each bullet texture is keyed as `bullet_{weaponId}` so `fireBullet()` can select the correct texture based on the active weapon
  - `createWeaponPickups()` generates per-weapon pickup textures (keyed `pickup_{weaponId}`) and iterates over all rooms calling `generateRoomWeapon()` from `@/src/systems/weapons.js` to place weapon pickups. Pickup sprites are stored in `this.weaponPickupSprites` and participate in the E-key interaction resolution system as candidates with type `'weapon'`
  - `createEnemyBullets()` creates a separate pooled physics group (`enemyBulletGroup`, `maxSize: 30`) with a red bullet texture, mirroring the player bullet pattern. Wires wall/furniture/door colliders and player overlap. `fireEnemyBullet()` is called from `updateEnemies()` when a spitter's AI state machine sets `wantsToFire = true`. `updateEnemyBullets()` expires enemy bullets using `SPITTER_PROJECTILE_RANGE`. `onEnemyBulletHitPlayer()` applies `SPITTER_PROJECTILE_DAMAGE` to the player's combat state (blocked while hiding)
  - `createExitZone()` places a 64x64 physics zone at the center-north of room 0 (position computed by `getExitPosition()` from `@/src/systems/startroom.js`) with a green visual indicator, wiring a player overlap callback (`onDayComplete`) for the day-cycle win condition. Also stores `this.exitPosition` for use by the minimap
  - `createCrackVisual()` draws a jagged crack line at the first doorway of room 0 using points from `generateCrackPoints()` in `@/src/systems/startroom.js`. The crack consists of a primary dark line (depth 2) and a thinner secondary line with a slight offset for a 3D effect. A glow rectangle (depth 1) behind the crack pulses with a sickly-yellow color (0xd4c073) via `updateCrackGlow()` each frame using a sine wave
  - `createStairs()` iterates over `this.level.stairs` (up to 2 stair connections, each using distinct rooms) and for each stair connection, draws a dark rectangle with step lines at both endpoints (the "from" side on floor 0 and the "to" side on floor 1). Each endpoint gets a Phaser physics zone (`STAIR_SIZE` x `STAIR_SIZE`) with `destX`, `destY`, and `destFloor` data properties. Overlap with the player triggers `onStairEnter()`. A separate `stairGlowGraphics` at depth 1 draws a pulsing purple glow (0x7733aa) behind both stair positions each frame via `updateStairGlow(time)`, using the same sine-wave pulse pattern as the crack glow
  - `createHUD()` creates a Graphics object with `setScrollFactor(0)` at depth 1000 so health and battery bars stay fixed on screen, plus text elements for battery count, treasure value, inventory capacity (`[itemCount/maxItems]` -- gray normally, turns red when full), and active weapon name (colored per-weapon with a `[Q]` switch indicator). Also creates a separate `minimapGraphics` Graphics object at depth 1000 with `setScrollFactor(0)` for the exploration minimap. A floor indicator text (`this.floorText`) shows "B1" or "B2" to indicate the player's current floor, updated each frame in `drawMinimap()`
- **update() frame loop**: early-returns if player is dead, `this.dayEnding` is true (exit transition in progress), or `this.isTeleporting` is true (stair transition in progress). Then: refreshes `pointer.worldX`/`worldY` via `pointer.updateWorldPoint(this.cameras.main)` so world coordinates stay current even when the camera moves without mouse input -> reads WASD key state -> if hiding and any movement key is pressed, exits hiding -> if hiding, zeroes velocity; otherwise, `calculateVelocity()` -> sets player velocity -> computes aim angle from mouse -> ticks combat cooldown (`updateCombat`), battery state (`updateBattery`), and fire cooldown (`updateFireCooldown`) -> checks for left-click + `canFire()` to fire bullets (blocked while hiding) -> `updateInteractPrompt()` resolves the nearest interactable (door, switch, hideable furniture, or weapon pickup by Euclidean distance) and shows/hides "Press E" -> checks Q key (`JustDown`) to switch weapons via `switchWeapon()` and `updateWeaponStats()` -> checks E key (`JustDown`) and dispatches to `onToggleDoor()`, `onToggleSwitch()`, `enterHiding()`, or `onWeaponPickup()` based on the resolved interactable type (pressing E while already hiding exits hiding) -> `updateBullets()` expires out-of-range player bullets (using per-bullet range from `getData('range')`) -> `updateEnemyBullets()` expires out-of-range enemy projectiles -> `updateEnemies(delta)` runs AI per enemy (with lit-room freeze check, `this.hidingState.isHiding` passed as the `playerHidden` flag, and type-specific chase speeds via `getScaledChaseSpeed()`; after AI update, checks `wantsToFire` flag to call `fireEnemyBullet()` for spitters) -> redraws player (with hiding, invulnerability, or normal visual state) -> recomputes darkness (flashlight cone + lit room rectangles drawn into BitmapMask) -> `updateCrackGlow(time)` animates the crack glow pulse -> `updateStairGlow(time)` animates the stair glow pulse -> `drawDoors()` renders closed doors -> `drawSwitches()` renders switch visuals (gray when off, yellow when on) -> updates exploration state via `updateExploration()` (tracking visited rooms) -> redraws HUD (health bar + battery bar + battery count + treasure value + weapon name) -> `drawMinimap()` renders the fog-of-war minimap (using per-floor bounds and floor filter). Right-click input triggers `onUseBattery()` outside the frame loop via a `pointerdown` event listener (blocked while hiding). Item pickup happens via Phaser overlap callbacks (auto-collected on contact)
- **Enemy integration pattern**: `GameScene` maintains a parallel `enemyStates` array alongside the Phaser physics sprites in `enemyGroup`. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, and standard AI state machine fields. Each frame, `updateEnemies()` syncs sprite positions into the state objects, calls the pure `updateEnemyAI()` function with the type-specific chase speed from `getScaledChaseSpeed()`, copies the returned state back, checks the `wantsToFire` flag for spitters (and calls `fireEnemyBullet()` if set), then applies velocity to the sprite. This keeps the AI logic testable in `@/src/systems/enemy.js` while letting Phaser handle physics resolution and projectile management
- **ShopScene lifecycle**: `init(data)` receives `treasureEarned` from GameScene, reads existing `shopState` from registry (or creates fresh state on first visit), adds the treasure as gold via `addGold()`, writes back to registry, and calls `saveGame()` to persist. `create()` builds the UI: title, treasure earned display, gold balance, upgrade rows, and "Enter Backrooms" button. Each upgrade row shows name, description, level dots (filled/empty circles), next cost, and a buy button. Purchases call `purchaseUpgrade()`, write back to registry, and call `saveGame()` to persist. The "Enter Backrooms" button writes shopState to registry, calls `saveGame()`, and then calls `this.scene.start('GameScene')` to transition back to gameplay. The save-at-every-mutation pattern ensures no progress is lost even if the browser crashes between user actions

### Things to Know
- **Depth layering order** controls what is visible through the darkness:

  | Layer | Depth | Purpose |
  |-------|-------|---------|
  | crack glow / stair glow graphics | 1 | Pulsing sickly-yellow glow behind the crack in room 0; pulsing purple glow behind stair zones |
  | crack line / stair step graphics | 2 | Jagged crack-in-the-wall lines at the first doorway of room 0; dark rectangles with step lines at stair positions |
  | item sprites | 5 | Collectibles hidden by darkness (only visible in flashlight) |
  | exit zone graphics | 10 | Green exit indicator visible under darkness |
  | furniture graphics | 50 | Furniture rendered below enemies |
  | door graphics | 50 | Closed doors rendered at same depth as furniture |
  | switch graphics | 50 | Light switches rendered at same depth as doors/furniture |
  | enemy sprites | 60 | Enemies hidden by darkness (below mask) |
  | `lightGraphics` | 99 | Faint yellow tint drawn under the flashlight cone |
  | `darkGraphics` | 100 | Black rectangle covering the entire level (the darkness) |
  | player bullet sprites | 150 | Player bullets visible above darkness but below player |
  | enemy bullet sprites | 150 | Enemy (spitter) projectiles, same depth as player bullets |
  | `playerGraphics` | 200 | Player always visible (above darkness) |
  | `hudGraphics` + HUD text | 1000 | Health bar, battery bar, battery count, treasure value, weapon name (above everything) |
  | `minimapGraphics` | 1000 | Fog-of-war minimap showing visited rooms, player dot, exit dot (above everything) |

  Items at depth 5, enemies at depth 60, and furniture at depth 50 are all hidden by the darkness overlay and only revealed by the flashlight. Bullets at depth 150 are always visible (above darkness), giving visual feedback even when firing into dark areas. The HUD uses `setScrollFactor(0)` so it does not move with the camera
- **Darkness system architecture**: the BitmapMask has `invertAlpha = true`, so white pixels in `maskGraphics` punch transparent holes in `darkGraphics`, revealing the room beneath. The `updateDarkness(time)` method first draws lit room rectangles (from `getLitRoomIds()`, which always includes room 0 since it is permanently lit) into both `maskGraphics` (white, to punch holes in darkness) and `lightGraphics` (faint yellow tint). Then it calls `getFlashlightConeAngle()` each frame to get a battery-adjusted cone angle and `shouldFlicker()` to determine if the flashlight should momentarily go dark. When the cone angle is zero (battery depleted) or flickering is active, the method returns early after drawing lit rooms but without drawing the flashlight polygon. Lit rooms remain visible even when the flashlight is dead -- they use the same BitmapMask mechanism but draw rectangles instead of the flashlight polygon
- **Enemy physics uses a dynamic group** (unlike walls and furniture which are static groups). Enemies collide with walls, furniture, and each other. Player-enemy interaction uses `overlap` (not `collider`), triggering `onEnemyContact(player, enemySprite)` which looks up the specific enemy state object to read its per-type `contactDamage` (falling back to `this.scaledEnemyDamage` if the state is not found), applies it via `applyDamage()`, triggers camera shake/flash, and checks for player death
- **Player bullet physics group** uses pooling (`maxSize: 30`). Bullet-wall/furniture uses `collider` (instant deactivation). Bullet-enemy uses `overlap` -- on hit, the bullet is deactivated and the enemy takes damage read from per-bullet data (`bullet.getData('damage')`, falling back to `this.bulletDamage`). Each bullet stores its origin, damage, and range via Phaser's `setData()` at spawn time in `spawnBullet()`. This per-bullet approach ensures that bullets fired with one weapon retain their stats even after the player switches weapons mid-flight. `fireBullet()` selects the correct texture (`bullet_{weaponId}`) based on the active weapon and dispatches to either `calculateShotgunSpread()` (for multi-pellet weapons with `bulletCount > 1`) or `calculateBulletVelocity()` (for single-bullet weapons). Fire cooldown resets to `this.fireRate` and range expiry checks per-bullet range -- both derived from the active weapon via `updateWeaponStats()`. **Enemy bullet physics group** (`enemyBulletGroup`) mirrors this pattern with `maxSize: 30`, using wall/furniture/door colliders and a player overlap callback (`onEnemyBulletHitPlayer`). Enemy bullets use `SPITTER_PROJECTILE_SPEED` and `SPITTER_PROJECTILE_RANGE` constants directly (no upgrades). Enemy projectile damage is blocked while the player is hiding
- **Player death flow**: when `combatState.isDead` becomes true, `onPlayerDeath()` stops the player body, disables physics, triggers a 500ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: 0 }`. The death penalty is losing all treasure collected that run -- gold already in the shop balance is not affected. The `update()` loop early-returns while dead, preventing further input processing
- **Stair teleportation flow**: when the player overlaps a stair zone, `onStairEnter()` sets `this.isTeleporting = true`, stops and disables the player body, triggers a 300ms camera fade-out. On fade-out complete, the player is teleported (`body.reset(destX, destY)`) to the matching stair endpoint on the other floor, body is re-enabled, `this.currentFloor` is updated to `destFloor`, and a 300ms fade-in begins. `isTeleporting` is cleared on fade-in complete. The `update()` loop early-returns while `isTeleporting` is true, freezing all game logic (enemies, bullets, battery drain) during the transition. This follows the same camera-fade + boolean-gate pattern as `dayEnding`
- **Day-complete flow (exit)**: when the player overlaps the exit zone, `onDayComplete()` sets `this.dayEnding = true`, stops the player, triggers a 1000ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: this.inventoryState.treasureValue }`. The player keeps all collected treasure. The `update()` loop early-returns while `dayEnding` is true, freezing game state during the transition
- **Visual feedback for combat, battery, hiding, inventory, and weapons**: taking damage triggers `cameras.main.shake()` and `cameras.main.flash()`. `drawPlayer()` selects from three visual states in priority order: hiding (muted green, alpha 0.3), invulnerability (red tint, alpha 0.5), or normal (green, alpha 1). The HUD health bar turns red when below 30% HP. The battery bar sits below the health bar and transitions from yellow to orange (at 50%) to red (at 25%). Below the bars, text elements display the current battery item count (`BAT xN`), accumulated treasure value (`$N`), and inventory capacity (`[itemCount/maxItems]` in gray, turning red when inventory is full). A weapon text element shows the active weapon name in the weapon's color, with a `[Q]` indicator when a second weapon is available for switching
- **E-key interaction resolution**: `updateInteractPrompt()` runs each frame and finds the nearest door, nearest switch, nearest hideable furniture, and nearest weapon pickup (within `WEAPON_PICKUP_RANGE` of 80px) independently. All candidates are collected into a candidates array, sorted by Euclidean distance to the player, and the closest one wins. The resolved interactable is stored in `this.nearestInteractable` (with `type` and `item` fields), and E-key press dispatches to `onToggleDoor()`, `onToggleSwitch()`, `enterHiding()`, or `onWeaponPickup()` based on the type. Pressing E while already hiding exits hiding regardless of nearby interactables. This candidates-array approach is extensible for future interactable types
- **Hiding mechanics**: when the player presses E near `canHide` furniture (tables and desks), `enterHiding()` from `@/src/systems/hiding.js` sets `this.hidingState.isHiding` to true. While hidden: movement is disabled (velocity set to 0), shooting is disabled, right-click battery use is disabled, the player is immune to enemy contact damage (`onEnemyContact()` early-returns), and enemies cannot detect the player (the `playerHidden` flag forces `canSee = false` in `updateEnemyAI()`). The player graphic renders semi-transparent (alpha 0.3) with muted green colors. The player exits hiding by pressing any WASD key or pressing E again
- **Exploration and minimap**: `GameScene` maintains `this.explorationState` (from `@/src/systems/exploration.js`) which tracks visited rooms as an immutable Set. Each frame, `updateExploration()` checks the player's position against all rooms and adds newly visited rooms to the set. `drawMinimap()` calls `getFloorBounds()` from `@/src/systems/stairs.js` to get the bounding box for only the current floor, then passes that plus `this.currentFloor` as the `floorFilter` to `getMinimapData()` -- this ensures the minimap only shows rooms on the player's current floor, not the entire multi-floor level. The minimap uses a separate `minimapGraphics` Graphics object (not the same as `hudGraphics`) so it can be cleared and redrawn independently. The exit position (`this.exitPosition`) is stored during `createExitZone()` and passed to `getMinimapData()` so the exit dot is always visible on the minimap regardless of fog-of-war state
- **Closable door toggle flow**: when the player presses E near a closable door, `onToggleDoor()` does three things: (1) if the door was closed, finds and splices its segment out of `wallSegments` via `indexOf`; (2) calls `toggleDoor()` from `doors.js` to produce new immutable door states; (3) enables/disables the Phaser physics body on the door zone. This ensures physics, raycasting, and enemy LOS all update atomically. Player, enemy, and bullet groups all have colliders against `doorGroup`, so closed doors block all three entity types
- **Light switch toggle flow**: when the player presses E near a switch, `onToggleSwitch()` calls `toggleSwitch()` from `lightswitch.js` to produce new immutable switch states. If the switch was toggled ON, it finds all enemies currently in the switch's room (via `isPointInRoom()`) and despawns them (deactivate sprite, stop body, remove from `enemyStates`). On subsequent frames, `updateEnemies()` checks `getLitRoomIds()` and freezes (zero velocity, skip AI) any enemy that is inside a lit room. This means enemies are despawned when the light first turns on, and any enemy that later wanders into the lit room is frozen but not destroyed
- **Starting room (room 0) as furniture store**: room 0 is visually distinct from all other rooms. It uses warm wood-tone colors (`STORE_FLOOR_COLOR`, `STORE_WALL_COLOR`) from `@/src/systems/startroom.js` instead of default grays, and hand-crafted store furniture (from `generateStoreLayout()`) instead of randomly generated furniture. Room 0 is permanently lit -- its ID is injected into the `litRoomIds` array in both `updateDarkness()` and `updateEnemies()`, reusing the same lit-room mechanism as light switches. This means room 0 has no darkness overlay, and any enemies that wander in are frozen (same behavior as a room with a light switch turned on). The exit zone is at the center-north of the room (representing the store entrance), positioned by `getExitPosition()`. A crack-in-the-wall visual at the first doorway of room 0 is drawn by `createCrackVisual()` with a pulsing glow updated each frame by `updateCrackGlow()`
- **Multi-room physics with door gaps**: `createRoomPhysics()` splits each wall into physics zones that skip over door offsets. This mirrors the raycasting segments produced by `createRoomWalls()` so that physics collision and flashlight occlusion agree on where doors are. Closable doors fill these gaps with their own physics zones that can be toggled
- **Right-click input**: `setupInput()` disables the browser context menu (`this.input.mouse.disableContextMenu()`) and listens for `pointerdown` events. Right-click triggers `onUseBattery()`, which calls `useBattery()` from `inventory.js` and, if successful, `rechargeBattery()` from `battery.js`. This input is guarded by `!isDead && !dayEnding` checks
- **Item pickup flow**: items are a static physics group with player overlap detection. On overlap, `onItemPickup()` first checks `canPickupItem(this.inventoryState)` from `@/src/systems/inventory.js` -- if the inventory is full, the function early-returns and the item remains on the ground for later pickup. Otherwise, it updates `this.inventoryState` via `pickupItem()` and destroys the item sprite. Items store their type and value via Phaser's `setData()` on the sprite. Items do not produce wall segments -- they are collectibles, not occluders
- **Wall dual-purpose**: walls exist as both Phaser Arcade Physics static zones (for player collision) and as line segments from `createRoomWalls()` (for raycasting). Light-blocking furniture (shelves, bookcases) and closable doors follow the same pattern -- each is both a physics zone and a raycasting segment. Non-blocking furniture (tables, desks) still has physics zones but does not produce raycasting segments (light passes over them). Maze internal walls also follow this dual pattern: they produce both physics zones and raycasting segments. Doors are unique in that they can toggle between active/inactive at runtime via `body.enable` and `wallSegments` push/splice. Enemies, items, and light switches do not produce segments; enemies are dynamic actors, items are small collectibles, and light switches are proximity-based interactables with no physics body
- **Player rendering**: the player is a Graphics object (`playerGraphics`) drawn at depth 200 (above the darkness overlay at depth 100), so the player is always visible regardless of flashlight direction
- Camera and physics bounds are computed dynamically from the level's room positions, so they adapt to however many rooms the level generator produces
