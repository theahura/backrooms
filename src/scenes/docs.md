# Noridoc: scenes

Path: @/src/scenes

### Overview
- Phaser Scene subclasses that wire pure game systems to Phaser's rendering, physics, and input APIs
- Contains two scenes: `GameScene` (gameplay: movement, flashlight/darkness, enemy AI, combat, healing via medkit pickup, shooting, weapon switching/pickups, battery drain, item collection, lore note collection, closable doors, light switches, furniture hiding, exploration minimap, multi-floor stairs with teleportation, location-themed starting room, alternate exit portals, exit condition, HUD, desktop keyboard/mouse and mobile twin-stick touch controls, procedural audio with ambient drone and event-driven sound effects) and `ShopScene` (between-run upgrade shop: displays gold balance, upgrade categories with buy buttons, starting location selector, lore journal overlay, "Enter Backrooms" button, finger-sized tap targets for mobile, and UI sound feedback)
- The two scenes alternate in a loop: GameScene -> ShopScene (on exit or death) -> GameScene (on "Enter Backrooms")

### How it fits into the larger codebase
- `@/src/main.js` registers scene classes with the Phaser.Game instance; GameScene is listed first (initial scene)
- Scenes import pure functions from `@/src/systems/` for game math (including maze generation from `@/src/systems/maze.js`, room visual theming from `@/src/systems/roomThemes.js`, lore spawning from `@/src/systems/lore.js`, location definitions/alternate exit spawning from `@/src/systems/locations.js`, BFS room graph pathfinding from `@/src/systems/pathfinding.js`, BFS entrance-distance loot scaling from `@/src/systems/distance.js`, time-based danger escalation from `@/src/systems/danger.js`, light spill gradient zones from `@/src/systems/lightSpill.js`, audio config and per-enemy-type sound logic from `@/src/systems/audio.js`, sound synthesis from `@/src/systems/audioEngine.js`, pixel art sprite definitions and character animation definitions from `@/src/systems/sprites.js`) and handle all Phaser-specific concerns: sprite creation, animation playback, physics bodies, input binding, camera, collision callbacks, Graphics drawing, and Web Audio integration
- Scenes are the only place in the codebase that depends on Phaser -- this boundary is intentional to keep game logic testable
- `GameScene` reads upgrade levels from `game.registry` in its `init()` method, using `getUpgradeValue()` from `@/src/systems/shop.js` to compute upgrade bonuses as deltas from pistol base values (e.g., `getUpgradeValue('weaponDamage', level) - 25` yields a +N damage bonus). These bonuses are stored in `this.upgradeBonuses` and applied additively to whatever weapon is active via `getEffectiveStats()` from `@/src/systems/weapons.js`. Binary unlock upgrades (`startingPistol`, `minimap`) are checked as boolean flags: `startingPistol` determines whether `createWeaponState({ startingPistol: true })` or `createWeaponState()` (unarmed) is called; `minimap` sets `this.hasMinimap` which gates minimap Graphics creation and rendering. When unarmed, `getActiveWeapon()` returns null and firing is disabled until the player picks up a weapon. Survival stats (max HP, max battery, flashlight angle, speed) are still computed as absolute values. It also calls scaling functions from `@/src/systems/scaling.js` to compute run-scaled enemy stats per type -- each enemy type (basic, crawler, spitter) has its own base HP, contact damage, and chase speed constants, and scaling is applied independently to each set (e.g., `this.scaledEnemyHP`, `this.scaledCrawlerHP`, `this.scaledSpitterHP`). These values are stored as instance properties and used throughout `create()` and `update()`. Upgrade level fallback uses `createShopState().upgrades` as defaults, ensuring forward compatibility when new upgrade keys are added to saves that predate them
- `ShopScene` reads/writes `shopState` from `game.registry`, using pure functions from `@/src/systems/shop.js` for all purchase logic. It receives `treasureEarned` and optionally `newLocation` as scene data from GameScene, converts treasure to gold via `addGold()`, and if `newLocation` is provided, adds it to `unlockedLocations` in the registry. ShopScene also calls `saveGame()` from `@/src/systems/persistence.js` at every point where state is mutated: after adding treasure gold in `init()`, before entering backrooms, and after purchasing an upgrade. All `saveGame()` calls pass `collectedLore`, `unlockedLocations`, and `activeLocation` from `game.registry` to ensure all cross-run progress is persisted. ShopScene imports `getLocation()` and `LOCATIONS` from `@/src/systems/locations.js` to display location names in the location selector UI. ShopScene imports `getJournalEntries()` and `getJournalPage()` from `@/src/systems/lore.js` to power the lore journal overlay, which lets players review collected and uncollected lore notes between runs. ShopScene imports `generateAllSounds()` from `@/src/systems/audioEngine.js` to initialize procedural audio, playing UI feedback sounds for purchases, denied purchases, and button clicks
- `GameScene` calls `saveGame()` from `@/src/systems/persistence.js` after incrementing `runCount` in `init()`, persisting the updated run counter, shop state, and location data to localStorage before the run begins
- Data flows between scenes via two mechanisms: `game.registry` for in-memory cross-scene state (shopState, runCount, unlockedLocations, activeLocation) and scene `init(data)` for per-transition data (treasureEarned, newLocation). Registry values are persisted to localStorage via `@/src/systems/persistence.js` so state survives browser reloads

### Core Implementation
- **GameScene lifecycle**: `init()` reads upgrade state from registry and computes per-run stats, `create()` sets up the level and all game objects, `update()` runs every frame
- **GameScene.init()**: reads `shopState`, `collectedLore`, `activeLocation`, and `unlockedLocations` from `game.registry` (converting `collectedLore` from a persisted array to a Set). Resolves the active location data via `getLocation()` from `@/src/systems/locations.js`, falling back to `'store'` if the location ID is not found. Spreads `createShopState().upgrades` as defaults before overlaying saved upgrade levels (for forward compatibility with saves predating new upgrades), then calls `getUpgradeValue()` for each upgrade to set survival stats (`this.maxCharge`, `this.flashlightAngle`, `this.maxHp`, `this.playerSpeed`), weapon upgrade bonuses (`this.upgradeBonuses = { damage, fireRate, range }` computed as deltas from pistol base values), and inventory capacity (`this.maxItems` from the backpack upgrade). Checks the `startingPistol` upgrade level and initializes `this.weaponState` via `createWeaponState({ startingPistol })` from `@/src/systems/weapons.js` -- when the upgrade is not purchased the player starts unarmed with empty slots and zero ammo. Sets `this.hasMinimap` from the `minimap` upgrade level (binary flag: true when level > 0) -- this gates both `minimapGraphics` creation in `createHUD()` and `drawMinimap()` rendering. Calls `this.updateWeaponStats()` to derive effective bullet stats from the active weapon plus bonuses. Also reads and increments `runCount` in the registry to derive a unique level seed for this run and stores `this.runCount` for use in `create()` (where it determines floor count via `getFloorRoomCounts()` from `@/src/systems/stairs.js`), and calls scaling functions from `@/src/systems/scaling.js` to precompute per-type scaled stats. Initializes `this.dangerState` via `createDangerState()` from `@/src/systems/danger.js` -- the danger state is per-run and not persisted. After incrementing, calls `saveGame()` to persist the updated `runCount`, `shopState`, `collectedLore`, `unlockedLocations`, and `activeLocation` to localStorage
- **create() setup sequence**: `generateSpriteTextures()` runs first, iterating all sprite keys from `@/src/systems/sprites.js` and calling `this.textures.generate()` for each, then iterating all animation keys from `ANIM_DEFS` and calling `this.anims.create()` for each -- this pre-generates all pixel art textures and animations before any other creation method references them. Then initializes level geometry (including per-room internal structure via `createRoomMaze()`), doors, switches, player, enemies, items, lore items, player bullets, enemy bullets, weapon pickups, exit zone, stairs, alternate exits, crack visual, input, camera, darkness overlay, HUD, and audio. Door creation happens after rooms but before player creation so that door colliders are ready. Switch creation happens after doors. Stairs and alternate exits are created after items. `createTouchControls()` is called between HUD and audio setup and is a no-op on desktop (it early-returns when `this.touchMode` is false), so it only builds the touch UI on touch devices. Audio is initialized last via `createSounds()`. Combat state is initialized with `createCombatState(this.maxHp)`, battery state with `createBatteryState(this.maxCharge)`, inventory state with `createInventoryState(this.maxItems)` (where `maxItems` comes from the backpack upgrade computed in `init()`), hiding state with `createHidingState()` from `@/src/systems/hiding.js`, and exploration state with `createExplorationState()` from `@/src/systems/exploration.js`. Two additional flags are initialized: `this.isTeleporting = false` (gates update loop during stair transitions) and `this.currentFloor = 0` (tracks which floor the player is currently on). Pathfinding state is initialized with `this.roomGraph = null`, `this.fullRoomGraph = null`, and `this.roomGraphDirty = true`, so both room graphs (standard and full) are lazily built on the first `updateEnemies()` call. Notable setup details:
  - `createEnemies()` passes `this.extraEnemyCount` and `this.runCount` to `generateRoomEnemies()` for run-scaled enemy density and type distribution. `getEnemyStats(type)` maps each spawn's type to the correct scaled HP, contact damage, texture key (via `getEnemyTextureKey()` from `@/src/systems/sprites.js`), and body size. Each enemy sprite plays its type-specific idle animation after creation. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, `spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`, and `soundCooldown` fields alongside the existing state machine fields. The room-tracking fields (`spawnRoomId`, `currentRoomId`) are initialized to the spawning room's ID; `roomTransitionCooldown` starts at 0 and `targetDoorway` starts null. `soundCooldown` is stagger-initialized per enemy to `1000 + Math.random() * 3000` so enemies do not emit ambient sounds simultaneously. Wires a player-enemy overlap callback (`onEnemyContact`) for per-enemy contact damage
  - `createItems()` creates a static physics group, computes per-room hop-distances from the entrance once via `computeRoomDistances(this.level.rooms, this.level.stairs)` from `@/src/systems/distance.js`, then calls `generateRoomItems()` per room -- passing `roomDistances.get(room.id) ?? 0` as the trailing `distance` argument so loot tiers scale with exploration depth (deeper rooms favor higher-value treasure; gems only appear far from the entrance). Places item sprites at depth 5 (below darkness) using pre-generated textures keyed as `item_{type}` from `@/src/systems/sprites.js` (including `item_medkit` for medkits). Wires a player-item overlap callback (`onItemPickup`) for auto-collection, where medkits are handled as a special case before the inventory capacity check (see Item pickup flow)
  - `createLoreItems()` creates a separate static physics group (`this.loreGroup`), calls `generateRoomLore()` from `@/src/systems/lore.js` per room (passing `this.collectedLore` to filter out already-collected entries), and places lore sprites at depth 5 using the pre-generated `lore_note` texture from `@/src/systems/sprites.js`. Wires a player-lore overlap callback (`onLorePickup`) that adds the entry ID to `this.collectedLore`, updates `game.registry`, destroys the sprite, and calls `showLorePopup()` to display the text. Lore items do NOT consume inventory space -- they bypass the inventory system entirely. `showLorePopup()` displays a tween-animated semi-transparent overlay at the bottom of the screen (depth 1001-1002) with the lore text in monospace font, auto-dismissing after 5 seconds with a fade-out. Only one popup can be active at a time (previous popup is destroyed if a new one triggers)
  - `createDoors()` calls `createDoorStates()` from `@/src/systems/doors.js` to determine which connections have closable doors. For each closable door, creates a static physics zone in `doorGroup` sized to fill the doorway gap and a Phaser sprite using the pre-generated `door_closed` texture (stored in `this.doorSprites` Map keyed by door ID, with `setDisplaySize()` to match the doorway dimensions). Closed doors have their segment pushed into `wallSegments`, physics body enabled, and sprite visible; open doors have `body.enable = false` and sprite hidden. Also creates a "Press E" interaction prompt text element (hidden by default)
  - `createSwitches()` calls `createSwitchStates()` from `@/src/systems/lightswitch.js` to place light switches in ~15% of non-starting rooms. For each switch, creates a Phaser sprite using the pre-generated `switch_off` or `switch_on` texture at depth 50, stored in `this.switchSprites` Map keyed by switch ID. Switches have no physics bodies -- they are purely visual objects interacted with via proximity checks
  - `createBullets()` creates a pooled physics group (`maxSize: 30`) and wires bullet-wall/furniture/door colliders and bullet-enemy overlap. Bullet textures (keyed `bullet_{weaponId}`) are pre-generated by `generateSpriteTextures()` from definitions in `@/src/systems/sprites.js`, so `fireBullet()` can select the correct texture based on the active weapon
  - `createWeaponPickups()` calls `generateLevelWeapons()` from `@/src/systems/weapons.js` to place weapon pickups using pre-generated pickup textures (keyed `pickup_{weaponId}`) from `@/src/systems/sprites.js` -- this guarantees one weapon spawn per floor (selecting a random non-starting room per floor). All weapon types including pistol can appear as floor drops. Pickup sprites are stored in `this.weaponPickupSprites` and participate in the E-key interaction resolution system as candidates with type `'weapon'`
  - `createEnemyBullets()` creates a separate pooled physics group (`enemyBulletGroup`, `maxSize: 30`), mirroring the player bullet pattern. The `enemyBullet` texture is pre-generated by `generateSpriteTextures()`. Wires wall/furniture/door colliders and player overlap. `fireEnemyBullet()` is called from `updateEnemies()` when a spitter's AI state machine sets `wantsToFire = true`. `updateEnemyBullets()` expires enemy bullets using `SPITTER_PROJECTILE_RANGE`. `onEnemyBulletHitPlayer()` applies `SPITTER_PROJECTILE_DAMAGE` to the player's combat state (blocked while hiding)
  - `createExitZone()` places a 64x64 physics zone at the west-center of room 0 (position computed by `getLocationExitPosition()` from `@/src/systems/locations.js`) with a green visual indicator, wiring a player overlap callback (`onDayComplete`) for the day-cycle win condition. Also stores `this.exitPosition` for use by the minimap
  - `createCrackVisual()` draws a jagged crack line at the first doorway of room 0 using points from `generateCrackPoints()` in `@/src/systems/startroom.js`. The crack consists of a primary dark line (depth 2) and a thinner secondary line with a slight offset for a 3D effect. A glow rectangle (depth 1) behind the crack pulses with a sickly-yellow color (0xd4c073) via `updateCrackGlow()` each frame using a sine wave
  - `createStairs()` iterates over `this.level.stairs` (up to 2 stair connections per adjacent floor pair, across all floor pairs generated by `getFloorRoomCounts(this.runCount)`). For each stair connection, draws a dark rectangle with step lines at both endpoints, using `stair.fromFloor` and `stair.toFloor` to determine visual direction ('down' when going to a higher floor, 'up' when going to a lower floor). Each endpoint gets a Phaser physics zone (`STAIR_SIZE` x `STAIR_SIZE`) with `destX`, `destY`, and `destFloor` data properties. Overlap with the player triggers `onStairEnter()`. A separate `stairGlowGraphics` at depth 1 draws a pulsing purple glow (0x7733aa) behind both stair positions each frame via `updateStairGlow(time)`, using the same sine-wave pulse pattern as the crack glow
  - `createAlternateExits()` calls `generateAlternateExit()` from `@/src/systems/locations.js` to potentially spawn a glowing alternate exit portal on a deeper floor (floor > 0). If an exit is generated (20% chance per run, only on floors with non-stair rooms, only when locked locations remain), a 60x60 blue-white physics zone is placed at the portal position with a label showing the destination location name. A pulsing blue glow (0x88aaff) is animated each frame via `updateAlternateExitGlow(time)`. When the player overlaps the zone, `onAlternateExitComplete()` adds the location ID to `unlockedLocations` in the registry, sets `activeLocation` to the new location, and calls `onDayComplete(locationId)` to transition to ShopScene with the `newLocation` data
  - `createHUD()` creates a Graphics object with `setScrollFactor(0)` at depth 1000 so health and battery bars stay fixed on screen, plus text elements for battery count, treasure value, inventory capacity (`[itemCount/maxItems]` -- gray normally, turns red when full), active weapon name (colored per-weapon with a `[Q]` switch indicator), and a red "DANGER N" indicator (visible only when danger waves have been spawned, showing the current wave count from `this.dangerState.waveCount`). The `minimapGraphics` Graphics object (depth 1000, `setScrollFactor(0)`) is only created if `this.hasMinimap` is true (i.e., the minimap shop upgrade has been purchased). A floor indicator text (`this.floorText`) shows the current floor label (e.g., "B1", "B2", up to "B4" on run 4+) to indicate the player's current floor, updated each frame in `drawMinimap()`
- **update() frame loop**: early-returns if player is dead, `this.dayEnding` is true (exit transition in progress), or `this.isTeleporting` is true (stair transition in progress). Then: refreshes `pointer.worldX`/`worldY` via `pointer.updateWorldPoint(this.cameras.main)` so world coordinates stay current even when the camera moves without mouse input -> computes the mouse pointer angle and, in touch mode, reads the aim-stick angle/force, then sets `this.playerAngle` via `resolveAimAngle()` from `@/src/systems/touchControls.js` (mouse on desktop; aim-stick angle while deflected, else last angle on touch) -> reads WASD key state and computes `keyboardVelocity` (at half speed while hiding via `HIDING_SPEED_MULTIPLIER`); in touch mode computes a force-scaled `stickVelocity` from the move stick via `getMoveVelocity()`, then picks between them via `resolveMoveVelocity()` -> sets player velocity -> ticks combat cooldown (`updateCombat`), battery state (`updateBattery`), and fire cooldown (`updateFireCooldown`) -> determines fire intent via `resolveFireIntent()` (left-click on desktop; the dedicated FIRE button, polled via `isFireHeld()`, on touch -- a generic screen touch never fires) then checks `canFire()` + `getActiveWeapon()` non-null + `hasAmmo()` to fire bullets (firing calls `consumeAmmo()` to decrement ammo; blocked while hiding or unarmed or out of ammo) -> `updateInteractPrompt()` resolves the nearest interactable (door, switch, hideable furniture, or weapon pickup by Euclidean distance) and shows/hides "Press E" -> `updateTouchButtons()` shows/hides the on-screen USE/WPN buttons and dims FIRE based on game state (touch only) -> checks Q key (`JustDown`) or scroll wheel (with 200ms cooldown) to switch weapons via `triggerWeaponSwitch()` -> checks E key (`JustDown`) and dispatches via `triggerInteract()` (door/switch/furniture/weapon, or `onExitHiding()` while already hiding) -> `updateBullets()` expires out-of-range player bullets (using per-bullet range from `getData('range')`) -> `updateEnemyBullets()` expires out-of-range enemy projectiles -> `updateEnemies(delta)` first updates the danger system: accumulates unsafe time via `updateDangerTime()` (safe rooms are room 0 and lit rooms), then spawns danger waves via a `while (shouldSpawnWave())` loop that calls `spawnDangerWave()` and `advanceWave()` for each triggered wave -- multiple waves can fire in a single tick if enough time has accumulated. Then runs AI per enemy (with lit-room freeze check, `this.hidingState.isHiding` passed as the `playerHidden` flag, and type-specific chase speeds via `getScaledChaseSpeed()`; after AI update, calls `getEnemySoundEvent()` from `@/src/systems/audio.js` to detect state transitions (alert sounds on idle->chase) and ambient sound cooldown expiry (per-type ambient sounds like zombie_growl, crawler_skitter, spitter_gurgle), playing any triggered sounds via `playSoundAtDistance()` for proximity-based volume attenuation; checks `wantsToOpenDoor` flag to call `onToggleDoor()` for crawlers; checks `wantsToFire` flag to call `fireEnemyBullet()` for spitters and plays `getEnemyFireSound()` at distance; also sets enemy sprite rotation from velocity direction via `Math.atan2` and switches animation via `getAnimKeyForState()` based on current AI state) -> redraws player (animation switching + tint/alpha for visual state) -> recomputes darkness (flashlight cone + lit room rectangles drawn into BitmapMask) -> `updateCrackGlow(time)` animates the crack glow pulse -> `updateStairGlow(time)` animates the stair glow pulse -> `updateAlternateExitGlow(time)` animates the alternate exit glow pulse -> `drawDoors()` toggles door sprite visibility (visible when closed, hidden when open) -> `drawSwitches()` swaps switch sprite textures (`switch_on`/`switch_off`) -> updates exploration state via `updateExploration()` (tracking visited rooms) -> redraws HUD (health bar + battery bar + battery count + treasure value + weapon name) -> `drawMinimap()` renders the fog-of-war minimap (using per-floor bounds and floor filter) -> audio timers: footstep sounds play at velocity-dependent intervals via `getFootstepInterval()` from `@/src/systems/audio.js`, and a battery warning beep plays every 3 seconds when battery charge is at or below 25% (non-zero). Right-click input triggers `onUseBattery()` outside the frame loop via a `pointerdown` event listener (blocked while hiding). Item pickup happens via Phaser overlap callbacks (auto-collected on contact)
- **Enemy integration pattern**: `GameScene` maintains a parallel `enemyStates` array alongside the Phaser physics sprites in `enemyGroup`. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, room-tracking fields (`spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`), door interaction fields (`doorOpenTimer`, `targetDoorId`), `soundCooldown` (for per-enemy ambient sound timing), and standard AI state machine fields. Enemies are added to `enemyStates` both at level creation (via `createEnemies()`) and dynamically during gameplay (via `spawnDangerWave()`, which creates new sprites in `enemyGroup` and pushes new state objects with the same structure, including stagger-initialized `soundCooldown`). Dynamic spawns automatically inherit existing colliders because Phaser group colliders cover all current and future group members. Each frame, `updateEnemies()` rebuilds both room graphs if dirty (standard graph via `buildRoomGraph()` and full graph via `buildFullRoomGraph()` from `@/src/systems/pathfinding.js`, cached and invalidated only when `onToggleDoor()` sets `roomGraphDirty = true`), runs BFS on both graphs from the player's current room, counts enemies per room, detects room transitions (updating `currentRoomId` and resetting cooldown/targetDoorway when an enemy enters a new room), evaluates idle doorway-seeking eligibility (gated by `DOORWAY_SEEK_CHANCE`, cooldown, `MAX_ROOM_DISTANCE` from spawn, and `MAX_ROOM_ENEMIES` cap), and constructs a `navContext` object per enemy. Crawlers use the full graph's BFS results for doorway routing (so they can path through closed doors), while basic/spitter enemies use the standard graph. For crawlers, `getClosedDoorOnPath()` is called to detect if a closed door sits on the next BFS step, and the result is included in `navContext.closedDoorOnPath`. The `navContext` carries `enemyRoomId`, `playerRoomId`, `doorway`, `seekDoorway`, `closedDoorOnPath` (crawler-only, null for other types), and the enemy's room-tracking state. This navContext is passed as the 7th argument to `updateEnemyAI()`. After the AI update, room-tracking fields (`roomTransitionCooldown`, `targetDoorway`) and door interaction fields (`doorOpenTimer`, `targetDoorId`) are copied back from the result. If `wantsToOpenDoor` is true on the result, GameScene calls `onToggleDoor(targetDoorId)` to open the door (guarded by a check that the door is still closed). This mirrors the spitter `wantsToFire` pattern -- pure function signals intent, scene executes side effect. After AI and side-effect processing, each enemy sprite's rotation is set from its velocity direction (`Math.atan2`), and its animation is switched via `getAnimKeyForState(type, state)` from `@/src/systems/sprites.js` -- this maps the AI state (idle/chase/search/attack) to the correct animation key (e.g., chase -> walk animation, attack -> spitter attack animation). The `play(key, true)` call with `ignoreIfPlaying=true` prevents animation restarts when the state hasn't changed. On death, `enemySprite.stop()` freezes the current animation frame before applying corpse visuals. This keeps the AI logic testable in `@/src/systems/enemy.js` while letting Phaser handle physics resolution, animation playback, and projectile management
- **ShopScene lifecycle**: `init(data)` receives `treasureEarned` and optionally `newLocation` from GameScene, reads existing `shopState` from registry (or creates fresh state on first visit), adds the treasure as gold via `addGold()`, writes back to registry. If `newLocation` is provided and not already unlocked, it is added to `unlockedLocations` in the registry. Calls `saveGame()` to persist all state including location data. Also reads `collectedLore` from the registry (as an array) and converts it to a Set for use by the journal. `create()` first calls `fitShopCamera()` to zoom-fit the design space into the viewport (see below), then builds the UI: title, treasure earned display, gold balance, upgrade rows, location selector (when multiple locations are unlocked), a "NOTES X/20" journal button (top-right), and "Enter Backrooms" button. If a `newLocation` was just discovered, a "New location discovered!" message is shown. When multiple locations are unlocked, `<` and `>` buttons let the player cycle through unlocked starting locations via `cycleLocation()`, which updates `activeLocation` in the registry. Each upgrade row shows name, description, level dots (filled/empty circles), next cost, and a buy button. Purchases call `purchaseUpgrade()`, write back to registry, and call `saveGame()` to persist. The "Enter Backrooms" button writes shopState and activeLocation to registry, calls `saveGame()`, and then calls `this.scene.start('GameScene')` to transition back to gameplay. The save-at-every-mutation pattern ensures no progress is lost even if the browser crashes between user actions. The **lore journal** is a full-screen overlay activated by clicking the "NOTES X/20" button. `showJournal()` creates a background Graphics at depth 2000 covering the full screen (1024x768), with all text elements at depth 2001 to layer above all shop content. The journal displays entries paginated at 5 per page via `getJournalPage()` from `@/src/systems/lore.js`. Each entry shows "Note #N" with the lore text (if collected) or "???" (if uncollected), using color to distinguish collected (white/gray) from uncollected (dark gray). Prev/Next buttons navigate pages; `getJournalPage()` handles page clamping so out-of-bounds page indices are safely corrected. A close button and ESC key handler (via `input.keyboard.on('keydown-ESC')`) call `hideJournal()`, which destroys all overlay objects. The journal is read-only -- it does not modify any game state

### Things to Know
- **Depth layering order** controls what is visible through the darkness:

  | Layer | Depth | Purpose |
  |-------|-------|---------|
  | crack glow / stair glow / alternate exit glow graphics | 1 | Pulsing sickly-yellow glow behind the crack in room 0; pulsing purple glow behind stair zones; pulsing blue glow behind alternate exit portals |
  | crack line / stair step / alternate exit border graphics | 2 | Jagged crack-in-the-wall lines at the first doorway of room 0; dark rectangles with step lines at stair positions; blue border and location label for alternate exit portals |
  | item sprites / enemy corpses | 5 | Collectibles and dead enemy corpses, hidden by darkness (only visible in flashlight). Corpses are inactive sprites with gray tint, faded alpha, and toppled angle |
  | exit zone graphics | 10 | Green exit indicator visible under darkness |
  | furniture sprites (with Graphics fallback) | 50 | Furniture rendered below enemies; standard types use pixel art sprites, location-specific types (e.g., counter, couch) fall back to Graphics |
  | door sprites | 50 | Closed doors rendered as pixel art sprites (visibility toggled, not redrawn per frame) |
  | switch sprites | 50 | Light switches rendered as pixel art sprites (texture swapped between `switch_on`/`switch_off`, not redrawn per frame) |
  | enemy sprites | 60 | Enemies hidden by darkness (below mask) |
  | `lightGraphics` | 99 | Faint yellow tint drawn under the flashlight cone |
  | `darkGraphics` | 100 | Black rectangle covering the entire level (the darkness) |
  | player bullet sprites | 150 | Player bullets visible above darkness but below player |
  | enemy bullet sprites | 150 | Enemy (spitter) projectiles, same depth as player bullets |
  | player sprite | 200 | Player always visible (above darkness); animated pixel art sprite with tint/alpha for visual states |
  | `hudGraphics` + HUD text | 1000 | Health bar, battery bar, battery count, treasure value, weapon name, danger indicator (above everything) |
  | `minimapGraphics` | 1000 | Fog-of-war minimap showing visited rooms, player dot, exit dot (above everything) |
  | lore popup background | 1001 | Semi-transparent black rounded rect for lore text overlay |
  | lore popup text | 1002 | Lore note text in monospace font (above popup background) |
  | touch control bases / thumbs / buttons | 1500-1501 | Virtual joystick bases, thumbs, FIRE button, and USE/WPN/BATT/FS tap buttons (touch devices only); `setScrollFactor(0)` so they stay fixed on screen (`TOUCH_UI_DEPTH = 1500`) |
  | journal overlay background | 2000 | Full-screen opaque black background for the ShopScene lore journal |
  | journal overlay text/controls | 2001 | Journal title, entry text, pagination buttons, close button (above journal background) |

  Items at depth 5, enemies at depth 60, and furniture at depth 50 are all hidden by the darkness overlay and only revealed by the flashlight. Bullets at depth 150 are always visible (above darkness), giving visual feedback even when firing into dark areas. The HUD uses `setScrollFactor(0)` so it does not move with the camera
- **Darkness system architecture**: the BitmapMask has `invertAlpha = true`, so white pixels in `maskGraphics` punch transparent holes in `darkGraphics`, revealing the room beneath. The `updateDarkness(time)` method first draws lit room rectangles (from `getLitRoomIds()`, which always includes room 0 since it is permanently lit) into both `maskGraphics` (white, to punch holes in darkness) and `lightGraphics` (faint yellow tint). After lit rooms, it computes light spill zones via `getLightSpillZones()` from `@/src/systems/lightSpill.js` and draws each zone as a gradient rectangle into both `maskGraphics` (using `fillGradientStyle` for per-corner alpha that creates gradient visibility through the BitmapMask) and `lightGraphics` (warm tint at 5% of the spill alpha). The gradient fades from full brightness at the doorway edge to zero over 150px (`LIGHT_SPILL_DEPTH`). Closed doors block spill entirely; both-lit room pairs are skipped (already fully revealed). Then it calls `getFlashlightConeAngle()` each frame to get a battery-adjusted cone angle and `shouldFlicker()` to determine if the flashlight should momentarily go dark. When the cone angle is zero (battery depleted) or flickering is active, the method returns early after drawing lit rooms and spill zones but without drawing the flashlight polygon. Lit rooms and their spill zones remain visible even when the flashlight is dead -- they use the same BitmapMask mechanism but draw rectangles instead of the flashlight polygon
- **Enemy physics uses a dynamic group** (unlike walls and furniture which are static groups). Enemies collide with walls, furniture, and each other. Player-enemy interaction uses `overlap` (not `collider`), triggering `onEnemyContact(player, enemySprite)` which looks up the specific enemy state object to read its per-type `contactDamage` (falling back to `this.scaledEnemyDamage` if the state is not found), applies it via `applyDamage()`, triggers camera shake/flash, and checks for player death
- **Player bullet physics group** uses pooling (`maxSize: 30`). Bullet-wall/furniture uses `collider` (instant deactivation). Bullet-enemy uses `overlap` -- on hit, the bullet is deactivated and the enemy takes damage read from per-bullet data (`bullet.getData('damage')`, falling back to `this.bulletDamage`). When an enemy dies, a per-type death sound is played via `playSoundAtDistance(getEnemyDeathSound(type), distance)` from `@/src/systems/audio.js` (e.g., zombie_death for basic, crawler_death for crawler, spitter_death for spitter), with volume attenuated by the enemy's distance from the player. The sprite is kept visible as a corpse: `setActive(false)` (removes from physics/overlap checks), `body.enable = false`, and corpse visual constants from `@/src/systems/combat.js` are applied (gray tint, faded alpha, toppled angle, depth 5). The sprite remains in `enemyGroup` but Phaser skips inactive sprites in overlap checks, so corpses cannot trigger contact damage. The enemy state is removed from `enemyStates`. Each bullet stores its origin, damage, and range via Phaser's `setData()` at spawn time in `spawnBullet()`. This per-bullet approach ensures that bullets fired with one weapon retain their stats even after the player switches weapons mid-flight. `fireBullet()` selects the correct texture (`bullet_{weaponId}`) based on the active weapon and dispatches to either `calculateShotgunSpread()` (for multi-pellet weapons with `bulletCount > 1`) or `calculateBulletVelocity()` (for single-bullet weapons). Fire cooldown resets to `this.fireRate` and range expiry checks per-bullet range -- both derived from the active weapon via `updateWeaponStats()`. **Enemy bullet physics group** (`enemyBulletGroup`) mirrors this pattern with `maxSize: 30`, using wall/furniture/door colliders and a player overlap callback (`onEnemyBulletHitPlayer`). Enemy bullets use `SPITTER_PROJECTILE_SPEED` and `SPITTER_PROJECTILE_RANGE` constants directly (no upgrades). Enemy projectile damage is blocked while the player is hiding
- **Player death flow**: when `combatState.isDead` becomes true, `onPlayerDeath()` stops the player body, disables physics, triggers a 500ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: 0 }`. The death penalty is losing all treasure collected that run -- gold already in the shop balance is not affected. The `update()` loop early-returns while dead, preventing further input processing
- **Stair teleportation flow**: when the player overlaps a stair zone, `onStairEnter()` sets `this.isTeleporting = true`, stops and disables the player body, triggers a 300ms camera fade-out. On fade-out complete, the player is teleported (`body.reset(destX, destY)`) to the matching stair endpoint on the other floor, body is re-enabled, `this.currentFloor` is updated to `destFloor`, and a 300ms fade-in begins. `isTeleporting` is cleared on fade-in complete. The `update()` loop early-returns while `isTeleporting` is true, freezing all game logic (enemies, bullets, battery drain) during the transition. This follows the same camera-fade + boolean-gate pattern as `dayEnding`
- **Day-complete flow (exit)**: when the player overlaps the exit zone or an alternate exit portal, `onDayComplete(newLocation)` sets `this.dayEnding = true`, stops the player, triggers a 1000ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: this.inventoryState.treasureValue, newLocation }`. When triggered by the normal exit zone, `newLocation` is null. When triggered by an alternate exit portal, `newLocation` is the unlocked location ID. The player keeps all collected treasure. The `update()` loop early-returns while `dayEnding` is true, freezing game state during the transition
- **Visual feedback for combat, battery, hiding, inventory, and weapons**: taking damage triggers `cameras.main.shake()` and `cameras.main.flash()`. `drawPlayer()` applies `setTint`/`setAlpha` for three visual states in priority order: hiding (green tint 0x336633, alpha 0.3), invulnerability (red tint 0xcc4444, alpha 0.5), or normal (no tint, alpha 1). The HUD health bar turns red when below 30% HP. The battery bar sits below the health bar and transitions from yellow to orange (at 50%) to red (at 25%). Below the bars, text elements display the current battery item count (`BAT xN`), accumulated treasure value (`$N`), and inventory capacity (`[itemCount/maxItems]` in gray, turning red when inventory is full). A weapon text element shows the active weapon name in the weapon's color, with a `[Q]` indicator when a second weapon is available for switching
- **E-key interaction resolution**: `updateInteractPrompt()` runs each frame and finds the nearest door, nearest switch, nearest hideable furniture, and nearest weapon pickup (within `WEAPON_PICKUP_RANGE` of 80px) independently. All candidates are collected into a candidates array, sorted by Euclidean distance to the player, and the closest one wins. The resolved interactable is stored in `this.nearestInteractable` (with `type` and `item` fields), and E-key press dispatches to `onToggleDoor()`, `onToggleSwitch()`, `onEnterHiding()`, or `onWeaponPickup()` based on the type. Pressing E while already hiding calls `onExitHiding()` regardless of nearby interactables. This candidates-array approach is extensible for future interactable types
- **Hiding mechanics**: when the player presses E near `canHide` furniture (any furniture type with `canHide: true` in `@/src/systems/furniture.js`), `onEnterHiding(furniture)` snaps the player to the furniture center via `body.reset()`, sets `body.setBoundsRectangle()` to constrain movement within the furniture area, enables `setCollideWorldBounds(true)`, and disables the player-furniture collider (`this.playerFurnitureCollider.active = false`) so physics does not push the player out. While hidden: the player moves at half speed (`HIDING_SPEED_MULTIPLIER = 0.5` from `@/src/systems/hiding.js`), shooting is disabled, right-click battery use is disabled, the player is immune to enemy contact damage (`onEnemyContact()` early-returns), and enemies cannot detect the player (the `playerHidden` flag forces `canSee = false` in `updateEnemyAI()`). The player graphic renders semi-transparent (alpha 0.3) with muted green colors. The player exits hiding only by pressing E, which calls `onExitHiding()` to restore `setCollideWorldBounds(false)` and re-enable the furniture collider
- **Exploration and minimap**: `GameScene` maintains `this.explorationState` (from `@/src/systems/exploration.js`) which tracks visited rooms as an immutable Set regardless of whether the minimap upgrade is purchased (future features may use visitation data). Each frame, `updateExploration()` checks the player's position against all rooms and adds newly visited rooms to the set. `drawMinimap()` early-returns if `this.hasMinimap` is false (the minimap shop upgrade has not been purchased -- cost 3000 gold, single tier). When enabled, it calls `getFloorBounds()` from `@/src/systems/stairs.js` to get the bounding box for only the current floor, then passes that plus `this.currentFloor` as the `floorFilter` to `getMinimapData()` -- this ensures the minimap only shows rooms on the player's current floor, not the entire multi-floor level. The minimap uses a separate `minimapGraphics` Graphics object (not the same as `hudGraphics`) so it can be cleared and redrawn independently. The exit position (`this.exitPosition`) is stored during `createExitZone()` and passed to `getMinimapData()` so the exit dot is always visible on the minimap regardless of fog-of-war state
- **Closable door toggle flow**: when the player (or a crawler enemy) triggers a door toggle, `onToggleDoor()` does four things: (1) if the door was closed, finds and splices its segment out of `wallSegments` via `indexOf`; (2) calls `toggleDoor()` from `doors.js` to produce new immutable door states; (3) enables/disables the Phaser physics body on the door zone; (4) sets `this.roomGraphDirty = true` to invalidate both cached pathfinding room graphs (standard and full), so the next `updateEnemies()` call rebuilds them with the updated door state. This ensures physics, raycasting, enemy LOS, and enemy pathfinding all update atomically. Player, enemy, and bullet groups all have colliders against `doorGroup`, so closed doors block all three entity types and exclude the connection from the standard pathfinding graph (making door-closing a defensive strategy against basic and spitter enemies, but not crawlers which use the full graph and can open doors)
- **Light switch toggle flow**: when the player presses E near a switch, `onToggleSwitch()` calls `toggleSwitch()` from `lightswitch.js` to produce new immutable switch states. The method is a pure state toggle with no enemy side effects. On subsequent frames, `updateEnemies()` checks `getLitRoomIds()` and freezes (zero velocity, skip AI via `continue`) any enemy that is inside a lit room. This means enemies in lit rooms are frozen in place but remain alive and visible -- if the switch is turned off, they resume AI. This is the sole mechanism for handling enemies in lit rooms (both when the light first turns on and when enemies later wander in)
- **Room visual theming**: every room has a distinct color palette. During level setup, `GameScene` builds a `this.roomThemes` Map keyed by room ID. Room 0's theme comes from `this.activeLocationData` (resolved from `@/src/systems/locations.js` in `init()`). Non-starting rooms derive their theme from `getRoomTheme(getRoomType(room.seed))` -- `getRoomType()` from `@/src/systems/maze.js` classifies the room by seed, and `getRoomTheme()` from `@/src/systems/roomThemes.js` maps that type to a `{ floorColor, wallColor }` object. The cached `roomThemes` Map is consumed by `drawRoom(gfx, room, theme)` (floor and wall fill), `drawDoorways()` (doorway floor fill via `this.roomThemes.get(room.id)`), and `createRoomMaze(gfx, room, theme)` (internal maze wall fill). `createRoomFurniture()` calls `getLocationLayout()` from `locations.js` for room 0 instead of `generateRoomFurniture()`. Room 0 is permanently lit -- its ID is injected into the `litRoomIds` array in both `updateDarkness()` and `updateEnemies()`, reusing the same lit-room mechanism as light switches. This means room 0 has no darkness overlay, and any enemies that wander in are frozen. The exit zone is at the west-center of the room, positioned by `getLocationExitPosition()`. A crack-in-the-wall visual at the first doorway of room 0 is drawn by `createCrackVisual()` using `generateCrackPoints()` from `@/src/systems/startroom.js` (the crack visual is identical across all locations)
- **Multi-room physics with door gaps**: `createRoomPhysics()` splits each wall into physics zones that skip over door offsets. This mirrors the raycasting segments produced by `createRoomWalls()` so that physics collision and flashlight occlusion agree on where doors are. Closable doors fill these gaps with their own physics zones that can be toggled
- **Touch controls wiring** (`GameScene`): `setupInput()` decides `this.touchMode` once via `detectTouchPrimary()` from `@/src/systems/touchControls.js` (coarse-pointer media query OR `navigator.maxTouchPoints > 0`) and, when true, calls `this.input.addPointer(3)` to enable multitouch (so a finger can hold a stick and tap a button simultaneously). `createTouchControls()` (no-op on desktop) builds two rex virtual joysticks fixed to the camera (`scrollFactor 0`, high depth `TOUCH_UI_DEPTH`), plus a polled FIRE button near the right thumb and tap buttons USE/WPN/BATT, plus an FS (fullscreen) button. Every position comes from the pure `computeTouchLayout({ width, height, stickRadius })` in `@/src/systems/touchControls.js` keyed on the live `this.scale.width`/`height` -- there are no hardcoded coordinates -- which centers both sticks vertically (move on the left, aim on the right) so the thumb can deflect fully downward without the mobile browser's bottom gesture zone eating the drag. The FIRE button is NOT a Phaser interactive object: `isFireHeld()` polls `pointer1..4` each frame and returns true when any down pointer falls within the fire-button radius -- this is what `resolveFireIntent` consumes, and is why touching a stick (reported as a left-button press) does not fire. `updateTouchButtons()` shows USE only when there is a nearby interactable (or while hiding), shows WPN only when more than one weapon slot is filled, and dims FIRE when there is no weapon/ammo. `triggerWeaponSwitch()` and `triggerInteract()` are extracted shared handlers so keyboard (Q/E), the scroll wheel, and the on-screen buttons all run identical logic; both guard against dead/dayEnding/teleporting state because the touch buttons fire outside the `update()` loop (via `pointerdown` callbacks) where the normal early-return gate does not apply
- **Touch controls respond to viewport changes** (`GameScene`): touch devices run under `Scale.RESIZE` (see `@/src/main.js`), so the canvas equals the device viewport and resizes on orientation flips, URL-bar hide/show, and fullscreen toggles. `createTouchControls()` registers a `this.scale.on('resize', ...)` handler that calls `layoutTouchControls()`, which recomputes `computeTouchLayout()` and repositions every stick, button (via the `moveTouchButton()` helper), and the right-anchored `floorText` HUD element. The handler is removed in the scene `shutdown` listener (alongside destroying the joysticks). To help hide the browser chrome on Android, `createTouchControls()` also registers `this.input.once('pointerdown', ...)` that calls `this.scale.startFullscreen()` on the first touch when the Fullscreen API is available; the FS button toggles fullscreen thereafter
- **ShopScene camera zoom-fit under RESIZE**: all shop content is placed in a fixed 1024x768 design space (e.g., the title at x=512, the journal overlay sized 1024x768). On desktop the scene runs under `Scale.FIT` so this already maps cleanly. On touch the game runs under `Scale.RESIZE` at native viewport size (see `@/src/main.js`), which would otherwise misalign or clip the design-space layout. `fitShopCamera()` solves this by setting the main camera zoom to `min(width/1024, height/768)` and centering it on `(512, 384)`, so the entire 1024x768 layout scales to fit the viewport (letterboxed, never clipped). It runs in `create()` and on every `this.scale` `resize` event (handler removed on `shutdown`). On desktop FIT it is a no-op (zoom 1, already centered)
- **ShopScene tap targets**: the shop already drove all buttons from `pointerdown`, so it worked on touch -- the only problem was that text buttons had hit areas exactly the size of their (small) glyphs. `enlargeHitArea(obj, padX, padY)` sets an oversized `Phaser.Geom.Rectangle` hit area padded beyond the text bounds (and sets a pointer cursor) so every interactive text button (location arrows, BUY, ENTER, NOTES, and journal PREV/NEXT/CLOSE) is finger-sized without changing its visual size
- **Right-click and scroll input**: `setupInput()` disables the browser context menu (`this.input.mouse.disableContextMenu()`) and listens for `pointerdown` and `wheel` events. Right-click triggers `onUseBattery()`, which calls `useBattery()` from `inventory.js` and, if successful, `rechargeBattery()` from `battery.js`. Scroll wheel triggers `switchWeapon()` from `@/src/systems/weapons.js` with a 200ms cooldown (`this.lastScrollTime`) to prevent rapid switching. Both inputs are guarded by `!isDead && !dayEnding` checks
- **Item pickup flow**: items are a static physics group with player overlap detection. On overlap, `onItemPickup()` first checks the item type. **Medkits** are handled before the inventory check: if the player is at full health, the function early-returns (medkit stays on the ground); otherwise, `applyHeal(this.combatState, MEDKIT_HEAL_AMOUNT)` from `@/src/systems/combat.js` heals the player, a `medkit_pickup` sound plays, a green camera flash fires (100ms, alpha 0.15), and the sprite is destroyed. Medkits bypass inventory entirely. For all other item types, `canPickupItem(this.inventoryState)` from `@/src/systems/inventory.js` is checked -- if the inventory is full, the function early-returns and the item remains on the ground for later pickup. Otherwise, it updates `this.inventoryState` via `pickupItem()` and destroys the item sprite. Items store their type and value via Phaser's `setData()` on the sprite. Items do not produce wall segments -- they are collectibles, not occluders. Lore items use a completely separate physics group (`this.loreGroup`) and pickup callback (`onLorePickup`) -- they bypass the inventory system entirely and are always collected on contact regardless of inventory capacity
- **Wall dual-purpose**: walls exist as both Phaser Arcade Physics static zones (for player collision) and as line segments from `createRoomWalls()` (for raycasting). Light-blocking furniture (any type with `blocksLight: true` in `@/src/systems/furniture.js`, e.g., shelf, bookcase, armoire, closet) and closable doors follow the same pattern -- each is both a physics zone and a raycasting segment. Non-blocking furniture (types with `blocksLight: false`, e.g., table, desk, bed, vent) still has physics zones but does not produce raycasting segments (light passes over them). Maze internal walls also follow this dual pattern: they produce both physics zones and raycasting segments. Doors are unique in that they can toggle between active/inactive at runtime via `body.enable` and `wallSegments` push/splice. Enemies, items, and light switches do not produce segments; enemies are dynamic actors, items are small collectibles, and light switches are proximity-based interactables with no physics body
- **Player rendering**: the player is an animated pixel art sprite at depth 200 (above the darkness overlay at depth 100), so the player is always visible regardless of flashlight direction. `drawPlayer()` switches between `player_walk` and `player_idle` animations based on velocity, sets rotation to the aim angle, and applies `setTint`/`setAlpha` for visual states: hiding (muted green tint 0x336633, alpha 0.3), invulnerability (red tint 0xcc4444, alpha 0.5), or normal (no tint, alpha 1)
- Camera and physics bounds are computed dynamically from the level's room positions, so they adapt to however many rooms the level generator produces
- **Audio integration**: both scenes initialize audio via a shared pattern: check for `this.sound.context`, handle browser autoplay policy (if `this.sound.locked`, defer initialization to the `'unlocked'` event), then call `generateAllSounds()` from `@/src/systems/audioEngine.js` to render all `SOUND_CONFIGS` from `@/src/systems/audio.js` into Phaser's audio cache via `OfflineAudioContext`. The `this.audioReady` flag gates all `playSound()` calls to prevent errors before sounds are rendered. `GameScene.createSounds()` additionally starts a continuous ambient drone via `createAmbientDrone()` and schedules recurring ambient sounds (distant_bang, drip, whisper) at pseudo-random intervals via `scheduleAmbientSound()`. Sound effects are wired to game events throughout GameScene (weapon fire, bullet hit, per-type enemy death/alert/ambient/fire sounds, player damage/death, item/lore/ammo pickup, battery recharge, door open/close, switch click, hiding enter/exit, stair transition, day complete, weapon switch) and ShopScene (purchase, denied, button clicks). `GameScene.playSoundAtDistance(key, distance)` is the proximity-based audio method: it reads the base gain from the sound's config, computes volume via `getDistanceVolume()` from `@/src/systems/audio.js` (linear falloff over `ENEMY_SOUND_RANGE` of 400px), and skips playback when volume is 0 (enemy is out of range). This is used for all per-enemy sounds (death, alert, ambient, fire) so players can identify enemies by audio proximity before visual contact. The ambient drone and ambient sound timer are stopped on scene exit (`stopAmbientAudio()` called in `onDayComplete()` and `onPlayerDeath()`). `GameScene.playSound()` plays at volume 1; `ShopScene.playSound()` plays at volume 0.5. All `playSound()` and `playSoundAtDistance()` calls are wrapped in try-catch as a system boundary safety measure
