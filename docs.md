# Noridoc: Backrooms (Project Root)

Path: @/

### Overview
- Survival horror top-down 2D twin-stick shooter built with Phaser 3, Vite, and plain JavaScript
- Based on the "Backrooms" creepypasta: player explores pitch-black rooms using only a flashlight, collects treasure, fights zombies, and must return to the exit before the battery dies
- Implements a Motherload-inspired roguelike loop: explore rooms, collect treasure, exit (or die), spend gold on upgrades in the shop, re-enter with stronger stats

### How it fits into the larger codebase
- `@/index.html` is the single HTML entry point, loading `@/src/main.js` as an ES module
- `@/src/main.js` creates the Phaser.Game instance and registers both scenes (GameScene and ShopScene)
- Game logic is split between pure systems (`@/src/systems/`) and Phaser scene classes (`@/src/scenes/`) to keep game math testable without a browser or Phaser runtime
- Cross-run state (gold balance, upgrade levels, run counter) lives in Phaser's `game.registry` for in-memory cross-scene access, and is backed by `localStorage` via `@/src/systems/persistence.js` for cross-session persistence. On boot, `@/src/main.js` loads saved state from localStorage into the registry before any scene starts
- `@/APPLICATION_SPEC.md` is the source of truth for game design requirements
- `@/CURRENT-PROGRESS.md` tracks what has been implemented vs. what remains
- `@/RESEARCH-NOTES.md` captures technology decisions and design rationale

### Core Implementation
- **Build tooling**: Vite dev server on port 8080, Vite build targeting ES2020, Phaser chunked separately via Rollup `manualChunks` for cache efficiency
- **Testing**: Vitest with Node environment -- systems are pure functions that do not require a DOM or Phaser
- **Phaser config**: 1024x768 canvas, Arcade physics with zero gravity (top-down game), auto-renderer selection
- **Controls**: WASD movement, mouse aim, left-click shoot, right-click use battery item, E key to interact with nearby doors, light switches, hideable furniture, or weapon pickups (nearest one wins by distance). E key exits hiding (WASD moves at half speed within the furniture while hidden). Q key switches between the two weapon slots (only works when both slots are occupied)
- **Game loop**: the game alternates between two scenes. `GameScene` is the gameplay scene where the player explores, fights, and collects treasure. On exit or death, the game transitions to `ShopScene` where the player spends accumulated gold on upgrades (battery capacity, flashlight width, max health, movement speed, weapon damage, fire rate, bullet range). Clicking "Enter Backrooms" transitions back to `GameScene` with upgraded stats. Enemy difficulty (HP, count, chase speed, contact damage) scales with `runCount` via `@/src/systems/scaling.js`, creating progression tension that the weapon upgrades help counter. Death forfeits all treasure earned that run; exiting keeps it
- **Level generation**: a seeded procedural level generator (`@/src/systems/level.js`) produces multiple connected rooms with doorways, including a post-placement pass that adds extra doors between grid-adjacent rooms (~50% chance per unconnected neighbor pair). The first room placed (room 1) is always east of room 0, so the crack/entry to the backrooms is deterministically on room 0's east wall regardless of seed. The seed increments each run via a `runCount` stored in `game.registry`, so each run produces a different layout. Non-starting rooms get internal maze walls or staggered columns via `@/src/systems/maze.js` (~40% open, ~30% maze, ~30% columns). `GameScene` renders and wires physics for the entire level at startup. Room 0 (the starting/spawn room) is special: it is rendered as a furniture store with warm wood-tone colors, hand-crafted store furniture (from `@/src/systems/startroom.js`), permanent lighting (no darkness overlay), a crack-in-the-wall visual at its first doorway with a pulsing glow, and the exit zone at its west-center position (representing the front entrance)
- **Closable doors**: approximately half of room connections (seeded random, 50% chance) have closable doors managed by `@/src/systems/doors.js`. Closable doors start closed, blocking movement (physics colliders), flashlight rays (wall segments), and enemy line-of-sight. The player can toggle them with the E key when within range. Opening a door removes its segment from the `wallSegments` array; closing it re-adds the segment -- following the same mutable segment-array pattern as walls and light-blocking furniture
- **Light switches**: ~15% of non-starting rooms get a light switch placed on a random wall, managed by `@/src/systems/lightswitch.js`. The player toggles switches with the E key (same as doors -- nearest interactable wins by Euclidean distance). Turning a switch on reveals the entire room by drawing a white rectangle into the BitmapMask (same system the flashlight uses), despawns all enemies currently in that room, and freezes any enemies that later wander in. Switches use a seeded PRNG with offset +40000
- **Enemy system**: enemies spawn in every room except the starting room, using a pure-function AI state machine (`@/src/systems/enemy.js`). Three enemy types exist: basic (standard zombie), crawler (fast melee, smaller, lower HP), and spitter (ranged, fires projectiles, has an `attack` AI state). Type distribution is run-gated: run 0 spawns only basic enemies, run 1 introduces crawlers, and run 2+ adds spitters. Each type carries its own speed constants, HP, and contact damage. The AI state machine has four states: idle/chase/search (shared by all types) and attack (spitter-only, fires projectiles when in range). Enemies detect the player via line-of-sight raycasting against the same wall segment array used by the flashlight system. Detection is suppressed when the player is hiding (via the `playerHidden` parameter on `updateEnemyAI()`). Room 0 is permanently included in the lit room list, so any enemies that wander in are frozen (same behavior as rooms with an active light switch). Enemy stats (HP, count per room, chase speed, contact damage) scale with `runCount` via pure functions in `@/src/systems/scaling.js` -- scaling is applied per-type using each type's base values, and all scaling is capped to prevent infinite growth
- **Hiding mechanics**: the player can hide inside `canHide` furniture (tables and desks) by pressing E when near them, managed by `@/src/systems/hiding.js`. While hidden: the player moves at half speed (`HIDING_SPEED_MULTIPLIER = 0.5`) constrained to the furniture bounds, shooting and battery use are disabled, the player is invisible to enemy line-of-sight detection and immune to contact damage, and the player graphic renders semi-transparent. Only the E key exits hiding (WASD moves the player within the furniture). The hiding system uses Phaser's `body.setBoundsRectangle()` to constrain the player within the furniture area, and the player-furniture collider is disabled while hidden (otherwise physics would push the player out of the furniture body). Hiding adds strategic depth by creating mobile safe spots within enemy-occupied rooms
- **Exploration and minimap**: a fog-of-war minimap renders in the top-right corner of the screen, managed by `@/src/systems/exploration.js`. The minimap is a shop upgrade (id: `minimap`, 1500 gold, single tier) -- it is not available by default. When purchased, rooms are revealed as the player visits them. The minimap shows visited rooms (dark gray), the current room (lighter gray), the player position (green dot), and the exit position (green dot). Exploration state tracks which rooms have been visited using `isPointInRoom()` from `@/src/systems/lightswitch.js` for room detection regardless of whether the minimap is purchased (future features may use visitation data). The minimap is drawn via Phaser Graphics at depth 1000 with `setScrollFactor(0)`, same as the HUD. Exploration state resets each run -- no persistence needed
- **Combat loop**: enemies damage the player on contact (with invulnerability cooldown) and spitter enemies additionally fire projectiles. Each enemy type has its own contact damage value stored on its state object, so `onEnemyContact` reads damage per-enemy rather than using a single global value. The player has a 2-slot weapon inventory (managed by `@/src/systems/weapons.js`) with three weapon types: Pistol (default, balanced), Shotgun (multi-pellet spread, close range, slow fire), and Rifle (high damage, long range, slow fire). Weapon pickups spawn in rooms and are collected via E-key. Q key switches weapons. Upgrade bonuses (damage, fire rate, range) from the shop are applied additively to whichever weapon is active. Bullets store per-bullet damage and range so in-flight bullets retain correct stats after weapon switches. Enemy projectiles use a separate physics group (`enemyBulletGroup`) that mirrors the player bullet pattern (pooling, wall/furniture collision, range expiry). Player death triggers scene restart. All combat/shooting/weapon math lives in pure-function systems (`@/src/systems/combat.js`, `@/src/systems/shooting.js`, `@/src/systems/weapons.js`), with Phaser wiring in `GameScene`
- **Survival loop**: the flashlight battery drains over 90 seconds, progressively narrowing the flashlight cone from 100% to 25% of its base angle. Below 25% charge, the flashlight flickers. At 0%, the flashlight goes completely dark. The player can collect battery items and use them (right-click) to recharge the flashlight by 30 charge. The player must reach the exit zone (in room 0) before the battery dies. Reaching the exit transitions to the shop with all treasure earned. Battery math lives in `@/src/systems/battery.js`, following the same pure-function immutable-state pattern as combat and shooting
- **Items and inventory**: collectible items (batteries, coins, gems) spawn in every room except the starting room via a seeded PRNG. Items are only collected on player overlap when inventory has capacity (controlled by `canPickupItem()` from `@/src/systems/inventory.js`) -- items remain on the ground when the inventory is full. Battery items are stored in inventory and consumed with right-click to recharge the flashlight (consuming a battery also frees an inventory slot). Treasure items (coins, gems) accumulate a value total that converts to gold in the shop. Inventory capacity defaults to 8 items and is increased by the "Backpack Size" shop upgrade (+4 per level, max 20). Item spawning lives in `@/src/systems/items.js`, inventory state management in `@/src/systems/inventory.js`, both following the pure-function pattern
- **Persistence**: game progress (shopState, runCount) is persisted to localStorage via `@/src/systems/persistence.js`, a pure function module that uses a versioned save format (`{ version, shopState, runCount }`) under the key `backrooms_save`. State is saved at every mutation point in both scenes (after adding gold, purchasing upgrades, incrementing runCount, and before entering backrooms) and also by browser lifecycle listeners (`visibilitychange`, `beforeunload`) installed in `@/src/main.js`. On boot, `main.js` loads saved state and populates `game.registry` before scenes start. All localStorage access is try-catch wrapped for graceful degradation
- **Shop and upgrades**: between runs, the player visits a shop (`@/src/scenes/ShopScene.js`) where treasure earned converts to gold. Gold can be spent on upgrades across several categories (survival: battery, flashlight, health, speed; weapon: damage, fire rate, bullet range; inventory: backpack size; unlocks: starting pistol, minimap) with three tiers each for most upgrades (costs: 100/300/800) and single-tier binary unlocks for others (starting pistol: 500, minimap: 1500). Upgrade definitions, pricing, and purchase logic live in the pure-function module `@/src/systems/shop.js`. The UPGRADES array is the single source of truth -- ShopScene auto-generates UI rows from it. Shop state persists in `game.registry` across scene transitions

### Things to Know
- The project uses Phaser 3 (~3.90.x) deliberately over Phaser 4, because Phaser 4's BitmapMask API changed with incomplete docs -- and the flashlight system is the core visual mechanic (see `@/RESEARCH-NOTES.md`)
- Systems in `@/src/systems/` are designed as pure functions with no Phaser dependency, so they can be tested with `vitest run` in a Node environment
- The `vite.config.js` test environment is set to `node`, not `jsdom`, because the testable code has no DOM dependencies
- `package.json` uses `"type": "module"` -- all imports use ES module syntax throughout
- All procedural generation (level layout, furniture placement, enemy spawning, item placement, door closability, light switch placement, weapon pickup placement, maze/column generation, extra door connections) uses a shared seeded PRNG (`@/src/systems/random.js`) so levels are deterministic for a given seed. Each system uses a different seed offset to avoid correlation: furniture uses +0, enemies use +10000, items use +20000, doors use +30000, light switches use +40000, stairs use +50000, weapons use +60000, maze uses +70000, extra doors use +80000. The seed changes each run via an incrementing `runCount` stored in `game.registry` (and persisted to localStorage via `@/src/systems/persistence.js`). The starting room system (`@/src/systems/startroom.js`), hiding system (`@/src/systems/hiding.js`), exploration system (`@/src/systems/exploration.js`), scaling system (`@/src/systems/scaling.js`), and persistence system (`@/src/systems/persistence.js`) do not use a seeded PRNG -- startroom uses a fixed hand-crafted layout, hiding operates on furniture objects already placed by the furniture system, exploration is deterministic based on player position, scaling applies deterministic formulas based on `runCount`, and persistence is a browser storage boundary
