# Noridoc: Backrooms (Project Root)

Path: @/

### Overview
- Survival horror top-down 2D twin-stick shooter built with Phaser 3, Vite, and plain JavaScript
- Playable on both desktop (keyboard + mouse) and mobile (full-screen landscape twin-stick touch controls); the two input modes share the same world/HUD coordinate space and game logic
- Based on the "Backrooms" creepypasta: player explores pitch-black rooms using only a flashlight, collects treasure, fights zombies, and must return to the exit before the battery dies
- Implements a Motherload-inspired roguelike loop: explore an effectively-unbounded, lazily-generated procedural world, collect treasure, exit (or die), spend gold on upgrades in the shop, re-enter with stronger stats
- Art is AI-generated offline ("nano banana") into committed PNGs with a procedural pixel-grid fallback, so no image API runs in-browser (see Build tooling / Offline AI-art pipeline)

### How it fits into the larger codebase
- `@/index.html` is the single HTML entry point, loading `@/src/main.js` as an ES module
- `@/src/main.js` creates the Phaser.Game instance and registers all scenes (TitleScene, GameScene, ShopScene, PauseScene, and RunSummaryScene); TitleScene is listed first so it auto-starts as the boot screen
- Game logic is split between pure systems (`@/src/systems/`) and Phaser scene classes (`@/src/scenes/`) to keep game math testable without a browser or Phaser runtime. Audio follows this same split: pure config and per-enemy-type sound logic in `@/src/systems/audio.js` (testable in Node), browser-dependent synthesis in `@/src/systems/audioEngine.js`, and volume settings state management in `@/src/systems/settings.js` (testable in Node) -- all sounds are procedurally generated via Web Audio API with no audio file assets. Volume uses exponential mapping for perceptual uniformity (OFF=0, LOW=0.05, MED=0.15, HIGH=0.4, FULL=1.0) and every audio call multiplies by `getEffectiveVolume(master, category)` which is `masterValue * categoryValue`. Each enemy type (basic/crawler/spitter) has distinct ambient, alert, death, and (for spitters) fire sounds, with proximity-based volume attenuation so players can identify enemies by audio before visual contact. Gunfire noise alerts (`@/src/systems/gunfireNoise.js`) also follow this split: pure logic (testable in Node) and Phaser physics/overlap wiring in `GameScene`. Pixel art sprites and character animations follow the same split: pure data/manifest definitions in `@/src/systems/sprites.js` and `@/src/systems/art.js` (both testable in Node) and Phaser texture/animation generation in `GameScene.generateSpriteTextures()`. Meaningful sprites ship as committed AI-generated PNGs under `@/src/assets/sprites/` (loaded in the scenes' `preload()`); for any key without a PNG, `generateSpriteTextures()` falls back to procedurally generating the texture via Phaser's `textures.generate()` data-array format. All character animations are created from `ANIM_DEFS` via `this.anims.create()` regardless of texture source. Per-room visual + furniture theming follows this pattern too: pure vibe data (palette + furniture profile) in `@/src/systems/roomVibes.js` (testable in Node), consumed by `GameScene` for rendering and furniture placement
- Cross-run state (gold balance, upgrade levels, run counter, collected lore IDs, unlocked locations, active starting location) lives in Phaser's `game.registry` for in-memory cross-scene access, and is backed by `localStorage` via `@/src/systems/persistence.js` for cross-session persistence. Audio settings (master, SFX, and ambient volume levels) also live in `game.registry` but are persisted to a separate localStorage key (`backrooms_settings`) via `@/src/systems/settings.js`, independent of game saves -- volume preferences survive save deletion. On boot, `@/src/main.js` loads both game state and audio settings from localStorage into the registry before any scene starts
- `@/APPLICATION_SPEC.md` is the source of truth for game design requirements
- `@/CURRENT-PROGRESS.md` tracks what has been implemented vs. what remains
- `@/RESEARCH-NOTES.md` captures technology decisions and design rationale

### Core Implementation
- **Build tooling**: Vite dev server on port 8080, Vite build targeting ES2020, Phaser chunked separately via Rollup `manualChunks` for cache efficiency
- **Offline AI-art pipeline**: meaningful pixel-art sprites and the shop background/icons are AI-generated rather than hand-drawn procedural grids. `@/scripts/generate-art.mjs` (run via `npm run generate-art`) is a build-time-only Node tool that calls Google's "nano banana" image model (`gemini-2.5-flash-image`) over the Gemini REST API, removes the (non-transparent) generated background by flood-filling the border-connected region to alpha, downscales with `jimp` (a devDependency), and writes committed PNGs into `@/src/assets/sprites/`. The model's prompts, target sizes, background-removal decision, and shop-icon mapping are the pure module `@/src/systems/art.js`. CRITICAL INVARIANT: the `GEMINI_KEY` (read from the repo-root `.env`) is build-time only and is NEVER shipped to the browser (it is not `VITE_`-prefixed); the game ships only the committed PNGs and CI/`vite build` runs with no key. The script supports `--force` (regenerate every entry), `--rekey` (re-run background removal over existing PNGs with no API calls), and positional manifest-`id` arguments (regenerate only those entries). At runtime each sprite key uses its PNG if present and otherwise falls back to the procedural pixel-grid texture from `@/src/systems/sprites.js`, so the game always renders even with no generated art
- **Testing**: Vitest with Node environment -- systems are pure functions that do not require a DOM or Phaser. Gameplay/visual behavior is verified by playtest, not headless tests: `@/scripts/playtest.mjs` is a Playwright script that drives the real game in Chromium against a running Vite dev server and captures screenshots; `@/src/main.js` exposes `window.__game` so the script can introspect live scene state
- **Phaser config**: fixed internal 1024x768 resolution with `Scale.FIT` + `CENTER_BOTH` (the canvas scales to fit any screen and letterboxes with black bars -- invisible against the black game), Arcade physics with zero gravity (top-down game), auto-renderer selection, `pixelArt: true` for nearest-neighbor texture sampling on all pixel art sprites. Using a fixed internal resolution means all existing world/HUD coordinates are unchanged across screen sizes. `@/src/main.js` also registers the `rexVirtualJoystick` global plugin (from the `phaser3-rex-plugins` dependency) used by the mobile touch sticks
- **Controls (desktop)**: WASD movement, mouse aim, left-click shoot, right-click use battery item, F key toggles the flashlight on/off (to conserve battery), E key to interact with nearby doors, light switches, hideable furniture, or weapon pickups (nearest one wins by distance). E key exits hiding (WASD moves at half speed within the furniture while hidden). Q key or scroll wheel switches between the two weapon slots (only works when both slots are occupied; scroll wheel has a 200ms cooldown). ESC key pauses the game, launching a PauseScene overlay that freezes GameScene and displays a controls reference
- **Controls (mobile/touch)**: a left virtual joystick moves the player (analog, force-scaled), a right virtual joystick aims the flashlight, a dedicated FIRE button shoots, and on-screen USE / WPN / BATT / LIGHT / FS / PAUSE buttons handle interact, weapon switch, battery use, flashlight toggle, fullscreen, and pause. The on-screen buttons reuse the same `triggerWeaponSwitch()`/`triggerInteract()`/`onUseBattery()` code paths as keyboard/mouse so behavior is identical. The pause button (`| |`) is positioned at the top-left of the screen via `computeTouchLayout().pauseButton` from `@/src/systems/touchControls.js`. `@/src/scenes/GameScene.js` detects touch at input setup and only builds the touch UI on touch devices -- desktop is unaffected. The game is landscape-only; in portrait a CSS-only overlay in `@/index.html` prompts the player to rotate
- **Game loop**: on boot, the game shows `TitleScene` (main menu), which displays the "BACKROOMS" title with atmospheric pulsing text and provides save-aware menu options: Continue (to ShopScene), New Game (clears save, to GameScene), and Delete Save (with confirmation), plus a Settings button that opens an audio volume controls overlay. Menu options are generated by `@/src/systems/titleMenu.js` based on whether a save exists (checked by calling `loadGame()` from `@/src/systems/persistence.js` directly). After leaving the title screen, the game alternates between primary scenes. `GameScene` is the gameplay scene where the player explores, fights, and collects treasure. During gameplay, the player can pause (ESC on desktop, pause button on mobile), which freezes GameScene via Phaser's `scene.pause()` and launches `PauseScene` as an overlay -- PauseScene displays a controls reference and audio volume controls (Master/SFX/Ambient cycle buttons that apply immediately to the running game) and resumes gameplay on dismissal (ESC or tap). On exit or death, the game transitions to `RunSummaryScene`, which displays run statistics (time survived, rooms explored, enemies killed, deepest floor, treasure earned, and HP remaining on survival) with animated reveals and a themed title ("RETURNED" in gold for survival, "YOU DIED" in red for death -- deliberately not "escaped", since the backrooms are explored by choice, and rooms explored is a raw count with no "X / total" ratio because the world is unbounded). After the player acknowledges (any key on desktop, tap on mobile), `RunSummaryScene` passes through to `ShopScene` where the player spends accumulated gold on upgrades (battery capacity, flashlight width, max health, movement speed, weapon damage, fire rate, bullet range). Clicking "Enter Backrooms" transitions back to `GameScene` with upgraded stats. Difficulty scales with **distance from the entrance**, not the day number: each room's `distance` (door-hop distance plus a fixed depth per floor descended) drives enemy HP, contact damage, per-room count, and type mix via `@/src/systems/scaling.js` and `@/src/systems/enemy.js` -- the same axis loot quality scales on, so risk and reward rise together. Per the binding spec feedback, `runCount` no longer affects combat difficulty at all; it only reseeds the world each day. The unbounded depth itself still raises exploration cost -- the deeper the player descends, the more battery drain to explore and return. Death forfeits all treasure earned that run; exiting keeps it. Players can cycle through unlocked starting locations in the shop screen (alternate exit portals were removed in the unbounded-world rewrite)
- **Level generation**: the world is an effectively-unbounded procedural grid produced by a coordinate-keyed deterministic generator (`@/src/systems/worldgen.js`). Each grid cell's content and doors derive from a hash of `(floorSeed, gx, gy)`, so a room is identical regardless of when it is generated and adjacent cells always agree on the shared door. `GameScene` generates ~2 rooms ahead of the player and activates each room exactly once; rooms stay static once generated. The seed increments each run via a `runCount` stored in `game.registry`, so each run produces a different world. The entrance cell `(0,0)` has EXACTLY one opening into the backrooms: the dimensional rift in its east wall (its other three edges are forced sealed by `@/src/systems/worldgen.js`). Non-entrance rooms get internal structure via `@/src/systems/maze.js` (six room types: open, maze, columns, corridor, storage, cubicles -- 'open' is deliberately rare, so most rooms are narrow winding passages); these interior maze walls are kept as solid rects (`this.mazeWallRects`) and the flashlight occludes only on their FAR faces (back-face culling via `wallRectOccluders()`) so the cone reaches across each wall band and lights its near surface -- so interior walls read as solid bands of visible thickness rather than infinitely-thin black lines (the fix for the spec's "walls have no thickness" bug; enemy line-of-sight still expands those rects to their full outlines so walls block sight from both sides). Each room also gets a per-room "vibe" via `@/src/systems/roomVibes.js`: each room is given a coherent-but-distinct identity (furniture store, library, office, derelict ship, overgrowth, warehouse), pairing a palette with a furniture profile. Every vibe palette stays desaturated/dim so the world still reads as liminal, but each vibe is its OWN dim, so adjacent rooms feel like different places rather than the same room twice (this reverses an earlier monotone single-palette approach that playtesters found "too samey"). The vibe is chosen on a seed axis ORTHOGONAL to the structural room type, so a library can be a maze and a forest can be a column hall. Rooms are deliberately small and densely furnished with each vibe's semantic clusters (office pods, forest groves, ship console banks) so the world feels claustrophobic and abandoned rather than empty. Floors are stacked as non-overlapping world-space bands and connected by deep-link stairs (down-stairs scattered per `hasStairDown()`, an up-stair at each non-entrance floor's entrance cell, navigated via a return stack). Room 0 (the entrance/spawn room) is special: its visual theme and furniture come from the active starting location via `@/src/systems/locations.js` (furniture store, office building, parking garage, hotel lobby, laundromat); it is permanently lit, and holds the only world exit zone (to the shop) at its west-center. Its east opening is the rift: rendered as a sizzling additive-blend tear of light (not a doorway), sealed against light in both directions (no door gap in the occluder segments, light-spill zones touching it are filtered), with a proximity crackle sound that grows louder on approach and a one-time white flash + camera shake on first crossing -- the backrooms are invisible until the player actually passes through
- **Closable doors**: approximately half of room connections (seeded random, 50% chance) have closable doors managed by `@/src/systems/doors.js`. Closable doors start closed, blocking movement (physics colliders), flashlight rays (wall segments), enemy line-of-sight, and the room connectivity graph (the room graph from `@/src/systems/pathfinding.js` excludes closed door connections). Closed doors block ALL enemy types -- since enemy pursuit is strictly sight-based, breaking LOS with a door ends the chase, making doors a universal defense (enemies cannot open doors; the former crawler door-opening ability was removed). Only the player can toggle them, with the E key when within range. Opening a door removes its segment from the `wallSegments` array; closing it re-adds the segment -- following the same mutable segment-array pattern as walls and light-blocking furniture. Toggling a door also invalidates the cached room graph so enemy wander filtering and sound-alert radii reflect the new connectivity
- **Light switches**: ~15% of non-starting rooms get a light switch placed on a random wall, managed by `@/src/systems/lightswitch.js`. The player toggles switches with the E key (same as doors -- nearest interactable wins by Euclidean distance). Turning a switch on reveals the entire room by drawing a white rectangle into the BitmapMask (same system the flashlight uses) and freezes any enemies currently in or later wandering into that room (velocity zeroed, AI skipped). Enemies in lit rooms remain alive and visible but frozen in place -- turning the switch off lets them resume AI. Switches use a seeded PRNG with offset +40000. Additionally, light spills through open doorways of lit rooms into adjacent dark rooms as a gradient (full brightness at the doorway, fading to dark over 150px), computed by `@/src/systems/lightSpill.js`. Closed doors block light spill completely
- **Enemy system**: enemies spawn in non-starting rooms (entrance-adjacent rooms may roll zero enemies and stay empty), using a pure-function AI state machine (`@/src/systems/enemy.js`). Three enemy types exist: basic (standard zombie), crawler (fast melee, smaller, lower HP), and spitter (ranged, fires projectiles, has an `attack` AI state). Type distribution is distance-gated: rooms near the entrance spawn only basic enemies, crawlers appear at mid distances, and spitters join deep in. Each type carries its own speed constants, HP, and contact damage. The AI state machine has four states: idle/chase/search (shared by all types) and attack (spitter-only, fires projectiles when in range). Enemies detect the player via line-of-sight raycasting against the same occluders the flashlight uses (perimeter walls, light-blocking furniture, closed doors, and the interior maze walls) -- but the maze walls are expanded differently per consumer: enemy LOS uses each maze wall's FULL outline (walls block sight from both sides) while the flashlight uses only its far faces, so the two share occluder geometry without sharing one array. Detection is suppressed when the player is hiding (via the `playerHidden` parameter on `updateEnemyAI()`). **Pursuit is strictly sight-based** (a binding spec rule: enemies must not follow the player once they lose line of sight, including through walls and while hiding): the AI stamps a last-seen position (`lastKnownX/Y`) on every frame it can see the player, and the moment LOS breaks -- wall, closed door, or hiding -- chase/attack drop to search, which walks to that stored position and gives up to idle on arrival or after a timeout. Enemies never route through doorways toward the live player and cannot open closed doors, so doors are a universal defense against every type. The one asymmetry is sound: gunfire alerts set an enemy to chase with `lastKnownX/Y` snapshotted at noise time, which resolves as a fixed-point investigation of the noise, not a live tracker. Room 0 is permanently included in the lit room list, so any enemies that wander in are frozen (same behavior as rooms with an active light switch). Enemy stats (HP, contact damage), per-room enemy count, and type mix scale with the spawn room's **distance from the entrance** (`room.distance`) via pure functions in `@/src/systems/scaling.js` -- a small safe band of rooms around the entrance stays at base difficulty permanently (and may be empty), scaling is applied per-type using each type's base values, all scaling is capped to prevent infinite growth, and movement speeds never scale (they are pure per-type constants). **Cross-room wandering** (not player tracking): idle enemies probabilistically (15% chance per wander tick) seek nearby doorways to migrate between rooms, constrained by a 10-second cooldown after room transitions, a maximum BFS distance of 2 from spawn (via the room graph from `@/src/systems/pathfinding.js`, which excludes closed doors), and a cap of 4 enemies per room. **Danger escalation**: beyond distance-based stat scaling and static enemy placement, a time-based danger system (`@/src/systems/danger.js`) spawns waves of enemies as the player spends time in unsafe rooms (any room that is not room 0 and not lit). The timer pauses but does not reset in safe rooms, so hiding in a safe room delays but cannot prevent escalation. Every 30 seconds of cumulative unsafe time triggers a wave, with enemy count and type composition shifting toward harder enemies as waves progress (basic-only early, mixed crawler/spitter late), up to a cap of 8 waves per run. Wave enemies spawn in a valid room on the same floor (not room 0, not lit, not the player's current room) and have the same state structure and behavior as pre-placed enemies. The HUD shows a red "DANGER N" indicator when waves have been spawned
- **Hiding mechanics**: the player can hide inside `canHide` furniture (any furniture type with `canHide: true` in `@/src/systems/furniture.js`, e.g., table, desk, bed, armoire, closet, vent) by pressing E when near them, managed by `@/src/systems/hiding.js`. While hidden: the player moves at half speed (`HIDING_SPEED_MULTIPLIER = 0.5`) constrained to the furniture bounds, shooting and battery use are disabled, the player is invisible to enemy line-of-sight detection and immune to contact damage, and the player graphic renders semi-transparent. Only the E key exits hiding (WASD moves the player within the furniture). The hiding system uses Phaser's `body.setBoundsRectangle()` to constrain the player within the furniture area, and the player-furniture collider is disabled while hidden (otherwise physics would push the player out of the furniture body). Hiding adds strategic depth by creating mobile safe spots within enemy-occupied rooms
- **Exploration and minimap**: a fog-of-war minimap renders in the top-right corner of the screen, managed by `@/src/systems/exploration.js`. The minimap is a shop upgrade (id: `minimap`, 3000 gold, single tier) -- it is not available by default. When purchased, rooms are revealed as the player visits them. The minimap shows visited rooms (dark gray), the current room (lighter gray), the player position (green dot), and the exit position (green dot). Exploration state tracks which rooms have been visited using `isPointInRoom()` from `@/src/systems/lightswitch.js` for room detection regardless of whether the minimap is purchased (future features may use visitation data). The minimap is drawn via Phaser Graphics at depth 1000 with `setScrollFactor(0)`, same as the HUD. Exploration state resets each run -- no persistence needed
- **Combat loop**: enemies damage the player on contact (with invulnerability cooldown) and spitter enemies additionally fire projectiles. Each enemy type has its own contact damage value stored on its state object, so `onEnemyContact` reads damage per-enemy rather than using a single global value. The player has a 2-slot weapon inventory (managed by `@/src/systems/weapons.js`) with three weapon types: Pistol (default, balanced), Shotgun (multi-pellet spread, close range, slow fire), and Rifle (high damage, long range, slow fire). Weapon pickups spawn in rooms and are collected via E-key. Q key or scroll wheel switches weapons. Upgrade bonuses (damage, fire rate, range) from the shop are applied additively to whichever weapon is active. Firing a weapon also produces gunfire noise (via `@/src/systems/gunfireNoise.js`) that alerts idle/search enemies within the weapon's `noiseRange` BFS hops (pistol/rifle: 1, shotgun: 2), setting them to chase state -- a 5-second cooldown prevents rapid re-alerting. Bullets store per-bullet damage and range so in-flight bullets retain correct stats after weapon switches. When an enemy dies, its sprite persists as a visible corpse (gray tint, faded alpha, toppled angle, depth 5) with physics disabled (`setActive(false)`, `body.enable = false`) -- corpses are safe because Phaser skips inactive sprites in overlap checks. Enemy projectiles use a separate physics group (`enemyBulletGroup`) that mirrors the player bullet pattern (pooling, wall/furniture collision, range expiry). Player death triggers scene restart. All combat/shooting/weapon/healing math lives in pure-function systems (`@/src/systems/combat.js`, `@/src/systems/shooting.js`, `@/src/systems/weapons.js`), with Phaser wiring in `GameScene`
- **Survival loop**: the flashlight battery drains over 150 seconds of lit time, progressively narrowing the flashlight cone from 100% to 25% of its base angle (base angle is PI/6 or 30 degrees at upgrade level 0). Below 25% charge, the flashlight flickers. At 0%, the flashlight goes completely dark. The flashlight can be toggled off (F key / touch LIGHT button) to conserve battery -- charge ONLY drains while the light is effectively lit -- and it auto-darkens while hiding (switch state preserved and restored on exit); this on/off state is the pure module `@/src/systems/flashlight.js`. The lit cone is also range-capped (`FLASHLIGHT_RANGE` in `@/src/systems/visibility.js`) so sight lines stay short and claustrophobic. The player can collect battery items and use them (right-click) to recharge the flashlight by 30 charge. Because the mechanic was unintuitive in playtests, the HUD labels the bars (HP / FLASHLIGHT [F]), shows contextual low-battery hints (`getBatteryHint()` in `@/src/systems/battery.js`), and displays a one-time first-run toast explaining drain/narrowing/recharge. The player must reach the exit zone (in room 0) before the battery dies. Reaching the exit transitions to the shop with all treasure earned. Battery math lives in `@/src/systems/battery.js`, following the same pure-function immutable-state pattern as combat and shooting
- **Items and inventory**: collectible items (batteries, coins, gems, medkits) spawn in non-starting rooms via a seeded PRNG. Crucially, both loot density and loot tiers scale with **distance from the entrance**: each room carries a `distance` (door-hop distance plus a fixed depth per floor descended, stamped at generation in `@/src/scenes/GameScene.js`), and `@/src/systems/items.js` uses it both to pick the per-room item count band (near-entrance rooms are generous -- rarely empty, up to 3 items -- with counts tapering until deep rooms are sparse, mostly empty with at most 2; per the binding spec feedback asking for more lower rewards early given high shop prices) and to bias item-type weights -- low-value coins dominate near the entrance while higher-value silver/gold/gems only become likely (gems only possible) deeper in. The result is a dense-but-cheap near zone and a sparse, rarity-driven deep economy. This rewards exploration, and enemy difficulty (`@/src/systems/scaling.js`) scales on the SAME axis, so deeper rooms pair better loot with harder fights. (The legacy `@/src/systems/distance.js` BFS computation of this value has no production callers -- it survives only in tests.) Most items are only collected on player overlap when inventory has capacity (controlled by `canPickupItem()` from `@/src/systems/inventory.js`) -- items remain on the ground when the inventory is full. Battery items are stored in inventory and consumed with right-click to recharge the flashlight (consuming a battery also frees an inventory slot). Treasure items (coins, gems) accumulate a value total that converts to gold in the shop. Medkits bypass the inventory entirely -- they are consumed immediately on contact, healing the player by `MEDKIT_HEAL_AMOUNT` (25 HP) via `applyHeal()` from `@/src/systems/combat.js`. If the player is already at full health, the medkit stays on the ground for later pickup. Inventory capacity defaults to 8 items and is increased by the "Backpack Size" shop upgrade (+4 per level, max 20). Item spawning lives in `@/src/systems/items.js`, inventory state management in `@/src/systems/inventory.js`, both following the pure-function pattern
- **Lore items**: collectible lore notes spawn in ~25% of non-starting rooms (max 1 per room) via `@/src/systems/lore.js`, a pure-function module using a seeded PRNG (seed offset +90000). Lore items are auto-collected on player overlap (they do NOT consume inventory space) and display a tween-animated text popup at the bottom of the screen that auto-dismisses after 5 seconds. Lore is a meta-progression collectible -- collected lore IDs are persisted across runs via `collectedLore` in `game.registry` and localStorage, and already-collected entries do not respawn. The module defines 20 Backrooms-themed lore entries (atmospheric/horror text). Lore items use a separate physics group (`loreGroup`) from regular items (`itemGroup`). Between runs, the ShopScene provides a **lore journal** overlay (activated via a "NOTES X/20" button) that lets players review all collected and uncollected lore entries with pagination. The journal data logic (`getJournalEntries`, `getJournalPage`) lives in `@/src/systems/lore.js` as pure functions, while the overlay rendering lives in `@/src/scenes/ShopScene.js`
- **Persistence**: game progress (shopState, runCount, collectedLore, unlockedLocations, activeLocation) is persisted to localStorage via `@/src/systems/persistence.js`, a pure function module that uses a versioned save format (`{ version: 3, shopState, runCount, collectedLore, unlockedLocations, activeLocation }`) under the key `backrooms_save`. State is saved at every mutation point in both scenes (after adding gold, purchasing upgrades, incrementing runCount, collecting lore, unlocking locations, and before entering backrooms) and also by browser lifecycle listeners (`visibilitychange`, `beforeunload`) installed in `@/src/main.js`. Audio settings (master, SFX, and ambient volume levels) are persisted separately via `@/src/systems/settings.js` under the key `backrooms_settings` -- this is intentionally a separate localStorage key so volume preferences are independent of game save lifecycle (creating/deleting saves does not affect volume preferences). Audio settings are saved immediately on each user interaction via `saveSettings()` (not by lifecycle listeners). On boot, `main.js` loads both game state and audio settings and populates `game.registry` before scenes start. All localStorage access is try-catch wrapped for graceful degradation. Older saves are backward-compatible -- `loadGame()` defaults missing fields to safe values (e.g., `collectedLore` to `[]`, `unlockedLocations` to `['store']`, `activeLocation` to `'store'`)
- **Shop and upgrades**: between runs, the player visits a shop (`@/src/scenes/ShopScene.js`) where treasure earned converts to gold. Gold can be spent on upgrades across several categories (survival: battery, flashlight, health, speed; weapon: damage, fire rate, bullet range; inventory: backpack size; unlocks: starting pistol, minimap) with three tiers each for most upgrades (costs: 200/600/1500) and single-tier binary unlocks for others (starting pistol: 1000, minimap: 3000). Upgrade definitions, pricing, and purchase logic live in the pure-function module `@/src/systems/shop.js`. The UPGRADES array is the single source of truth -- ShopScene auto-generates UI rows from it. Shop state persists in `game.registry` across scene transitions. The shop also displays a lore journal button ("NOTES X/20") that opens a full-screen overlay for reviewing collected lore entries (see Lore items section)

### Things to Know
- The project uses Phaser 3 (~3.90.x) deliberately over Phaser 4, because Phaser 4's BitmapMask API changed with incomplete docs -- and the flashlight system is the core visual mechanic (see `@/RESEARCH-NOTES.md`)
- Run statistics are tracked per-run via a pure module `@/src/systems/runStats.js` following the same immutable-state pattern as combat, battery, etc. The `runStats` state object (`{ enemiesKilled, timeElapsed, maxFloor }`) is initialized in `GameScene.init()` and updated each frame (time) and on game events (kills, floor transitions). It is passed to `RunSummaryScene` at end-of-run for display
- **Touch vs desktop is a single boolean (`touchMode`)**, decided once at input setup by `detectTouchPrimary()` from `@/src/systems/touchControls.js`. All input-mode forks (aim source, move source, fire source, bullet aim point) route through pure resolver functions in `touchControls.js` so the branching is unit-tested without Phaser. The critical invariant lives in `resolveFireIntent`: in touch mode the game fires ONLY from the dedicated FIRE button, never from a generic screen touch -- the browser reports touching the move/aim sticks as a left-button press, so naively reusing the desktop left-click path would make moving fire the weapon
- Systems in `@/src/systems/` are designed as pure functions with no Phaser dependency, so they can be tested with `vitest run` in a Node environment
- The `vite.config.js` test environment is set to `node`, not `jsdom`, because the testable code has no DOM dependencies
- `package.json` uses `"type": "module"` -- all imports use ES module syntax throughout
- All procedural generation is deterministic for a given seed. The unbounded world generator (`@/src/systems/worldgen.js`) hashes the per-floor seed with cell coordinates so room/door/stair generation is order-independent (the player reaching a cell by any path yields the same room). Per-room content (furniture +0, enemies +10000, items +20000, weapon +60000, maze room type +70000, lore +90000, room vibe +130000) and per-cell decisions made in `GameScene` (door closability, switch placement) likewise seed from the room's stable cell coordinates and the shared `@/src/systems/random.js` PRNG, so adding/removing items in one system does not change another's sequence. The room vibe (`@/src/systems/roomVibes.js`, +130000) is a separate seed axis from the maze room type (+70000), so visual/furniture identity and structural layout vary independently. The per-floor seed is `levelSeed + floor * 1000`. The seed changes each run via an incrementing `runCount` stored in `game.registry` (and persisted to localStorage via `@/src/systems/persistence.js`). The legacy whole-level offsets (stairs +50000, extra doors +80000, alternate exits +100000) remain only in their now-unused source modules. The starting room system (`@/src/systems/startroom.js`), locations system (`@/src/systems/locations.js` for static location definitions), hiding system (`@/src/systems/hiding.js`), exploration system (`@/src/systems/exploration.js`), scaling system (`@/src/systems/scaling.js`), danger system (`@/src/systems/danger.js`), pathfinding system (`@/src/systems/pathfinding.js`), distance system (`@/src/systems/distance.js`, a pure BFS over the door/stair graph), persistence system (`@/src/systems/persistence.js`), light spill system (`@/src/systems/lightSpill.js`, a pure geometric computation from lit rooms and door states), pause controls data (`@/src/systems/pause.js`, a pure data lookup), title menu data (`@/src/systems/titleMenu.js`, a pure data lookup), audio systems (`@/src/systems/audio.js`, `@/src/systems/audioEngine.js`), and audio settings (`@/src/systems/settings.js`) do not use a seeded PRNG -- startroom uses a fixed hand-crafted layout, location definitions are static data, hiding operates on furniture objects already placed by the furniture system, exploration is deterministic based on player position, scaling applies deterministic formulas based on a room's distance from the entrance, danger uses `Math.random()` for spawn positions because dynamic mid-run spawns do not need cross-session reproducibility, persistence is a browser storage boundary, and audio uses a Knuth multiplicative hash of game time for ambient scheduling and a deterministic hash for per-enemy ambient sound cooldown resets (no effect on level determinism). The `generateAlternateExit()` function in `locations.js` is the one part of that module that does use a seeded PRNG (offset +100000)
