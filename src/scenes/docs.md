# Noridoc: scenes

Path: @/src/scenes

### Overview
- Phaser Scene subclasses that wire pure game systems to Phaser's rendering, physics, and input APIs
- Contains five scenes: `TitleScene` (main menu: displays "BACKROOMS" title with atmospheric pulsing text, provides continue/new game/delete save options based on save state, ambient drone audio, and a settings overlay for audio volume controls), `GameScene` (gameplay: movement, flashlight/darkness with a range-capped cone and an on/off toggle (F key / LIGHT button, auto-off while hiding), enemy AI, combat, healing via medkit pickup, shooting, weapon switching/pickups, battery drain (only while the flashlight is lit), item collection, enemy loot drops on death, lore note collection, closable doors, light switches, furniture hiding, exploration minimap, an effectively-unbounded lazily-generated procedural world with deep-link stairs between floors, location-themed entrance room whose only opening into the backrooms is a sizzling dimensional rift (sealed against light, with proximity crackle audio and a first-crossing flash), exit condition, HUD, desktop keyboard/mouse and mobile twin-stick touch controls, procedural audio with ambient drone and event-driven sound effects with volume-controlled playback, per-run statistics tracking), `RunSummaryScene` (end-of-run stats display: shows "RETURNED" or "YOU DIED" with animated stat reveals, supports both desktop and mobile continue input, passes through to ShopScene), `ShopScene` (between-run upgrade shop: displays gold balance, upgrade categories with buy buttons, starting location selector, lore journal overlay, "Enter Backrooms" button, finger-sized tap targets for mobile, and volume-controlled UI sound feedback), and `PauseScene` (overlay scene launched on top of a frozen GameScene, displaying a controls reference, audio volume controls, and resume instructions)
- The game boots into TitleScene (main menu), which routes to ShopScene (continue with existing save) or GameScene (new game with cleared state). The primary gameplay loop then cycles through GameScene -> RunSummaryScene (on exit or death) -> ShopScene (on continue) -> GameScene (on "Enter Backrooms"). PauseScene is an overlay that pauses and resumes GameScene without interrupting this loop

### How it fits into the larger codebase
- `@/src/main.js` registers scene classes with the Phaser.Game instance (TitleScene, GameScene, ShopScene, PauseScene, RunSummaryScene); TitleScene is listed first (auto-starts as the boot screen). PauseScene and RunSummaryScene are registered but never auto-started -- PauseScene is launched on demand by GameScene via `scene.launch('PauseScene')`, and RunSummaryScene is transitioned to by GameScene via `scene.start('RunSummaryScene')` on exit or death
- Scenes import pure functions from `@/src/systems/` for game math (including maze generation from `@/src/systems/maze.js`, per-room visual + furniture vibes from `@/src/systems/roomVibes.js`, lore spawning from `@/src/systems/lore.js`, enemy loot drops from `@/src/systems/lootDrops.js`, gunfire noise alerts from `@/src/systems/gunfireNoise.js`, coordinate-keyed unbounded room generation from `@/src/systems/worldgen.js`, AI-art helpers (texture-source selection and shop-icon mapping) from `@/src/systems/art.js`, location definitions from `@/src/systems/locations.js`, BFS room graph pathfinding from `@/src/systems/pathfinding.js`, time-based danger escalation from `@/src/systems/danger.js`, light spill gradient zones from `@/src/systems/lightSpill.js`, pause controls data from `@/src/systems/pause.js`, run statistics tracking and summary formatting from `@/src/systems/runStats.js`, audio config and per-enemy-type sound logic from `@/src/systems/audio.js`, sound synthesis from `@/src/systems/audioEngine.js`, audio settings and volume computation from `@/src/systems/settings.js`, pixel art sprite definitions and character animation definitions from `@/src/systems/sprites.js`) and handle all Phaser-specific concerns: sprite creation, animation playback, physics bodies, input binding, camera, collision callbacks, Graphics drawing, and Web Audio integration
- **AI-generated art vs procedural fallback**: `GameScene` and `ShopScene` each have a `preload()` that loads the committed AI-generated PNGs under `@/src/assets/sprites/` (discovered at build time via Vite `import.meta.glob`). `GameScene` loads every non-`shop_` PNG; `ShopScene` loads only `shop_`-prefixed PNGs (the shop background and per-upgrade icons). These PNGs are produced offline by `@/scripts/generate-art.mjs` and committed -- the game itself never calls any image API and ships no API key. For every texture key, the PNG (if present) wins; any missing key transparently falls back to the procedural pixel-grid texture from `@/src/systems/sprites.js`, so the game is always renderable even with no generated art (see `getTextureSource` in `@/src/systems/art.js`)
- Scenes are the only place in the codebase that depends on Phaser -- this boundary is intentional to keep game logic testable
- `GameScene` reads upgrade levels from `game.registry` in its `init()` method, using `getUpgradeValue()` from `@/src/systems/shop.js` to compute upgrade bonuses as deltas from pistol base values (e.g., `getUpgradeValue('weaponDamage', level) - 25` yields a +N damage bonus). These bonuses are stored in `this.upgradeBonuses` and applied additively to whatever weapon is active via `getEffectiveStats()` from `@/src/systems/weapons.js`. Binary unlock upgrades (`startingPistol`, `startingShotgun`, `startingRifle`, `minimap`) are checked as boolean flags: the three starting weapon flags are passed to `createWeaponState()`, which uses priority-based slot filling (rifle > shotgun > pistol) to equip up to 2 weapon slots -- when none are purchased the player starts unarmed, when all three are purchased the cheapest (pistol) is dropped; `minimap` sets `this.hasMinimap` which gates minimap Graphics creation and rendering. When unarmed, `getActiveWeapon()` returns null and firing is disabled until the player picks up a weapon. Survival stats (max HP, max battery, flashlight angle, speed) are still computed as absolute values. It also calls scaling functions from `@/src/systems/scaling.js` to compute run-scaled enemy stats per type -- each enemy type (basic, crawler, spitter) has its own base HP, contact damage, and chase speed constants, and scaling is applied independently to each set (e.g., `this.scaledEnemyHP`, `this.scaledCrawlerHP`, `this.scaledSpitterHP`). These values are stored as instance properties and used throughout `create()` and `update()`. Upgrade level fallback uses `createShopState().upgrades` as defaults, ensuring forward compatibility when new upgrade keys are added to saves that predate them
- `ShopScene` reads/writes `shopState` from `game.registry`, using pure functions from `@/src/systems/shop.js` for all purchase logic. It receives `treasureEarned` and optionally `newLocation` as scene data from GameScene, converts treasure to gold via `addGold()`, and if `newLocation` is provided, adds it to `unlockedLocations` in the registry. ShopScene also calls `saveGame()` from `@/src/systems/persistence.js` at every point where state is mutated: after adding treasure gold in `init()`, before entering backrooms, and after purchasing an upgrade. All `saveGame()` calls pass `collectedLore`, `unlockedLocations`, and `activeLocation` from `game.registry` to ensure all cross-run progress is persisted. ShopScene imports `getLocation()` and `LOCATIONS` from `@/src/systems/locations.js` to display location names in the location selector UI. ShopScene imports `getJournalEntries()` and `getJournalPage()` from `@/src/systems/lore.js` to power the lore journal overlay, which lets players review collected and uncollected lore notes between runs. ShopScene imports `generateAllSounds()` from `@/src/systems/audioEngine.js` to initialize procedural audio, playing UI feedback sounds for purchases, denied purchases, and button clicks
- `GameScene` calls `saveGame()` from `@/src/systems/persistence.js` after incrementing `runCount` in `init()`, persisting the updated run counter, shop state, and location data to localStorage before the run begins
- Data flows between scenes via two mechanisms: `game.registry` for in-memory cross-scene state (shopState, runCount, unlockedLocations, activeLocation) and scene `init(data)` for per-transition data. GameScene passes run statistics and outcome data to RunSummaryScene (`survived`, `treasureEarned`, `runStats`, `roomsExplored`, `hp`, `maxHp`, `newLocation`, `touchMode`). RunSummaryScene passes through `treasureEarned` and `newLocation` to ShopScene unchanged. Registry values are persisted to localStorage via `@/src/systems/persistence.js` so state survives browser reloads

### Core Implementation
- **TitleScene lifecycle**: `create()` fades in the camera, renders the "BACKROOMS" title text with a pulsing alpha tween (yellow #D4C073, oscillating between 1.0 and 0.3 over 3 seconds), a subtitle, and interactive menu options. Calls `loadGame()` from `@/src/systems/persistence.js` directly (not via registry) to check for an existing save, then `getMenuOptions(hasSave)` from `@/src/systems/titleMenu.js` to determine which menu items to display. When a save exists: Continue, New Game, and Delete Save are shown. Without a save: only New Game appears. A "SETTINGS" button is always shown below the dynamic menu options. Continue fades out and transitions to `ShopScene` (the returning player picks up from the shop). New Game calls `clearSave()` from persistence, resets all registry values (`shopState`, `runCount`, `collectedLore`, `unlockedLocations`, `activeLocation`) to defaults, and transitions to `GameScene`. Delete Save shows a confirmation dialog overlay; on confirmation, calls `clearSave()`, resets the registry, and rebuilds the menu (now showing only New Game). Settings opens a full-screen overlay with three volume cycle buttons (Master, SFX, Ambient) that read and write `audioSettings` from the registry and persist changes immediately to localStorage via `saveSettings()` from `@/src/systems/settings.js`. Volume changes immediately update the title screen's ambient drone gain via `setVolume()`. The overlay uses the same depth-based modal pattern as the delete confirmation (depth 100-101 with an interactive background that blocks clicks through to the menu). Menu selection is blocked while either overlay is open (`this.confirmOverlay || this.settingsOverlay`). Uses the same responsive camera pattern as ShopScene (`fitCamera()` with `min(width/1024, height/768)` zoom), the same `enlargeHitArea()` pattern for finger-sized tap targets, and initializes ambient drone audio via `generateAllSounds()` and `createAmbientDrone()` from `@/src/systems/audioEngine.js`. The drone volume is set on creation using `getEffectiveVolume()` from `@/src/systems/settings.js` with the current master and ambient settings. The drone is stopped on scene shutdown. The `isTransitioning` flag prevents double-transitions (same pattern as RunSummaryScene)
- **GameScene lifecycle**: `init()` reads upgrade state from registry and computes per-run stats, `create()` sets up the level and all game objects, `update()` runs every frame
- **GameScene.init()**: reads `shopState`, `collectedLore`, `activeLocation`, `unlockedLocations`, and `audioSettings` from `game.registry` (converting `collectedLore` from a persisted array to a Set; falling back to `createDefaultSettings()` from `@/src/systems/settings.js` for audioSettings). Resolves the active location data via `getLocation()` from `@/src/systems/locations.js`, falling back to `'store'` if the location ID is not found. Spreads `createShopState().upgrades` as defaults before overlaying saved upgrade levels (for forward compatibility with saves predating new upgrades), then calls `getUpgradeValue()` for each upgrade to set survival stats (`this.maxCharge`, `this.flashlightAngle`, `this.maxHp`, `this.playerSpeed`), weapon upgrade bonuses (`this.upgradeBonuses = { damage, fireRate, range }` computed as deltas from pistol base values), and inventory capacity (`this.maxItems` from the backpack upgrade). Checks the three starting weapon upgrade levels (`startingPistol`, `startingShotgun`, `startingRifle`) and initializes `this.weaponState` via `createWeaponState({ startingPistol, startingShotgun, startingRifle })` from `@/src/systems/weapons.js` -- when no upgrades are purchased the player starts unarmed with empty slots and zero ammo; purchased weapons fill up to 2 slots by priority (rifle > shotgun > pistol). Sets `this.hasMinimap` from the `minimap` upgrade level (binary flag: true when level > 0) -- this gates both `minimapGraphics` creation in `createHUD()` and `drawMinimap()` rendering. Calls `this.updateWeaponStats()` to derive effective bullet stats from the active weapon plus bonuses. Also reads and increments `runCount` in the registry to derive a unique level seed for this run (the seed for the lazy world generator) and stores `this.runCount`, and calls scaling functions from `@/src/systems/scaling.js` to precompute per-type scaled stats. Initializes `this.dangerState` via `createDangerState()` from `@/src/systems/danger.js` -- the danger state is per-run and not persisted. Initializes `this.runStats` via `createRunStats()` from `@/src/systems/runStats.js` -- the run stats state is per-run and not persisted, tracking kills, elapsed time, and deepest floor reached for the end-of-run summary screen. Initializes `this.gunshotAlertCooldown = 0` -- tracks the cooldown between gunfire noise alerts (consumed by `@/src/systems/gunfireNoise.js` integration in the update loop). After incrementing, calls `saveGame()` to persist the updated `runCount`, `shopState`, `collectedLore`, `unlockedLocations`, and `activeLocation` to localStorage
- **Lazy world generation (replaces whole-level setup)**: `GameScene` no longer builds the entire multi-floor level in `create()`. Instead it keeps a growing `this.level = { rooms, stairs }` (only activated rooms) plus lazy-generation bookkeeping: `cellId` (Map from `"floor:gx,gy"` to a numeric id), `nextRoomId`, `cellRooms` (generated room cache), `roomsById`, `activatedRoomIds` (Set), `doorConnectionKeys` (Set for door dedup), `returnStack` (stair return path), and `weaponedFloors` (per-floor weapon guard). `floorSeed(floor)` is `levelSeed + floor * 1000`. `genRoom(floor, gx, gy)` builds a room via `getRoomAt()` from `@/src/systems/worldgen.js`, then re-stamps it with a numeric `id` (via `idForCell()`, assigned on first sight so the entrance cell `(0,0)` on floor 0 is always id 0), this floor's world-space Y (`floor * FLOOR_Y_OFFSET + gy * ROOM_HEIGHT`, `FLOOR_Y_OFFSET = 5_000_000`), a `distance = floor * FLOOR_DEPTH + localDistance(gx, gy)` (`FLOOR_DEPTH = 6`, the per-room exploration depth that replaces the old `computeRoomDistances`/`this.roomDistances`), and door `targetRoomId`s rewritten from worldgen string ids to numeric ids. `activateRoom(room)` runs all per-room scene wiring exactly once (guarded by `activatedRoomIds`): appends to `this.level.rooms`/`roomsById`, draws floor/walls/doorways, builds room physics, pushes the room's perimeter occluder segments into `wallSegments` via `createRoomWalls()` from `@/src/systems/room.js` (with gaps at the doors -- EXCEPT the floor-0 rift, identified by `isRiftDoor()` from `@/src/systems/worldgen.js`, which is left out of the gap list so no light passes through it in either direction), then generates the internal maze rects FIRST (via `generateMazeWalls()`, skipped for id 0, and for rooms that have a stair passing BOTH a `stairKeepOut()` zone and a `stairLandingKeepOut()` zone from `@/src/systems/stairs.js` so the maze carves around both the centered stair AND the spot below it where the player materializes after a transition), then furniture -- `createRoomFurniture()` is handed `[...mazeRects, ...doorEntryZones(room), ...landingKeep]` as obstacles so furniture never spawns inside an internal wall, a doorway, or on the stair landing (the landing is ~140px below center, outside the furniture spawn zone, so it must be reserved explicitly; see **Intra-room placement clearance** in the systems docs) -- then renders the precomputed maze rects via `renderRoomMaze()` (which replaced the old `createRoomMaze()` that re-generated its own rects), then per-room doors, switch, items, lore, enemies, and weapon -- the three ground-pickup activators (`activateRoomItems/Lore/Weapon`, which all accept `(room, mazeRects = [])`) are each handed the same `mazeRects` and forward it as the generator's `obstacles` arg, so coins/notes/weapons are rejected out of solid maze walls where they would be uncollectable (maze walls only, not door zones -- doorways are walkable; see **Intra-room placement clearance** in the systems docs) -- and finally sets `roomGraphDirty = true`. Whether a room has a stair (and which way) is decided once by the shared `roomStairDirection(room)` helper, used by both the keep-out computation here (gating BOTH the stair and the landing keep-outs) and `activateRoomStairs()`, so the two never disagree. `ensureRoomsAround(floor, gx, gy)` activates every room returned by `roomsWithinRadius(floorSeed, gx, gy, 2)`; `ensureRoomsAroundPlayer()` (called each frame) recomputes the player's grid cell on the current floor and re-runs activation only when the player crosses into a new cell. Doors are deduped per shared edge (created when the second endpoint room activates, keyed by `doorConnectionKeys`); door closability and switch placement are seeded from canonical cell coordinates so they are independent of the order the player reaches a cell -- preserving determinism (same `runCount` produces the same world)
- **create() setup sequence**: `generateSpriteTextures()` runs first, iterating all sprite keys from `@/src/systems/sprites.js` and calling `this.textures.generate()` for each **only when `this.textures.exists(key)` is false** -- so any AI-art PNG already loaded in `preload()` is kept and only the missing keys get a procedural fallback texture. It then iterates all animation keys from `ANIM_DEFS` and calls `this.anims.create()` for each -- this pre-generates all pixel art textures and animations before any other creation method references them. Texture KEYS are identical whether sourced from PNG or procedural generation, so no gameplay/rendering code downstream changes. Then it creates the shared scene **infrastructure** once (room/furniture graphics, door group + interact prompt, switch sprite map, item/lore static groups, enemy group + colliders, weapon-pickup list, stair glow graphics), calls `ensureRoomsAround(0, 0, 0)` to generate and activate the entrance plus everything within 2 door-hops, captures `this.entranceRoom` (id 0), then creates the player (at the entrance center, registering player-vs-item/lore/enemy overlaps), player bullets, enemy bullets, exit zone, crack visual, input, camera, darkness overlay, HUD, touch controls, and audio. `createTouchControls()` is called between HUD and audio setup and is a no-op on desktop (it early-returns when `this.touchMode` is false), so it only builds the touch UI on touch devices. Audio is initialized last via `createSounds()`. Alternate exit portals were removed (the lazy world's only world-exit is the floor-0 entrance zone); the shop's location cycling is unaffected. Combat state is initialized with `createCombatState(this.maxHp)`, battery state with `createBatteryState(this.maxCharge)`, inventory state with `createInventoryState(this.maxItems)` (where `maxItems` comes from the backpack upgrade computed in `init()`), hiding state with `createHidingState()` from `@/src/systems/hiding.js`, exploration state with `createExplorationState()` from `@/src/systems/exploration.js`, and run stats state with `createRunStats()` from `@/src/systems/runStats.js` (initialized in `init()` rather than `create()`). Two additional flags are initialized: `this.isTeleporting = false` (gates update loop during stair transitions) and `this.currentFloor = 0` (tracks which floor the player is currently on). Pathfinding state is initialized with `this.roomGraph = null` and `this.roomGraphDirty = true`, so the single closed-door-respecting room graph is lazily built on the first `updateEnemies()` call. Notable setup details:
  - `createEnemies()` creates the `enemyGroup` and registers its wall/furniture/door/self colliders once; `activateRoomEnemies(room)` passes `this.extraEnemyCount` and `this.runCount` to `generateRoomEnemies()` for that room (run-scaled enemy density and type distribution). Dynamically added enemies inherit the group colliders automatically. `getEnemyStats(type)` maps each spawn's type to the correct scaled HP, contact damage, texture key (via `getEnemyTextureKey()` from `@/src/systems/sprites.js`), and body size. Each enemy sprite plays its type-specific idle animation after creation. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, `spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`, and `soundCooldown` fields alongside the existing state machine fields. The room-tracking fields (`spawnRoomId`, `currentRoomId`) are initialized to the spawning room's ID; `roomTransitionCooldown` starts at 0 and `targetDoorway` starts null. `soundCooldown` is stagger-initialized per enemy to `1000 + Math.random() * 3000` so enemies do not emit ambient sounds simultaneously. Wires a player-enemy overlap callback (`onEnemyContact`) for per-enemy contact damage
  - `createItems()` creates the `itemGroup` static physics group once; `activateRoomItems(room, mazeRects = [])` (called from `activateRoom`) calls `generateRoomItems()` for that room, passing `room.distance` as the trailing `distance` argument so loot tiers scale with exploration depth (deeper rooms favor higher-value treasure; gems only appear far from the entrance), and forwarding `mazeRects` as the generator's `obstacles` so a coin never spawns inside a solid maze wall. Places item sprites at depth 5 (below darkness) using pre-generated textures keyed as `item_{type}` from `@/src/systems/sprites.js` (including `item_medkit` for medkits). The player-item overlap callback (`onItemPickup`, registered in `createPlayer()`) auto-collects, with medkits handled as a special case before the inventory capacity check (see Item pickup flow). The per-room `distance` (computed in `genRoom()`) replaces the old `computeRoomDistances`/`this.roomDistances`
  - `createLoreItems()` creates a separate static physics group (`this.loreGroup`) once; `activateRoomLore(room, mazeRects = [])` calls `generateRoomLore()` from `@/src/systems/lore.js` for that room (passing `this.collectedLore` to filter out already-collected entries, and forwarding `mazeRects` as the generator's `obstacles` so a note never spawns inside a solid maze wall) and places lore sprites at depth 5 using the pre-generated `lore_note` texture from `@/src/systems/sprites.js`. The player-lore overlap callback (`onLorePickup`, registered in `createPlayer()`) that adds the entry ID to `this.collectedLore`, updates `game.registry`, destroys the sprite, and calls `showLorePopup()` to display the text. Lore items do NOT consume inventory space -- they bypass the inventory system entirely. `showLorePopup()` displays a tween-animated semi-transparent overlay at the bottom of the screen (depth 1001-1002) with the lore text in monospace font, auto-dismissing after 5 seconds with a fade-out. Only one popup can be active at a time (previous popup is destroyed if a new one triggers)
  - `createDoorInfra()` sets up the empty `doorStates` array, `doorGroup` static group, sprite/zone maps, and the one-time "Press E" interaction prompt text (hidden by default). `activateRoomDoors(room)` creates door states for the room's shared edges (one per connection, made only when the second endpoint room is already activated, deduped via `doorConnectionKeys`, closability seeded from the connection's canonical cell coordinates) and `createDoorSprite(door)` builds each door's static physics zone sized to fill the doorway gap plus a Phaser sprite using the pre-generated `door_closed` texture (stored in `this.doorSprites` keyed by door ID, `setDisplaySize()` to match the gap). Closed doors push their segment into `wallSegments`, enable the body, and show the sprite; open doors disable the body and hide the sprite
  - `createSwitchInfra()` sets up the empty `switchStates` array and `switchSprites` map. `activateRoomSwitch(room)` places a light switch in ~15% of non-entrance rooms (seeded from the room's cell coordinates) and creates a Phaser sprite using the pre-generated `switch_off`/`switch_on` texture at depth 50, keyed by switch ID. Switches have no physics bodies -- they are purely visual objects interacted with via proximity checks
  - `createBullets()` creates a pooled physics group (`maxSize: 30`) and wires bullet-wall/furniture/door colliders and bullet-enemy overlap. The bullet-furniture collider passes `this.bulletHitsFurniture` as its Arcade `processCallback` (4th arg) so gunfire flies over low furniture but is stopped by tall cover (see **Selective bullet-vs-furniture collision** under Things to Know). Bullet textures (keyed `bullet_{weaponId}`) are pre-generated by `generateSpriteTextures()` from definitions in `@/src/systems/sprites.js`, so `fireBullet()` can select the correct texture based on the active weapon
  - `createWeaponPickups()` initializes the empty `this.weaponPickupSprites` list; `activateRoomWeapon(room, mazeRects = [])` calls `generateRoomWeapon()` from `@/src/systems/weapons.js` for each non-entrance room and spawns at most one weapon per floor (first valid placement in activation order, guarded by `this.weaponedFloors`), forwarding `mazeRects` as the generator's `obstacles` so a dropped weapon never spawns inside a solid maze wall where it could never be picked up, using pre-generated pickup textures (keyed `pickup_{weaponId}`) from `@/src/systems/sprites.js`. All weapon types including pistol can appear as floor drops. Pickup sprites are stored in `this.weaponPickupSprites` and participate in the E-key interaction resolution system as candidates with type `'weapon'`
  - `createEnemyBullets()` creates a separate pooled physics group (`enemyBulletGroup`, `maxSize: 30`), mirroring the player bullet pattern. The `enemyBullet` texture is pre-generated by `generateSpriteTextures()`. Wires wall/furniture/door colliders and player overlap, with the furniture collider using the same `this.bulletHitsFurniture` process callback as player bullets (enemy/spitter projectiles also pass over low furniture and stop at tall cover). `fireEnemyBullet()` is called from `updateEnemies()` when a spitter's AI state machine sets `wantsToFire = true`. `updateEnemyBullets()` expires enemy bullets using `SPITTER_PROJECTILE_RANGE`. `onEnemyBulletHitPlayer()` applies `SPITTER_PROJECTILE_DAMAGE` to the player's combat state (blocked while hiding)
  - `createExitZone()` places a 64x64 physics zone at the west-center of the entrance room (`this.entranceRoom`, id 0; position computed by `getLocationExitPosition()` from `@/src/systems/locations.js`) with a green visual indicator, wiring a player overlap callback (`onDayComplete`) for the day-cycle win condition. This is the only world exit to the shop. Also stores `this.exitPosition` for use by the minimap
  - `createCrackVisual()` draws the rift -- the single entrance to the backrooms -- at the entrance room's east opening using jagged points from `generateCrackPoints()` in `@/src/systems/startroom.js`. The rift is deliberately NOT presented as a doorway: an additive-blend glow graphics (depth 95, just under the darkness overlay) sizzles each frame via `updateCrackGlow()` (combined sine waves plus per-frame random flicker, a brighter core band, and short random spark lines), so it reads as a tear of light in the wall. `createCrackVisual()` also stores `this.riftRect`, the world-space rectangle of the opening, used by `zoneTouchesRift()` to filter light-spill zones (no light leaks through the rift in either direction -- the backrooms stay invisible until crossed) and by `updateRiftCrackle()` to drive the proximity crackle volume. `updateRiftCrossing()` (frame loop) detects the player's first crossing past the entrance room's east edge and fires a one-time white camera flash + shake + transition sound (`this.riftCrossed` flag), selling the dimensional transition
  - `createStairInfra()` sets up the `stairGlowGraphics` and empty `stairZones` list once. Stairs are deep-links discovered lazily as rooms activate: `activateRoomStairs(room)` resolves direction via the shared `roomStairDirection(room)` helper (DOWN for a non-entrance room with `hasStairDown(floorSeed, gx, gy)`, UP for a non-entrance floor's entrance cell `(0,0)`) -- the same helper `activateRoom()` uses to decide whether to carve a maze keep-out, so stair placement and keep-out never disagree. Each stair gets a `STAIR_SIZE` x `STAIR_SIZE` physics zone at the canonical centered position from `getStairTopLeft()` (imported from `@/src/systems/stairs.js`, which replaced the removed local `stairTopLeft()` method) carrying `direction`, `floor`, `gx`, `gy`, and glow position data, and a dark rectangle with step lines drawn in the correct visual direction. Because the stair is centered (not searched for a clear spot), the maze in that room is carved away from a `stairKeepOut()` square around it so the stair never sits inside a wall, and a separate `stairLandingKeepOut()` square (around `getStairLandingPosition()`, where the player materializes after a transition) keeps both maze walls and furniture off the landing. Overlap with the player triggers `onStairEnter()`. A separate `stairGlowGraphics` at depth 1 draws a pulsing purple glow (0x7733aa) behind every active stair zone each frame via `updateStairGlow(time)`, using the same sine-wave pulse pattern as the crack glow
  - Alternate exit portals were removed in the lazy-world rewrite. The only world exit is the floor-0 entrance zone (`createExitZone()`); deeper floors are reached and returned from purely via stairs. The shop's starting-location cycling is unaffected (it operates on `unlockedLocations`, which can still grow through other means)
  - `createHUD()` creates a Graphics object with `setScrollFactor(0)` at depth 1000 so health and battery bars stay fixed on screen, plus text elements for battery count, treasure value, inventory capacity (`[itemCount/maxItems]` -- gray normally, turns red when full), active weapon name (colored per-weapon with a `[Q]` switch indicator), and a red "DANGER N" indicator (visible only when danger waves have been spawned, showing the current wave count from `this.dangerState.waveCount`). The `minimapGraphics` Graphics object (depth 1000, `setScrollFactor(0)`) is only created if `this.hasMinimap` is true (i.e., the minimap shop upgrade has been purchased). A floor indicator text (`this.floorText`) shows the current floor label (e.g., "B1", "B2", up to "B4" on run 4+) to indicate the player's current floor, updated each frame in `drawMinimap()`
  - `setupCameraLayers()` is the LAST step of `create()` (immediately after `createTouchControls()`) and installs the two-camera architecture (see **Two-camera zoom / HUD split** under Things to Know). - **update() frame loop**: early-returns if player is dead, `this.dayEnding` is true (exit transition in progress), or `this.isTeleporting` is true (stair transition in progress). This gunfire alert is now the ONLY place in the codebase where enemy AI state is mutated from outside the `updateEnemyAI()` function; under the sight-based AI an alerted enemy without LOS falls through to search toward that snapshot, investigating the noise rather than tracking the live player. A successful alert resets `this.gunshotAlertCooldown` to `GUNSHOT_ALERT_COOLDOWN` (5000ms), preventing rapid re-alerting. The cooldown is decremented by `delta` each frame before the fire check -> `updateInteractPrompt()` resolves the nearest interactable (door, switch, hideable furniture, or weapon pickup by Euclidean distance) and shows/hides "Press E" -> `updateTouchButtons()` shows/hides the on-screen USE/WPN buttons and dims FIRE based on game state (touch only) -> checks Q key (`JustDown`) or scroll wheel (with 200ms cooldown) to switch weapons via `triggerWeaponSwitch()` -> checks E key (`JustDown`) and dispatches via `triggerInteract()` (door/switch/furniture/weapon, or `onExitHiding()` while already hiding) -> checks F key (`JustDown`) to toggle the flashlight via `toggleFlashlightAction()` (shared with the touch LIGHT button) -> `updateRiftCrossing()` fires the one-time rift-crossing flash/shake on first passage into the backrooms -> `updateBullets()` expires out-of-range player bullets (using per-bullet range from `getData('range')`) -> `updateEnemyBullets()` expires out-of-range enemy projectiles -> `updateEnemies(delta)` first updates the danger system: accumulates unsafe time via `updateDangerTime()` (safe rooms are room 0 and lit rooms), then spawns danger waves via a `while (shouldSpawnWave())` loop that calls `spawnDangerWave()` and `advanceWave()` for each triggered wave -- multiple waves can fire in a single tick if enough time has accumulated. Then runs AI per enemy (against a once-per-frame `filterSegmentsNear()`-prefiltered occluder set sized to the enemy `DETECTION_RANGE` -- enemies share the same `wallSegments` the flashlight uses, so they cannot see through walls, and the prefilter keeps LOS raycasts cheap as occluders accumulate in the unbounded world; Right-click input triggers `onUseBattery()` outside the frame loop via a `pointerdown` event listener (blocked while hiding). Item pickup happens via Phaser overlap callbacks (auto-collected on contact)
- **Enemy integration pattern**: `GameScene` maintains a parallel `enemyStates` array alongside the Phaser physics sprites in `enemyGroup`. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, room-tracking fields (`spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`), `soundCooldown` (for per-enemy ambient sound timing), and standard AI state machine fields including the `lastKnownX/Y` last-seen-position memory. Enemies are added to `enemyStates` both at level creation (via `createEnemies()`) and dynamically during gameplay (via `spawnDangerWave()`, which creates new sprites in `enemyGroup` and pushes new state objects with the same structure, including stagger-initialized `soundCooldown`). Dynamic spawns automatically inherit existing colliders because Phaser group colliders cover all current and future group members. Each frame, `updateEnemies()` rebuilds the single room graph if dirty (via `buildRoomGraph()` from `@/src/systems/pathfinding.js`, cached and invalidated when `onToggleDoor()` or `activateRoom()` sets `roomGraphDirty = true`), counts enemies per room, detects room transitions (updating `currentRoomId` and resetting cooldown/targetDoorway when an enemy enters a new room), evaluates idle doorway-seeking eligibility (gated by `DOORWAY_SEEK_CHANCE`, cooldown, `MAX_ROOM_DISTANCE` from spawn via `getRoomDistance()`, and `MAX_ROOM_ENEMIES` cap), and constructs a slim per-enemy `navContext` of `{ roomTransitionCooldown, targetDoorway, doorway, seekDoorway }` -- this carries ONLY idle-wander data, passed as the 7th argument to `updateEnemyAI()`. The room graph is never consulted for chase routing: pursuit is strictly sight-based inside `updateEnemyAI()` (the AI stamps `lastKnownX/Y` while it has LOS and searches toward that stored point when LOS breaks -- see `@/src/systems/enemy.js` and its docs). After the AI update, state machine and room-tracking fields (including `lastKnownX/Y`, `roomTransitionCooldown`, `targetDoorway`) are copied back from the result. If `wantsToFire` is true on the result, GameScene calls `fireEnemyBullet()` -- pure function signals intent, scene executes side effect. After AI and side-effect processing, each enemy sprite's rotation is set from its velocity direction (`Math.atan2`), and its animation is switched via `getAnimKeyForState(type, state)` from `@/src/systems/sprites.js` -- this maps the AI state (idle/chase/search/attack) to the correct animation key (e.g., chase -> walk animation, attack -> spitter attack animation). The `play(key, true)` call with `ignoreIfPlaying=true` prevents animation restarts when the state hasn't changed. On death, `enemySprite.stop()` freezes the current animation frame before applying corpse visuals. This keeps the AI logic testable in `@/src/systems/enemy.js` while letting Phaser handle physics resolution, animation playback, and projectile management
- **ShopScene lifecycle**: `init(data)` receives `treasureEarned` and optionally `newLocation` from GameScene, reads existing `shopState` from registry (or creates fresh state on first visit), adds the treasure as gold via `addGold()`, writes back to registry. - **RunSummaryScene lifecycle**: `init(data)` receives run data from GameScene (`survived`, `treasureEarned`, `runStats`, `roomsExplored`, `hp`, `maxHp`, `newLocation`, `touchMode`), with safe defaults for all fields. `create()` calls `getSummaryData()` from `@/src/systems/runStats.js` to compute the display data (title, colors, stat lines), then renders a black background with the title ("RETURNED" in gold or "YOU DIED" in red -- never "escaped", and never an "X / total rooms" ratio, since the unbounded world has no total) faded in over 800ms.   Items at depth 5, enemies at depth 60, and furniture at depth 50 are all hidden by the darkness overlay and only revealed by the flashlight. Bullets at depth 150 are always visible (above darkness), giving visual feedback even when firing into dark areas. The HUD uses `setScrollFactor(0)` so it does not move with the camera AND is rendered by the separate un-zoomed `this.uiCamera` (see **Two-camera zoom / HUD split**) so the `CAMERA_ZOOM` applied to the world camera does not also scale it
- **Two-camera zoom / HUD split**: `GameScene` renders through TWO cameras. `cameras.main` is the world camera, zoomed in to `CAMERA_ZOOM` (`2.0`, from `@/src/systems/camera.js`) so the player and world read bigger and "closer" -- the deliberate fix for the binding spec's repeated "make the player bigger / rooms smaller / everything closer / claustrophobic" feedback. `this.uiCamera` is a second, un-zoomed camera added at viewport size that renders the HUD at 1:1. A dedicated UI camera is REQUIRED because Phaser's `setZoom()` also scales `scrollFactor(0)` objects, so HUD elements would otherwise zoom along with the world. `setupCameraLayers()` seeds the split: `partitionSceneObjects(this.children.list)` from `@/src/systems/camera.js` classifies every existing object (HUD = pinned on BOTH axes; everything else = world), then `cameras.main.ignore(hud)` and `uiCamera.ignore(world)`. The classification invariant is **scrollFactor**, not depth: the only one-axis-pinned WORLD object is the "Press E" `interactText` (`scrollFactorX = 1`, follows the world even though it sits at a HUD-like depth), which therefore renders on the zoomed main camera. Because `camera.ignore(group)` does NOT cover objects created later, an `ADDED_TO_SCENE` listener (`this._routeNewObject`, registered on `this.events`, removed on `shutdown`) routes every newly-created object -- pooled bullets, danger-wave enemies, loot drops, the lore popup -- to the correct camera. The listener keys off a `this._buildingHud` flag (default false, so new objects default to world) rather than reading `scrollFactor`, because `ADDED_TO_SCENE` fires BEFORE chained `.setScrollFactor()` runs and the value is not yet readable; `showLorePopup()` wraps its two `this.add.*` calls in `this._buildingHud = true/false` so the dynamic lore popup routes to the UI camera
- **Camera effects span both cameras vs. one**: the three scene-transition fades -- stair teleport (`onStairEnter()` fadeOut/fadeIn), day-complete exit (`onDayComplete()` fadeOut), and death (`onPlayerDeath()` fadeOut) -- are mirrored onto `this.uiCamera` so the HUD fades together with the world during transitions. Camera shake and flash (combat damage, the rift-crossing flash, the medkit green flash) stay on `cameras.main` ONLY, so they shake/flash the world without disturbing the fixed HUD
- **Darkness system architecture**: the BitmapMask has `invertAlpha = true`, so white pixels in `maskGraphics` punch transparent holes in `darkGraphics`, revealing the room beneath. The `updateDarkness(time)` method first draws lit room rectangles (from `getLitRoomIds()`, which always includes room 0 since it is permanently lit) into both `maskGraphics` (white, to punch holes in darkness) and `lightGraphics` (faint yellow tint). After lit rooms, it computes light spill zones via `getLightSpillZones()` from `@/src/systems/lightSpill.js` and draws each zone as a gradient rectangle into both `maskGraphics` (using `fillGradientStyle` for per-corner alpha that creates gradient visibility through the BitmapMask) and `lightGraphics` (warm tint at 5% of the spill alpha). The gradient fades from full brightness at the doorway edge to zero over 150px (`LIGHT_SPILL_DEPTH`). Closed doors block spill entirely; both-lit room pairs are skipped (already fully revealed); and any spill zone touching `this.riftRect` is filtered out via `zoneTouchesRift()`, so no light leaks through the rift in either direction (the permanently-lit store would otherwise reveal the first backroom before the player ever crosses). Then the flashlight polygon is drawn only if `isFlashlightLit()` from `@/src/systems/flashlight.js` is true (light switched on and not hiding); it calls `getFlashlightConeAngle()` for the battery-adjusted cone angle and `shouldFlicker()` for low-battery flicker, returning early without the polygon when the light is off, depleted, or flickering. The polygon itself comes from `getFlashlightPolygon()` with the occluder array prefiltered by `filterSegmentsNear()` and capped at `FLASHLIGHT_RANGE` (both from `@/src/systems/visibility.js`), so sight is range-limited and the raycast cost stays bounded as the world grows. Lit rooms and their spill zones remain visible even when the flashlight is dark -- they use the same BitmapMask mechanism but draw rectangles instead of the flashlight polygon
- **Enemy physics uses a dynamic group** (unlike walls and furniture which are static groups). Enemies collide with walls, furniture, and each other. Player-enemy interaction uses `overlap` (not `collider`), triggering `onEnemyContact(player, enemySprite)` which looks up the specific enemy state object to read its per-type `contactDamage` (falling back to `this.scaledEnemyDamage` if the state is not found), applies it via `applyDamage()`, triggers camera shake/flash, and checks for player death
- **Player bullet physics group** uses pooling (`maxSize: 30`). Bullet-wall and bullet-door use `collider` (instant deactivation via `onBulletHitWall`). Bullet-furniture also uses `collider` but is gated by the `this.bulletHitsFurniture` process callback so only tall/solid furniture stops it (see **Selective bullet-vs-furniture collision**). Bullet-enemy uses `overlap` -- on hit, the bullet is deactivated and the enemy takes damage read from per-bullet data (`bullet.getData('damage')`, falling back to `this.bulletDamage`). When an enemy dies, the kill is recorded via `recordKill()` from `@/src/systems/runStats.js`, and a per-type death sound is played via `playSoundAtDistance(getEnemyDeathSound(type), distance)` from `@/src/systems/audio.js` (e.g., zombie_death for basic, crawler_death for crawler, spitter_death for spitter), with volume attenuated by the enemy's distance from the player. The sprite is kept visible as a corpse: `setActive(false)` (removes from physics/overlap checks), `body.enable = false`, and corpse visual constants from `@/src/systems/combat.js` are applied (gray tint, faded alpha, toppled angle, depth 5). After corpse visuals, `rollEnemyDrop()` from `@/src/systems/lootDrops.js` is called with the enemy's type, the room's exploration depth (the per-room `room.distance` field stamped during lazy generation -- see `@/src/scenes/GameScene.js` `genRoom()`), and `Math.random`. If a drop is returned, a sprite is created in `this.itemGroup` at the enemy's position with the item's type and value set via `setData()` -- the existing `onItemPickup` overlap collider auto-registers the dropped item for pickup with no additional wiring. A `loot_drop` sound plays at the death distance. The sprite remains in `enemyGroup` but Phaser skips inactive sprites in overlap checks, so corpses cannot trigger contact damage. The enemy state is removed from `enemyStates`. Each bullet stores its origin, damage, and range via Phaser's `setData()` at spawn time in `spawnBullet()`. This per-bullet approach ensures that bullets fired with one weapon retain their stats even after the player switches weapons mid-flight. `fireBullet()` selects the correct texture (`bullet_{weaponId}`) based on the active weapon and dispatches to either `calculateShotgunSpread()` (for multi-pellet weapons with `bulletCount > 1`) or `calculateBulletVelocity()` (for single-bullet weapons). Fire cooldown resets to `this.fireRate` and range expiry checks per-bullet range -- both derived from the active weapon via `updateWeaponStats()`. **Enemy bullet physics group** (`enemyBulletGroup`) mirrors this pattern with `maxSize: 30`, using wall/furniture/door colliders and a player overlap callback (`onEnemyBulletHitPlayer`). Enemy bullets use `SPITTER_PROJECTILE_SPEED` and `SPITTER_PROJECTILE_RANGE` constants directly (no upgrades). Enemy projectile damage is blocked while the player is hiding
- **Selective bullet-vs-furniture collision** (the spec's binding "gunfire goes over low furniture, blocked by armoires" feedback): all furniture stays in the single static `furnitureGroup`, but each furniture zone is tagged at creation in `createRoomFurniture()` via `zone.setData('blocksBullets', blocksBullets(item.type))` (from `@/src/systems/furniture.js`, true iff the type also blocks light/LoS). Both bullet-vs-furniture colliders (player `bulletGroup` and `enemyBulletGroup`) pass `this.bulletHitsFurniture` as the Arcade physics `processCallback` (the collider's 4th arg). `bulletHitsFurniture(bullet, furnitureZone)` returns `furnitureZone.getData('blocksBullets')`: returning a truthy value lets `World.separate()` proceed (the bullet is separated/stopped and `onBulletHitWall` fires), returning `false` makes Arcade skip separation entirely so the bullet flies straight over low furniture (tables, desks, beds) without deactivating. This is the only `processCallback` in the codebase -- it is the chosen mechanism because it preserves the single-group invariant (furniture stays in one group, so the UNCHANGED player-body collider `this.playerFurnitureCollider` and the enemy-body furniture collider still block ALL furniture -- the player still cannot walk through a table; only bullets pass selectively). Phaser's strict `=== false` check means the callback must return a real boolean, which `getData()` provides (`true`/`false`)
- **Player death flow**: when `combatState.isDead` becomes true, `onPlayerDeath()` stops the player body, disables physics, triggers a 500ms camera fade-out (mirrored onto `this.uiCamera` so the HUD fades with the world -- see **Two-camera zoom / HUD split**), and transitions to RunSummaryScene with `{ survived: false, treasureEarned: 0, runStats, roomsExplored, hp: 0, maxHp, newLocation: null, touchMode }`. RunSummaryScene then passes through to ShopScene with `{ treasureEarned: 0 }`. The death penalty is losing all treasure collected that run -- gold already in the shop balance is not affected. The `update()` loop early-returns while dead, preventing further input processing
- **Stair teleportation flow (deep-link + return stack)**: stairs are deep-links rather than fixed pairs. On overlap, `onStairEnter()` reads the zone's `direction`. A DOWN stair pushes `{ floor, gx, gy }` (its own room) onto `this.returnStack` and targets floor `+1` cell `(0,0)`; an UP stair (at a floor's `(0,0)`) pops the stack and targets that recorded floor/cell (defensively falling back to the floor-0 entrance if the stack is empty). It then sets `this.isTeleporting = true`, stops/disables the player body, and triggers a 300ms camera fade-out (mirrored onto `this.uiCamera`). On fade-out complete, it sets `this.currentFloor` to the destination floor, calls `ensureRoomsAround(destFloor, destGx, destGy)` to lazily generate the destination neighborhood, teleports the player (`body.reset()`) to `getStairLandingPosition(destRoom)` from `@/src/systems/stairs.js` (single-sourced -- the same point `activateRoom()` reserves via `stairLandingKeepOut()`, ~140px below the centered stair so the overlap does not re-fire), re-enables the body, updates `this.runStats` via `updateMaxFloor()` from `@/src/systems/runStats.js`, and begins a 300ms fade-in. `isTeleporting` is cleared on fade-in complete. The `update()` loop early-returns while `isTeleporting` is true, freezing all game logic during the transition. This follows the same camera-fade + boolean-gate pattern as `dayEnding`
- **Day-complete flow (exit)**: when the player overlaps the floor-0 entrance exit zone, `onDayComplete(newLocation)` sets `this.dayEnding = true`, stops the player, triggers a 1000ms camera fade-out (mirrored onto `this.uiCamera`), and transitions to RunSummaryScene with `{ survived: true, treasureEarned, runStats, roomsExplored, hp, maxHp, newLocation, touchMode }`. RunSummaryScene then passes through to ShopScene with `{ treasureEarned, newLocation }`. `newLocation` is null for the normal exit (the `newLocation` parameter remains in the signature but alternate exit portals were removed, so it is always null at runtime). The player keeps all collected treasure. The `update()` loop early-returns while `dayEnding` is true, freezing game state during the transition
- **Visual feedback for combat, battery, hiding, inventory, and weapons**: taking damage triggers `cameras.main.shake()` and `cameras.main.flash()`. `drawPlayer()` applies visual states in priority order: hiding (muted green tint, alpha 0.3), hurt (a high-contrast white blink driven by `getHurtFlash()` from `@/src/systems/combat.js`, alternating `setTintFill(white)`/clear with alpha never below 0.7 -- the old constant half-alpha dark-red tint made the player vanish over dark floors), or normal (no tint, alpha 1). The HUD bars are labeled (`HP`, `FLASHLIGHT [F]`); the health bar turns red when below 30% HP, and the battery bar transitions yellow -> orange -> red as it drains, graying out entirely when the flashlight is switched off. A hint line below the HUD explains the battery mechanic contextually: "FLASHLIGHT OFF" when toggled off, "Hiding -- flashlight dark" while hiding, and `getBatteryHint()` from `@/src/systems/battery.js` at half charge or lower; a one-time toast on the first run (runCount 0) spells out the drain/narrow/recharge mechanic up front. Below the bars, text elements display the current battery item count (`BAT xN`), accumulated treasure value (`$N`), and inventory capacity (`[itemCount/maxItems]` in gray, turning red when inventory is full). A weapon text element shows the active weapon name in the weapon's color, with a `[Q]` indicator when a second weapon is available for switching
- **E-key interaction resolution**: `updateInteractPrompt()` runs each frame and finds the nearest door, nearest switch, nearest hideable furniture, and nearest weapon pickup (within `WEAPON_PICKUP_RANGE` of 80px) independently. All candidates are collected into a candidates array, sorted by Euclidean distance to the player, and the closest one wins. The resolved interactable is stored in `this.nearestInteractable` (with `type` and `item` fields), and E-key press dispatches to `onToggleDoor()`, `onToggleSwitch()`, `onEnterHiding()`, or `onWeaponPickup()` based on the type. Pressing E while already hiding calls `onExitHiding()` regardless of nearby interactables. This candidates-array approach is extensible for future interactable types
- **Hiding mechanics**: when the player presses E near `canHide` furniture (any furniture type with `canHide: true` in `@/src/systems/furniture.js`), `onEnterHiding(furniture)` snaps the player to the furniture center via `body.reset()`, sets `body.setBoundsRectangle()` to constrain movement within the furniture area, enables `setCollideWorldBounds(true)`, and disables the player-furniture collider (`this.playerFurnitureCollider.active = false`) so physics does not push the player out. While hidden: the player moves at half speed (`HIDING_SPEED_MULTIPLIER = 0.5` from `@/src/systems/hiding.js`), shooting is disabled, right-click battery use is disabled, the player is immune to enemy contact damage (`onEnemyContact()` early-returns), and enemies cannot detect the player (the `playerHidden` flag forces `canSee = false` in `updateEnemyAI()`). The flashlight is auto-darkened via `setFlashlightHiding(true)` from `@/src/systems/flashlight.js` -- no cone is drawn and the battery stops draining while hidden, but the on/off switch state is preserved and restored on exit. The player graphic renders semi-transparent (alpha 0.3) with muted green colors. The player exits hiding only by pressing E, which calls `onExitHiding()` to restore `setCollideWorldBounds(false)`, re-enable the furniture collider, and clear the flashlight hiding flag
- **Exploration and minimap**: `GameScene` maintains `this.explorationState` (from `@/src/systems/exploration.js`) which tracks visited rooms as an immutable Set regardless of whether the minimap upgrade is purchased (future features may use visitation data). Each frame, `updateExploration()` checks the player's position against all rooms and adds newly visited rooms to the set. `drawMinimap()` early-returns if `this.hasMinimap` is false (the minimap shop upgrade has not been purchased -- cost 3000 gold, single tier). For the unbounded world it calls `getWindowedMinimapData()` from `@/src/systems/exploration.js` with the player position, exit position, screen width, `this.currentFloor`, `FLOOR_Y_OFFSET`, and `MINIMAP_WINDOW_CELLS` -- this renders a player-centered grid window (+/- 4 cells by default) of visited rooms on the current floor at a fixed scale, instead of scaling a whole-floor bounding box (which an unbounded world does not have). The minimap uses a separate `minimapGraphics` Graphics object (not the same as `hudGraphics`) so it can be cleared and redrawn independently. The exit position (`this.exitPosition`) is stored during `createExitZone()` and passed through so the exit dot is visible when within the window on floor 0
- **Closable door toggle flow**: when the player toggles a door, `onToggleDoor()` does four things: (1) if the door was closed, finds and splices its segment out of `wallSegments` via `indexOf`; (2) calls `toggleDoor()` from `doors.js` to produce new immutable door states; (3) enables/disables the Phaser physics body on the door zone; (4) sets `this.roomGraphDirty = true` to invalidate the cached pathfinding room graph, so the next `updateEnemies()` call rebuilds it with the updated door state. This ensures physics, raycasting, enemy LOS, and the wander/alert graph all update atomically. Player, enemy, and bullet groups all have colliders against `doorGroup`, so closed doors block all three entity types, break enemy line of sight (the door's segment is an occluder), and exclude the connection from the pathfinding graph -- making door-closing a universal defense against every enemy type. Only the player can toggle doors; enemies cannot open them (the former crawler door-opening behavior was removed under the sight-based pursuit rule)
- **Light switch toggle flow**: when the player presses E near a switch, `onToggleSwitch()` calls `toggleSwitch()` from `lightswitch.js` to produce new immutable switch states. The method is a pure state toggle with no enemy side effects. On subsequent frames, `updateEnemies()` checks `getLitRoomIds()` and freezes (zero velocity, skip AI via `continue`) any enemy that is inside a lit room. This means enemies in lit rooms are frozen in place but remain alive and visible -- if the switch is turned off, they resume AI. This is the sole mechanism for handling enemies in lit rooms (both when the light first turns on and when enemies later wander in)
- **Room visual + furniture vibes**: each non-starting room has a coherent-but-distinct identity (palette + furniture profile) from `@/src/systems/roomVibes.js` -- every vibe stays desaturated/dim so the world still reads as liminal, but adjacent rooms feel like different places (furniture store, library, derelict ship, overgrowth, ...). During activation, `GameScene` stores the vibe in the `this.roomThemes` Map keyed by room ID (the Map name is unchanged; it now holds vibe objects). Room 0's entry comes from `this.activeLocationData` (resolved from `@/src/systems/locations.js` in `init()`). Non-starting rooms derive their vibe from `getRoomVibe(room.seed)` -- this is a seed axis ORTHOGONAL to `getRoomType()` from `@/src/systems/maze.js` (vibe +130000 vs. structural type +70000), so a library can be a maze and a forest can be a column hall. The cached `roomThemes` Map is consumed by `drawRoom(gfx, room, theme)` (floor and wall fill), `drawRoomDoorways(gfx, room, theme)` (doorway floor fill, called per room during activation), and `renderRoomMaze(gfx, mazeRects, theme)` (internal maze wall fill from precomputed rects). `createRoomFurniture(gfx, room, vibe, obstacles)` passes `vibe.furniture` as the profile and the maze rects + door entry zones as `obstacles` to `generateRoomFurniture()` for non-starting rooms (so each vibe leaves behind its own kind of furniture, kept clear of internal walls and doorways), and calls `getLocationLayout()` from `locations.js` for room 0 instead. Room 0 is permanently lit -- its ID is injected into the `litRoomIds` array in both `updateDarkness()` and `updateEnemies()`, reusing the same lit-room mechanism as light switches. This means room 0 has no darkness overlay, and any enemies that wander in are frozen. The exit zone is at the west-center of the room, positioned by `getLocationExitPosition()`. The rift visual at the entrance room's east opening is drawn by `createCrackVisual()` using `generateCrackPoints()` from `@/src/systems/startroom.js` (the rift is identical across all locations -- see the rift bullet in Core Implementation)
- **Multi-room physics with door gaps**: `createRoomPhysics()` splits each wall into physics zones that skip over door offsets. This mirrors the raycasting segments produced by `createRoomWalls()` so that physics collision and flashlight occlusion agree on where doors are. Closable doors fill these gaps with their own physics zones that can be toggled
- **Touch controls wiring** (`GameScene`): `setupInput()` decides `this.touchMode` once via `detectTouchPrimary()` from `@/src/systems/touchControls.js` (coarse-pointer media query OR `navigator.maxTouchPoints > 0`) and, when true, calls `this.input.addPointer(3)` to enable multitouch (so a finger can hold a stick and tap a button simultaneously). `createTouchControls()` (no-op on desktop) builds two rex virtual joysticks fixed to the camera (`scrollFactor 0`, high depth `TOUCH_UI_DEPTH`), plus a polled FIRE button near the right thumb and tap buttons USE/WPN/BATT/LIGHT (LIGHT calls the same `toggleFlashlightAction()` as the desktop F key), plus an FS (fullscreen) button. Every position comes from the pure `computeTouchLayout({ width, height, stickRadius })` in `@/src/systems/touchControls.js` keyed on the live `this.scale.width`/`height` -- there are no hardcoded coordinates -- which centers both sticks vertically (move on the left, aim on the right) so the thumb can deflect fully downward without the mobile browser's bottom gesture zone eating the drag. The FIRE button is NOT a Phaser interactive object: `isFireHeld()` polls `pointer1..4` each frame and returns true when any down pointer falls within the fire-button radius -- this is what `resolveFireIntent` consumes, and is why touching a stick (reported as a left-button press) does not fire. `updateTouchButtons()` shows USE only when there is a nearby interactable (or while hiding), shows WPN only when more than one weapon slot is filled, and dims FIRE when there is no weapon/ammo. `triggerWeaponSwitch()` and `triggerInteract()` are extracted shared handlers so keyboard (Q/E), the scroll wheel, and the on-screen buttons all run identical logic; both guard against dead/dayEnding/teleporting state because the touch buttons fire outside the `update()` loop (via `pointerdown` callbacks) where the normal early-return gate does not apply
- **Touch controls respond to viewport changes** (`GameScene`): touch devices run under `Scale.RESIZE` (see `@/src/main.js`), so the canvas equals the device viewport and resizes on orientation flips, URL-bar hide/show, and fullscreen toggles. `createTouchControls()` registers a `this.scale.on('resize', ...)` handler that calls `layoutTouchControls()`, which recomputes `computeTouchLayout()` and repositions every stick, button (via the `moveTouchButton()` helper), and the right-anchored `floorText` HUD element. The handler is removed in the scene `shutdown` listener (alongside destroying the joysticks). To help hide the browser chrome on Android, `createTouchControls()` also registers `this.input.once('pointerdown', ...)` that calls `this.scale.startFullscreen()` on the first touch when the Fullscreen API is available; the FS button toggles fullscreen thereafter
- **ShopScene camera zoom-fit under RESIZE**: all shop content is placed in a fixed 1024x768 design space (e.g., the title at x=512, the journal overlay sized 1024x768). On desktop the scene runs under `Scale.FIT` so this already maps cleanly. On touch the game runs under `Scale.RESIZE` at native viewport size (see `@/src/main.js`), which would otherwise misalign or clip the design-space layout. `fitShopCamera()` solves this by setting the main camera zoom to `min(width/1024, height/768)` and centering it on `(512, 384)`, so the entire 1024x768 layout scales to fit the viewport (letterboxed, never clipped). It runs in `create()` and on every `this.scale` `resize` event (handler removed on `shutdown`). On desktop FIT it is a no-op (zoom 1, already centered)
- **ShopScene tap targets**: the shop already drove all buttons from `pointerdown`, so it worked on touch -- the only problem was that text buttons had hit areas exactly the size of their (small) glyphs. `enlargeHitArea(obj, padX, padY)` sets an oversized `Phaser.Geom.Rectangle` hit area padded beyond the text bounds (and sets a pointer cursor) so every interactive text button (location arrows, BUY, ENTER, NOTES, and journal PREV/NEXT/CLOSE) is finger-sized without changing its visual size
- **Right-click and scroll input**: `setupInput()` disables the browser context menu (`this.input.mouse.disableContextMenu()`) and listens for `pointerdown` and `wheel` events. Right-click triggers `onUseBattery()`, which calls `useBattery()` from `inventory.js` and, if successful, `rechargeBattery()` from `battery.js`. Scroll wheel triggers `switchWeapon()` from `@/src/systems/weapons.js` with a 200ms cooldown (`this.lastScrollTime`) to prevent rapid switching. Both inputs are guarded by `!isDead && !dayEnding` checks
- **Item pickup flow**: items are a static physics group with player overlap detection. On overlap, `onItemPickup()` first checks the item type. **Medkits** are handled before the inventory check: if the player is at full health, the function early-returns (medkit stays on the ground); otherwise, `applyHeal(this.combatState, MEDKIT_HEAL_AMOUNT)` from `@/src/systems/combat.js` heals the player, a `medkit_pickup` sound plays, a green camera flash fires (100ms, alpha 0.15), and the sprite is destroyed. Medkits bypass inventory entirely. For all other item types, `canPickupItem(this.inventoryState)` from `@/src/systems/inventory.js` is checked -- if the inventory is full, the function early-returns and the item remains on the ground for later pickup. Otherwise, it updates `this.inventoryState` via `pickupItem()` and destroys the item sprite. Items store their type and value via Phaser's `setData()` on the sprite. Items do not produce wall segments -- they are collectibles, not occluders. Lore items use a completely separate physics group (`this.loreGroup`) and pickup callback (`onLorePickup`) -- they bypass the inventory system entirely and are always collected on contact regardless of inventory capacity
- **Wall dual-purpose**: walls exist as both Phaser Arcade Physics static zones (for player collision) and as line segments from `createRoomWalls()` (for raycasting). Light-blocking furniture (any type with `blocksLight: true` in `@/src/systems/furniture.js`, e.g., shelf, bookcase, armoire, closet) and closable doors follow the same pattern -- each is both a physics zone and a raycasting segment. Non-blocking furniture (types with `blocksLight: false`, e.g., table, desk, bed, vent) still has physics zones (so the player and enemies cannot walk through it) but does not produce raycasting segments -- and, via the same `blocksLight` classification surfaced as `blocksBullets()`, both light AND gunfire pass over it (see **Selective bullet-vs-furniture collision**). Maze internal walls are solid rects from `generateMazeWalls()` in `@/src/systems/maze.js`: `activateRoom()` generates the rects (passing stair and stair-landing keep-out zones where applicable) and `renderRoomMaze()` draws each rect exactly with `fillRect`, adds a single physics zone matching the rect, and pushes the rect's four outline segments (via `wallRectSegments()` from the same module) into `wallSegments` -- so collision, the drawn wall, and light occlusion are all the same rectangle. (Previously the scene drew and collided thick bands centered on zero-width occluder segments, which let the flashlight leak through wall edges and rendered/collided box obstacles slightly proud of their occluders; box obstacles now appear at their true footprints.) Doors are unique in that they can toggle between active/inactive at runtime via `body.enable` and `wallSegments` push/splice. Enemies, items, and light switches do not produce segments; enemies are dynamic actors, items are small collectibles using staticGroup + overlap, and light switches are proximity-based interactables with no physics body
- **Player rendering**: the player is an animated pixel art sprite at depth 200 (above the darkness overlay at depth 100), so the player is always visible regardless of flashlight direction. The sprite's display size (40px, with a smaller physics body) was deliberately scaled up in the readability pass -- alongside larger enemy display sizes and scaled-up item/weapon-pickup sprites -- so entities are tellable apart at a glance in the smaller liminal rooms. `drawPlayer()` switches between `player_walk` and `player_idle` animations based on velocity, sets rotation to the aim angle, and applies visual states: hiding (muted green tint, alpha 0.3), hurt (white blink via `getHurtFlash()` from `@/src/systems/combat.js`), or normal (no tint, alpha 1)
- Camera and physics world bounds are a large fixed region (`WORLD_BOUND`, a `GameScene` constant) rather than computed from room positions, because the lazily-generated world is effectively unbounded and has no finite extent. The darkness overlay and its flashlight BitmapMask are sized from `cameras.main.worldView` each frame for the same reason. `worldView` already accounts for the main camera's `CAMERA_ZOOM`, so the darkness/flashlight mask -- which draws in WORLD coordinates -- stays correctly aligned under zoom with no zoom-specific code (this is why the camera zoom required no change to the darkness system)
- **Audio integration**: both scenes initialize audio via a shared pattern: check for `this.sound.context`, handle browser autoplay policy (if `this.sound.locked`, defer initialization to the `'unlocked'` event), then call `generateAllSounds()` from `@/src/systems/audioEngine.js` to render all `SOUND_CONFIGS` from `@/src/systems/audio.js` into Phaser's audio cache via `OfflineAudioContext`. The `this.audioReady` flag gates all `playSound()` calls to prevent errors before sounds are rendered. `GameScene.createSounds()` additionally starts a continuous ambient drone via `createAmbientDrone()`, starts the looping rift static via `createRiftCrackle()` (volume driven per-frame by `updateRiftCrackle()` from player distance to the rift, so it grows louder on approach), and schedules recurring ambient sounds (distant_bang, drip, whisper) at pseudo-random intervals via `scheduleAmbientSound()`. Sound effects are wired to game events throughout GameScene (weapon fire, bullet hit, per-type enemy death/alert/ambient/fire sounds, player damage/death, item/lore/ammo pickup, battery recharge, door open/close, switch click, hiding enter/exit, stair transition, day complete, weapon switch) and ShopScene (purchase, denied, button clicks). All audio playback is modulated by audio settings from `@/src/systems/settings.js`: `GameScene.playSound(key)` multiplies by `getEffectiveVolume(master, sfx)` and skips playback when the result is zero. `GameScene.playSoundAtDistance(key, distance)` computes distance-based volume via `getDistanceVolume()` from `@/src/systems/audio.js` (linear falloff over `ENEMY_SOUND_RANGE` of 400px), then multiplies by `getEffectiveVolume(master, sfx)` and skips playback when either factor is zero. `ShopScene.playSound()` multiplies `0.5 * getEffectiveVolume(master, sfx)`. The ambient drone's gain is set on creation and updated at runtime via `setVolume(getEffectiveVolume(master, ambient))`. Scheduled ambient sounds pass `getEffectiveVolume(master, ambient)` as the `volumeMultiplier` to `playAmbientSound()`. `GameScene.applyVolumeSettings()` re-reads audioSettings from the registry and updates the drone gain -- this is called by PauseScene when the player changes volume mid-game. TitleScene applies the same volume pattern to its own drone instance. The ambient drone, rift crackle, and ambient sound timer are stopped on scene exit (`stopAmbientAudio()` called in `onDayComplete()` and `onPlayerDeath()`). All `playSound()` and `playSoundAtDistance()` calls are wrapped in try-catch as a system boundary safety measure
