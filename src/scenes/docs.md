# Noridoc: scenes

Path: @/src/scenes

### Overview
- Phaser Scene subclasses that wire pure game systems to Phaser's rendering, physics, and input APIs
- Contains two scenes: `GameScene` (gameplay: movement, flashlight/darkness, enemy AI, combat, shooting, weapon switching/pickups, battery drain, item collection, lore note collection, closable doors, light switches, furniture hiding, exploration minimap, multi-floor stairs with teleportation, location-themed starting room, alternate exit portals, exit condition, HUD) and `ShopScene` (between-run upgrade shop: displays gold balance, upgrade categories with buy buttons, starting location selector, lore journal overlay, and "Enter Backrooms" button)
- The two scenes alternate in a loop: GameScene -> ShopScene (on exit or death) -> GameScene (on "Enter Backrooms")

### How it fits into the larger codebase
- `@/src/main.js` registers scene classes with the Phaser.Game instance; GameScene is listed first (initial scene)
- Scenes import pure functions from `@/src/systems/` for game math (including maze generation from `@/src/systems/maze.js`, lore spawning from `@/src/systems/lore.js`, location definitions/alternate exit spawning from `@/src/systems/locations.js`, BFS room graph pathfinding from `@/src/systems/pathfinding.js`, and time-based danger escalation from `@/src/systems/danger.js`) and handle all Phaser-specific concerns: sprite creation, physics bodies, input binding, camera, collision callbacks, and Graphics drawing
- Scenes are the only place in the codebase that depends on Phaser -- this boundary is intentional to keep game logic testable
- `GameScene` reads upgrade levels from `game.registry` in its `init()` method, using `getUpgradeValue()` from `@/src/systems/shop.js` to compute upgrade bonuses as deltas from pistol base values (e.g., `getUpgradeValue('weaponDamage', level) - 25` yields a +N damage bonus). These bonuses are stored in `this.upgradeBonuses` and applied additively to whatever weapon is active via `getEffectiveStats()` from `@/src/systems/weapons.js`. Binary unlock upgrades (`startingPistol`, `minimap`) are checked as boolean flags: `startingPistol` determines whether `createWeaponState({ startingPistol: true })` or `createWeaponState()` (unarmed) is called; `minimap` sets `this.hasMinimap` which gates minimap Graphics creation and rendering. When unarmed, `getActiveWeapon()` returns null and firing is disabled until the player picks up a weapon. Survival stats (max HP, max battery, flashlight angle, speed) are still computed as absolute values. It also calls scaling functions from `@/src/systems/scaling.js` to compute run-scaled enemy stats per type -- each enemy type (basic, crawler, spitter) has its own base HP, contact damage, and chase speed constants, and scaling is applied independently to each set (e.g., `this.scaledEnemyHP`, `this.scaledCrawlerHP`, `this.scaledSpitterHP`). These values are stored as instance properties and used throughout `create()` and `update()`. Upgrade level fallback uses `createShopState().upgrades` as defaults, ensuring forward compatibility when new upgrade keys are added to saves that predate them
- `ShopScene` reads/writes `shopState` from `game.registry`, using pure functions from `@/src/systems/shop.js` for all purchase logic. It receives `treasureEarned` and optionally `newLocation` as scene data from GameScene, converts treasure to gold via `addGold()`, and if `newLocation` is provided, adds it to `unlockedLocations` in the registry. ShopScene also calls `saveGame()` from `@/src/systems/persistence.js` at every point where state is mutated: after adding treasure gold in `init()`, before entering backrooms, and after purchasing an upgrade. All `saveGame()` calls pass `collectedLore`, `unlockedLocations`, and `activeLocation` from `game.registry` to ensure all cross-run progress is persisted. ShopScene imports `getLocation()` and `LOCATIONS` from `@/src/systems/locations.js` to display location names in the location selector UI. ShopScene imports `getJournalEntries()` and `getJournalPage()` from `@/src/systems/lore.js` to power the lore journal overlay, which lets players review collected and uncollected lore notes between runs
- `GameScene` calls `saveGame()` from `@/src/systems/persistence.js` after incrementing `runCount` in `init()`, persisting the updated run counter, shop state, and location data to localStorage before the run begins
- Data flows between scenes via two mechanisms: `game.registry` for in-memory cross-scene state (shopState, runCount, unlockedLocations, activeLocation) and scene `init(data)` for per-transition data (treasureEarned, newLocation). Registry values are persisted to localStorage via `@/src/systems/persistence.js` so state survives browser reloads

### Core Implementation
- **GameScene lifecycle**: `init()` reads upgrade state from registry and computes per-run stats, `create()` sets up the level and all game objects, `update()` runs every frame
- **GameScene.init()**: reads `shopState`, `collectedLore`, `activeLocation`, and `unlockedLocations` from `game.registry` (converting `collectedLore` from a persisted array to a Set). Resolves the active location data via `getLocation()` from `@/src/systems/locations.js`, falling back to `'store'` if the location ID is not found. Spreads `createShopState().upgrades` as defaults before overlaying saved upgrade levels (for forward compatibility with saves predating new upgrades), then calls `getUpgradeValue()` for each upgrade to set survival stats (`this.maxCharge`, `this.flashlightAngle`, `this.maxHp`, `this.playerSpeed`), weapon upgrade bonuses (`this.upgradeBonuses = { damage, fireRate, range }` computed as deltas from pistol base values), and inventory capacity (`this.maxItems` from the backpack upgrade). Checks the `startingPistol` upgrade level and initializes `this.weaponState` via `createWeaponState({ startingPistol })` from `@/src/systems/weapons.js` -- when the upgrade is not purchased the player starts unarmed with empty slots and zero ammo. Sets `this.hasMinimap` from the `minimap` upgrade level (binary flag: true when level > 0) -- this gates both `minimapGraphics` creation in `createHUD()` and `drawMinimap()` rendering. Calls `this.updateWeaponStats()` to derive effective bullet stats from the active weapon plus bonuses. Also reads and increments `runCount` in the registry to derive a unique level seed for this run and stores `this.runCount` for use in `create()` (where it determines floor count via `getFloorRoomCounts()` from `@/src/systems/stairs.js`), and calls scaling functions from `@/src/systems/scaling.js` to precompute per-type scaled stats. Initializes `this.dangerState` via `createDangerState()` from `@/src/systems/danger.js` -- the danger state is per-run and not persisted. After incrementing, calls `saveGame()` to persist the updated `runCount`, `shopState`, `collectedLore`, `unlockedLocations`, and `activeLocation` to localStorage
- **create() setup sequence**: initializes level geometry (including per-room maze walls/columns via `createRoomMaze()`), doors, switches, player, enemies, item textures, items, lore items, player bullets, enemy bullets, weapon pickups, exit zone, stairs, alternate exits, crack visual, input, camera, darkness overlay, and HUD. Door creation happens after rooms but before player creation so that door colliders are ready. Switch creation happens after doors. Stairs and alternate exits are created after items. Combat state is initialized with `createCombatState(this.maxHp)`, battery state with `createBatteryState(this.maxCharge)`, inventory state with `createInventoryState(this.maxItems)` (where `maxItems` comes from the backpack upgrade computed in `init()`), hiding state with `createHidingState()` from `@/src/systems/hiding.js`, and exploration state with `createExplorationState()` from `@/src/systems/exploration.js`. Two additional flags are initialized: `this.isTeleporting = false` (gates update loop during stair transitions) and `this.currentFloor = 0` (tracks which floor the player is currently on). Pathfinding state is initialized with `this.roomGraph = null` and `this.roomGraphDirty = true`, so the room graph is lazily built on the first `updateEnemies()` call. Notable setup details:
  - `createEnemyTextures()` generates distinct textures for each enemy type: basic (red circle with dark "eyes"), crawler (smaller purple/magenta circle), and spitter (blue/cyan circle with a "mouth" marking)
  - `createEnemies()` passes `this.extraEnemyCount` and `this.runCount` to `generateRoomEnemies()` for run-scaled enemy density and type distribution. `getEnemyStats(type)` maps each spawn's type to the correct scaled HP, contact damage, texture key, and body size. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, `spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, and `targetDoorway` fields alongside the existing state machine fields. The room-tracking fields (`spawnRoomId`, `currentRoomId`) are initialized to the spawning room's ID; `roomTransitionCooldown` starts at 0 and `targetDoorway` starts null. Wires a player-enemy overlap callback (`onEnemyContact`) for per-enemy contact damage
  - `createItemTextures()` generates a Phaser texture for each item type (rectangles for coins/batteries, triangle for gems), keyed as `item_{type}`
  - `createItems()` creates a static physics group, calls `generateRoomItems()` per room, and places item sprites at depth 5 (below darkness). Wires a player-item overlap callback (`onItemPickup`) for auto-collection
  - `createLoreItems()` generates a parchment-colored note texture (`lore_note`, 12x10px with horizontal lines), creates a separate static physics group (`this.loreGroup`), calls `generateRoomLore()` from `@/src/systems/lore.js` per room (passing `this.collectedLore` to filter out already-collected entries), and places lore sprites at depth 5. Wires a player-lore overlap callback (`onLorePickup`) that adds the entry ID to `this.collectedLore`, updates `game.registry`, destroys the sprite, and calls `showLorePopup()` to display the text. Lore items do NOT consume inventory space -- they bypass the inventory system entirely. `showLorePopup()` displays a tween-animated semi-transparent overlay at the bottom of the screen (depth 1001-1002) with the lore text in monospace font, auto-dismissing after 5 seconds with a fade-out. Only one popup can be active at a time (previous popup is destroyed if a new one triggers)
  - `createDoors()` calls `createDoorStates()` from `@/src/systems/doors.js` to determine which connections have closable doors. For each closable door, creates a static physics zone in `doorGroup` sized to fill the doorway gap. Closed doors have their segment pushed into `wallSegments` and their physics body enabled; open doors have `body.enable = false`. Also creates a "Press E" interaction prompt text element (hidden by default)
  - `createSwitches()` calls `createSwitchStates()` from `@/src/systems/lightswitch.js` to place light switches in ~15% of non-starting rooms. Creates a Graphics object at depth 50 for switch rendering. Switches have no physics bodies -- they are purely visual objects interacted with via proximity checks
  - `createBullets()` creates a pooled physics group (`maxSize: 30`), generates per-weapon bullet textures (pistol: yellow 3px, shotgun: orange 2px, rifle: cyan 3px), and wires bullet-wall/furniture/door colliders and bullet-enemy overlap. Each bullet texture is keyed as `bullet_{weaponId}` so `fireBullet()` can select the correct texture based on the active weapon
  - `createWeaponPickups()` generates per-weapon pickup textures (keyed `pickup_{weaponId}`) and calls `generateLevelWeapons()` from `@/src/systems/weapons.js` to place weapon pickups -- this guarantees one weapon spawn per floor (selecting a random non-starting room per floor). All weapon types including pistol can appear as floor drops. Pickup sprites are stored in `this.weaponPickupSprites` and participate in the E-key interaction resolution system as candidates with type `'weapon'`
  - `createEnemyBullets()` creates a separate pooled physics group (`enemyBulletGroup`, `maxSize: 30`) with a red bullet texture, mirroring the player bullet pattern. Wires wall/furniture/door colliders and player overlap. `fireEnemyBullet()` is called from `updateEnemies()` when a spitter's AI state machine sets `wantsToFire = true`. `updateEnemyBullets()` expires enemy bullets using `SPITTER_PROJECTILE_RANGE`. `onEnemyBulletHitPlayer()` applies `SPITTER_PROJECTILE_DAMAGE` to the player's combat state (blocked while hiding)
  - `createExitZone()` places a 64x64 physics zone at the west-center of room 0 (position computed by `getLocationExitPosition()` from `@/src/systems/locations.js`) with a green visual indicator, wiring a player overlap callback (`onDayComplete`) for the day-cycle win condition. Also stores `this.exitPosition` for use by the minimap
  - `createCrackVisual()` draws a jagged crack line at the first doorway of room 0 using points from `generateCrackPoints()` in `@/src/systems/startroom.js`. The crack consists of a primary dark line (depth 2) and a thinner secondary line with a slight offset for a 3D effect. A glow rectangle (depth 1) behind the crack pulses with a sickly-yellow color (0xd4c073) via `updateCrackGlow()` each frame using a sine wave
  - `createStairs()` iterates over `this.level.stairs` (up to 2 stair connections per adjacent floor pair, across all floor pairs generated by `getFloorRoomCounts(this.runCount)`). For each stair connection, draws a dark rectangle with step lines at both endpoints, using `stair.fromFloor` and `stair.toFloor` to determine visual direction ('down' when going to a higher floor, 'up' when going to a lower floor). Each endpoint gets a Phaser physics zone (`STAIR_SIZE` x `STAIR_SIZE`) with `destX`, `destY`, and `destFloor` data properties. Overlap with the player triggers `onStairEnter()`. A separate `stairGlowGraphics` at depth 1 draws a pulsing purple glow (0x7733aa) behind both stair positions each frame via `updateStairGlow(time)`, using the same sine-wave pulse pattern as the crack glow
  - `createAlternateExits()` calls `generateAlternateExit()` from `@/src/systems/locations.js` to potentially spawn a glowing alternate exit portal on a deeper floor (floor > 0). If an exit is generated (20% chance per run, only on floors with non-stair rooms, only when locked locations remain), a 60x60 blue-white physics zone is placed at the portal position with a label showing the destination location name. A pulsing blue glow (0x88aaff) is animated each frame via `updateAlternateExitGlow(time)`. When the player overlaps the zone, `onAlternateExitComplete()` adds the location ID to `unlockedLocations` in the registry, sets `activeLocation` to the new location, and calls `onDayComplete(locationId)` to transition to ShopScene with the `newLocation` data
  - `createHUD()` creates a Graphics object with `setScrollFactor(0)` at depth 1000 so health and battery bars stay fixed on screen, plus text elements for battery count, treasure value, inventory capacity (`[itemCount/maxItems]` -- gray normally, turns red when full), active weapon name (colored per-weapon with a `[Q]` switch indicator), and a red "DANGER N" indicator (visible only when danger waves have been spawned, showing the current wave count from `this.dangerState.waveCount`). The `minimapGraphics` Graphics object (depth 1000, `setScrollFactor(0)`) is only created if `this.hasMinimap` is true (i.e., the minimap shop upgrade has been purchased). A floor indicator text (`this.floorText`) shows the current floor label (e.g., "B1", "B2", up to "B4" on run 4+) to indicate the player's current floor, updated each frame in `drawMinimap()`
- **update() frame loop**: early-returns if player is dead, `this.dayEnding` is true (exit transition in progress), or `this.isTeleporting` is true (stair transition in progress). Then: refreshes `pointer.worldX`/`worldY` via `pointer.updateWorldPoint(this.cameras.main)` so world coordinates stay current even when the camera moves without mouse input -> reads WASD key state -> if hiding, `calculateVelocity(keyState, speed * HIDING_SPEED_MULTIPLIER)` for half-speed crawling within furniture bounds; otherwise, `calculateVelocity()` at full speed -> sets player velocity -> computes aim angle from mouse -> ticks combat cooldown (`updateCombat`), battery state (`updateBattery`), and fire cooldown (`updateFireCooldown`) -> checks for left-click + `canFire()` + `getActiveWeapon()` non-null + `hasAmmo()` to fire bullets (firing calls `consumeAmmo()` to decrement ammo; blocked while hiding or unarmed or out of ammo) -> `updateInteractPrompt()` resolves the nearest interactable (door, switch, hideable furniture, or weapon pickup by Euclidean distance) and shows/hides "Press E" -> checks Q key (`JustDown`) or scroll wheel (with 200ms cooldown) to switch weapons via `switchWeapon()` and `updateWeaponStats()` -> checks E key (`JustDown`) and dispatches to `onToggleDoor()`, `onToggleSwitch()`, `onEnterHiding()`, or `onWeaponPickup()` based on the resolved interactable type (pressing E while already hiding calls `onExitHiding()`) -> `updateBullets()` expires out-of-range player bullets (using per-bullet range from `getData('range')`) -> `updateEnemyBullets()` expires out-of-range enemy projectiles -> `updateEnemies(delta)` first updates the danger system: accumulates unsafe time via `updateDangerTime()` (safe rooms are room 0 and lit rooms), then spawns danger waves via a `while (shouldSpawnWave())` loop that calls `spawnDangerWave()` and `advanceWave()` for each triggered wave -- multiple waves can fire in a single tick if enough time has accumulated. Then runs AI per enemy (with lit-room freeze check, `this.hidingState.isHiding` passed as the `playerHidden` flag, and type-specific chase speeds via `getScaledChaseSpeed()`; after AI update, checks `wantsToFire` flag to call `fireEnemyBullet()` for spitters) -> redraws player (with hiding, invulnerability, or normal visual state) -> recomputes darkness (flashlight cone + lit room rectangles drawn into BitmapMask) -> `updateCrackGlow(time)` animates the crack glow pulse -> `updateStairGlow(time)` animates the stair glow pulse -> `updateAlternateExitGlow(time)` animates the alternate exit glow pulse -> `drawDoors()` renders closed doors -> `drawSwitches()` renders switch visuals (gray when off, yellow when on) -> updates exploration state via `updateExploration()` (tracking visited rooms) -> redraws HUD (health bar + battery bar + battery count + treasure value + weapon name) -> `drawMinimap()` renders the fog-of-war minimap (using per-floor bounds and floor filter). Right-click input triggers `onUseBattery()` outside the frame loop via a `pointerdown` event listener (blocked while hiding). Item pickup happens via Phaser overlap callbacks (auto-collected on contact)
- **Enemy integration pattern**: `GameScene` maintains a parallel `enemyStates` array alongside the Phaser physics sprites in `enemyGroup`. Each enemy state object carries `type`, `contactDamage`, `attackCooldown`, room-tracking fields (`spawnRoomId`, `currentRoomId`, `roomTransitionCooldown`, `targetDoorway`), and standard AI state machine fields. Enemies are added to `enemyStates` both at level creation (via `createEnemies()`) and dynamically during gameplay (via `spawnDangerWave()`, which creates new sprites in `enemyGroup` and pushes new state objects with the same structure). Dynamic spawns automatically inherit existing colliders because Phaser group colliders cover all current and future group members. Each frame, `updateEnemies()` rebuilds the room graph if dirty (via `buildRoomGraph()` from `@/src/systems/pathfinding.js`, cached and invalidated only when `onToggleDoor()` sets `roomGraphDirty = true`), runs BFS from the player's current room, counts enemies per room, detects room transitions (updating `currentRoomId` and resetting cooldown/targetDoorway when an enemy enters a new room), evaluates idle doorway-seeking eligibility (gated by `DOORWAY_SEEK_CHANCE`, cooldown, `MAX_ROOM_DISTANCE` from spawn, and `MAX_ROOM_ENEMIES` cap), and constructs a `navContext` object per enemy. The `navContext` carries `enemyRoomId`, `playerRoomId`, `doorway` (the next doorway toward the player or a randomly chosen eligible doorway), `seekDoorway` (whether the idle enemy should move toward a doorway), and the enemy's room-tracking state. This navContext is passed as the 7th argument to `updateEnemyAI()`. After the AI update, room-tracking fields (`roomTransitionCooldown`, `targetDoorway`) are copied back from the result alongside all other state fields. This keeps the AI logic testable in `@/src/systems/enemy.js` while letting Phaser handle physics resolution and projectile management
- **ShopScene lifecycle**: `init(data)` receives `treasureEarned` and optionally `newLocation` from GameScene, reads existing `shopState` from registry (or creates fresh state on first visit), adds the treasure as gold via `addGold()`, writes back to registry. If `newLocation` is provided and not already unlocked, it is added to `unlockedLocations` in the registry. Calls `saveGame()` to persist all state including location data. Also reads `collectedLore` from the registry (as an array) and converts it to a Set for use by the journal. `create()` builds the UI: title, treasure earned display, gold balance, upgrade rows, location selector (when multiple locations are unlocked), a "NOTES X/20" journal button (top-right), and "Enter Backrooms" button. If a `newLocation` was just discovered, a "New location discovered!" message is shown. When multiple locations are unlocked, `<` and `>` buttons let the player cycle through unlocked starting locations via `cycleLocation()`, which updates `activeLocation` in the registry. Each upgrade row shows name, description, level dots (filled/empty circles), next cost, and a buy button. Purchases call `purchaseUpgrade()`, write back to registry, and call `saveGame()` to persist. The "Enter Backrooms" button writes shopState and activeLocation to registry, calls `saveGame()`, and then calls `this.scene.start('GameScene')` to transition back to gameplay. The save-at-every-mutation pattern ensures no progress is lost even if the browser crashes between user actions. The **lore journal** is a full-screen overlay activated by clicking the "NOTES X/20" button. `showJournal()` creates a background Graphics at depth 2000 covering the full screen (1024x768), with all text elements at depth 2001 to layer above all shop content. The journal displays entries paginated at 5 per page via `getJournalPage()` from `@/src/systems/lore.js`. Each entry shows "Note #N" with the lore text (if collected) or "???" (if uncollected), using color to distinguish collected (white/gray) from uncollected (dark gray). Prev/Next buttons navigate pages; `getJournalPage()` handles page clamping so out-of-bounds page indices are safely corrected. A close button and ESC key handler (via `input.keyboard.on('keydown-ESC')`) call `hideJournal()`, which destroys all overlay objects. The journal is read-only -- it does not modify any game state

### Things to Know
- **Depth layering order** controls what is visible through the darkness:

  | Layer | Depth | Purpose |
  |-------|-------|---------|
  | crack glow / stair glow / alternate exit glow graphics | 1 | Pulsing sickly-yellow glow behind the crack in room 0; pulsing purple glow behind stair zones; pulsing blue glow behind alternate exit portals |
  | crack line / stair step / alternate exit border graphics | 2 | Jagged crack-in-the-wall lines at the first doorway of room 0; dark rectangles with step lines at stair positions; blue border and location label for alternate exit portals |
  | item sprites / enemy corpses | 5 | Collectibles and dead enemy corpses, hidden by darkness (only visible in flashlight). Corpses are inactive sprites with gray tint, faded alpha, and toppled angle |
  | exit zone graphics | 10 | Green exit indicator visible under darkness |
  | furniture graphics | 50 | Furniture rendered below enemies |
  | door graphics | 50 | Closed doors rendered at same depth as furniture |
  | switch graphics | 50 | Light switches rendered at same depth as doors/furniture |
  | enemy sprites | 60 | Enemies hidden by darkness (below mask) |
  | `lightGraphics` | 99 | Faint yellow tint drawn under the flashlight cone |
  | `darkGraphics` | 100 | Black rectangle covering the entire level (the darkness) |
  | player bullet sprites | 150 | Player bullets visible above darkness but below player |
  | enemy bullet sprites | 150 | Enemy (spitter) projectiles, same depth as player bullets |
  | `playerGraphics` | 200 | Player always visible (above darkness) |
  | `hudGraphics` + HUD text | 1000 | Health bar, battery bar, battery count, treasure value, weapon name, danger indicator (above everything) |
  | `minimapGraphics` | 1000 | Fog-of-war minimap showing visited rooms, player dot, exit dot (above everything) |
  | lore popup background | 1001 | Semi-transparent black rounded rect for lore text overlay |
  | lore popup text | 1002 | Lore note text in monospace font (above popup background) |
  | journal overlay background | 2000 | Full-screen opaque black background for the ShopScene lore journal |
  | journal overlay text/controls | 2001 | Journal title, entry text, pagination buttons, close button (above journal background) |

  Items at depth 5, enemies at depth 60, and furniture at depth 50 are all hidden by the darkness overlay and only revealed by the flashlight. Bullets at depth 150 are always visible (above darkness), giving visual feedback even when firing into dark areas. The HUD uses `setScrollFactor(0)` so it does not move with the camera
- **Darkness system architecture**: the BitmapMask has `invertAlpha = true`, so white pixels in `maskGraphics` punch transparent holes in `darkGraphics`, revealing the room beneath. The `updateDarkness(time)` method first draws lit room rectangles (from `getLitRoomIds()`, which always includes room 0 since it is permanently lit) into both `maskGraphics` (white, to punch holes in darkness) and `lightGraphics` (faint yellow tint). Then it calls `getFlashlightConeAngle()` each frame to get a battery-adjusted cone angle and `shouldFlicker()` to determine if the flashlight should momentarily go dark. When the cone angle is zero (battery depleted) or flickering is active, the method returns early after drawing lit rooms but without drawing the flashlight polygon. Lit rooms remain visible even when the flashlight is dead -- they use the same BitmapMask mechanism but draw rectangles instead of the flashlight polygon
- **Enemy physics uses a dynamic group** (unlike walls and furniture which are static groups). Enemies collide with walls, furniture, and each other. Player-enemy interaction uses `overlap` (not `collider`), triggering `onEnemyContact(player, enemySprite)` which looks up the specific enemy state object to read its per-type `contactDamage` (falling back to `this.scaledEnemyDamage` if the state is not found), applies it via `applyDamage()`, triggers camera shake/flash, and checks for player death
- **Player bullet physics group** uses pooling (`maxSize: 30`). Bullet-wall/furniture uses `collider` (instant deactivation). Bullet-enemy uses `overlap` -- on hit, the bullet is deactivated and the enemy takes damage read from per-bullet data (`bullet.getData('damage')`, falling back to `this.bulletDamage`). When an enemy dies, its sprite is kept visible as a corpse: `setActive(false)` (removes from physics/overlap checks), `body.enable = false`, and corpse visual constants from `@/src/systems/combat.js` are applied (gray tint, faded alpha, toppled angle, depth 5). The sprite remains in `enemyGroup` but Phaser skips inactive sprites in overlap checks, so corpses cannot trigger contact damage. The enemy state is removed from `enemyStates`. Each bullet stores its origin, damage, and range via Phaser's `setData()` at spawn time in `spawnBullet()`. This per-bullet approach ensures that bullets fired with one weapon retain their stats even after the player switches weapons mid-flight. `fireBullet()` selects the correct texture (`bullet_{weaponId}`) based on the active weapon and dispatches to either `calculateShotgunSpread()` (for multi-pellet weapons with `bulletCount > 1`) or `calculateBulletVelocity()` (for single-bullet weapons). Fire cooldown resets to `this.fireRate` and range expiry checks per-bullet range -- both derived from the active weapon via `updateWeaponStats()`. **Enemy bullet physics group** (`enemyBulletGroup`) mirrors this pattern with `maxSize: 30`, using wall/furniture/door colliders and a player overlap callback (`onEnemyBulletHitPlayer`). Enemy bullets use `SPITTER_PROJECTILE_SPEED` and `SPITTER_PROJECTILE_RANGE` constants directly (no upgrades). Enemy projectile damage is blocked while the player is hiding
- **Player death flow**: when `combatState.isDead` becomes true, `onPlayerDeath()` stops the player body, disables physics, triggers a 500ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: 0 }`. The death penalty is losing all treasure collected that run -- gold already in the shop balance is not affected. The `update()` loop early-returns while dead, preventing further input processing
- **Stair teleportation flow**: when the player overlaps a stair zone, `onStairEnter()` sets `this.isTeleporting = true`, stops and disables the player body, triggers a 300ms camera fade-out. On fade-out complete, the player is teleported (`body.reset(destX, destY)`) to the matching stair endpoint on the other floor, body is re-enabled, `this.currentFloor` is updated to `destFloor`, and a 300ms fade-in begins. `isTeleporting` is cleared on fade-in complete. The `update()` loop early-returns while `isTeleporting` is true, freezing all game logic (enemies, bullets, battery drain) during the transition. This follows the same camera-fade + boolean-gate pattern as `dayEnding`
- **Day-complete flow (exit)**: when the player overlaps the exit zone or an alternate exit portal, `onDayComplete(newLocation)` sets `this.dayEnding = true`, stops the player, triggers a 1000ms camera fade-out, and transitions to ShopScene with `{ treasureEarned: this.inventoryState.treasureValue, newLocation }`. When triggered by the normal exit zone, `newLocation` is null. When triggered by an alternate exit portal, `newLocation` is the unlocked location ID. The player keeps all collected treasure. The `update()` loop early-returns while `dayEnding` is true, freezing game state during the transition
- **Visual feedback for combat, battery, hiding, inventory, and weapons**: taking damage triggers `cameras.main.shake()` and `cameras.main.flash()`. `drawPlayer()` selects from three visual states in priority order: hiding (muted green, alpha 0.3), invulnerability (red tint, alpha 0.5), or normal (green, alpha 1). The HUD health bar turns red when below 30% HP. The battery bar sits below the health bar and transitions from yellow to orange (at 50%) to red (at 25%). Below the bars, text elements display the current battery item count (`BAT xN`), accumulated treasure value (`$N`), and inventory capacity (`[itemCount/maxItems]` in gray, turning red when inventory is full). A weapon text element shows the active weapon name in the weapon's color, with a `[Q]` indicator when a second weapon is available for switching
- **E-key interaction resolution**: `updateInteractPrompt()` runs each frame and finds the nearest door, nearest switch, nearest hideable furniture, and nearest weapon pickup (within `WEAPON_PICKUP_RANGE` of 80px) independently. All candidates are collected into a candidates array, sorted by Euclidean distance to the player, and the closest one wins. The resolved interactable is stored in `this.nearestInteractable` (with `type` and `item` fields), and E-key press dispatches to `onToggleDoor()`, `onToggleSwitch()`, `onEnterHiding()`, or `onWeaponPickup()` based on the type. Pressing E while already hiding calls `onExitHiding()` regardless of nearby interactables. This candidates-array approach is extensible for future interactable types
- **Hiding mechanics**: when the player presses E near `canHide` furniture (any furniture type with `canHide: true` in `@/src/systems/furniture.js`), `onEnterHiding(furniture)` snaps the player to the furniture center via `body.reset()`, sets `body.setBoundsRectangle()` to constrain movement within the furniture area, enables `setCollideWorldBounds(true)`, and disables the player-furniture collider (`this.playerFurnitureCollider.active = false`) so physics does not push the player out. While hidden: the player moves at half speed (`HIDING_SPEED_MULTIPLIER = 0.5` from `@/src/systems/hiding.js`), shooting is disabled, right-click battery use is disabled, the player is immune to enemy contact damage (`onEnemyContact()` early-returns), and enemies cannot detect the player (the `playerHidden` flag forces `canSee = false` in `updateEnemyAI()`). The player graphic renders semi-transparent (alpha 0.3) with muted green colors. The player exits hiding only by pressing E, which calls `onExitHiding()` to restore `setCollideWorldBounds(false)` and re-enable the furniture collider
- **Exploration and minimap**: `GameScene` maintains `this.explorationState` (from `@/src/systems/exploration.js`) which tracks visited rooms as an immutable Set regardless of whether the minimap upgrade is purchased (future features may use visitation data). Each frame, `updateExploration()` checks the player's position against all rooms and adds newly visited rooms to the set. `drawMinimap()` early-returns if `this.hasMinimap` is false (the minimap shop upgrade has not been purchased -- cost 3000 gold, single tier). When enabled, it calls `getFloorBounds()` from `@/src/systems/stairs.js` to get the bounding box for only the current floor, then passes that plus `this.currentFloor` as the `floorFilter` to `getMinimapData()` -- this ensures the minimap only shows rooms on the player's current floor, not the entire multi-floor level. The minimap uses a separate `minimapGraphics` Graphics object (not the same as `hudGraphics`) so it can be cleared and redrawn independently. The exit position (`this.exitPosition`) is stored during `createExitZone()` and passed to `getMinimapData()` so the exit dot is always visible on the minimap regardless of fog-of-war state
- **Closable door toggle flow**: when the player presses E near a closable door, `onToggleDoor()` does four things: (1) if the door was closed, finds and splices its segment out of `wallSegments` via `indexOf`; (2) calls `toggleDoor()` from `doors.js` to produce new immutable door states; (3) enables/disables the Phaser physics body on the door zone; (4) sets `this.roomGraphDirty = true` to invalidate the cached pathfinding room graph, so the next `updateEnemies()` call rebuilds it with the updated door state. This ensures physics, raycasting, enemy LOS, and enemy pathfinding all update atomically. Player, enemy, and bullet groups all have colliders against `doorGroup`, so closed doors block all three entity types and also exclude the connection from the pathfinding graph (making door-closing a defensive strategy against cross-room enemy pursuit)
- **Light switch toggle flow**: when the player presses E near a switch, `onToggleSwitch()` calls `toggleSwitch()` from `lightswitch.js` to produce new immutable switch states. The method is a pure state toggle with no enemy side effects. On subsequent frames, `updateEnemies()` checks `getLitRoomIds()` and freezes (zero velocity, skip AI via `continue`) any enemy that is inside a lit room. This means enemies in lit rooms are frozen in place but remain alive and visible -- if the switch is turned off, they resume AI. This is the sole mechanism for handling enemies in lit rooms (both when the light first turns on and when enemies later wander in)
- **Starting room (room 0) as themed location**: room 0 is visually distinct from all other rooms. Its appearance (floor color, wall color, furniture layout) is determined by the active starting location via `this.activeLocationData` (resolved from `@/src/systems/locations.js` in `init()`). The default location is the furniture store (`'store'`), but the player can unlock and select other locations (office building, parking garage, hotel lobby, laundromat). `drawRoom()` and `drawDoorways()` use `this.activeLocationData.floorColor` and `this.activeLocationData.wallColor` for room 0 instead of default grays. `createRoomFurniture()` calls `getLocationLayout()` from `locations.js` for room 0 instead of `generateRoomFurniture()`. Room 0 is permanently lit -- its ID is injected into the `litRoomIds` array in both `updateDarkness()` and `updateEnemies()`, reusing the same lit-room mechanism as light switches. This means room 0 has no darkness overlay, and any enemies that wander in are frozen. The exit zone is at the west-center of the room, positioned by `getLocationExitPosition()`. A crack-in-the-wall visual at the first doorway of room 0 is drawn by `createCrackVisual()` using `generateCrackPoints()` from `@/src/systems/startroom.js` (the crack visual is identical across all locations)
- **Multi-room physics with door gaps**: `createRoomPhysics()` splits each wall into physics zones that skip over door offsets. This mirrors the raycasting segments produced by `createRoomWalls()` so that physics collision and flashlight occlusion agree on where doors are. Closable doors fill these gaps with their own physics zones that can be toggled
- **Right-click and scroll input**: `setupInput()` disables the browser context menu (`this.input.mouse.disableContextMenu()`) and listens for `pointerdown` and `wheel` events. Right-click triggers `onUseBattery()`, which calls `useBattery()` from `inventory.js` and, if successful, `rechargeBattery()` from `battery.js`. Scroll wheel triggers `switchWeapon()` from `@/src/systems/weapons.js` with a 200ms cooldown (`this.lastScrollTime`) to prevent rapid switching. Both inputs are guarded by `!isDead && !dayEnding` checks
- **Item pickup flow**: items are a static physics group with player overlap detection. On overlap, `onItemPickup()` first checks `canPickupItem(this.inventoryState)` from `@/src/systems/inventory.js` -- if the inventory is full, the function early-returns and the item remains on the ground for later pickup. Otherwise, it updates `this.inventoryState` via `pickupItem()` and destroys the item sprite. Items store their type and value via Phaser's `setData()` on the sprite. Items do not produce wall segments -- they are collectibles, not occluders. Lore items use a completely separate physics group (`this.loreGroup`) and pickup callback (`onLorePickup`) -- they bypass the inventory system entirely and are always collected on contact regardless of inventory capacity
- **Wall dual-purpose**: walls exist as both Phaser Arcade Physics static zones (for player collision) and as line segments from `createRoomWalls()` (for raycasting). Light-blocking furniture (any type with `blocksLight: true` in `@/src/systems/furniture.js`, e.g., shelf, bookcase, armoire, closet) and closable doors follow the same pattern -- each is both a physics zone and a raycasting segment. Non-blocking furniture (types with `blocksLight: false`, e.g., table, desk, bed, vent) still has physics zones but does not produce raycasting segments (light passes over them). Maze internal walls also follow this dual pattern: they produce both physics zones and raycasting segments. Doors are unique in that they can toggle between active/inactive at runtime via `body.enable` and `wallSegments` push/splice. Enemies, items, and light switches do not produce segments; enemies are dynamic actors, items are small collectibles, and light switches are proximity-based interactables with no physics body
- **Player rendering**: the player is a Graphics object (`playerGraphics`) drawn at depth 200 (above the darkness overlay at depth 100), so the player is always visible regardless of flashlight direction
- Camera and physics bounds are computed dynamically from the level's room positions, so they adapt to however many rooms the level generator produces
